## Story

**As an** admin,  
**I want** to define reusable task templates per tenant,  
**so that** tasks are consistently created across workflows in my company.

---

## Acceptance Criteria

1. A task_template entity exists with tenant_id
2. A task_instance entity exists for runtime execution
3. Each task template includes: title, description, default_due_days, assignee_role (nullable)
4. Task templates are visible only within the same tenant_id
5. Runtime tasks are linked to: process_instance, step_instance

### Data Models

[Source: architecture/data-models.md, architecture/workflow-engine-schema.md]

**Design Principles Applied to Task Tables:**
- **Normalized schema** for data integrity
- **UUID primary keys** for security and distributed systems
- **Soft deletes** (`deleted_at`) for audit trail preservation
- **JSONB fields** for flexible metadata without schema changes
- **Enum types** for status fields (enforced at application level, documented in shared types)
- **Foreign key constraints** with CASCADE for tenant/process hierarchy, RESTRICT for user attribution, SET NULL for template reference

**Tenant Isolation Pattern:**
```typescript
// All task tables must include tenant_id
export interface TaskTemplate {
  id: string; // UUID
  tenantId: string; // FK to tenants - CASCADE delete
  name: string; // Unique within tenant (for template lookup)
  title: string; // Display title for the task
  description: string; // Detailed instructions
  defaultDueDays: number; // Default days to complete
  assigneeRole: string | null; // Role to auto-assign task to (nullable)
  status: TaskTemplateStatus; // 'active' | 'archived'
  metadata: Record<string, any>; // Flexible configuration
  createdBy: string; // User ID - RESTRICT delete
  createdAt: Date;
  updatedAt: Date;
  deletedAt: Date | null;
}

export interface TaskInstance {
  id: string; // UUID
  tenantId: string; // FK to tenants - CASCADE delete
  taskTemplateId: string | null; // FK to task_template - SET NULL
  processInstanceId: string; // FK to process_instance - CASCADE delete
  stepInstanceId: string | null; // FK to step_instance - CASCADE delete
  title: string; // Runtime title (copied from template or custom)
  description: string; // Runtime description (copied from template or custom)
  assignedTo: string; // User ID - RESTRICT delete
  status: TaskInstanceStatus; // 'pending' | 'in_progress' | 'completed' | 'cancelled'
  dueDate: Date | null; // Calculated from defaultDueDays or custom
  completedBy: string | null; // User ID - RESTRICT delete
  completedAt: Date | null;
  metadata: Record<string, any>; // Runtime task data
  createdAt: Date;
  updatedAt: Date;
  deletedAt: Date | null;
}
```

**Task Template vs Task Instance:**
- **task_template**: Reusable definition of a task (like a blueprint)
  - Created by admins in backoffice (Story 2.2.7)
  - Tenant-scoped (only visible within same tenant_id)
  - Can be archived (status = 'archived') but runtime tasks continue
  - Examples: "Review Supplier Documentation", "Verify ISO 9001 Certificate", "Approve Qualification"
  
- **task_instance**: Runtime execution of a task
  - Created by workflow engine when process starts (Story 2.2.8)
  - Linked to specific process_instance and optionally step_instance
  - May or may not reference a task_template (task_template_id nullable)
  - Assigned to specific user (assigned_to)
  - Has concrete due date calculated from template's default_due_days
  - Independent lifecycle: If template deleted (SET NULL), task continues

**Workflow Integration:**
```typescript
// Task instances are created as part of workflow execution
// Example: Supplier qualification process creates multiple task instances

// Process: Supplier Qualification
// └─ Step 1: Initial Review (step_instance)
//    ├─ Task Instance: "Review Supplier Profile" (from template)
//    └─ Task Instance: "Verify Tax ID" (from template)
// └─ Step 2: Document Review (step_instance)
//    ├─ Task Instance: "Check ISO 9001" (from template)
//    └─ Task Instance: "Review Financial Statements" (custom)

// Task instances can exist:
// 1. At process level (no step_instance_id) - general process tasks
// 2. At step level (with step_instance_id) - step-specific tasks
```

```sql
CREATE TABLE task_template (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    name VARCHAR(200) NOT NULL, -- Internal identifier (unique within tenant)
    title VARCHAR(300) NOT NULL, -- Display title
    description TEXT,
    default_due_days INTEGER NOT NULL DEFAULT 7,

    status VARCHAR(50) NOT NULL DEFAULT 'active', -- 'active' or 'archived'
```

**Table 2: task_instance**
```sql
CREATE TABLE task_instance (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    task_template_id UUID REFERENCES task_template(id) ON DELETE SET NULL, -- Nullable, SET NULL preserves runtime tasks
    process_instance_id UUID NOT NULL REFERENCES process_instance(id) ON DELETE CASCADE,
    step_instance_id UUID REFERENCES step_instance(id) ON DELETE CASCADE, -- Nullable: task may belong to process without specific step
    title VARCHAR(300) NOT NULL, -- Runtime title (copied from template or custom)
    description TEXT, -- Runtime description (copied from template or custom)
    assigned_to UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    status VARCHAR(50) NOT NULL DEFAULT 'pending', -- 'pending', 'in_progress', 'completed', 'cancelled'
    due_date TIMESTAMP WITH TIME ZONE, -- Calculated from default_due_days or set manually
    completed_by UUID REFERENCES users(id) ON DELETE RESTRICT,
    completed_at TIMESTAMP WITH TIME ZONE,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMP WITH TIME ZONE
);

-- User task lists (active/pending tasks assigned to user)
CREATE INDEX idx_task_instance_tenant_assigned_status 
    ON task_instance(tenant_id, assigned_to, status) 
    WHERE status IN ('pending', 'in_progress') AND deleted_at IS NULL;

-- Workflow task retrieval (tasks for specific process/step)
CREATE INDEX idx_task_instance_process_step 
    ON task_instance(process_instance_id, step_instance_id) 
    WHERE deleted_at IS NULL;

-- Overdue task detection
CREATE INDEX idx_task_instance_tenant_due_date 
    ON task_instance(tenant_id, due_date) 
    WHERE status IN ('pending', 'in_progress') AND deleted_at IS NULL;
```

**Tenant Isolation Strategy:**
- Both tables include `tenant_id` with CASCADE delete when tenant is removed
- All queries must filter by `tenant_id` (enforced in application layer)
- Future stories will add Row Level Security (RLS) policies for database-level enforcement
- Indexes include `tenant_id` as first column for optimal query performance

**Foreign Key Cascade Rules:**
- `tenant_id` → CASCADE: When tenant deleted, all templates and instances deleted
- `process_instance_id` → CASCADE: When process deleted, all tasks deleted
- `step_instance_id` → CASCADE: When step deleted, all step-specific tasks deleted
- `task_template_id` → SET NULL: When template deleted, runtime tasks continue (preserves audit trail)
- `created_by`, `assigned_to`, `completed_by` → RESTRICT: Cannot delete users who participated in tasks (audit trail preservation)

**SET NULL Rationale for task_template_id:**
- Runtime task instances should survive template deletion/archival
- Historical tasks maintain their data even if template is archived
- Allows admins to clean up unused templates without affecting active workflows
- Task instance contains denormalized title/description copied from template at creation

### Technical Constraints

[Source: architecture/coding-standards.md, architecture/tech-stack.md]

**Drizzle ORM Patterns:**
- **Table Definition:** Use `pgTable` from `drizzle-orm/pg-core`
- **Column Types:** 
  - `uuid()` for all IDs and foreign keys
  - `varchar()` with explicit length for text fields
  - `text()` for long-form descriptions
  - `integer()` for default_due_days
  - `timestamp()` with `{ withTimezone: true, mode: "date" }` for all dates
  - `jsonb()` for flexible metadata fields
- **Indexes:** Use composite indexes starting with `tenant_id` for multi-tenant queries
- **Partial Indexes:** Use `.where()` clause for conditional indexes (e.g., deleted_at IS NULL)
- **Relations:** Define using `relations()` from Drizzle ORM for type-safe joins
- **Naming:** 
  - Tables: snake_case (e.g., `task_template`, `task_instance`)
  - Schema files: kebab-case (e.g., `task-template.ts`, `task-instance.ts`)
  - TypeScript interfaces: PascalCase (e.g., `TaskTemplate`, `TaskInstance`)

**Drizzle Relations Example:**
