## Story

**As a** the system,  
**I want** to execute workflow templates with fill/validate modes and decline loops,  
**so that** users can submit, validate, request changes, and re-submit with comments.

---

## Acceptance Criteria

1. Admin selects workflow from dropdown of published workflow templates filtered by tenant_id
2. A process_instance is created with tenant_id from the workflow template
3. step_instance records inherit tenant_id
4. Only the first step is created in active state
5. Subsequent steps are created in blocked state
6. When a step becomes active:
   - The system creates one or more task_instance records for that step (based on approver configuration)
7. Form behavior:
   - If form_action_mode = fill_out:
     - The assigned user must fill and submit the form
     - After submission, the workflow proceeds to the next step
   - If form_action_mode = validate:
     - The form is read-only
     - The step UI shows approve and decline actions
     - On decline:
       - A comment is required
       - The workflow returns to the previous step
       - The previous step user can update the form and respond to the comment
8. Document behavior:
   - If document_action_mode = upload:
     - The assigned user must upload required documents and submit
     - After submission, the workflow proceeds to the next step
   - If document_action_mode = validate:
     - The validator sees documents in read-only mode
     - The step UI shows approve and decline actions
     - On decline:
       - A comment is required
       - The workflow returns to the previous step
       - The previous step user can re-upload documents and respond to the comment
9. Commenting model:
   - Decline comments are stored and linked to:
     - process_instance_id
     - step_instance_id
     - entity_type = form or document
   - Responses to decline comments are supported and stored in the same thread
10. Tenant isolation applies to all runtime records:
    - process_instance, step_instance, task_instance, submissions, documents, comments

### Task 4: Create Step Completion API Endpoint (AC: 7, 8)

- [x] Create API route: `apps/api/src/routes/workflows/steps/:stepInstanceId/complete.ts`
- [x] Endpoint: `POST /api/workflows/steps/:stepInstanceId/complete`
- [x] Request body validation:
  - action (enum: 'submit', 'approve', 'decline')
  - comment (string, optional for submit/approve, required for decline)
- [x] Implementation logic:
  - Get authenticated user
  - Query step_instance by ID (filter by tenant_id)
  - Query process_instance and workflow_step_template
  - Validate user has permission to complete (check assignee or approver)
  - **If action = 'submit':**
    - Verify form_action_mode = 'fill_out' OR document_action_mode = 'upload'
    - Verify form/document is submitted:
      - If form: Query form_submission table for step_instance_id (ensure record exists)
      - If document: Query document storage for step_instance_id (ensure documents uploaded)
    - Update step_instance: status = 'completed', completed_by = user.id, completed_date = NOW()
    - Mark associated task_instance as completed
    - Call transitionToNextStep() helper
  - **If action = 'approve':**
    - Verify form_action_mode = 'validate' OR document_action_mode = 'validate'
    - If multi_approver = true:
      - Mark current user's task as completed
      - Count completed tasks
      - If count >= approver_count: proceed to next step
      - Else: return (wait for more approvals)
    - If multi_approver = false:
      - Update step_instance: status = 'completed'
      - Mark task_instance as completed
      - Call transitionToNextStep() helper
  - **If action = 'decline':**
    - Verify comment is provided
    - Create comment_thread record:
      - entity_type = 'form' or 'document' (from step config)
      - comment_text = request.comment
      - commented_by = user.id
    - Update step_instance: status = 'declined'
    - Mark task_instance as completed (with declined metadata)
    - Call returnToPreviousStep() helper
  - Return response with updated status

### State Machine

**Workflow Process Status:**
```
active → completed
active → cancelled
```

**Step Instance Status:**
```
blocked → active → completed
blocked → active → declined → (return to previous step) → active → completed
blocked → skipped
```

**Task Instance Status:**
```
open → completed
```

**Multi-Approver Logic:**
- Create one task per approver
- Step completes when approver_count tasks are completed
- Remaining tasks auto-cancelled when threshold reached

**Decline Behavior:**
- Current step transitions to 'declined'
- Calculate target step: current_step_order - decline_returns_to_step_offset
- Target step transitions to 'active'
- New tasks created for target step
- Comment thread created with decline reason

   - Form fill_out mode
   - Document upload mode
   - Transition to next step
   - Task completion

3. **Step Completion - Approve:**
   - Form validate mode (single approver)
   - Document validate mode (single approver)
   - Multi-approver partial completion
   - Multi-approver full completion
   - Transition to next step

4. **Step Completion - Decline:**
   - Decline with comment
   - Comment thread creation
   - Return to previous step
   - New tasks created for returned step

5. **Multi-Approver Logic:**
   - Multiple tasks created
   - Partial approvals don't complete step
   - Threshold completion triggers next step
   - Tenant isolation per approver

6. **Comment Threading:**
   - Create decline comment
   - Reply to comment (parent_comment_id)
   - Query threaded comments
   - Tenant filtering

7. **Tenant Isolation:**
   - Cannot query processes from other tenants
   - Cannot complete steps from other tenants
   - Cannot see comments from other tenants
   - All queries include tenant filter

8. **Edge Cases:**
   - Decline offset larger than available steps
   - Complete already completed step
   - Approve without permission
   - Missing required comment on decline
