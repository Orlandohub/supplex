## Story

**As an** admin,  
**I want** to configure validation as a checkbox property on workflow steps,  
**so that** validation tasks are automatically created at runtime without requiring manual validation step creation.

---

## Context

### User Feedback (from Story 2.2.14 Testing)

During user acceptance testing of the workflow template builder, users reported that creating validation steps is cumbersome:

> "Currently when creating workflow templates I have to manually add separate validation step types. Instead, I should be able to check 'Requires Validation?' on any step, select approver roles, and let the system create validation tasks automatically during workflow execution."

### Current vs. Proposed Approach

**Current (Manual) - Example Workflow:**
```
Step 1: Submit Supplier Profile (formActionMode='fill_out')
Step 2: Validate Supplier Profile (formActionMode='validate') ← MANUAL!
Step 3: Upload Financial Documents (documentActionMode='upload')
Step 4: Validate Financial Documents (documentActionMode='validate') ← MANUAL!
Step 5: Final Approval
```
**Problems:**
- Users must remember to create validation steps manually
- Template becomes cluttered with extra steps
- Error-prone (easy to forget validation steps)
- Not intuitive (validation feels like it should be a property, not a separate step)

**Proposed (Automatic) - Same Workflow:**
```
Step 1: Submit Supplier Profile 
  - formActionMode='fill_out'
  - requiresValidation=true ← CHECKBOX!
  - validatorRoles=['quality_manager']
  → Runtime: System auto-creates validation task for quality_manager

Step 2: Upload Financial Documents
  - documentActionMode='upload'
  - requiresValidation=true ← CHECKBOX!
  - validatorRoles=['procurement_manager']
  → Runtime: System auto-creates validation task for procurement_manager

Step 3: Final Approval
```
**Benefits:**
- ✅ Fewer steps in workflow template (simpler)
- ✅ Validation configured where it belongs (as step property)
- ✅ System handles task creation automatically
- ✅ Prevents user errors

### Workflow Engine - Auto-Validation Logic (AC 7-10)

7. When step completes and `requires_validation=true`, workflow engine creates validation task(s) BEFORE activating next step
8. Validation tasks are created for each role in `validationConfig.approverRoles`

10. Next step activates only after all required validation tasks are approved

**Runtime Flow:**
```
User completes step (form/document submission)
  ↓
System checks: step.requiresValidation?
  ↓ YES
System creates validation task(s) for configured roles
  ↓

Next step activates
```

### Validation Config Data Structure

**TypeScript Interface:**
```typescript
interface ValidationConfig {
  approverRoles: string[];          // Required: Roles that can approve validation
  requireAllApprovals?: boolean;    // Optional: Future enhancement (default: false)
}
```

**Example JSONB:**
```json
{
  "approverRoles": ["quality_manager", "procurement_manager"],
  "requireAllApprovals": false
}
```

**Validation Rules:**
- If `requiresValidation=true`, `approverRoles` MUST be non-empty array
- Each role in `approverRoles` must be valid tenant role
- Empty validation config (`{}`) is allowed when `requiresValidation=false`

### Workflow Engine Logic (Pseudocode)

**File:** `apps/api/src/lib/workflow-engine/complete-step.ts`

```typescript
async function completeStep(stepInstanceId: string, userId: string) {
  // 1. Mark step as completed
  await db.update(stepInstance)
    .set({ 
      status: 'completed', 
      completedAt: new Date(),
      completedBy: userId 
    })
    .where(eq(stepInstance.id, stepInstanceId));

  // 2. Fetch step template to check validation requirements
  const stepTemplate = await db.query.workflowStepTemplate.findFirst({
    where: eq(workflowStepTemplate.id, stepInstance.workflowStepTemplateId)
  });

  // 3. Check if validation required
  if (stepTemplate.requiresValidation) {
    // 3a. Parse validation config
    const validationConfig = stepTemplate.validationConfig as ValidationConfig;
    
    // 3b. Create validation task for each approver role
    for (const role of validationConfig.approverRoles) {
      await createTaskInstance({
        stepInstanceId,
        assigneeType: 'role',
        assigneeRole: role,
        title: `Validate: ${stepTemplate.name}`,
        description: 'Review and approve or decline this step',
        status: 'open',
        // Form/document automatically in validate mode
        metadata: {
          isValidationTask: true,
          originalStepName: stepTemplate.name
        }
      });
      
      console.log(`[Validation] Created validation task for role: ${role}, step: ${stepTemplate.name}`);
    }
    
    // 3c. Set step status to awaiting validation
    await db.update(stepInstance)
      .set({ status: 'awaiting_validation' });
    
    // 3d. Log event for audit trail
    await logWorkflowEvent({
      processInstanceId: stepInstance.processInstanceId,
      eventType: 'validation_tasks_created',
      details: { stepInstanceId, approverRoles: validationConfig.approverRoles }
    });
    
    // DO NOT activate next step yet - wait for validation approval
    return;
  }

  // 4. No validation required - proceed to next step immediately
  await activateNextStep(stepInstanceId);
}
```

**File:** `apps/api/src/lib/workflow-engine/approve-validation.ts` (may need creation)

```typescript
async function approveValidationTask(taskInstanceId: string, userId: string) {
  // 1. Mark validation task as completed
  await db.update(taskInstance)
    .set({ 
      status: 'completed', 
      completedAt: new Date(),
      completedBy: userId 
    })
    .where(eq(taskInstance.id, taskInstanceId));

  // 2. Get step instance
  const task = await db.query.taskInstance.findFirst({
    where: eq(taskInstance.id, taskInstanceId)
  });
  
  const stepInstanceId = task.stepInstanceId;

  // 3. Check if all validation tasks for this step are completed
  const allValidationTasks = await db.query.taskInstance.findMany({
    where: and(
      eq(taskInstance.stepInstanceId, stepInstanceId),
      eq(taskInstance.metadata.isValidationTask, true)
    )
  });

  const allComplete = allValidationTasks.every(t => t.status === 'completed');

  // 4. If all validations complete, activate next step
  if (allComplete) {
    await db.update(stepInstance)
      .set({ status: 'validated' })
      .where(eq(stepInstance.id, stepInstanceId));
    
    await activateNextStep(stepInstanceId);
    
    console.log(`[Validation] All validations complete, activating next step`);
  }
}
```

### Performance Considerations

**Database Queries:**
- Index on `requires_validation` ensures fast filtering during step completion
- Validation config stored as JSONB (efficient storage, flexible structure)
- Additional task creation overhead: ~10-50ms per validation task (minimal)

**Workflow Execution:**
- Validation task creation is async (non-blocking)
- Typical workflow has 1-3 validation tasks per step (low overhead)
- No performance impact on workflows without validation

**API Response Times:**
- Step CRUD operations: < 100ms (no significant change)
- Step completion with validation: +50-150ms (acceptable)

### Security Considerations

**Tenant Isolation:**
- All validation tasks enforce tenant_id filtering
- Approver roles validated against tenant's available roles
- No cross-tenant validation possible

**Access Control:**
- Only users with matching role can see validation tasks
- Admin role cannot be excluded from validation approval
- Validation config only editable by admins

**Data Validation:**
- TypeBox validation on API (server-side)
- Zod validation on frontend (client-side)
- JSONB structure validated before database insert

### Edge Cases & Error Handling

**1. Empty Approver Roles:**
- **Scenario:** User checks validation checkbox but doesn't select any roles
- **Handling:** Form validation prevents submission, show error: "Select at least one approver role"

**2. Invalid Role in Config:**
- **Scenario:** Validation config contains role that doesn't exist in tenant
- **Handling:** API validation rejects request, return 400 error

**3. Approver Role Deleted After Config:**
- **Scenario:** Role was deleted from tenant after workflow template created
- **Handling:** Runtime falls back to admin role, log warning

**4. No Users With Approver Role:**
- **Scenario:** Validation requires `quality_manager` but no user has that role
- **Handling:** Task created but unassigned, admin receives notification

**5. Concurrent Validation Approvals:**
- **Scenario:** Multiple approvers approve validation simultaneously
- **Handling:** Use database transactions to prevent race conditions

**6. Step Completed But Validation Not Yet Approved:**
- **Scenario:** User tries to view next step before validation complete
- **Handling:** Next step shows "Blocked - awaiting validation" status
