## Story

**As a** compliance officer,  
**I want** to view complete audit history for qualifications,  
**so that** we can demonstrate compliance during audits.

---

## Acceptance Criteria

1. Workflow detail page includes "History" tab showing complete audit trail
2. Timeline view displays all events in reverse chronological order (newest first)
3. Each event displays: Event type icon, Event description, Actor (user), Timestamp, Comments/notes
4. Event types tracked: Workflow initiated, Document uploaded, Document removed, Stage submitted, Stage approved, Stage rejected, Risk score changed, Comments added
5. Document events show document name and type
6. Stage events show reviewer name and decision
7. Each event is immutable (cannot be edited or deleted)
8. Events display user's full name and role at time of action
9. Timestamps display in user's local timezone with UTC timestamp on hover
10. Can filter timeline by event type: All events, Approvals, Rejections, Documents, Comments
11. Timeline is printable with "Print Audit Trail" button generating PDF
12. PDF includes: Supplier name, Workflow ID, Print date, Complete timeline with all events
13. Database audit trail table indexed for fast queries (workflow_id, timestamp)
14. Mobile-responsive timeline with touch-friendly expand/collapse for event details

### Previous Story Insights

[From Story 2.9 - Qualification Workflow List & Filtering]

- **Database Performance**: Existing indexes on `(tenant_id, status)` and `(tenant_id, supplier_id)` ensure fast filtering. For Story 2.10, we need similar indexes on `workflow_events` table: `(workflow_id, created_at)` for timeline queries.
- **Tenant Isolation**: Strictly enforce tenant filtering in all queries. All workflow event queries MUST include `eq(workflowEvents.tenantId, tenantId)`.
- **Server-side Pagination Pattern**: Story 2.9 uses server-side pagination. For Story 2.10 timeline, consider client-side rendering since audit trails are typically <1000 events per workflow. If performance becomes an issue, implement server-side pagination later.
- **Eden Treaty API Client**: Use `createEdenTreatyClient(token)` for all API calls from frontend. Pattern: `client.api.workflows[id].timeline.get()`.
- **Remix Loader Pattern**: Fetch ALL data in loader using `Promise.all` for parallel requests. Pass data as props to components. Use `useRevalidator()` for mutations.

**New `workflow_events` Schema Design**:
```typescript
export const workflowEvents = pgTable("workflow_events", {
  id: uuid("id").primaryKey().defaultRandom(),
  tenantId: uuid("tenant_id").notNull().references(() => tenants.id, { onDelete: "cascade" }),
  workflowId: uuid("workflow_id").notNull().references(() => qualificationWorkflows.id, { onDelete: "cascade" }),
  eventType: varchar("event_type", { length: 100 }).notNull(), // WORKFLOW_INITIATED, DOCUMENT_UPLOADED, etc.
  eventDescription: text("event_description").notNull(),
  actorUserId: uuid("actor_user_id").notNull().references(() => users.id, { onDelete: "set null" }),
  actorName: varchar("actor_name", { length: 200 }).notNull(), // Denormalized for immutability
  actorRole: varchar("actor_role", { length: 50 }).notNull(), // Denormalized to capture role at time of action
  targetDocumentId: uuid("target_document_id").references(() => documents.id, { onDelete: "set null" }),
  targetDocumentName: varchar("target_document_name", { length: 255 }),
  targetStageNumber: integer("target_stage_number"),
  targetReviewerName: varchar("target_reviewer_name", { length: 200 }),
  comments: text("comments"),
  metadata: jsonb("metadata").default({}), // Flexible field for event-specific data
  createdAt: timestamp("created_at", { withTimezone: true, mode: "date" }).notNull().defaultNow(),
}, (table) => ({
  workflowCreatedIdx: index("idx_workflow_events_workflow_created").on(table.workflowId, table.createdAt),
  tenantWorkflowIdx: index("idx_workflow_events_tenant_workflow").on(table.tenantId, table.workflowId),
  eventTypeIdx: index("idx_workflow_events_type").on(table.eventType),
}));
```

**Why Denormalize Actor Name and Role?**
- Audit trails must be immutable and reflect the state at the time of the action.
- If a user's name or role changes later, the audit trail should show the original name/role.
- This is a compliance requirement for audit trails (AC 8).

### Frontend Components

**Remix Loader Pattern** (CRITICAL):
```typescript
// apps/web/app/routes/_app.workflows.$id.tsx

export async function loader(args: LoaderFunctionArgs) {
  const { session } = await requireAuth(args);
  const { id } = args.params;
  const token = session?.access_token;
  const client = createEdenTreatyClient(token);
  
  // Fetch timeline events in parallel with existing workflow data
  const [workflowResponse, timelineResponse] = await Promise.all([
    client.api.workflows[id].get(),
    client.api.workflows[id].timeline.get({ query: { eventType: "all" } }),
  ]);
  
  return json({
    workflow: workflowResponse.data.workflow,
    timelineEvents: timelineResponse.data?.events || [],
    token,
  });
}

// shouldRevalidate: Return false to prevent revalidation on tab switches
export function shouldRevalidate({ currentUrl, nextUrl, defaultShouldRevalidate }) {
  if (currentUrl.pathname === nextUrl.pathname) {
    if (currentUrl.searchParams.toString() !== nextUrl.searchParams.toString()) {
      return false; // Don't revalidate on URL param changes (tab switches)
    }
  }
  return defaultShouldRevalidate;
}
```

### Technical Constraints

[Source: docs/architecture/coding-standards.md]

**Critical Fullstack Rules**:
1. **Type Sharing**: Define types in `packages/types` and import from there. Create `packages/types/src/workflow-events.ts`.
2. **API Calls**: Use Eden Treaty client - never direct HTTP calls. Pattern: `client.api.workflows[id].timeline.get()`.
3. **Environment Variables**: Frontend uses `config.apiUrl` - never access env vars directly.
4. **Error Handling**: All API routes must use standard error handler with `success: false, error: { code, message, timestamp }`.
5. **Tenant Isolation**: ALL database queries MUST include tenant filter: `eq(workflowEvents.tenantId, tenantId)`.
6. **Authentication Middleware**: Use `authenticate` middleware directly on routes. Perform role checks inside handlers: `if (!user?.role || ...)`.
7. **Database Schema**: Verify field names against actual schema files before writing queries. Add null checks for joined data: `actor?.fullName`.
8. **ElysiaJS Validation**: Use TypeBox (`t.*`) exclusively for route validation, never Zod.
9. **ElysiaJS Route Organization**: Only parent aggregator (`index.ts`) should have prefix. Child routes must NOT have prefixes.
10. **Remix Data Loading**: ALL data fetching must be in loaders (server-side), never in useEffect. Use `Promise.all` for parallel fetching.
11. **Remix Mutations**: Use `useRevalidator()` for mutations, never manual state updates.

**Naming Conventions**:
- Components: PascalCase (`WorkflowTimeline.tsx`)
- Hooks: camelCase with 'use' (`useAuth.ts`)
- Services: PascalCase + 'Service' (`workflow-event-logger.ts`)
- API Routes: kebab-case (`timeline.ts`, `export-timeline-pdf.ts`)
- Database Tables: snake_case (`workflow_events`)

### Security & Performance

[Source: docs/architecture/security-and-performance.md]

**Security Requirements**:
- **Authentication**: All endpoints require JWT Bearer token via `authenticate` middleware
- **Tenant Isolation**: NEVER allow cross-tenant data access. Always filter by `tenantId`.
- **Immutable Audit Trail**: Workflow events CANNOT be edited or deleted (AC 7). No UPDATE or DELETE routes for events.
- **SQL Injection Protection**: Use Drizzle ORM parameterized queries - never string concatenation
- **Input Validation**: Use TypeBox schemas for all route parameters and query params

**Performance Optimization**:
- **Database Indexes**: Create indexes on `(workflow_id, created_at DESC)` for fast timeline queries
- **Denormalized Data**: Store actor name and role in events table to avoid JOIN on users table
- **Parallel Queries**: Use `Promise.all` in loader to fetch workflow and timeline in parallel
- **Client-side Filtering**: Filter timeline events in React (not server) since audit trails are typically <1000 events
- **PDF Generation**: Generate PDFs asynchronously if possible, show loading spinner during generation

**Estimated Performance**:
- Timeline query: <150ms (with indexes)
- PDF generation: <2s for 500 events
- Client-side filtering: <50ms (instant)
