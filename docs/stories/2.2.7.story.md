## Story

**As an** admin,  
**I want** to configure workflow steps including approvers and form/document action modes,  
**so that** the workflow matches how my organization works.

---

## Acceptance Criteria

1. Workflow builder UI exists
2. admin can create workflow templates scoped to tenant_id
3. For each step, admin can configure:
   - Step name
   - Task title
   - Task description
   - Due days (optional)
4. For each step, admin can optionally attach a form and set:
   - form_template_version_id
   - form_action_mode = fill_out or validate
5. For each step, admin can optionally attach a document template and set:
   - document_template_id
   - document_action_mode = upload or validate
6. For multi-approver steps, admin can configure:
   - Multi-approver flag
   - Number of approvers
   - For each approver, admin can select:
     - A role (tenant-wide), or
     - A specific user (within the tenant)
7. If a role is selected:
   - All users in the same tenant_id with that role will see and be able to complete the task
8. Workflow templates can be published, creating immutable versions

- Pass data to components via props
- Add `shouldRevalidate` for routes with URL state (tabs, filters, sorting)

**ElysiaJS Route Organization** [Source: architecture/coding-standards.md#ElysiaJS Route Organization]:
- ONLY parent aggregator (`index.ts`) should have a prefix (`/workflow-templates`)
- Child routes must NOT have prefixes - parent provides them
- Use consistent parameter names: `workflowId`, `versionId`, `stepId`, `approverId`
- Use TypeBox (`t.*`) for validation, never Zod
- For enums, use `t.Union([t.Literal('value1'), t.Literal('value2')])`

**Remix Routing Conventions** [Source: architecture/routing-conventions.md]:
- Use sibling route pattern for edit pages: `$id_.edit.tsx` (trailing underscore)
- Edit pages rarely share layout with detail pages
- Sibling routes are independent, no `<Outlet />` needed
- Child routes (no trailing underscore) require parent to have `<Outlet />`

**Authentication & Permissions** [Source: architecture/coding-standards.md#Authentication Middleware]:
- Use `authenticate` middleware directly on routes

- Frontend: Use SSR permissions from `_app` loader for UI visibility (no flash)
- Only admin role can manage workflow templates

**Tenant Isolation** [Source: architecture/high-level-architecture.md]:
- All database queries MUST include tenant filter

- Middleware enforces tenant context on all operations

### Data Models

**Workflow Template** [Source: architecture/workflow-template-schema.md]:

```typescript
// Tables
workflow_template {
  id: UUID (PK)
  tenant_id: UUID (FK → tenants, CASCADE)
  process_type: VARCHAR (e.g., 'supplier_qualification', 'sourcing')
  name: VARCHAR
  description: TEXT
  status: VARCHAR ('draft', 'published', 'archived')
  created_by: UUID (FK → users, RESTRICT)
  created_at, updated_at, deleted_at
}

workflow_template_version {
  id: UUID (PK)
  workflow_template_id: UUID (FK → workflow_template, CASCADE)
  tenant_id: UUID (FK → tenants, CASCADE)
  version: INTEGER (1, 2, 3, ...)
  status: VARCHAR ('draft', 'published', 'archived')
  is_published: BOOLEAN
  UNIQUE (workflow_template_id, version)
  CHECK (is_published = false OR (is_published = true AND status = 'published'))
}

workflow_step_template {
  id: UUID (PK)
  workflow_template_version_id: UUID (FK → workflow_template_version, CASCADE)
  tenant_id: UUID (FK → tenants, CASCADE)
  step_order: INTEGER
  name: VARCHAR
  step_type: VARCHAR ('form', 'approval', 'document', 'task')
  task_title: VARCHAR
  task_description: TEXT
  due_days: INTEGER (nullable)
  assignee_type: VARCHAR ('role', 'user')
  assignee_role: VARCHAR (nullable)
  assignee_user_id: UUID (nullable, FK → users, RESTRICT)
  form_template_version_id: UUID (nullable, FK → form_template_version, RESTRICT)
  form_action_mode: VARCHAR ('fill_out', 'validate')
  document_template_id: UUID (nullable)
  document_action_mode: VARCHAR ('upload', 'validate')
  multi_approver: BOOLEAN
  approver_count: INTEGER (nullable)
  decline_returns_to_step_offset: INTEGER (default: 1)
  metadata: JSONB (extensible)
}

step_approver {
  id: UUID (PK)
  workflow_step_template_id: UUID (FK → workflow_step_template, CASCADE)
  tenant_id: UUID (FK → tenants, CASCADE)
  approver_order: INTEGER
  approver_type: VARCHAR ('role', 'user')
  approver_role: VARCHAR (nullable)
  approver_user_id: UUID (nullable, FK → users, RESTRICT)
}
```

### Common Query Examples

**Drizzle ORM Query Patterns** [Source: architecture/workflow-template-schema.md]:

```typescript
import { db } from "@supplex/db";
import { 
  workflowTemplate, 
  workflowTemplateVersion, 
  workflowStepTemplate,
  stepApprover 
} from "@supplex/db";
import { eq, and, isNull, desc, asc } from "drizzle-orm";

// Get all templates for tenant
const templates = await db.query.workflowTemplate.findMany({
  where: and(
    eq(workflowTemplate.tenantId, tenantId),
    isNull(workflowTemplate.deletedAt)
  ),
  orderBy: [desc(workflowTemplate.updatedAt)],
});

// Get template with latest version
const template = await db.query.workflowTemplate.findFirst({
  where: and(
    eq(workflowTemplate.id, templateId),
    eq(workflowTemplate.tenantId, tenantId),
    isNull(workflowTemplate.deletedAt)
  ),
  with: {
    versions: {
      where: isNull(workflowTemplateVersion.deletedAt),
      orderBy: [desc(workflowTemplateVersion.version)],
      limit: 1,
    },
  },
});

// Get steps with approvers
const steps = await db.query.workflowStepTemplate.findMany({
  where: and(
    eq(workflowStepTemplate.tenantId, tenantId),
    eq(workflowStepTemplate.workflowTemplateVersionId, versionId),
    isNull(workflowStepTemplate.deletedAt)
  ),
  with: {
    approvers: {
      where: isNull(stepApprover.deletedAt),
      orderBy: [asc(stepApprover.approverOrder)],
    },
  },
  orderBy: [asc(workflowStepTemplate.stepOrder)],
});

// Create workflow template with version and steps
await db.transaction(async (tx) => {
  // 1. Create template
  const [template] = await tx.insert(workflowTemplate).values({
    tenantId,
    processType: "supplier_qualification",
    name: "Supplier Qualification v2",
    description: "Enhanced qualification process",
    status: "draft",
    createdBy: userId,
  }).returning();

  // 2. Create version
  const [version] = await tx.insert(workflowTemplateVersion).values({
    workflowTemplateId: template.id,
    tenantId,
    version: 1,
    status: "draft",
    isPublished: false,
  }).returning();

  // 3. Create steps
  const step1 = await tx.insert(workflowStepTemplate).values({
    workflowTemplateVersionId: version.id,
    tenantId,
    stepOrder: 1,
    name: "Fill Out Supplier Profile",
    stepType: "form",
    taskTitle: "Complete supplier information",
    taskDescription: "Provide basic company details.",
    dueDays: 7,
    assigneeType: "role",
    assigneeRole: "supplier_contact",
    formTemplateVersionId: formVersionId,
    formActionMode: "fill_out",
    multiApprover: false,
    declineReturnsToStepOffset: 1,
  }).returning();

  // 4. Create multi-approver step with approvers
  const step2 = await tx.insert(workflowStepTemplate).values({
    workflowTemplateVersionId: version.id,
    tenantId,
    stepOrder: 2,
    name: "Review Supplier Profile",
    stepType: "approval",
    taskTitle: "Review supplier submission",
    taskDescription: "Review for completeness.",
    dueDays: 3,
    formTemplateVersionId: formVersionId,
    formActionMode: "validate",
    multiApprover: true,
    approverCount: 2,
    declineReturnsToStepOffset: 1,
  }).returning();

  // 5. Add approvers
  await tx.insert(stepApprover).values([
    {
      workflowStepTemplateId: step2[0].id,
      tenantId,
      approverOrder: 1,
      approverType: "role",
      approverRole: "procurement_manager",
    },
    {
      workflowStepTemplateId: step2[0].id,
      tenantId,
      approverOrder: 2,
      approverType: "role",

    },
  ]);

```

### Versioning and Immutability Rules

**Version Lifecycle** [Source: architecture/workflow-template-schema.md#Versioning and Immutability]:

```
Draft → Published → Archived
  ↓         ↓          ↓
Mutable  Immutable  Immutable
```

**Enforcement**:
- **Draft**: Can edit/delete freely
- **Published**: Immutable (enforced at application level, not database)
  - Reject PUT/DELETE requests with 400 error
  - Error message: "Cannot modify published versions"
- **Archived**: Immutable (same as published)
  - Only transition allowed: none (end state)

**Publishing Logic**:
```typescript
// Validate version is draft
if (version.status !== 'draft') {
  throw new Error('Only draft versions can be published');
}

// Update to published
await db.update(workflowTemplateVersion)
  .set({
    status: 'published',
    isPublished: true,
    updatedAt: new Date(),
  })
  .where(eq(workflowTemplateVersion.id, versionId));
```

### Multi-Approver Business Logic

**Configuration Rules** [Source: architecture/workflow-template-schema.md#Multi-Approver Configuration]:

1. `multi_approver = true` requires `approver_count > 0`
2. Number of approvers in `step_approver` table must be >= `approver_count`
3. Each approver must have either:
   - `approver_type = 'role'` and `approver_role` set, OR
   - `approver_type = 'user'` and `approver_user_id` set
4. At runtime, workflow engine creates one `task_instance` per approver
5. Step completes when `approver_count` tasks reach "completed"

**Example**:
- Step requires 2 approvals (`approver_count = 2`)
- 3 approvers defined (procurement_manager, quality_manager, CEO)
- Any 2 of the 3 can approve
- Once 2 approve, step completes

### Form and Document Integration

**Form Action Modes** [Source: architecture/workflow-template-schema.md#Form and Document Integration]:

1. **fill_out**: User fills form, can save draft, submits when complete
   - Use case: Initial data entry, questionnaires
   - UI: Editable form fields, "Save Draft" + "Submit" buttons

2. **validate**: User reviews read-only form, approves or declines
   - Use case: Manager review, compliance validation
   - UI: Read-only form fields, "Approve" + "Decline" buttons
   - Decline requires comment, returns to previous step

**Document Action Modes**:

1. **upload**: User uploads required documents
   - Use case: Supplier uploads certifications, financial statements
   - UI: File upload zone, document checklist

2. **validate**: User reviews documents, approves or declines
   - Use case: Quality manager reviews certifications
   - UI: Document viewer, "Approve" + "Decline" buttons
   - Decline requires comment, returns to previous step

**UI Considerations**:
- Form/document sections are optional (can have neither, one, or both)
- When form attached: Show form template name in step card
- When document attached: Show document template name in step card
- Action mode badge: "Fill Out" vs "Validate" (different colors)
