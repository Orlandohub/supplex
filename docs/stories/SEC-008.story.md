# Story SEC-008: Standardize Admin Enforcement & Remove Dead Admin Code

<!-- Powered by BMAD™ Core -->

## Status

**Done**

## Story

**As a** platform operator,
**I want** all admin-only API routes enforced via `requireAdmin` / `requireRole` middleware instead of inline handler checks, and dead admin route code removed,
**so that** authorization is declarative, auditable at the route level, consistent across all modules, and impossible to bypass by forgetting an inline check.

---

## Context

The auth analysis found that admin enforcement is split between two patterns: some routes use `requireRole`/`requireAdmin` middleware (the correct pattern), while others use inline `if (user.role !== UserRole.ADMIN)` checks inside handlers. The inline pattern is error-prone — it is easy to forget, produces inconsistent error responses, and cannot be audited by inspecting route declarations alone.

Additionally, the `admin/index.ts` aggregator has a misleading comment claiming "All routes require Admin" when enforcement is actually per-child via inline checks, and a dead admin route file (`workflow-statuses.ts`) exists on disk but is not registered.

After SEC-005 adds missing role gates and SEC-008 converts inline checks to middleware, all admin routes will use declarative middleware enforcement.

### Analysis Report Items Addressed

| ID | Finding | Severity |
|----|---------|----------|
| M1 | Admin enforcement split between `requireRole([ADMIN])` middleware and inline `if (user.role !== ADMIN)` checks | Medium |

| M4 | `admin/workflow-statuses.ts` implemented but not registered in `admin/index.ts` — dead code | Low |

- **Independent of SEC-001 through SEC-004 and SEC-006/007.** This is a code consistency refactor, not a security-critical change.

### Why Group-Level for Admin, Per-Route for Templates

**Admin routes** (`/api/admin/*`): Every route in this group requires admin. Group-level enforcement is the cleanest approach — one `.use(requireAdmin)` protects all current and future admin routes. A new admin route added without an inline check is automatically protected.

**Template routes** (`/api/workflow-templates/*`, `/api/form-templates/*`): Some routes allow non-admin access (list, get, get-published). Group-level enforcement would block these. Per-route middleware is required.

### ElysiaJS Plugin Deduplication

`requireAdmin` and `requireRole` internally call `.use(authenticate)`. ElysiaJS deduplicates named plugins — the `authenticate` plugin (with `name: "authenticate"`) is only applied once even if multiple middleware layers reference it. This means:
- Group-level `requireAdmin` + child route `.use(authenticate)` → no double auth. But the child `.use(authenticate)` is redundant and can be removed for clarity.
- Per-route `.use(requireAdmin)` fully replaces `.use(authenticate)` + inline check.

### Error Response Shape Change

The inline admin checks use this pattern:
```typescript
if (!user?.role || user.role !== UserRole.ADMIN) {
  set.status = 403;
  return {
    success: false,
    error: {
      code: "FORBIDDEN",
      message: "Access denied. Required role: Admin",
      timestamp: new Date().toISOString(),
    },
  };
}
```

The `requireAdmin` middleware produces (thrown as `new Error(JSON.stringify(...))` in `onBeforeHandle`):
```typescript
{
  error: {
    code: "FORBIDDEN",
    message: "Access denied. Required role(s): admin. Your role: viewer",
    timestamp: "2026-04-06T..."
  }
}
```

Key differences:
- No `success` wrapper in middleware response (inline returns `{ success: false, error: ... }`)
- `message` includes the user's actual role (useful for debugging)
- Middleware throws an `Error` with JSON-stringified body; inline checks use `set.status = 403; return {...}`

Note: some admin route inline checks use a simpler error shape (`{ success: false, error: "Access denied. Admin role required." }`) rather than the structured `{ code, message, timestamp }` object shown above. Both patterns are removed by this story.

This is an acceptable minor change. The HTTP status (403) is unchanged. Consumers checking for 403 status are unaffected. Tests that assert on the exact response body will need updating.

### Implementation Guardrails

1. **Do not convert routes that intentionally allow non-admin access.** These must keep `.use(authenticate)` — see the "Intentionally Keeping `authenticate`" table above.
2. **Do not change route registration, URL patterns, or core business logic.** Only change the authorization mechanism.
3. **After removing inline checks, verify that the handler no longer references `UserRole` for authorization purposes.** If `UserRole` was imported only for the inline check, remove the unused import.
4. **The admin group-level middleware must be applied BEFORE child routes are mounted.** In ElysiaJS: `.use(requireAdmin).use(childRoute)`, not `.use(childRoute).use(requireAdmin)`.
5. **Do not modify the `requireAdmin` or `requireRole` middleware implementations.** They are correct as-is.
6. **Once group-level `requireAdmin` is applied in `admin/index.ts`, all child admin routes must remove redundant child-level `authenticate` / `requireRole([ADMIN])` middleware.** The final state must be uniform: group-level middleware is the single authorization layer. If any child admin route retains its own admin middleware, a route-specific exception must be explicitly documented and justified in completion notes.

### Testing [Source: architecture/testing-strategy.md]

- **Backend tests:** Bun test runner (`bun test`)
- **Key verification:**
  - Non-admin request to any `/api/admin/*` endpoint → 403
  - Non-admin request to admin-only template endpoint → 403
  - Admin request to admin-only endpoint → success
  - Non-admin request to `list.ts` (templates) → success (filtered results)
  - Non-admin request to `get-published-by-tenant.ts` → success
- **Test updates:** Tests that assert on the specific inline error response body (`{ success: false, error: { code: "FORBIDDEN", ... } }`) must be updated to match the middleware response shape.

### Coding Standards [Source: architecture/coding-standards.md]

- Authentication middleware: use `authenticate` directly on routes, not nested in wrappers
- `requireRole` / `requireAdmin` includes `authenticate` — do not chain both
- Admin routes: `requireAdmin` or `requireRole([UserRole.ADMIN])` — both are correct; `requireAdmin` is preferred for readability
- Error responses: middleware handles the 403 response — no handler code needed
