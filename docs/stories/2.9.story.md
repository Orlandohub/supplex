## Story

**As a** procurement manager,  
**I want** to see all qualification workflows with filtering,  
**so that** I can track progress and identify bottlenecks.

---

## Acceptance Criteria

1. "Qualifications" page displays list of all workflows in tenant
2. Table columns: Supplier Name, Status (badge with color), Current Stage, Initiated By, Initiated Date, Days In Progress, Risk Score
3. Status filter dropdown: All, Draft, In Progress (Stage 1-3), Approved, Rejected
4. Stage filter: All, Stage 1, Stage 2, Stage 3
5. Risk filter: All, Low, Medium, High
6. Search bar filters by supplier name
7. Sort by: Initiated date (newest/oldest), Days in progress (longest/shortest), Risk score (high/low)
8. Clicking workflow row navigates to workflow detail page
9. "My Tasks" tab shows only workflows assigned to current user for review
10. "My Initiated" tab shows only workflows initiated by current user
11. Pagination for lists over 20 workflows
12. Empty state displays "No qualifications found" with "Start New Qualification" CTA
13. Mobile view converts to card layout with swipe actions
14. Export to CSV button exports current filtered view

### Database Query Patterns

[Source: architecture/database-schema.md#SQL-DDL]

**List Workflows with Filters:**
```typescript
// Drizzle ORM query pattern
const workflows = await db
  .select({
    id: qualificationWorkflows.id,
    supplierName: suppliers.name,
    supplierId: qualificationWorkflows.supplierId,
    status: qualificationWorkflows.status,
    currentStage: qualificationWorkflows.currentStage,
    initiatedBy: users.fullName,
    initiatedDate: qualificationWorkflows.initiatedDate,
    riskScore: qualificationWorkflows.riskScore,
  })
  .from(qualificationWorkflows)
  .leftJoin(suppliers, eq(qualificationWorkflows.supplierId, suppliers.id))
  .leftJoin(users, eq(qualificationWorkflows.initiatedBy, users.id))
  .where(
    and(
      eq(qualificationWorkflows.tenantId, tenantId), // Tenant isolation
      isNull(qualificationWorkflows.deletedAt), // Exclude soft-deleted
      // Apply filters conditionally
      status ? eq(qualificationWorkflows.status, status) : undefined,
      stage ? eq(qualificationWorkflows.currentStage, stage) : undefined,
      search ? like(suppliers.name, `%${search}%`) : undefined,
      // Risk level filter
      riskLevel === 'Low' ? lt(qualificationWorkflows.riskScore, sql`3.0`) : undefined,
      riskLevel === 'Medium' ? and(gte(qualificationWorkflows.riskScore, sql`3.0`), lte(qualificationWorkflows.riskScore, sql`6.0`)) : undefined,
      riskLevel === 'High' ? gt(qualificationWorkflows.riskScore, sql`6.0`) : undefined,
    )
  )
  .orderBy(
    sortOrder === 'desc'
      ? desc(qualificationWorkflows[sortBy])
      : asc(qualificationWorkflows[sortBy])
  )
  .limit(limit)
  .offset((page - 1) * limit);

// Calculate days in progress (server-side)
const workflowsWithDays = workflows.map(w => ({
  ...w,
  daysInProgress: Math.floor(
    (Date.now() - new Date(w.initiatedDate).getTime()) / (1000 * 60 * 60 * 24)
  ),
}));
```

### Coding Standards Compliance

[Source: architecture/coding-standards.md#Critical-Fullstack-Rules]

**CRITICAL RULES:**

1. **ElysiaJS Route Organization:**
   - Only parent aggregator (`workflows/index.ts`) has prefix
   - Child routes (`list-qualifications.ts`, `export-qualifications.ts`) must NOT have prefixes
   - Use consistent parameter naming

2. **ElysiaJS Validation:**
   - Use TypeBox (`t.*`) exclusively, never Zod (`z.*`)
   - For enums, use `t.Union([t.Literal(...), ...])`

3. **Authentication Middleware:**
   - Use `authenticate` middleware directly on routes
   - Perform role checks inside handlers with null checks: `if (!user?.role || ...)`

4. **Database Schema:**
   - Verify field names against actual schema files
   - Add null checks for joined data: `suppliers?.name`, `users?.fullName`

5. **Remix Data Loading:**
   - ALL data fetching in loaders (server-side), never useEffect
   - Use `Promise.all` for parallel data fetching
   - Pass data via props, not by fetching in components

6. **Remix shouldRevalidate:**
   - MUST add `shouldRevalidate` function to prevent revalidation on filter/tab URL changes
   - Only revalidate on actual route changes or explicit actions

7. **Tenant Isolation:**
   - All database queries must include tenant filter
   - Enforce in all route handlers and services

8. **Type Sharing:**
   - Define types in `packages/types` and import from there

9. **Error Handling:**
   - All API routes must use standard error handler
   - Return proper HTTP status codes (404, 403, 500)

10. **URL-Based State:**
    - Use `useSearchParams` for all filters, sorting, pagination, tabs
    - Enables shareable/bookmarkable URLs

### Remix Pattern Requirements

[Source: architecture/remix-patterns.md#Server-Side-Data-Loading]

**CRITICAL PATTERNS:**

1. **Server-Side Data Loading:**
   - Fetch ALL data in loader (no useEffect)
   - Return token for client-side mutations
   - Use `Promise.all` for parallel fetching

2. **shouldRevalidate Implementation:**
   ```typescript
   export function shouldRevalidate({
     currentUrl,
     nextUrl,
     defaultShouldRevalidate,
   }) {
     // Don't revalidate on search param changes (filters, tabs, sorting)
     if (currentUrl.pathname === nextUrl.pathname) {
       if (currentUrl.searchParams.toString() !== nextUrl.searchParams.toString()) {
         return false; // ✅ No revalidation needed
       }
     }
     
     // Route change or action = use default behavior
     return defaultShouldRevalidate;
   }
   ```

3. **Props-Based Components:**
   - Pass data via props (not by fetching in children)
   - Use `useRevalidator()` for mutations

4. **URL-Based State:**
   - Use `useSearchParams` for filters, tabs, sorting, pagination
   - Shareable and bookmarkable URLs

5. **Error Handling:**
   - Proper error boundaries
   - Return proper HTTP status codes in loaders
