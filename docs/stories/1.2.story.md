### Data Models & Schema Design

[Source: architecture/data-models.md]

**Design Principles:**
- **UUID primary keys** for security (no sequential ID guessing)
- **Soft deletes** with `deleted_at` timestamp for audit trail
- **JSONB fields** for flexible metadata and extensibility
- **Enum types** for status fields (enforced at database level)
- **Foreign key constraints** with CASCADE/RESTRICT based on business rules
- **Normalized schema** for data integrity

#### Model 1: Tenant

**Purpose:** Multi-tenant isolation root entity

**Key Fields:**
- `id` (UUID, PK)
- `name` (VARCHAR 200, NOT NULL) - Organization name
- `slug` (VARCHAR 100, UNIQUE) - URL-friendly identifier
- `status` (ENUM: active, suspended, cancelled)
- `plan` (ENUM: starter, professional, enterprise)
- `settings` (JSONB) - Tenant-specific configuration
- `subscription_ends_at` (TIMESTAMP WITH TIME ZONE)
- `created_at`, `updated_at` (TIMESTAMP WITH TIME ZONE)

**TypeScript Interface:**
```typescript
export enum TenantStatus {
  ACTIVE = "active",
  SUSPENDED = "suspended",
  CANCELLED = "cancelled",
}

export enum TenantPlan {
  STARTER = "starter",
  PROFESSIONAL = "professional",
  ENTERPRISE = "enterprise",
}

export interface Tenant {
  id: string; // UUID
  name: string;
  slug: string;
  status: TenantStatus;
  plan: TenantPlan;
  settings: {
    evaluationFrequency?: "monthly" | "quarterly" | "annually";
    notificationEmail?: string;
    customFields?: Record<string, any>;
    qualificationRequirements?: string[];

  subscriptionEndsAt: Date | null;
  createdAt: Date;
  updatedAt: Date;
}
```

### Database Schema DDL

[Source: architecture/database-schema.md#sql-ddl-postgresql]

**Key SQL Patterns:**

```sql
-- Enable UUID extension
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Tenants Table
CREATE TABLE tenants (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name VARCHAR(200) NOT NULL,
    slug VARCHAR(100) NOT NULL UNIQUE,
    status VARCHAR(50) NOT NULL DEFAULT 'active',
    plan VARCHAR(50) NOT NULL DEFAULT 'starter',
    settings JSONB DEFAULT '{}',
    subscription_ends_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL
);

-- Users Table (references Supabase auth.users)
CREATE TABLE users (
    id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    email VARCHAR(255) NOT NULL,
    full_name VARCHAR(200) NOT NULL,
    role VARCHAR(50) NOT NULL,
    avatar_url VARCHAR(500),
    is_active BOOLEAN DEFAULT TRUE NOT NULL,
    last_login_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,
    UNIQUE(tenant_id, email)
);

-- Suppliers Table
CREATE TABLE suppliers (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    name VARCHAR(200) NOT NULL,
    tax_id VARCHAR(50) NOT NULL,
    category VARCHAR(50) NOT NULL,
    status VARCHAR(50) NOT NULL DEFAULT 'prospect',
    performance_score NUMERIC(3, 2),
    contact_name VARCHAR(200) NOT NULL,
    contact_email VARCHAR(255) NOT NULL,
    contact_phone VARCHAR(50),
    address JSONB NOT NULL,
    certifications JSONB DEFAULT '[]',
    metadata JSONB DEFAULT '{}',
    risk_score NUMERIC(4, 2),
    created_by UUID NOT NULL REFERENCES users(id),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL,
    deleted_at TIMESTAMP WITH TIME ZONE,
    UNIQUE(tenant_id, tax_id)
);

-- Performance Indexes
CREATE INDEX idx_suppliers_tenant_id ON suppliers(tenant_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_suppliers_status ON suppliers(tenant_id, status) WHERE deleted_at IS NULL;
CREATE INDEX idx_suppliers_name ON suppliers(tenant_id, name) WHERE deleted_at IS NULL;
```

### Multi-Tenancy Architecture

[Source: architecture/high-level-architecture.md#architectural-patterns]

**Multi-tenant with Row Level Security (RLS):**
- Database-enforced tenant isolation using PostgreSQL RLS policies
- Application-level checks as additional layer (defense-in-depth)
- All queries automatically filtered by tenant_id from JWT
- Required for compliance (SOC 2, GDPR)

**RLS Policy Pattern:**
```sql
-- Enable RLS
ALTER TABLE suppliers ENABLE ROW LEVEL SECURITY;

-- Policy: Users can only see suppliers in their tenant
CREATE POLICY tenant_isolation_suppliers ON suppliers
  USING (tenant_id = auth.jwt() ->> 'tenant_id');
```

**Hybrid Query Strategy:**
- **Read operations:** Supabase SDK (fast, RLS-protected) - Used in Remix loaders
- **Write operations:** ElysiaJS + Drizzle (transactional, business logic) - Used in API endpoints

### Coding Standards

[Source: architecture/coding-standards.md#critical-fullstack-rules]

**Database-Specific Rules:**
- **Tenant Isolation:** ALL database queries MUST include tenant filter (no exceptions)
- **Type Sharing:** Always define types in `packages/types` and import from there
- **No Direct SQL:** Use Drizzle ORM exclusively (except for RLS policies)
- **Naming:** Database tables use `snake_case`, TypeScript uses `camelCase`

**Table Naming:**
| Element | Convention | Example |
|---------|------------|---------|
| Database Tables | snake_case | `supplier_evaluations` |
| TypeScript Types | PascalCase | `SupplierEvaluation` |
| Fields (DB) | snake_case | `created_at` |
| Fields (TS) | camelCase | `createdAt` |

### Security Considerations

[Source: architecture/security-and-performance.md]

**For this story:**
- **Never skip tenant_id filter** - All queries must enforce tenant isolation
- **RLS policies are mandatory** - Database-level defense against bugs
- **UUID primary keys** - Prevents ID guessing attacks
- **Connection string security** - Store in `.env`, never commit to git
- **Validate tenant_id from JWT** - Never trust client-sent tenant IDs

### Security Review

The implementation uses a three-layer security approach:

1. **Database Layer (RLS Policies)**
   - PostgreSQL RLS enabled on all tables
   - Policies automatically filter by tenant_id from JWT
   - Separate policies for SELECT, INSERT, UPDATE, DELETE
   - Soft-delete handling in policies

2. **Application Layer (Helper Functions)**
   - `withTenantId()` enforces tenant filtering in queries
   - `withTenantIdAndNotDeleted()` combines isolation with soft-delete checks
   - `TenantContextError` provides clear error messages
   - UUID validation prevents injection attacks

   - `requireTenant()` middleware extracts tenant from JWT
   - `extractTenantContext()` validates JWT structure
   - Automatic tenant context injection into request handlers

### Performance Considerations

**Status: VERY GOOD ✅**

**Index Optimization:**

The schema includes well-designed indexes for tenant-based queries:

1. **Composite Indexes (tenant_id first)**
   - `idx_suppliers_tenant_status` on (tenant_id, status)
   - `idx_suppliers_tenant_name` on (tenant_id, name)
   - Tenant-first ordering optimal for multi-tenant queries

2. **Partial Indexes (performance optimization)**
   - `WHERE deleted_at IS NULL` excludes soft-deleted records
   - Reduces index size and improves query performance
   - Matches common query patterns

3. **Single-Column Indexes**
   - `idx_users_tenant_id` for user lookups
   - `idx_contacts_supplier_id` for contact lookups
   - `idx_contacts_tenant_id` for tenant-based contact queries
   - `idx_documents_tenant_id` for document lookups
   - `idx_documents_supplier_id` for supplier document queries

**Connection Pooling:**

- Max 20 connections (appropriate for Supabase Starter plan)
- 60-second idle timeout (prevents connection leaks)
- 10-second connect timeout (fast failure detection)

**Query Performance:**

Most common query patterns are optimized:
- Tenant-based supplier listing (uses tenant_status index)
- Supplier name search (uses tenant_name index)
- Supplier contacts (uses supplier_id index)
- Document lookups (uses tenant_id and supplier_id indexes)
