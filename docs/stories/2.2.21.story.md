## Story

**As a** platform operator,
**I want** all API requests and workflow engine operations to emit structured JSON logs with correlation IDs, and a health-check endpoint to detect stuck or inconsistent workflows,
**so that** production issues can be diagnosed quickly, operations can be correlated end-to-end, and inconsistent workflow states are detected before users report them.

---

## Context

The architecture assessment (`.cursor/plans/architecture_assessment_report_5c337427.plan.md`, Section 4.4) found that the system has **zero operational observability infrastructure**:

- **0 structured log calls** — all logging is `console.log`/`console.warn`/`console.error` with free-form string concatenation (~28 calls across 7 engine files, ~80+ calls across all route files).
- **0 correlation IDs** — no request ID is generated or propagated. A step completion that triggers transition, task creation, document seeding, and event logging cannot be correlated across log lines.
- **0 health-check queries** — no mechanism to detect stuck workflows (`in_progress` with no active step), orphaned tasks, or state mismatches.

The tech stack specifies **Axiom** for structured logging (SaaS) and pino is the industry standard JSON logger for Node.js/Bun that integrates with Axiom out of the box.

### Assessment Items Addressed

| ID | Finding |
|----|---------|

| M8 | Fire-and-forget audit logging with no completeness guarantee |

This story introduces the **logging foundation** scoped to workflow observability. It does NOT cover:
- Replacing `console.*` in non-workflow route files (suppliers, users, form-templates, auth, admin, etc.) — deferred to a future API-wide logging cleanup story
- Prometheus metrics/counters (future story)
- Alerting rules (depends on deployment platform)
- Dead-letter pattern for failed transitions (future story)
- Full observability dashboard (Axiom configuration)

## Acceptance Criteria

### AC 1: Structured Logger Module

1. A new `apps/api/src/lib/logger.ts` module exports a pino logger instance configured for JSON output with base fields: `{ service: "supplex-api", environment: process.env.NODE_ENV }`.
2. The logger supports child loggers via `logger.child({ correlationId, tenantId })` for request-scoped context.
3. Log level is configurable via `LOG_LEVEL` environment variable (default: `info` in production, `debug` in development).
4. In development mode, the logger uses `pino-pretty` for human-readable output. **Important:** Pino's `transport` option (worker_threads-based) is incompatible with Bun. Use pipe-based pretty printing instead — see Dev Notes for the recommended pattern.

### AC 2: Request-Scoped Correlation ID Middleware

5. A new Elysia middleware (`apps/api/src/lib/correlation-id.ts`) generates a UUID v4 correlation ID for each incoming request.
6. If the request includes an `X-Correlation-ID` header, that value is used instead (for client-initiated tracing).
7. The correlation ID is attached to the Elysia request context (e.g., `correlationId` field on the derive context) so all route handlers can access it.
8. The correlation ID is included in all response headers as `X-Correlation-ID`.
9. The middleware creates a child logger (`requestLogger`) with `{ correlationId, method, path, tenantId }` and attaches it to the context so route handlers use `requestLogger.info(...)` instead of `console.log(...)`.

### AC 3: Replace `console.*` in Workflow Engine

10. All `console.log`, `console.warn`, and `console.error` calls in the 7 workflow engine files are replaced with the structured logger:
    - `complete-step.ts` (4 calls)
    - `approve-validation-task.ts` (5 calls)
    - `create-tasks-for-step.ts` (11 calls)
    - `create-validation-tasks.ts` (5 calls)
    - `transition-to-next-step.ts` (1 call)
    - `instantiate-workflow.ts` (1 call)
    - `seed-step-documents.ts` (1 call)
11. Each engine function accepts an optional `logger` parameter (a pino child logger). If not provided, it uses the module-level logger as fallback.
12. Log entries include contextual fields: `{ tenantId, processId, stepId, taskId, action, durationMs }` as structured key-value pairs — not string-interpolated messages.

### AC 4: Replace `console.*` in Global Handlers & Workflow Route Files

13. The `onRequest` middleware in `index.ts` uses the structured logger instead of `console.log` for request logging.
14. The `onError` handler in `index.ts` uses `logger.error(...)` instead of `console.error(...)`.
15. `console.*` calls in **workflow-related route handlers** are replaced with `requestLogger.error({ err, ... }, "message")`. The in-scope route files are:
    - `routes/workflows/instantiate.ts`
    - `routes/workflows/steps/complete.ts`
    - `routes/workflows/steps/get.ts`
    - `routes/workflows/steps/documents/review.ts`
    - `routes/workflows/steps/documents/upload.ts`
    - `routes/workflows/steps/documents/view.ts`
    - `routes/workflows/processes/get.ts`
    - `routes/workflows/processes/list.ts`
    - `routes/workflows/processes/send-reminder.ts`
    - `routes/workflows/my-tasks.ts`
    - `routes/workflows/my-tasks-count.ts`
    - `routes/workflows/supplier-processes.ts`
    - `routes/workflows/comments/create.ts`
    - `routes/workflows/comments/get-by-step.ts`
    - `routes/documents/upload.ts`
    - `routes/documents/download.ts`
16. Replacing `console.*` in **non-workflow route files** (suppliers, users, form-templates, form-submissions, auth, admin, etc.) is **out of scope** for this story. Those files will be migrated in a future API-wide logging cleanup story.

### AC 5: Correlation ID in Workflow Events

17. The `logWorkflowEvent()` function accepts an optional `correlationId` parameter.
18. When provided, the `correlationId` is stored in the `metadata` JSONB field of the `workflow_event` row (e.g., `metadata: { ...existing, correlationId }`).
19. Route handlers pass the request's `correlationId` to `logWorkflowEvent()` calls.

### AC 6: Request Timing

20. The correlation ID middleware records `startTime = Date.now()` on request entry.
21. An `onAfterHandle` hook logs the completed request with `{ correlationId, method, path, statusCode, durationMs }` at `info` level.
22. Requests taking longer than 5 seconds are logged at `warn` level with a `"slow_request"` flag.

### AC 7: Workflow Health-Check Endpoint (Diagnostic Only)

23. A new `GET /api/admin/workflow-health` endpoint (restricted to `admin` role) returns a **read-only diagnostic report** of inconsistent workflow states:
    - **Stuck processes:** `process_instance WHERE status = 'in_progress' AND current_step_instance_id IS NULL`
    - **Orphaned tasks:** `task_instance WHERE status = 'pending' AND step_instance_id NOT IN (SELECT id FROM step_instance WHERE status = 'active' OR status = 'awaiting_validation')`
    - **State mismatches:** `step_instance WHERE status IN ('completed', 'validated') AND process_instance.status = 'in_progress' AND step_order = (SELECT MAX(step_order) FROM step_instance WHERE process_instance_id = ...)`
24. The endpoint returns counts and a limited sample of IDs (up to 10 per category) for each inconsistency type. It does **not** trigger repairs, retries, mutations, or any side effects — it is purely diagnostic.
25. The endpoint is logged at `info` level with the results.
26. The endpoint must remain lightweight and safe for repeated admin use. It is not a reconciliation engine or repair job.

### AC 8: Auth Middleware Logger Migration

27. The custom `authLogger` object in `middleware.ts` (the const with manual `debug`/`info`/`warn`/`error` wrappers around `console.*`) is replaced with a pino child logger `logger.child({ module: "auth" })`, removing the manual log-level checking logic.

### AC 9: Backward Compatibility

28. All existing functionality continues to work. The only visible change is:
    - Log output is JSON instead of free-form text (in production)
    - A new `X-Correlation-ID` header appears in all responses
    - A new `/api/admin/workflow-health` endpoint is available

## Implementation Guardrails

The following constraints **must not** be violated during implementation:

1. **This story introduces logging, correlation, and diagnostic visibility only.** It must not change workflow business behavior, route request/response semantics, or engine control flow — except where strictly required to thread logger or correlation context through function parameters.
2. **No changes to RBAC middleware or permission matrix** — only the `authLogger` object is replaced; `authenticate`, `requireRole`, and `requirePermission` functions are untouched.
3. **No changes to database schema** — the only new dependency is `pino`/`pino-pretty`; no migrations are added.
4. **No new routes beyond `/api/admin/workflow-health`** — the correlation ID middleware is an enhancement to existing request flow, not a new endpoint.
5. **Health-check endpoint is read-only and diagnostic only** — SELECT queries only. No repairs, retries, mutations, or side effects. It is not a reconciliation engine.
6. **Engine function signatures: additive only** — the `logger` parameter is optional and added at the end; existing callers must continue to work without changes.
7. **`console.*` replacement is scoped to workflow paths** — do not migrate `console.*` in non-workflow route files (suppliers, users, form-templates, auth, admin, etc.). Do not replace `console.*` in test files or test fixtures.
8. **No unrelated refactors** — do not restructure, rename, or reorganize code beyond what is needed for the logging migration. If a file is not listed in the in-scope route list (AC 15), leave it untouched.

## Dev Notes

### Pino Logger Setup [Source: architecture/tech-stack.md — Axiom for structured logging]

Pino is the standard JSON logger for Node.js/Bun. It outputs one JSON line per log entry, which is directly consumable by Axiom, Datadog, ELK, etc.

```typescript
// apps/api/src/lib/logger.ts
import pino from "pino";

const isDev = process.env.NODE_ENV === "development";

export const logger = pino({
  level: process.env.LOG_LEVEL || (isDev ? "debug" : "info"),
  base: { service: "supplex-api", env: process.env.NODE_ENV },
});

export default logger;
```

> **Bun Compatibility Note:** Do NOT use `transport: { target: "pino-pretty" }`. Pino transports rely on Node.js `worker_threads`, which are incompatible with Bun ([pinojs/pino-pretty#448](https://github.com/pinojs/pino-pretty/issues/448), [oven-sh/bun#11401](https://github.com/oven-sh/bun/issues/11401)). Instead, pipe output through pino-pretty in the dev script: `"dev": "bun run --watch src/index.ts | npx pino-pretty"`. In production, JSON output goes directly to Axiom with no transport needed.

### Correlation ID Middleware Pattern

```typescript
// apps/api/src/lib/correlation-id.ts
import { Elysia } from "elysia";
import { randomUUID } from "crypto";
import logger from "./logger";

export const correlationId = new Elysia({ name: "correlation-id" })
  .derive({ as: "global" }, ({ headers, request }) => {
    const id = headers["x-correlation-id"] || randomUUID();
    const url = new URL(request.url);
    const requestLogger = logger.child({
      correlationId: id,
      method: request.method,
      path: url.pathname,
    });
    return { correlationId: id, requestLogger, _requestStart: Date.now() };
  })
  .onAfterHandle(({ correlationId, requestLogger, _requestStart, set }) => {
    set.headers["x-correlation-id"] = correlationId;
    const durationMs = Date.now() - _requestStart;
    const logData = { durationMs, statusCode: set.status || 200 };
    if (durationMs > 5000) {
      requestLogger.warn({ ...logData, slowRequest: true }, "slow request");
    } else {
      requestLogger.info(logData, "request completed");
    }
  });
```

The middleware must be registered in `index.ts` BEFORE route groups so all routes inherit the context:
```typescript
const app = new Elysia()
  .use(cors({ ... }))
  .use(correlationId)   // <-- Add here, before routes
  .onError(...)
  // ... routes
```

### Global Error Handler Context Limitation

The `.onError()` handler in `index.ts` is registered at the app level **before** the correlation ID plugin's `.derive()`. In Elysia, global error handlers do NOT have access to derived context (`requestLogger`, `correlationId`). Task 4.2 must use the **module-level** `logger` (not `requestLogger`). If correlation is needed in the error handler, manually extract it from the request headers:

```typescript
.onError(({ code, error, set, request }) => {
  const corrId = request.headers.get("x-correlation-id") || "unknown";
  logger.error({ err: error, code, correlationId: corrId }, "unhandled error");
  // ... existing error response logic ...
})
```

### Health-Check Queries

```sql
-- Stuck processes: in_progress but no active step pointer
SELECT id, tenant_id, workflow_name, status, updated_at
FROM process_instance
WHERE status = 'in_progress'
  AND current_step_instance_id IS NULL
  AND deleted_at IS NULL
LIMIT 10;

-- Orphaned tasks: pending but step is not active/awaiting_validation
SELECT ti.id, ti.step_instance_id, ti.title, si.status as step_status
FROM task_instance ti
JOIN step_instance si ON si.id = ti.step_instance_id
WHERE ti.status = 'pending'
  AND ti.deleted_at IS NULL
  AND si.status NOT IN ('active', 'awaiting_validation')
LIMIT 10;

-- State mismatch: final step completed but process still in_progress
SELECT pi.id, pi.workflow_name, pi.status, MAX(si.step_order) as max_step
FROM process_instance pi
JOIN step_instance si ON si.process_instance_id = pi.id
WHERE pi.status = 'in_progress'
  AND pi.deleted_at IS NULL
  AND NOT EXISTS (
    SELECT 1 FROM step_instance s2
    WHERE s2.process_instance_id = pi.id
      AND s2.status IN ('active', 'blocked', 'pending')
  )
GROUP BY pi.id
LIMIT 10;
```
