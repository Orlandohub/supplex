## Story

**As an** admin or procurement_manager,  
**I want** to see which users are assigned to workflow tasks, have workflows show both completed and ongoing processes with status information, be able to assign workflow steps to supplier users automatically, and reorder workflow steps when building workflows,  
**so that** I have clear visibility into task ownership and workflow progress, suppliers can participate in workflows through their platform access contact, and I can organize workflow steps in the correct sequence.

---

## Acceptance Criteria

### Process Detail Page - Task Assignment Display (AC 1-3)

1. Process detail page (`/workflows/processes/:id`) displays a "Task Assignments" section
2. For each active step, show all users assigned to tasks for that step
3. Display user full name, email, role.

### Supplier Detail Page - Workflows Tab Enhancement (AC 4-7)

4. Workflows tab shows both completed and ongoing workflow processes
5. For each workflow, display the current status (from `process_instance.status`)
6. Status reflects the most recently completed step's `completion_status`
7. Table includes columns: Process Type, Current Status, Initiated Date, Completed Date (if applicable), Actions(only available for admin and procurement manager, or if the current step is assigned to the logged in user)

### Workflow Step Creation - Remove User Selection (AC 8-10)

8. Step creation form removes the "Assignee User" dropdown entirely
9. Only "Assignee Role" dropdown remains for assignment configuration
10. Multi-approver configuration also removes user-specific selection, keeping only role-based approvers and will be called multiple roles

### Supplier User Role - Auto-Assignment (AC 11-15)

11. "Assignee Role" dropdown includes new option: "Supplier User"
12. When "Supplier User" is selected, UI shows helper text: "Tasks will be auto-assigned to the supplier's platform access contact"
13. At runtime, when a step with `assignee_role='supplier_user'` becomes active:
    - Workflow engine fetches the supplier user from the supplier associated with the process
    - Task is assigned to `supplierUser.id` (from Platform Access Contact)
14. If no supplier user exists when step activates, task is assigned to the procurement manager of the tenant to whom that supplier belongs (fallback)
15. Supplier User role assignment only works for `entity_type='supplier'` workflows

### Workflow Step Reordering (AC 16-20)

16. Step list displays drag handle icons (or up/down arrow buttons) for each step
17. User can drag and drop steps to reorder them (or use up/down buttons)
18. Step order updates automatically maintain sequential numbering (1, 2, 3, ...)
19. Reordering is only allowed for draft workflow template versions
20. Published workflow versions display steps in order but cannot be reordered

### Supplier User Role - Auto-Assignment Logic

**Source: [architecture/data-models.md](#user-model)**

**Assignment Resolution Flow:**
```typescript
// When step with assignee_role='supplier_user' activates:

1. Fetch process_instance to get entity_type, entity_id, and tenant_id
   
2. Verify entity_type === 'supplier'
   - If not supplier, log error and assign to initiator (fallback)

3. Query users table for supplier user:
   ```sql
   SELECT * FROM users 
   WHERE tenant_id = $1 
     AND supplier_id = $2 
     AND role = 'supplier_user'
     AND is_active = true
   LIMIT 1;
   ```

4. If supplier user found:
   - Assign task to supplierUser.id
   - Log: "Task assigned to supplier user [email]"

5. If NO supplier user found (FALLBACK):
   - Query for procurement manager:
     ```sql
     SELECT * FROM users
     WHERE tenant_id = $1
       AND role = 'procurement_manager'
       AND is_active = true
     LIMIT 1;
     ```
   - If procurement manager found: Assign to procurementManager.id
   - Log: "No supplier user found, assigning to procurement manager [email]"

6. If NO procurement manager found (LAST RESORT):
   - Assign task to process_instance.initiated_by
   - Log error: "No procurement manager found, assigning to initiator [email]"

7. Create task_instance:
   assignee_type: 'user'
   assignee_role: NULL
   assignee_user_id: <resolved_user_id>
```

### Step Reordering Implementation

**Drag and Drop Options:**

**Option A: Using @dnd-kit/core (Recommended)**
- Modern, accessible drag-and-drop library
- Better performance than react-dnd
- Built-in keyboard navigation support
- Already used in shadcn/ui components

**Option B: Simple Up/Down Buttons**
- Simpler implementation, no external library
- More accessible by default
- Better for mobile/touch devices
- Good fallback if drag-and-drop issues occur

**Reorder Algorithm:**
```typescript
function reorderSteps(steps: Step[], fromIndex: number, toIndex: number) {
  // Create copy of steps array
  const reordered = Array.from(steps);
  
  // Remove step from old position
  const [movedStep] = reordered.splice(fromIndex, 1);
  
  // Insert at new position
  reordered.splice(toIndex, 0, movedStep);
  
  // Update step_order for all steps
  const updates = reordered.map((step, index) => ({
    id: step.id,
    step_order: index + 1 // 1-indexed
  }));
  
  return updates;
}
```

**API Request:**
```typescript
// PATCH /api/workflow-templates/:templateId/versions/:versionId/steps/reorder
{
  "steps": [
    { "stepId": "uuid-1", "newOrder": 1 },
    { "stepId": "uuid-2", "newOrder": 2 },
    { "stepId": "uuid-3", "newOrder": 3 }
  ]
}
```

**Database Update (Transaction):**
```typescript
await db.transaction(async (tx) => {
  // Verify version is draft
  const version = await tx.query.workflowTemplateVersion.findFirst({
    where: eq(workflowTemplateVersion.id, versionId)
  });
  
  if (version.status !== 'draft') {
    throw new Error('Cannot reorder steps in published workflow');
  }
  
  // Update all steps in batch
  for (const update of stepUpdates) {
    await tx.update(workflowStepTemplate)
      .set({ step_order: update.newOrder })
      .where(eq(workflowStepTemplate.id, update.stepId));
  }
});
```

### Edge Cases

**1. Supplier User Deactivated:**
- Check `is_active = true` in query
- Fall back to initiator if user deactivated
- Log warning for visibility

**2. Multiple Supplier Users:**
- Use `LIMIT 1` to select one user
- Document that first active user is selected
- Future enhancement: Allow selection or assign to all

**3. Non-Supplier Workflow:**
- Log error if supplier_user role used for non-supplier entity
- Fall back to initiator
- UI should warn when creating such steps

**4. Supplier Without Contact:**
- Common during onboarding
- Falls back to procurement manager of the tenant (preferred fallback)
- If no procurement manager, falls back to initiator (last resort)
- Ensures workflow never blocks
- Admin can reassign task manually after contact added

**Tenant Isolation:**
- All queries MUST filter by tenant_id
- Supplier user resolution enforces tenant boundaries
- Process access checks tenant ownership

**Access Control:**
- Only users with appropriate permissions can view process details
- Supplier users see only their own tasks
- Internal users see all supplier tasks for their tenant

**Data Privacy:**
- User email/name displayed only to authorized users
- Supplier user info not exposed cross-tenant

- Validate version is in 'draft' status before allowing reorder
