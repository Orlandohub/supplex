## Story

**As an** admin,  
**I want** to create and manage document templates that define required documents for workflow steps,  
**so that** I can specify what documents need to be collected or validated in any workflow process.

---

## Acceptance Criteria

### Database Schema (AC 1-5)

1. A `document_template` table exists with tenant_id, template_name, required_documents (JSONB), status, is_default, created_at, updated_at, deleted_at
2. The `required_documents` JSONB field stores an array of document definitions with structure: `[{ name, description, required, type }]`
3. The `status` field supports 'draft', 'published', 'archived' values (default: 'published')
4. Composite index exists on `(tenant_id, status)` for efficient filtering
5. Soft delete support via `deleted_at` field

### Workflow Integration (AC 6-8)

6. `workflow_step_template.document_template_id` field references `document_template.id` via FK constraint
7. FK constraint uses ON DELETE RESTRICT (cannot delete templates in use by workflow steps)
8. Index exists on `workflow_step_template.document_template_id` for usage tracking

### API Endpoints (AC 9-14)

9. `GET /api/document-templates` returns all document templates for tenant (supports status filter)
10. `GET /api/document-templates/published` returns only published templates for tenant (used in workflow builder dropdowns)
11. `POST /api/document-templates` creates new document template (admin only)
12. `PUT /api/document-templates/:id` updates existing document template (admin only)
13. `DELETE /api/document-templates/:id` soft-deletes template (fails if referenced by workflow steps)
14. All endpoints enforce tenant isolation and admin-only access

### Backoffice UI (AC 15-20)

15. Document Templates management page exists at `/settings/document-templates`
16. Admin can view list of document templates (table shows: name, status, default flag, created date, actions)
17. Admin can create new document template with: template name, required documents list, default flag, status
18. Admin can edit existing document templates (published templates show warning)
19. Admin can delete document templates (shows error if in use by workflow steps)
20. Admin can set one template as default per tenant

### Workflow Builder Integration (AC 21-23)

21. When creating workflow step with `step_type='document'`, admin selects document template from dropdown
22. Dropdown shows only published document templates for tenant (filtered by: tenant_id, status='published', deleted_at IS NULL)
23. Dropdown displays template_name, sorted alphabetically

### Tenant Isolation (AC 24-25)

24. All queries filter by `tenant_id` (enforced in API middleware)
25. Cross-tenant access is prevented at database and application layers

## Dev Notes

### Context: Why This Story Exists

During the implementation of **Story 2.2.9** (Supplier Workflow Integration), the legacy qualification system was removed via **Sprint Change Proposal SCP-2026-01-31-001**. This included dropping the `qualification_templates` table via migration `0017_drop_legacy_qualification_system.sql`.

However, the `qualification_templates` table contained valuable **document template functionality** that should NOT have been removed. It provided:
- Reusable document checklist templates
- JSONB-based document requirements definition
- Integration with workflow steps via `document_template_id` FK
- Tenant-isolated template management

This story **restores** that functionality under the new name **"Document Templates"** to align with the generalized workflow system.

### Previous Implementation (Reference Only)

**Original Table Structure (from migration 0000, renamed in migration 0005):**
```sql
CREATE TABLE document_checklists (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  template_name VARCHAR(200) NOT NULL,
  required_documents JSONB DEFAULT '[]'::jsonb NOT NULL,
  is_default BOOLEAN DEFAULT false NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT now() NOT NULL,
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT now() NOT NULL,
  deleted_at TIMESTAMP WITH TIME ZONE
);
```

**Status Field Added (migration 0014):**
```sql
ALTER TABLE qualification_templates
ADD COLUMN status VARCHAR(50) NOT NULL DEFAULT 'published';
```

**Required Documents JSONB Structure:**
```json

  {

  },

  }
]
```

### Database Schema Details

[Source: packages/db/migrations/0014_add_status_to_qualification_templates.sql]

**Indexes Required:**
```sql
-- Tenant + status filtering (published templates)
CREATE INDEX idx_document_template_tenant_status
ON document_template(tenant_id, status)
WHERE deleted_at IS NULL;

-- Default template lookup
CREATE INDEX idx_document_template_tenant_default
ON document_template(tenant_id, is_default);

-- Workflow step usage tracking
CREATE INDEX idx_workflow_step_template_document_template
ON workflow_step_template(document_template_id)
WHERE deleted_at IS NULL AND document_template_id IS NOT NULL;
```

**FK Constraints:**
```sql
-- Template belongs to tenant (cascade delete)
ALTER TABLE document_template
ADD CONSTRAINT fk_document_template_tenant
FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE;

-- Workflow step references template (restrict delete)
ALTER TABLE workflow_step_template
ADD CONSTRAINT fk_workflow_step_document_template
FOREIGN KEY (document_template_id)
REFERENCES document_template(id)
ON DELETE RESTRICT;
```

### Performance Considerations

**Database Queries:**
- Composite indexes enable fast filtering: (tenant_id, status)
- Partial indexes (WHERE deleted_at IS NULL) improve performance
- FK constraint check on delete is O(n) but expected to be fast (low row count)

**API Response Times:**
- List templates: < 50ms (indexed query)
- Get published templates: < 30ms (highly selective index)
- Create/update: < 100ms (single insert/update)
- Delete: < 200ms (FK check + soft delete)

**Frontend Loading:**
- Templates list cached by Remix loader
- Dropdown data fetched once per workflow builder session
- Optimistic UI updates for create/edit/delete

### Security Considerations

**Tenant Isolation:**
- All queries filter by tenant_id (enforced in API middleware)
- FK constraints ensure templates only reference entities within same tenant
- No cross-tenant template sharing

**Access Control:**
- Admin role required for create/update/delete
- All authenticated users can view published templates (for workflow builder)
- Soft delete preserves audit trail

**Data Validation:**
- TypeBox validation on API (server-side)
- Zod validation on frontend (client-side)
- JSONB structure validated for required_documents field
- Template name limited to 200 characters

### Edge Cases

**1. Delete Template In Use:**
- FK constraint prevents hard delete
- API returns 400 error with list of workflows using template
- Frontend shows error message with helpful instructions

**2. Multiple Default Templates:**
- When setting isDefault = true, unset other defaults for tenant
- Transaction ensures only one default at a time
- UI shows current default with star icon

**3. Published Template Modification:**
- Show warning in UI: "This template is published and may affect workflows"
- Allow editing (no immutability constraint for document templates)
- Consider versioning in future story (Phase 2 enhancement)

**4. Empty Required Documents:**
- Allow empty array (template with no required documents)
- Workflow step can still use template (no documents to upload)
- Useful for optional document steps

**5. Soft Deleted Templates:**
- Not visible in UI (filtered out)
- FK references still valid (not orphaned)
- Can be restored by setting deleted_at = NULL (admin utility)
