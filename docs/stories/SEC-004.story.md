# Story SEC-004: Entity Authorization for Supplier-User Routes

## Story

**As a** platform operator,
**I want** all routes accessible by `supplier_user` that accept entity identifiers (process, step, supplier, document) to verify that the requesting supplier user owns the referenced entity,
**so that** a supplier user cannot view, modify, or upload data belonging to another supplier in the same tenant.

---

## Context

The entity-authorization helpers (`verifyProcessAccess`, `verifyStepProcessAccess`, `verifyDocumentAccess`) already exist and are used in some routes (e.g., `steps/documents/view.ts`). However, 6 routes that accept entity identifiers accessible to `supplier_user` do not call these helpers.

The vulnerability: within a single tenant, a supplier user who knows (or guesses) another supplier's process/step UUID can access that supplier's workflow data, comments, events, documents, and form submissions. Tenant isolation alone is insufficient — entity-level ownership must be enforced.

Internal roles (`admin`, `procurement_manager`, `quality_manager`, `viewer`) are allowed to see all entities within their tenant. The entity authorization helpers already handle this: they short-circuit to `allowed: true` for non-`supplier_user` roles.

### Analysis Report Items Addressed

| ID | Finding | Route |
|----|---------|-------|
| A1 | `workflows/supplier-processes.ts` — no `verifyProcessAccess`; any tenant user can request any `supplierId` | Supplier can view other suppliers' processes |
| A2 | `workflows/processes/events.ts` — no entity authorization; supplier can read any process's audit events | Supplier can read other suppliers' event logs |
| A3 | `workflows/comments/create.ts` — no `verifyProcessAccess`; supplier can post comments on any process | Supplier can comment on other suppliers' workflows |
| A4 | `workflows/comments/get-by-step.ts` — no entity check; supplier can read comments for any step | Supplier can read other suppliers' comments |
| A5 | `workflows/steps/documents/upload.ts` — no `verifyStepProcessAccess`; supplier can upload to any active step | Supplier can upload documents to other suppliers' steps |
| A6 | `form-submissions/create-draft.ts` — no process access verification; supplier can link drafts to arbitrary processes | Supplier can create form drafts on other suppliers' workflows |
| F1 | Same route as A5 — `steps/documents/upload.ts` missing entity auth | (duplicate of A5) |

### Dependencies

- **None** — this story is independent. The entity authorization helpers already exist.
- The helpers use the `suppliers.supplierUserId` column to resolve which supplier a `supplier_user` belongs to. This column must already be populated (it has been since Story 1.x).

## Acceptance Criteria

### AC 1: Supplier-Processes Route Protected (A1)

1. `workflows/supplier-processes.ts` verifies that a `supplier_user` requesting `/supplier/:supplierId/processes` is associated with the specified `supplierId`.
2. If the user is `supplier_user` and `supplierId` does not match their linked supplier, the route returns 403.
3. Internal roles (`admin`, `procurement_manager`, `quality_manager`, `viewer`) can still access any `supplierId` in their tenant.

### AC 2: Process Events Route Protected (A2)

4. `workflows/processes/events.ts` loads the process and calls `verifyProcessAccess` before returning events.
5. If the user is `supplier_user` and the process does not belong to their supplier, the route returns 403.

### AC 3: Comment Create Route Protected (A3)

6. `workflows/comments/create.ts` calls `verifyProcessAccess` on the resolved process before inserting the comment.
7. If the user is `supplier_user` and the process does not belong to their supplier, the route returns 403.

### AC 4: Comment Get-By-Step Route Protected (A4)

8. `workflows/comments/get-by-step.ts` calls `verifyStepProcessAccess` before returning comments.
9. If the user is `supplier_user` and the step's process does not belong to their supplier, the route returns 403.

### AC 5: Step Document Upload Route Protected (A5 / F1)

10. `workflows/steps/documents/upload.ts` calls `verifyStepProcessAccess` before processing the upload.
11. If the user is `supplier_user` and the step's process does not belong to their supplier, the route returns 403.
12. The authorization check must occur **before** any file upload processing — rejected requests must not trigger file storage operations.

### AC 6: Form Draft Create Route Protected (A6)

13. `form-submissions/create-draft.ts`, when `processInstanceId` is provided, loads the process and calls `verifyProcessAccess` before creating the draft.
14. If the user is `supplier_user` and the process does not belong to their supplier, the route returns 403.
15. If `processInstanceId` is not provided (standalone draft), no entity check is needed.

### AC 7: Consistent Error Response

16. All 403 responses from entity authorization use a consistent shape: `{ success: false, error: "Access denied: [reason from helper]" }` with HTTP status 403.
17. The error message must not leak entity details (e.g., should not reveal that the entity exists for another supplier).

### AC 8: Automated Tests

18. Each of the 6 routes has at least one test verifying that a `supplier_user` is denied access to another supplier's entity (403 response).
19. Each of the 6 routes has at least one test verifying that an internal role (e.g., `admin`) can access any entity in the tenant.

### Comment Create — Step-to-Process Binding Fix (A3 Extra)

The analysis noted that `comments/create.ts` loads the step by `stepInstanceId` + `tenantId` without verifying that the step belongs to the specified `processInstanceId`. This allows a client to pair an arbitrary step with a different process. Fix: add `eq(stepInstance.processInstanceId, processInstanceId)` to the step lookup query. This is a small but important consistency fix bundled with the entity auth work.

### Implementation Guardrails

1. **Do not modify the entity authorization helpers themselves.** They are already tested and correct. This story only calls them from new locations.
2. **The 403 response must not reveal entity existence** for other suppliers. Use generic "Access denied" messages, not "Process belongs to Supplier X".
3. **The upload route auth check must be before file processing.** A rejected upload must not touch storage.
4. **Internal roles must not be affected.** All helpers return `allowed: true` for non-`supplier_user` roles. Do not add additional role checks that could block internal users.
5. **Do not change route registration, URL patterns, or request/response shapes.** Only add authorization checks to existing handlers.
6. **The step-to-process binding fix in `comments/create.ts`** is a small scope addition. If it causes unexpected complexity, it can be deferred — but flag it in completion notes.

### Testing [Source: architecture/testing-strategy.md]

- **Backend tests:** Bun test runner (`bun test`)
- **Test strategy:** Route-level or handler-level tests are the preferred approach. The helpers themselves are already tested in `__tests__/entity-authorization.test.ts` — this story's tests must prove that each route is correctly wired to call the helpers before sensitive work. Helper-only tests are acceptable as a fallback only if route-level testing is genuinely impractical (see Task 7.2 constraints).
- **Key test scenarios per route:**
  - `supplier_user` for Supplier A → access Supplier A's entity → 200 (allowed)
  - `supplier_user` for Supplier A → access Supplier B's entity → 403 (denied)
  - `admin` → access Supplier B's entity → 200 (allowed)
- **Minimum test count:** 6 routes × 1 deny test = 6 mandatory deny tests. Allow and internal-role tests are strongly recommended. At least one route must have a full route-level integration test regardless of the approach used for other routes.

### Coding Standards [Source: architecture/coding-standards.md]

- Authentication middleware: use `authenticate` directly on routes, not nested in wrappers
- Perform role checks inside handlers with null checks: `if (!user?.role || ...)`
- Error responses: `{ success: false, error: "..." }` with appropriate HTTP status
- ElysiaJS validation uses TypeBox (`t.*`), not Zod
