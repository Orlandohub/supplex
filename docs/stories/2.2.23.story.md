## Story

**As a** developer,
**I want** type inconsistencies, dead code from removed features, validation drift between client and server, and schema mismatches to be cleaned up,
**so that** the codebase is internally consistent, new contributors can trust the type system, and future features build on a clean foundation.

## Context

The architecture assessment (`.cursor/plans/architecture_assessment_report_5c337427.plan.md`, Tier 4 — Maintainability and Type Safety) identified several tech debt items that don't affect correctness or security individually but compound into developer friction and latent bugs.

## Acceptance Criteria

### AC 1: Extract Shared Validation Module (M1)

1. A new `packages/types/src/validation/form-field-validation.ts` module contains a single `validateFieldValue(value: string, field: FieldDefinition): string | null` function that covers all field types: `text`, `textarea`, `number`, `date`, `dropdown`, `multi_select`, `checkbox`.
2. The server-side `apps/api/src/lib/validation/form-answer-validation.ts` imports and re-exports from the shared module (thin wrapper to maintain API compatibility).
3. The client-side `FormRenderer.tsx` `validateFieldValue()` function (lines 305+) is replaced with an import from the shared module, adapting the return type (`string | true` → `string | null` with a small wrapper).
4. Both client and server validate the exact same field types with identical rules.

### AC 2: Migrate `step_instance.status` to PG Enum (M2)

5. A new migration creates a `step_instance_status` PG enum with values: `pending`, `active`, `completed`, `blocked`, `skipped`, `awaiting_validation`, `validated`, `declined`.
6. The migration converts the `step_instance.status` `varchar(50)` column to use the new enum.
7. The Drizzle schema `step-instance.ts` is updated: replace `varchar("status", { length: 50 })` with `stepInstanceStatusEnum("status")`.
8. The `StepStatus` TS constant in `step-instance.ts` is updated to include all runtime values: `AWAITING_VALIDATION`, `VALIDATED`, `DECLINED`.

### AC 3: Fix `DocumentType` Enum Drift (M3)

9. Add `WORKFLOW_DOCUMENT = "workflow_document"` to the `DocumentType` enum in `packages/types/src/models/document.ts`.
10. Update the `DocumentTypeSchema` Zod schema to include the new value.
11. The workflow step upload route (`steps/documents/upload.ts`) now uses `DocumentType.WORKFLOW_DOCUMENT` instead of the hardcoded string `"workflow_document"`.

### AC 4: Type JSONB Fields (M4)

12. Define typed interfaces for the JSONB fields in `packages/types`:
    - `ValidationRules` — `{ min?: number; max?: number; minLength?: number; maxLength?: number; pattern?: string; customMessage?: string }`
    - `FieldOptions` — `{ choices?: Array<{ label: string; value: string }> }`
    - `RequiredDocumentItem` — `{ name: string; description?: string; required?: boolean; type?: string }` (already partially defined in `seed-step-documents.ts`)
    - `ValidationConfig` — already exists, verify completeness
13. The shared validation function (AC 1) uses `ValidationRules` and `FieldOptions` types instead of `any`.
14. The `field` parameter type in `validateAnswerFormat()` is updated from `any` to a typed `FieldDefinition` interface.

### AC 5: Remove Legacy Versioning Type Names (M6)

15. In `packages/types/src/models/form-template-ui.ts`:
    - Rename `FormTemplateVersionWithStructure` → `FormTemplateWithStructure` (the "version" concept was removed in Story 2.2.14).
    - Rename `FormSectionWithFields.formTemplateVersionId` → `FormSectionWithFields.formTemplateId`.
    - Rename `FormTemplateListItem.versionCount` → `FormTemplateListItem.templateCount` (or remove if unused).
    - Update `FormTemplateListItem.latestVersion` → `FormTemplateListItem.latestTemplate` (or remove if unused).
    - Rename `FormTemplateWithDetails.versions` → `FormTemplateWithDetails.templates` (or remove if unused).
    - Remove or rename any other `*Version*` references that no longer correspond to a real versioning concept.
16. Update all import sites across the web app to use the new names.
17. The old names are exported as deprecated type aliases for one release cycle (optional — check usage count; if <3 usages, just rename directly).

### AC 6: Fix Migration Numbering Collision (M10)

18. The two migration files sharing prefix `0020_` are **documented but not renamed**. Since both have already been applied to production, renaming them would break drizzle-kit's migration tracking. A `README.md` in the migrations folder documents the collision and explains why the files are left as-is.

### AC 7: Update Stale Documentation (M9)

19. Review `docs/prd/epic-2.2-dynamic-workflows.md` and update references to:
    - Remove `multi_approver` and `approver_count` from Story 2.2.6 and 2.2.7 descriptions (replaced by auto-validation in 2.2.15, fully removed in 2.2.18).
    - Add note that Story 2.2.15 (auto-validation) supersedes the multi-approver approach.
    - Update Story 2.2.14 description to reference "copy" functionality (versioning removed).
20. Verify `docs/architecture/` files (if any reference versioning or multi-approver) and update accordingly.

### Client vs Server Validation Drift (M1 Detail)

**Client** (`FormRenderer.tsx` → `validateFieldValue`):
- Handles: `number` (min/max), `date` (YYYY-MM-DD), `text`/`textarea` (minLength/maxLength/pattern)
- Missing: `dropdown`, `multi_select`, `checkbox`
- Return type: `string | true`

**Server** (`form-answer-validation.ts` → `validateAnswerFormat`):
- Handles: `number`, `date`, `dropdown`, `multi_select`, `checkbox`, `text`/`textarea`
- Complete coverage
- Return type: `string | null`

The shared module should have the server's complete coverage with the `string | null` return type. The client wraps it to return `string | true` for React Hook Form compatibility.

### Step Status Values in Practice

Currently used across the codebase:
- `pending` — initial state (unused? steps start as `blocked`)
- `active` — current step accepting input
- `completed` — step finished (pre-validation or non-validated step)
- `blocked` — future step not yet activated
- `skipped` — (defined in type but not used in engine)
- `awaiting_validation` — step submitted, waiting for validators (2.2.15)
- `validated` — all validation tasks approved (2.2.15)
- `declined` — step declined by validator (2.2.15)

The `StepStatus` TS constant (lines 33-39 of `step-instance.ts`) only defines `PENDING`, `ACTIVE`, `COMPLETED`, `BLOCKED`, `SKIPPED`. The missing values `awaiting_validation`, `validated`, `declined` are used as string literals throughout the engine code.

### `RequiredDocumentItem` Consolidation

Currently defined locally in `seed-step-documents.ts`:
```typescript
interface RequiredDocumentItem {
  name: string;
  description?: string;
  required?: boolean;
```

## Implementation Guardrails

1. **Each cleanup item is independent.** Do not introduce cross-dependencies between the separate ACs (validation module, PG enum, document type, JSONB typing, legacy names, migration docs, stale docs). If one item encounters an unexpected blocker, the others can still be completed and merged.
2. **Do not change workflow engine behavior, route semantics, or API response shapes.** This is a cleanup story — functional behavior must remain identical.
3. **Do not change comment access behavior.** Comment visibility / internal-vs-external filtering is out of scope and deferred to a separate story.
4. **The PG enum migration must not change runtime behavior.** Existing string values must map 1:1 to enum values. If any unexpected status value is found in the database, surface it as a blocker.
5. **Type renames (AC 5) must be find-and-replace only.** Do not restructure, merge, or split the interfaces — just rename to remove stale "version" references.
6. **The shared validation module must produce identical results** to the existing server-side validation for all inputs. The merge is a consolidation, not a rewrite.
7. **Do not refactor unrelated code** touched during these changes. Keep diffs minimal and focused on the specific cleanup item.
