## Story

**As a** platform operator,
**I want** the admin configuration endpoints to enforce referential integrity against the workflow system,
**so that** deleting a supplier status referenced by a workflow type, or changing a workflow type's completion-status mapping while processes are in flight, cannot silently break workflow completion behavior.

---

## Context

Two admin configuration endpoints allow mutations that can silently break the workflow completion path. When a workflow completes, `transitionToNextStep` reads the chain `processInstance → workflowTemplate → workflowType → supplierStatus` via INNER JOINs to determine which status to set on the supplier. If any link in that chain is broken or changed, the status update either silently skips or applies an unexpected value.

**Problem 1 — Supplier status delete breaks workflow type mapping (`DELETE /api/admin/supplier-statuses/:id`).** The route currently blocks deletion if any `suppliers` row references the status, but does NOT check if any `workflowType` references it via `supplierStatusId`. The DB FK has `onDelete: "set null"`, so deleting the status silently nullifies the workflow type's mapping. Any workflow of that type that subsequently completes will skip the supplier status update entirely — the INNER JOIN in `transitionToNextStep` (lines 152-167) returns no match, and the supplier status is never set. The admin gets no indication this happened.

**Problem 2 — Workflow type status mapping change affects in-flight processes (`PATCH /api/admin/workflow-types/:id`).** When an admin changes `supplierStatusId` on a workflow type, all in-flight processes using templates of that type will pick up the NEW mapping when they complete — not the mapping that was in effect when the process was started. There is no check, no warning, and no audit trail. The admin may not realize that changing the mapping retroactively affects running workflows.

Both routes are in `apps/api/src/routes/admin/`. Neither performs any cross-reference check against the workflow system.

## Acceptance Criteria

### AC 1: Supplier Status Delete Blocked When Referenced by Workflow Type

1. `DELETE /api/admin/supplier-statuses/:id` checks if any `workflowType` row references the status via `supplierStatusId`.
2. If one or more workflow types reference the status, the delete is **blocked** with 409 Conflict.
3. The error message includes the count of referencing workflow types (e.g., `"Cannot delete: status is referenced by 2 workflow type(s)."`).
4. The existing suppliers-in-use check (lines 100-108 of `supplier-statuses.ts`) remains unchanged. Both checks must pass for the delete to proceed — the new check is additive, not a replacement.
5. If no workflow types reference the status (and no suppliers use it), the delete proceeds as before.

### AC 2: Workflow Type Status Mapping Change Blocked When Active Processes Exist (With Force Override)

6. `PATCH /api/admin/workflow-types/:id` checks for active processes when `supplierStatusId` is being changed in the request body.
7. "Active processes" means: `processInstance` rows whose `workflowTemplate.workflowTypeId` matches the workflow type being updated, with `status NOT IN ('complete', 'cancelled')` and `deletedAt IS NULL`.
8. If active processes exist and the request does NOT include `force: true`, the status mapping change is **blocked** with 409 Conflict. The error message includes the active process count.
9. If active processes exist and the request includes `force: true`, the status mapping change **proceeds past the active-process conflict guard**. `force: true` does NOT bypass any other validation — normal tenant isolation, "workflow type exists" check, Elysia schema validation, and DB FK constraints all still apply. If the new `supplierStatusId` value is invalid, cross-tenant, or fails any existing check, the request still fails normally regardless of `force`. No audit log is required for this story — this is a configuration change, not a dangerous override like WFH-004's supplier status mutation.
10. If no active processes exist, the change proceeds as before (no change in behavior).
11. If `supplierStatusId` is NOT present in the request body (e.g., name-only update), the active process check is **skipped entirely** — no unnecessary blocking.
12. The `force` field is optional in the request body and defaults to `false`. Existing API consumers that do not send `force` see blocking behavior when conflicts exist.

### AC 3: Existing Behavior Preserved for Non-Conflicting Cases

13. Supplier status delete with no workflow type references (and no supplier usage) works exactly as before.
14. Workflow type update with no active processes works exactly as before.
15. Workflow type `name` update (without `supplierStatusId` change) is never blocked by the active process check.
16. All other admin routes (GET, POST, email logs, email settings, workflow health) are unaffected.
17. Existing tests continue to pass.

### AC 4: Route-Level Tests Verify Guard Behavior

18. Route-level test: DELETE supplier status that is referenced by a workflow type → 409 Conflict, status NOT deleted.
19. Route-level test: DELETE supplier status with no workflow type references and no supplier usage → 200, status deleted.
20. Route-level test: PATCH workflow type `supplierStatusId` with active processes and no `force` → 409 Conflict, mapping unchanged.
21. Route-level test: PATCH workflow type `supplierStatusId` with active processes and `force: true` → 200, mapping changed.
22. Route-level test: PATCH workflow type `supplierStatusId` with no active processes → 200, mapping changed.
23. Route-level test: PATCH workflow type `name` only (no `supplierStatusId`) with active processes → 200, name changed (no blocking).

## Dev Notes

### How Workflow Completion Uses the Status Mapping

When a workflow completes (all steps done), `transitionToNextStep` in `apps/api/src/lib/workflow-engine/transition-to-next-step.ts` (lines 152-182) reads the chain:

```
processInstance → workflowTemplate → workflowType → supplierStatus
```

via INNER JOINs. If every link resolves, the supplier's status is updated to the mapped value. If any link is broken (e.g., `workflowType.supplierStatusId` is null because the supplier status was deleted), the INNER JOIN returns no match, `statusMapping` is null, and the supplier status update is **silently skipped**.

This is why both guards in this story matter:
- **Guard 1** prevents the chain from being broken by deleting a supplier status that a workflow type still references.
- **Guard 2** warns when changing the mapping for a type that has in-flight processes, since those processes will pick up the new mapping on completion — not the mapping that existed when they started.

### Why Supplier Status DELETE Is a Hard Block

The `supplierStatus` FK on `workflowType` has `onDelete: "set null"`. If the application allows the delete, the DB will silently null out `workflowType.supplierStatusId`, breaking the completion chain. There is no legitimate reason to delete a supplier status that a workflow type actively maps to — the admin should first update the workflow type to use a different status (or remove its mapping), then delete the old status.

This guard is additive: the existing check for supplier-row usage remains. The delete only proceeds if BOTH checks pass.

### Why Workflow Type Status Mapping Change Uses `force` Override

Changing `supplierStatusId` on a workflow type is not destructive in the same way as WFH-004's supplier status mutation. It's a configuration change that affects future completions. However, it can cause unexpected behavior for in-flight processes: the admin expects them to complete with the old mapping, but they'll get the new one.

The `force` pattern is consistent with WFH-004 and gives the admin an explicit acknowledgment path. No audit log is required because this is a config change, not a direct entity override. If the team later wants audit coverage for all admin config changes, that's a separate concern.

### Scope of `force: true`

`force: true` has exactly one effect: it bypasses the active-process conflict guard added in this story (the `if (activeProcesses > 0 && !body.force)` check). It does NOT bypass:

- **Elysia schema validation** — `supplierStatusId` must be a valid UUID string or `null` per the `t.Object` schema
- **"Workflow type exists" check** — the route's existing 404 guard still runs
- **Tenant isolation** — all queries still filter by `user.tenantId`
- **DB FK constraints** — if `supplierStatusId` references a non-existent `supplier_status` row, the DB write will fail with a FK violation (this is the existing behavior, unchanged)
- **Any other existing validation** in the route

The dev agent must place the active-process check AFTER the existing "workflow type exists" check and BEFORE the update write. This ensures the normal validation pipeline runs first, and `force` only controls the new guard.

### `force` Flag — Backward Compatibility

Same pattern as WFH-004: the `force` field is `t.Optional(t.Boolean())`. Existing API consumers that do not send `force` will start seeing 409 errors when they try to change a workflow type's status mapping while active processes exist. This is intentional — it surfaces conflicts that were previously silent.

### `supplierStatusId` Change Detection

The active process check triggers only when `body.supplierStatusId !== undefined`. This means:
- `{ name: "New Name" }` → no check (name-only update)
- `{ supplierStatusId: "new-uuid" }` → check (mapping change)
- `{ supplierStatusId: null }` → check (mapping removal — also affects completion behavior)
- `{ name: "New Name", supplierStatusId: "new-uuid" }` → check (mixed update that includes mapping change)

### Query Performance

**Guard 1 (supplier status delete):** Simple `COUNT(*)` on `workflowType` filtered by `supplierStatusId` and `tenantId`. Uses the `idx_workflow_type_tenant` index. Fast — typically 0-5 workflow types per tenant.

**Guard 2 (workflow type status mapping change):** `COUNT(*)` on `processInstance` joined to `workflowTemplate` filtered by `workflowTypeId`, tenant, and status. The number of templates per workflow type is small (typically 1-3), and the number of active processes is small. No additional index needed.

### Error Response Contract (409 Conflict)

Same contract as WFH-004:

| Requirement | Required? | Notes |
|-------------|-----------|-------|
| HTTP 409 status code | **Yes** | Standard conflict response |
| `error.message` includes reference/process count | **Yes** | e.g., `"referenced by 2 workflow type(s)"` or `"3 active workflow process(es)"` |
| Structured details array in body | **No** | Not required — message-only |
| Modifications to `ApiError` / error infra | **No** | Do NOT modify |

### Implementation Guardrails

1. **Do not modify the workflow engine.** This story adds guards to admin configuration routes only.
2. **Do not modify any DB schema, FKs, or indexes.** The existing schema is sufficient.
3. **Do not modify `ApiError` or the error response infrastructure.** Work within the existing error shape.
4. **Do not add cascade behavior.** Deleting a supplier status does NOT auto-remove workflow type references. The admin must update the workflow type first.
5. **Do not add audit logging in this story.** The `force` override for a config change does not warrant an audit entry. If comprehensive admin config audit logging is desired, that is a separate concern.
6. **The active process check applies only to `supplierStatusId` changes, not `name` changes.**
7. **All DB queries must include tenant isolation.**
8. **`force: true` only bypasses the active-process conflict guard.** It must not skip or weaken any other validation (schema, tenant isolation, FK constraints, existence checks).

- **For the "no active processes" test (3.6):** Update the process to `complete` before the request.
- **For the "name only" test (3.7):** Ensure the process is still active.
