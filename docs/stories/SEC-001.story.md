# Story SEC-001: Authorization Claims Model Hardening

## Story

**As a** platform operator,
**I want** authorization claims (role, tenant_id) moved from `user_metadata` to `app_metadata` in all Supabase Auth interactions, and invalid/missing roles to be rejected instead of silently downgraded,
**so that** users cannot self-modify their own authorization claims, tenant isolation is enforced at the JWT level, and the system fails safely on corrupted auth state.

---

## Context

This is the highest-priority story in the auth/authorization refactor program (P0 ‚ÄĒ must complete before any production exposure). It addresses the most critical finding from the [Auth Architecture Analysis](../implementation-reports/auth-architecture-analysis-final.md):

> **C3 (Most Critical Finding):** Authorization claims (`role`, `tenant_id`) are stored in `user_metadata`, which Supabase allows any authenticated user to self-modify from the client SDK. This is a privilege-escalation and tenant-switching vulnerability.

Supabase's default behavior allows any authenticated user to call `auth.updateUser({ data: { role: "admin", tenant_id: "..." } })`. The `data` parameter maps to `user_metadata`. On the next token refresh, the modified claims appear in the JWT.

`app_metadata` is only writable via the Supabase Admin API (service_role key), which is never exposed to the browser. Moving authorization claims there closes this attack vector.

### Analysis Report Items Addressed

| ID | Finding | Severity |
|----|---------|----------|
| C3 | Role and tenant_id stored in `user_metadata` ‚ÄĒ users can self-modify authorization claims | **Critical** |
| S5 | Three different JWT claim paths for `tenant_id` across the codebase | High |
| T3 | RLS policy JWT claim path mismatch (legacy policies reference `app_metadata` but code writes to `user_metadata`) | High |
| M5 | `extractRoleFromMetadata` silently defaults to VIEWER on missing/invalid role | Medium |
| M9 | Three different JWT claim paths for `tenant_id` (same as S5) | Medium |

### Dependencies

- **None** ‚ÄĒ this story is the foundation for the entire auth refactor program.
- **Downstream dependents**: SEC-003A (RLS), SEC-009 (Auth Hook), SEC-010 (Cleanup) all depend on this being complete and stable.

## Acceptance Criteria

### AC 1: All Write Paths Use `app_metadata`

1. Every Supabase Admin API call that previously wrote `role` and/or `tenant_id` to `user_metadata` now writes them to `app_metadata` instead.
2. Non-authorization fields (`full_name`, `invited`, `invited_at`, `invitation_message`, `email_verified`, `tenant_name`) remain in `user_metadata` ‚ÄĒ they are profile data, not authorization claims.
3. The `createUserMetadata` helper in `packages/types` is split into two functions: one for authorization claims (`app_metadata`) and one for profile data (`user_metadata`).

### AC 2: Middleware Reads from `app_metadata`

4. The `authenticate` middleware in `apps/api/src/lib/rbac/middleware.ts` reads `role` and `tenant_id` from `jwtPayload.app_metadata` instead of `jwtPayload.user_metadata`.
5. The `SupabaseJWTPayload` interface in `jwt-verifier.ts` is updated so that `app_metadata` contains `role` and `tenant_id`, and `user_metadata` only contains profile fields.

### AC 3: Missing/Invalid Role Is Rejected (M5)

6. `extractRoleFromMetadata` throws a descriptive error when `role` is missing or not a valid `UserRole` value, instead of silently defaulting to `VIEWER`.
7. The `authenticate` middleware catches this error and returns a 401 with error code `INVALID_ROLE`.

### AC 4: Existing Users Backfilled

8. A backfill SQL script copies `role` and `tenant_id` from `raw_user_meta_data` to `raw_app_meta_data` in `auth.users` for all existing users.
9. The backfill is idempotent (safe to run multiple times).
10. The backfill script includes a pre-check query that reports how many users need migration and a post-check query that verifies all users have `app_metadata.role` and `app_metadata.tenant_id` set.

### AC 5: JWT Claim Path Standardized

11. After this change, the canonical claim path for authorization data is `app_metadata.role` and `app_metadata.tenant_id` across the entire system.
12. The existing legacy RLS policies in `rls-policies.sql` (which already reference `auth.jwt() -> 'app_metadata' ->> 'tenant_id'`) will read from a populated claim path after this change (they were previously reading from an empty path). However, this story only aligns the JWT claims model required for RLS evaluation ‚ÄĒ correctness, completeness, and rollout safety of RLS policies are explicitly deferred to SEC-003A / SEC-003B. This story does not validate or certify any RLS policies as production-ready.
13. The storage RLS policy claim path mismatch (`current_setting('request.jwt.claims')::json->>'tenant_id'`) is documented as a known issue for SEC-003A ‚ÄĒ this story does not change storage policies.

### AC 6: Seeding and Test Scripts Updated

14. `packages/db/src/create-test-users.ts` and `packages/db/src/create-auth-users.ts` write authorization claims to `app_metadata`.
15. All test files that mock JWT payloads or create test users via the Admin API are updated to use `app_metadata` for `role` and `tenant_id`.
16. The dev-login route (`apps/api/src/routes/auth/dev-login.ts`) continues to work correctly (it does not write metadata ‚ÄĒ no changes needed, but verify).

### AC 7: No Functional Regressions

17. The authentication flow (fast path via cache + slow path via Supabase) continues to work identically.
18. All existing auth-related tests pass after the changes.
19. The two-tier cache (`auth-cache.ts`) is unaffected ‚ÄĒ it caches the resolved user context, not raw JWT claims.
20. After backfill and code deployment, fresh JWT issuance (login or token refresh) produces tokens containing `app_metadata.role` and `app_metadata.tenant_id`.

## Dev Notes

### Architecture Context

**Supabase Metadata Model:**
- `user_metadata` (`raw_user_meta_data` in DB) ‚ÄĒ writable by the user themselves via client SDK (`auth.updateUser({ data: {...} })`). Intended for profile preferences.
- `app_metadata` (`raw_app_meta_data` in DB) ‚ÄĒ only writable via Admin/service_role API. Intended for authorization claims, provider info, and system-managed fields.
- Both are embedded in the JWT issued by Supabase Auth, at paths `user_metadata` and `app_metadata` respectively.
[Source: Supabase docs ‚ÄĒ Custom Claims & RBAC]

**Current State (BEFORE this story):**
- Authorization claims (`role`, `tenant_id`) are written to `user_metadata` by all user-creation and role-update paths.
- The `authenticate` middleware reads from `user_metadata` in the JWT.
- The legacy RLS policies (in `rls-policies.sql`) reference `app_metadata` ‚ÄĒ they are non-functional because `app_metadata` is empty.
- The storage RLS policy references `current_setting('request.jwt.claims')::json->>'tenant_id'` ‚ÄĒ a third, different path.
[Source: `auth-architecture-analysis-final.md` ‚ÄĒ Findings C3, S5, T3]

**Target State (AFTER this story):**
- Authorization claims (`role`, `tenant_id`) are written to `app_metadata` only.
- The `authenticate` middleware reads from `app_metadata` in the JWT.
- Profile data (`full_name`, `invited`, etc.) remains in `user_metadata`.
- The canonical JWT claim path for authorization is `app_metadata.role` and `app_metadata.tenant_id` everywhere.
- Legacy RLS policies will read from a populated claim path (correctness, completeness, and rollout safety of RLS are deferred to SEC-003A / SEC-003B).
- Storage RLS claim path fix is deferred to SEC-003A.

### Supabase Admin API Differences

When using `supabaseAdmin.auth.admin.createUser()` or `updateUserById()`:
- `{ user_metadata: {...} }` writes to `raw_user_meta_data`
- `{ app_metadata: {...} }` writes to `raw_app_meta_data`

Both can be passed simultaneously:
```typescript
await supabaseAdmin.auth.admin.updateUserById(userId, {
  app_metadata: { role: "admin", tenant_id: tenantId },
  user_metadata: { full_name: "John Doe" },
});
```

### `extractRoleFromMetadata` Behavior Change (M5)

**Before:** Missing or invalid role ‚Üí silently returns `UserRole.VIEWER`
**After:** Missing or invalid role ‚Üí throws Error with descriptive message

This is safe to do in this story because:
1. After this change, `app_metadata` is only written by server-side admin API calls ‚ÄĒ if role is missing, it's a system bug, not a user issue.
2. The middleware already handles auth errors and returns 401 ‚ÄĒ the new throw is caught and surfaced properly.
3. Silent downgrade to VIEWER is dangerous ‚ÄĒ a misconfigured user could silently access protected routes at a reduced privilege level without anyone noticing.

### Implementation Guardrails

1. **Do not remove `user_metadata` entirely.** Profile fields (`full_name`, etc.) legitimately belong there. Only authorization claims (`role`, `tenant_id`) move out.
2. **Do not change the cache layer.** `auth-cache.ts` caches the resolved user context object, not raw JWT claims. It is unaffected.
3. **Do not modify RLS policies.** The legacy `rls-policies.sql` already references `app_metadata` ‚ÄĒ this story aligns the claim path they depend on, but RLS correctness and rollout are deferred to SEC-003A / SEC-003B. Storage RLS is also deferred to SEC-003A.
4. **Do not clean up `user_metadata` on existing users.** Removing `role`/`tenant_id` from `user_metadata` is a future cleanup task. The backfill only adds to `app_metadata`; it does not remove from `user_metadata`. This avoids any risk of breaking during rollback.
5. **Do not implement the Custom Access Token Auth Hook.** SEC-001 establishes the interim secure claims model using `app_metadata`. The Auth Hook (a PostgreSQL function that injects claims at JWT issuance time) is a separate, later story. This story must not expand into token-hook implementation work.
6. **The backfill migration modifies `auth.users` ‚ÄĒ treat as operationally sensitive.** `auth.users` is a Supabase-managed table. The backfill only adds JSONB keys to `raw_app_meta_data` ‚ÄĒ it does not alter schema, remove data, or modify `raw_user_meta_data`. It must be idempotent and non-destructive. The operator must review and approve the SQL before execution. Pre-check and post-check verification queries are mandatory, not optional. If there is any uncertainty about migration runner compatibility with `auth.users`, the operator-reviewed manual SQL path (Supabase SQL Editor) takes precedence over automated migration execution.

## Operator Manual Steps

These steps must be performed by the operator (you) at specific points during deployment. The dev agent cannot perform these.

### Before Deployment

**Step 1: Pre-check existing users (Supabase SQL Editor)**

Run this query in the Supabase Dashboard ‚Üí SQL Editor to understand the current state:

```sql
-- Count users with auth claims in user_metadata
SELECT
  count(*) FILTER (WHERE raw_user_meta_data->>'role' IS NOT NULL) AS users_with_role_in_user_metadata,
  count(*) FILTER (WHERE raw_app_meta_data->>'role' IS NOT NULL) AS users_with_role_in_app_metadata,
  count(*) AS total_users
FROM auth.users;
```

Expected result: all roles in `user_metadata`, none (or few) in `app_metadata`.

### During Deployment

**Step 2: Apply the backfill migration**

The migration file `0036_backfill_app_metadata.sql` will run as part of the normal migration process. If migrations are applied manually, run it in the Supabase SQL Editor:

```sql
UPDATE auth.users
SET raw_app_meta_data = COALESCE(raw_app_meta_data, '{}'::jsonb)
  || jsonb_build_object(
       'role', raw_user_meta_data->>'role',
       'tenant_id', raw_user_meta_data->>'tenant_id'
     )
WHERE raw_user_meta_data->>'role' IS NOT NULL
  AND raw_user_meta_data->>'tenant_id' IS NOT NULL;
```

**Step 3: Verify backfill success (Supabase SQL Editor)**

```sql
-- All users should now have role and tenant_id in app_metadata
SELECT
  count(*) FILTER (WHERE raw_app_meta_data->>'role' IS NOT NULL AND raw_app_meta_data->>'tenant_id' IS NOT NULL) AS migrated_ok,
  count(*) FILTER (WHERE raw_app_meta_data->>'role' IS NULL OR raw_app_meta_data->>'tenant_id' IS NULL) AS still_missing,
  count(*) AS total_users
FROM auth.users;
```

Expected: `still_missing = 0` (or only system/anonymous accounts without roles).

**Step 4: Spot-check a user (Supabase Dashboard)**

1. Go to Supabase Dashboard ‚Üí Authentication ‚Üí Users
2. Click on any user
3. Verify that `app_metadata` now contains `role` and `tenant_id`
4. Verify that `user_metadata` still contains `full_name` (profile data preserved)

### After Deployment

**Step 5: Verify auth flow and JWT claim shape**

1. Log in to the application (fresh login ‚ÄĒ not a cached session)
2. Navigate to a protected page
3. Check browser DevTools ‚Üí Network ‚Üí any API call ‚Üí verify the request succeeds with 200 (not 401)
4. If using dev quick login, test with different roles (admin, viewer, supplier_user)
5. Decode the JWT (e.g., jwt.io or browser DevTools ‚Üí Application ‚Üí Cookies) and verify that `app_metadata` contains `role` and `tenant_id`
6. Test token refresh: wait for automatic token refresh or trigger it manually, then verify the refreshed JWT still contains `app_metadata.role` and `app_metadata.tenant_id`

**Step 5a: Session refresh expectations**

After deployment, existing users may still hold JWTs minted with the old claim shape (claims in `user_metadata` only, `app_metadata` empty for role/tenant). These sessions will continue to work via the auth cache until cache expiry. Be aware of the following:

- Some users may need to re-login or wait for automatic token refresh before the new `app_metadata` claim path is reflected in their JWT
- The middleware reads from `app_metadata` after this change ‚ÄĒ if a stale JWT has claims only in `user_metadata`, a cache-miss will fail to extract role/tenant and return 401
- **Mitigation**: The backfill ensures `app_metadata` is populated in the database. On the next token refresh, Supabase will mint a JWT with the updated `app_metadata`. Users who encounter a 401 should simply re-login
- This is expected behavior during the rollout window and is not a bug

**Step 6: Verify legacy RLS policies (if applied)**

If the legacy `rls-policies.sql` has been applied to the database, the policies now reference the correct claim path. You can verify:

```sql
SELECT tablename, policyname, qual
FROM pg_policies
WHERE schemaname = 'public'
  AND qual LIKE '%app_metadata%';
```

This should show policies for `tenants`, `users`, `suppliers`, `contacts`, `documents`, etc.
