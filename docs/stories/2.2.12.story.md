# Story 2.2.12: Immutable Audit Event Log for Templates and Workflow Execution

<!-- Powered by BMAD™ Core -->

**Ready for Development**

## Story

**As the** system,  
**I want** all template changes and workflow execution actions recorded in an immutable, append-only event log,  
**so that** compliance, traceability, and accurate workflow history display are ensured without relying on mutable task/step records.

---

## Context

### Current Problem

The workflow history tab in `WorkflowProcessDetailPage.tsx` is derived at render time from three mutable data sources:

1. **Step instances** (`step_instance`) — `completedDate`/`completedBy` are overwritten when a step is resubmitted after validation decline
2. **Task instances** (`task_instance`) — `updatedAt` changes whenever a task is modified; new rows are created per cycle but lack explicit event semantics
3. **Comment threads** (`comment_thread`) — only captures decline comments, not the full action context

**Problems caused by this approach:**
- When a form is resubmitted after rejection, the previous submission's date in history is lost (overwritten)
- History ordering becomes incorrect after resubmission cycles
- `completedBy` on steps reflects the last actor, not a per-event record
- No way to reconstruct the full sequence of events accurately
- No template change audit trail exists
- Querying "what happened and when" requires joining multiple mutable tables and inferring events from state changes
- Poor query performance for timeline/history views (requires multi-table joins with business logic filtering)

### Legacy Infrastructure

The legacy qualification system had a `workflow_events` table (dropped in migration 0017) and corresponding:
- `packages/types/src/workflow-events.ts` — `WorkflowEventType` enum and `TimelineEvent` interface
- `apps/api/src/services/workflow-event-logger.ts` — `logWorkflowEvent()` service

These were designed for the old qualification-only model (`workflowId` references `qualification_process`). The new dynamic workflow engine (Stories 2.2.1–2.2.17) never created its own event log. This story replaces the legacy event infrastructure with a new, engine-aligned event table.

### Design Principles

1. **Append-only** — Events are INSERT-only, never updated or deleted
2. **Denormalized actor data** — Store `actor_name` and `actor_role` at write time so history is accurate even if the user changes roles later
3. **Single source of truth** — Workflow history UI reads exclusively from the event log, not from step/task/comment records
4. **Fire-and-forget** — Event logging must never block or fail the primary workflow action; errors are logged but swallowed
5. **Tenant-isolated** — All events scoped by `tenant_id`
6. **Index-optimized** — Composite indexes designed for the two primary access patterns: (a) all events for a process, (b) all events for a tenant sorted by time

## Acceptance Criteria

### Task 1: Database Schema (AC 1–5)

1. New `workflow_event` table created via SQL migration with the following columns:

| Column | Type | Nullable | Description |
|--------|------|----------|-------------|
| `id` | UUID PK | No | `gen_random_uuid()` |
| `tenant_id` | UUID FK → tenants | No | Tenant isolation |
| `process_instance_id` | UUID FK → process_instance | Yes | NULL for template-level events |
| `step_instance_id` | UUID FK → step_instance | Yes | NULL for process-level events |
| `task_instance_id` | UUID FK → task_instance | Yes | NULL for step/process-level events |
| `event_type` | VARCHAR(50) | No | Enum-like string (see Event Types below) |
| `event_description` | VARCHAR(500) | No | Human-readable description |
| `actor_user_id` | UUID FK → users | No | Who performed the action |
| `actor_name` | VARCHAR(255) | No | Denormalized full name at event time |
| `actor_role` | VARCHAR(50) | No | Denormalized role at event time |
| `entity_type` | VARCHAR(50) | Yes | `form`, `document`, `step`, `task`, `template`, etc. |
| `entity_id` | UUID | Yes | ID of the affected entity |
| `comment` | TEXT | Yes | Decline reason, review comment, etc. |
| `metadata` | JSONB | No (default `{}`) | Additional structured data |
| `created_at` | TIMESTAMPTZ | No | `DEFAULT NOW()`, immutable event time |

2. Composite index: `(tenant_id, process_instance_id, created_at)` — primary query path for workflow history
3. Index: `(tenant_id, created_at DESC)` — for tenant-wide audit log views
4. Index: `(event_type)` — for filtering by event type
5. No `updated_at` column — events are immutable

10. New service: `apps/api/src/services/workflow-event-logger.ts` (replace existing legacy file) with signature:

```typescript
interface LogEventParams {
  tenantId: string;
  processInstanceId?: string | null;
  stepInstanceId?: string | null;
  taskInstanceId?: string | null;
  eventType: WorkflowEventType;
  eventDescription: string;
  actorUserId: string;
  actorName: string;
  actorRole: string;

  entityId?: string | null;
  comment?: string | null;
  metadata?: Record<string, any>;
```

11. The logger wraps the INSERT in a try/catch — on failure it logs an error to `console.error` but **does not throw** (fire-and-forget)

13. Caller must provide `actorName` and `actorRole` at call time (denormalized, not looked up by the logger)

30. New endpoint: `GET /api/workflows/processes/:processInstanceId/events`

    - Tenant-isolated via JWT `tenantId`

31. Response shape:

```typescript

  success: true,
  data: {
    events: WorkflowEvent[],
    total: number,
  }
}
```

32. No joins needed — all display data is denormalized in the event row
33. Endpoint accessible to all authenticated users of the tenant (not admin-only — users need to see history of processes they participate in)

34. `WorkflowProcessDetailPage.tsx` — History tab fetches from the new events API endpoint instead of deriving from step/task/comment data

### Task 8: Tenant-Wide Audit Log Page (AC 39–42)

39. New route: `/settings/audit-log` — admin-only page showing all `workflow_event` rows for the tenant
40. Table columns: Date, Event Type, Description, Actor, Process (link to detail if `process_instance_id` is set)
41. Filters: event type dropdown, date range, actor search
42. Paginated (server-side) with 50 rows per page default

### Non-Functional Requirements (AC 43–47)

43. Event logging adds less than 5ms overhead per workflow action (single INSERT, no joins, no transaction coordination)
44. History API query uses the `(tenant_id, process_instance_id, created_at)` composite index — no sequential scans
45. No data migration needed — existing process instances will have empty history (events are logged going forward only)
46. Legacy `workflow-event-logger.ts` and `workflow-events.ts` are replaced, not extended
47. The `workflow_event` table has no `ON DELETE CASCADE` from `process_instance` — events persist even if the process is deleted (audit compliance)

## Technical Notes

### Why Not Extend the Existing `audit_logs` Table?

The existing `audit_logs` table (`packages/db/src/schema/audit-logs.ts`) is designed for user management actions (invitations, role changes). It has `target_user_id` and `ip_address` columns specific to that domain. Workflow events have different semantics (process/step/task references, denormalized actor data for historical accuracy, entity type context). A dedicated table keeps both domains clean, queryable, and independently indexable.

### Why Denormalize Actor Data?

If a user changes roles (e.g., from `procurement_manager` to `admin`) or their name changes, the historical events must still show what role they had *at the time of the action*. Joining to the `users` table at read time would show their current data, not the historical data. Denormalization costs a few extra bytes per row but guarantees accuracy.

### Why Fire-and-Forget?

The primary workflow action (completing a step, creating a task) must never fail because of an event logging issue. The event log is a secondary concern. If the INSERT fails (e.g., connection timeout), the workflow action has already succeeded, and the missing event can be reconstructed from the system state if needed.

### Migration Strategy

- No backfill of existing workflow history — events are logged going forward only
- Existing processes will show empty history tabs until new actions are performed on them
- The old derived history code is removed entirely once this story is implemented

## Risk Assessment

- **Low risk**: Append-only table with fire-and-forget logging cannot break existing workflow logic
- **Low risk**: No data migration — existing history will be empty, not incorrect
- **Medium risk**: Ensuring all action points are instrumented — missing an event type means a gap in history. Mitigated by the comprehensive list in AC 14–29 and code review
- **Low risk**: Performance — single INSERT per action with no joins; composite index covers the read path
