## Story

**As an** admin,  
**I want** to configure options (choices) for dropdown and multi-select fields when building form templates,  
**so that** users can select from predefined values when filling out forms.

---

## Context

This is a **bug-fix story** for Story 2.2.3 (Form Builder UI).

**Issue Discovered**: Post-implementation review of Story 2.2.3 revealed that the Form Builder UI lacks the ability to configure the `options` field for dropdown, multi-select, and checkbox field types.

**Current State**:
- ‚úÖ Database schema includes `form_field.options` JSONB column
- ‚úÖ API endpoints accept and persist options data
- ‚ĚĆ UI modals (AddFieldModal, EditFieldModal) do not provide options editor
- ‚ö†ÔłŹ **Validation Gap Discovered**: API endpoints lack server-side validation for options structure (identified during PO validation review)

**Impact**: 
- Admins cannot create functional dropdown or multi-select fields, making the form builder incomplete for real-world questionnaires
- Without server-side validation, invalid options could be persisted via direct API calls (security/data integrity risk)

**Resolution Approach**: 
1. Add options configuration UI to existing modal components following established patterns
2. Add server-side validation to API endpoints to enforce options structure and constraints

---

## Acceptance Criteria

1. When creating a new field with `field_type` = "dropdown" or "multi_select", an "Options" configuration section appears in AddFieldModal
2. The options editor allows adding, editing, and removing choice items (each with value + label)
3. Validation enforces that dropdown and multi-select fields have at least one option before saving
4. When editing an existing dropdown/multi-select field, the options editor pre-populates with existing options and allows modification
5. Options data is persisted correctly in the `form_field.options` JSONB column
6. The options configuration UI follows the same design patterns as other form builder components (shadcn/ui, consistent styling)

### Server-Side Validation (CRITICAL)

**‚ö†ÔłŹ VALIDATION GAP IDENTIFIED:** Post-implementation review of Story 2.2.3 revealed that the API endpoints do NOT validate the `options` field structure.

**Current State:**
- `apps/api/src/routes/form-templates/fields/create.ts` - Line 150: `options: t.Optional(t.Any())`
- `apps/api/src/routes/form-templates/fields/update.ts` - Line 168: `options: t.Optional(t.Any())`
- **NO validation** of options structure, option count, or field content
- Client-side validation can be bypassed via direct API calls

**Security/Data Integrity Risk:**
- Users could create dropdown fields with zero options (unusable forms)
- Invalid option structures could break form rendering
- Malicious payloads could store huge option arrays (performance/storage abuse)
- No protection against empty values/labels

**Required Server-Side Validation Rules:**

1. **Structure Validation:**
   - Options must be an object with `choices` array
   - Error code: `INVALID_OPTIONS`
   - Message: "Dropdown and multi-select fields must have an options object with a choices array"

2. **Minimum Count Validation:**
   - At least 1 option required for dropdown/multi_select
   - Error code: `EMPTY_OPTIONS`
   - Message: "Dropdown and multi-select fields must have at least one option"

3. **Maximum Count Validation:**
   - Maximum 100 options per field (prevent abuse)
   - Error code: `TOO_MANY_OPTIONS`
   - Message: "Fields can have a maximum of 100 options"

4. **Choice Format Validation:**
   - Each choice must be an object with `value` and `label` properties
   - Error code: `INVALID_OPTION_FORMAT`
   - Message: "Option at index {index} must be an object with value and label"

5. **Value Validation:**
   - Must be non-empty string
   - Maximum length: 255 characters
   - Error codes: `INVALID_OPTION_VALUE`, `OPTION_VALUE_TOO_LONG`

6. **Label Validation:**
   - Must be non-empty string
   - Maximum length: 255 characters
   - Error codes: `INVALID_OPTION_LABEL`, `OPTION_LABEL_TOO_LONG`

**TypeBox Schema Enhancement:**

Replace `t.Any()` with structured validation:

```typescript
options: t.Optional(
  t.Object({
    choices: t.Array(
      t.Object({
        value: t.String({ minLength: 1, maxLength: 255 }),
        label: t.String({ minLength: 1, maxLength: 255 }),
      }),
      { minItems: 1, maxItems: 100 } // Only enforced for dropdown/multi_select via handler logic
    ),
  })
),
```

**Implementation Notes:**
- Validation logic must be added to BOTH create and update endpoints
- Only validate when `fieldType` is "dropdown" or "multi_select"
- For update endpoint, only validate if fieldType is being changed OR options are being updated
- Other field types (text, textarea, number, date, checkbox) can have empty/null options
