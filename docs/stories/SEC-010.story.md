## Story

**As a** frontend developer and API consumer,
**I want** every API error response to use a single, predictable JSON shape with correct HTTP status codes,
**so that** the global frontend error handler (SEC-007) can reliably process all errors, developers don't need to guess the error format per route, and 200-with-error responses no longer mask failures from monitoring.

---

## Context

The auth analysis identified at least 6 distinct error response shapes across the API:

| # | Pattern | Approx. Count | Shape |
|---|---------|--------------|-------|
| 1 | Route structured error | ~240 | `{ success: false, error: { code, message, timestamp } }` |
| 2 | Route string error | ~110 | `{ success: false, error: "string" }` |
| 3 | Middleware thrown JSON | ~12 | `throw new Error(JSON.stringify({ error: { code, message, timestamp } }))` |
| 4 | Global `onError` handler | all uncaught | `{ error: "Error Type", message: "description" }` |
| 5 | Dev routes | ~2 | `{ error: "string" }` (no `success` wrapper) |
| 6 | 200 + `success: false` | widespread | Correct error body but HTTP 200 status (no `set.status` call) |

The two most damaging problems are: (a) routes returning HTTP 200 with `{ success: false }` — this makes failures invisible to monitoring, caching, and the SEC-007 global handler; and (b) the middleware using `throw new Error(JSON.stringify(...))` which is fragile, hard to type, and produces different shapes from route-level errors.

### Analysis Report Items Addressed

| ID | Finding | Category |
|----|---------|----------|
| M2 | 3+ different error response shapes across routes | **Consistency / Maintainability** |
| C.5 | Required: single standardized API error shape; proper HTTP status codes; `throw new Error(JSON.stringify(...))` must be replaced | **Required Architecture Direction** |

### Dependencies

- **Independent of all other SEC stories.** This is a code quality / consistency refactor.
- **Complements SEC-007** — the frontend global 401/403 handler relies on correct HTTP status codes. Routes returning 200 for errors will bypass the handler.
- **After SEC-008** — SEC-008 standardizes admin enforcement to middleware. After SEC-008, the inline admin checks (which had their own error shapes) are gone. SEC-010 then only needs to standardize the middleware error shape, not both middleware + inline.

## Acceptance Criteria

### AC 1: Error Utility Created

1. An `ApiError` class exists in `apps/api/src/lib/errors.ts`.
2. `ApiError` accepts `statusCode`, `code`, and `message` parameters.
3. `ApiError` extends `Error` and can be thrown from any route handler or middleware.
4. A set of factory functions exists for common cases: `notFound(message)`, `forbidden(message)`, `unauthorized(code, message)`, `badRequest(message)`, `conflict(message)`, `internalError(message)`.

### AC 2: Global Error Handler Updated

5. The `onError` handler in `apps/api/src/index.ts` detects `ApiError` instances and returns the standard shape with the correct HTTP status:
   ```json
   {
     "success": false,
     "error": {
       "code": "NOT_FOUND",
       "message": "Supplier not found"
     }
   }
   ```
6. For non-`ApiError` errors (e.g., ElysiaJS `VALIDATION`, `PARSE`, `NOT_FOUND`), the handler returns the same standard shape with an appropriate code.
7. In production (`NODE_ENV=production`), error messages for 500-level errors are sanitized to "An unexpected error occurred." In non-production, the original message is included for debugging.
8. The `correlationId` is included in structured log output (already present) but is NOT included in the response body — correlation IDs are internal.

### AC 3: Middleware Errors Standardized

9. The `authenticate` middleware throws `ApiError` instead of `throw new Error(JSON.stringify(...))`. Status codes: 401 for auth failures, 403 for deactivated users.
10. The `requireRole` / `requireAdmin` / `requirePermission` middlewares throw `ApiError` with status 403.
11. The `createErrorResponse` helper in `middleware.ts` is removed (replaced by `ApiError`).
12. Middleware errors pass through the global `onError` handler and produce the standard response shape.

### AC 4: Route Errors Use Proper HTTP Status Codes

13. No route returns HTTP 200 with `{ success: false }`. Every error return either throws `ApiError` (preferred) or sets `set.status` explicitly.
14. The standard error HTTP codes are used consistently:
    - 400 — validation failures, bad input
    - 401 — authentication failures (missing token, invalid token, expired)
    - 403 — authorization failures (wrong role, no permission, entity access denied)
    - 404 — resource not found
    - 409 — conflict (duplicate, state conflict)
    - 429 — rate limit
    - 500 — unexpected server errors

### AC 5: Route Error Shape Standardized

15. All route error responses use the standard shape: `{ success: false, error: { code: string, message: string } }`.
16. Routes that currently return `{ success: false, error: "string" }` (Pattern C) are converted to throw `ApiError` with an appropriate code.
17. Routes that currently return `{ success: false, error: { code, message, timestamp } }` (Pattern B) are converted to throw `ApiError`. The `timestamp` field is dropped from the response body — it adds no value to the consumer and is already in structured server logs.

### AC 6: Success Responses Unchanged

18. Success response shapes (`{ success: true, data: ... }` or `{ success: true, ...fields }`) are NOT changed by this story. Only error paths are standardized.

### AC 7: Legacy Fallback Removed

19. The temporary JSON-parse fallback in the global `onError` handler (Task 2.2) is removed before story completion.
20. No `throw new Error(JSON.stringify(...))` pattern remains anywhere in the API codebase. A codebase search for this pattern must return zero matches.

### AC 8: No Functional Regressions

21. All existing tests pass (after updating error assertions to match the new shape).
22. The SEC-007 global frontend handler correctly processes the standardized error responses.
23. Non-JSON routes (e.g., `unsubscribe.ts` returning HTML) are excluded from standardization.

```typescript
    new ApiError(404, code, message),

  conflict: (message: string, code = "CONFLICT") =>
    new ApiError(409, code, message),

  internal: (message: string, code = "INTERNAL_SERVER_ERROR") =>
    new ApiError(500, code, message),
};
```

## Dev Notes

### Target Error Response Shape

**Every API error response MUST conform to this shape:**

```typescript
{
  success: false,
  error: {
    code: string,   // Machine-readable code: "NOT_FOUND", "FORBIDDEN", "VALIDATION_ERROR", etc.
    message: string  // Human-readable description
  }
}
```

**HTTP status code** is set on the response (not in the body). The `code` field provides machine-readable classification beyond the HTTP status.

**Fields intentionally excluded:**
- `timestamp` — already in structured server logs; adds no value to API consumers
- `correlationId` — internal debugging field; exposed via `X-Correlation-ID` response header, not body
- `details` / `errors` array — may be added in a future story for validation error detail; out of scope here

### `ApiError` Class Design

```typescript
class ApiError extends Error {
  statusCode: number;  // HTTP status code
  code: string;        // Machine-readable error code
  message: string;     // Human-readable message (inherited from Error)
}
```

The factory functions (`Errors.notFound`, `Errors.forbidden`, etc.) provide convenient defaults:
- `Errors.notFound("Supplier not found")` → `ApiError(404, "NOT_FOUND", "Supplier not found")`
- `Errors.forbidden("Access denied")` → `ApiError(403, "FORBIDDEN", "Access denied")`
- `Errors.unauthorized("Token expired", "TOKEN_EXPIRED")` → `ApiError(401, "TOKEN_EXPIRED", "Token expired")`

The second argument (`code`) is optional for factories — it defaults to a sensible code based on the status. For auth errors, the specific code is important (`TOKEN_EXPIRED`, `MISSING_TOKEN`, `INVALID_ROLE`) because the frontend may need to differentiate.

### Conversion Patterns

**Pattern C (string error) → ApiError:**
```typescript
// BEFORE
if (!user?.id) {
  return { success: false, error: "Unauthorized" };
}

// AFTER
if (!user?.id) {
  throw Errors.unauthorized("Unauthorized");
}
```

**Pattern B (structured error) → ApiError:**
```typescript
// BEFORE
if (!template) {
  set.status = 404;
  return {
    success: false,
    error: {
      code: "NOT_FOUND",
      message: "Template not found",
      timestamp: new Date().toISOString(),
    },
  };
}

// AFTER
if (!template) {
  throw Errors.notFound("Template not found");
}
```

**Pattern D (200 with error) → ApiError:**
```typescript
// BEFORE (returns 200 because no set.status)
try {
  const result = await doSomething();
  return { success: true, data: result };
} catch (error) {
  return { success: false, error: error.message };
}

// AFTER
try {
  const result = await doSomething();
  return { success: true, data: result };
} catch (error) {
  throw Errors.internal(error instanceof Error ? error.message : "An unexpected error occurred");
}
```

**Catch blocks — re-throwing known errors:**
```typescript
try {
  // ...
} catch (error) {
  if (error instanceof ApiError) throw error; // re-throw known errors
  throw Errors.internal(error instanceof Error ? error.message : "An unexpected error occurred");
}
```

### Middleware Conversion

```typescript
// BEFORE (middleware.ts)
set.status = 401;
throw new Error(
  JSON.stringify({
    error: {
      code: "MISSING_TOKEN",
      message: "Missing or invalid authorization token",
      timestamp: new Date().toISOString(),
    },
  })
);

// AFTER
throw Errors.unauthorized("Missing or invalid authorization token", "MISSING_TOKEN");
```

The `set.status` line is removed — the global handler reads `ApiError.statusCode` and sets it.

### Workflow Engine Library Functions — NOT Converted

The workflow engine library (`apps/api/src/lib/workflow-engine/`) returns result objects like:
```typescript
return { success: false, error: "Step not found" };
```

These are **internal function return values**, not HTTP responses. They must NOT be converted to `throw ApiError`. The calling route handler translates engine results to API responses:

```typescript
const result = await completeStep(db, params);
if (!result.success) {
  throw Errors.badRequest(result.error);
}
return { success: true, data: result.data };
```

### Implementation Guardrails

1. **Do not change success response shapes.** Only error paths are standardized. `{ success: true, data: ... }` remains unchanged.
2. **Do not convert workflow engine library functions** (`apps/api/src/lib/workflow-engine/`). These return result objects, not HTTP responses.
3. **Preserve the error `code` values** already used by the frontend. In particular, `TOKEN_EXPIRED`, `MISSING_TOKEN`, and `FORBIDDEN` are used by the SEC-007 global handler for routing decisions.
4. **Always re-throw `ApiError` in catch blocks.** If a catch block receives an `ApiError` (from a called function or middleware), it must re-throw it, not wrap it in a new error.
5. **Use the factory functions** (`Errors.notFound`, etc.) instead of `new ApiError(...)` wherever possible for consistency. Use `new ApiError(...)` directly only when a custom status+code combination is needed.
6. **Remove `set.status` calls** that precede `throw new ApiError(...)` — the global handler sets the status. Having both is redundant and confusing.
7. **The `timestamp` field is intentionally removed from error responses.** Do not add it back. It is available in structured server logs.
8. **Non-JSON routes are excluded** from standardization. Do not force JSON error shapes on HTML endpoints.
9. **The legacy JSON-parse fallback (Task 2.2) is transition-only.** It exists solely to allow file-by-file migration without breakage. It must be removed (Task 10) before the story is marked complete. Do not rely on it as a permanent error-handling path.

### Implementation Order

The recommended order minimizes risk:

1. **Task 1** (Create `ApiError`) — no functional change, just adds the utility
2. **Task 2** (Update global handler) — backwards-compatible, catches both old and new patterns
3. **Task 3** (Convert middleware) — highest impact, standardizes all 401/403
4. **Tasks 4-7** (Convert routes) — can be done module-by-module; each module is independently deployable
5. **Task 8** (Verify engine library) — audit only, no changes expected
6. **Task 9** (Rate limiter) — small, independent
7. **Task 10** (Remove legacy fallback) — cleanup after all conversions are done
8. **Task 11** (Verify) — run tests, fix assertions
