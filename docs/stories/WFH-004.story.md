# Story WFH-004: Workflow-Aware Supplier Mutation Guards

<!-- Powered by BMAD™ Core -->

## Status

**Approved**

Date Created: April 8, 2026

## Story

**As a** platform operator,
**I want** the supplier status change and supplier delete endpoints to check for active workflow processes before mutating supplier state,
**so that** an admin cannot accidentally override an in-progress workflow outcome or orphan active workflow processes by deleting a supplier.

---

## Context

Two supplier mutation endpoints currently bypass the workflow engine entirely:

**Problem 1 — Manual supplier status override (`PATCH /api/suppliers/:id/status`).** An admin can set any supplier status at any time, regardless of active workflow processes. If a qualification workflow is in progress and the admin manually sets the supplier to "approved," the workflow will continue running and may later set the status again on completion — or the workflow may complete and try to set a status that conflicts with the manual override. There is no warning, no audit trail, and no visibility into this conflict.

**Problem 2 — Supplier soft-delete with active processes (`DELETE /api/suppliers/:id`).** A supplier can be soft-deleted while workflow processes are in-progress for that supplier. The process instances reference the supplier via `entityId`, and after deletion the step task assignment logic (`resolveSupplierUser`) will fail or assign tasks incorrectly. The processes become orphaned with no clear recovery path.

Both routes are in `apps/api/src/routes/suppliers/detail.ts`. Neither performs any check against `processInstance`.

## Acceptance Criteria

### AC 1: Supplier Delete Blocked When Active Processes Exist

1. `DELETE /api/suppliers/:id` checks for active (non-`complete`, non-`cancelled`) workflow processes where `entityType = 'supplier'` and `entityId = supplier.id`.
2. If one or more active processes exist, the delete is **blocked** with 409 Conflict.
3. The error message string includes the active process count (e.g., `"Cannot delete supplier: 2 active workflow process(es) exist."`). Process IDs may be included in the message if practical, but a structured `activeProcesses` array in the error body is NOT required.
4. If no active processes exist, the delete proceeds as before (soft-delete).
5. Completed and cancelled processes do not block deletion.

### AC 2: Supplier Status Change Blocked When Active Processes Exist (With Force Override)

6. `PATCH /api/suppliers/:id/status` checks for active (non-`complete`, non-`cancelled`) workflow processes where `entityType = 'supplier'` and `entityId = supplier.id`.
7. If active processes exist and the request does NOT include `force: true`, the status change is **blocked** with 409 Conflict. The error message string includes the active process count. Same contract as AC 1 — no structured array required.
8. If active processes exist and the request includes `force: true`, the status change **proceeds** — but an audit log entry is created recording the manual override (see AC 3).
9. If no active processes exist, the status change proceeds as before (no change in behavior).
10. The `force` field is optional in the request body and defaults to `false`. Existing API consumers that do not send `force` see blocking behavior when conflicts exist — this is intentional and the safer default.

### AC 3: Audit Log Entry for Manual Status Override

11. When a supplier status change is performed with `force: true` while active processes exist, an audit log entry is created via `logAuditEvent` with:
    - `action: AuditAction.SUPPLIER_UPDATED`
    - `details` including: `manualStatusOverride: true`, `oldStatus`, `newStatus`, `activeProcessIds` (array of process instance IDs), `activeProcessCount`
12. The audit log entry is fire-and-forget (same pattern as all other audit log calls — errors logged but not thrown).

### AC 4: Active Process Query Is Efficient

13. The active process check uses a single query against `processInstance` with the existing index `idx_process_instance_tenant_entity` on `(tenant_id, entity_type, entity_id)`.
14. The query filters: `tenantId`, `entityType = 'supplier'`, `entityId = supplierId`, `status NOT IN ('complete', 'cancelled')`, `deletedAt IS NULL`.
15. The query returns only `id` and `status` (no full row scan). No `LIMIT` is needed — the number of active processes per supplier is small (typically 1-5). The `id` values are used for the audit log `details`; only the count is required in the error message.

### AC 5: Existing Behavior Preserved for Non-Conflicting Cases

16. Supplier delete with no active processes works exactly as before.
17. Supplier status change with no active processes works exactly as before.
18. All other supplier routes (GET, PUT, add-contact, update-contact) are unaffected.
19. Existing tests continue to pass.

### AC 6: Route-Level Tests Verify Guard Behavior

20. Route-level test: DELETE supplier with active process → 409 Conflict, supplier NOT deleted.
21. Route-level test: DELETE supplier with no active processes → 200, supplier soft-deleted.
22. Route-level test: PATCH status with active process and no `force` → 409 Conflict, status unchanged.
23. Route-level test: PATCH status with active process and `force: true` → 200, status changed, audit log entry created.
24. Route-level test: PATCH status with no active processes → 200, status changed (no blocking, no audit override entry).

### Task 1: Create Active Process Check Helper (AC: 4)

- [ ] **1.1** Create a reusable helper function, either in `detail.ts` itself or in a shared location like `apps/api/src/lib/rbac/entity-authorization.ts`. Suggested signature:

  ```typescript
  async function getActiveProcessesForSupplier(
    supplierId: string,
    tenantId: string,
    db: DbOrTx
  ): Promise<{ id: string; status: string }[]>
  ```

  The return type is minimal: `id` is needed for audit log `details.activeProcessIds`; `status` is available for diagnostic purposes. `workflowName` is NOT required — it is not used in the error message or audit log.

- [ ] **1.2** Implement the query:

  ```typescript
  import { processInstance } from "@supplex/db";
  import { eq, and, notInArray, isNull } from "drizzle-orm";

  const activeProcesses = await db
    .select({
      id: processInstance.id,
      status: processInstance.status,
    })
    .from(processInstance)
    .where(
      and(
        eq(processInstance.tenantId, tenantId),
        eq(processInstance.entityType, "supplier"),
        eq(processInstance.entityId, supplierId),
        notInArray(processInstance.status, ["complete", "cancelled"]),
        isNull(processInstance.deletedAt)
      )
    );
  ```

  This uses the existing `idx_process_instance_tenant_entity` composite index.

- [ ] **1.3** Place this helper where it makes sense for reuse. If placed in `entity-authorization.ts`, it complements the existing `verifyProcessAccess` and `verifyStepProcessAccess` helpers. If placed in the route file, that's also acceptable for this story's scope — it can be extracted later if WFH-005 or other stories need it.

## Dev Notes

### Active Process Statuses

The `processInstance.status` enum values are:
- `in_progress` — actively executing steps
- `pending_validation` — waiting for a validator to approve/decline
- `declined_resubmit` — a step was declined and awaiting resubmission
- `complete` — workflow finished successfully
- `cancelled` — workflow was cancelled

**Active** means NOT `complete` and NOT `cancelled`. The first three statuses (`in_progress`, `pending_validation`, `declined_resubmit`) all represent live, in-flight work. All three should block delete and trigger the status change guard.

### Why DELETE Is a Hard Block but PATCH Has Force Override

**DELETE** orphans processes irrecoverably — the task assignment logic (`resolveSupplierUser`) joins against `suppliers` and will fail for a deleted supplier. There is no legitimate use case for deleting a supplier mid-workflow. The admin must cancel or wait for the processes to complete.

**PATCH status** is a softer conflict — the workflow will still complete correctly and will try to set the supplier status on completion. The manual status may be overwritten by the workflow or vice versa. An admin may have a legitimate reason to override (e.g., a process is stuck and won't complete, or they need to block a supplier immediately for compliance reasons). The `force` flag gives them this escape hatch while ensuring they acknowledged the conflict.

### `force` Flag — Backward Compatibility

The `force` field is `t.Optional(t.Boolean())` in the request body. Existing API consumers that do not send `force` will get `undefined`, which is treated as `false`. This means:

- **Existing clients** that never send `force` will start seeing 409 errors when they try to change a supplier's status while processes are active. This is the desired behavior — it surfaces conflicts that were previously silent.
- **No breaking change** to the API contract — the field is optional, and the 200 success path is unchanged when no conflicts exist.

### Database Index Coverage

The query `SELECT FROM process_instance WHERE tenant_id = ? AND entity_type = 'supplier' AND entity_id = ? AND status NOT IN ('complete','cancelled') AND deleted_at IS NULL` is covered by the existing composite index:

```sql
idx_process_instance_tenant_entity ON (tenant_id, entity_type, entity_id) WHERE deleted_at IS NULL
```

The `status NOT IN` filter happens as a post-index scan, which is fine since the number of processes per supplier is small (typically 1-5).

### Audit Log Pattern

The codebase already has a working audit log system:
- `logAuditEvent()` in `apps/api/src/lib/audit/logger.ts` — fire-and-forget, errors swallowed
- `AuditAction.SUPPLIER_UPDATED` already exists in the enum — no new enum value needed
- The `details` field is `Record<string, any>`, so `manualStatusOverride: true` plus process IDs can be added without schema changes

The supplier routes already have `// TODO: Record status change in audit history` comments. This story implements the audit entry for the forced override case only. Full audit coverage for all supplier status changes is a separate concern.

### Error Response Contract (409 Conflict)

The 409 response uses the standard `Errors.conflict(message)` pattern. The contract for this story is:

| Requirement | Required? | Notes |
|-------------|-----------|-------|
| HTTP 409 status code | **Yes** | Standard conflict response |
| `error.message` includes active process count | **Yes** | e.g., `"2 active workflow process(es) exist"` |
| Process IDs in the message string | Nice-to-have | Append as `"Process IDs: abc, def"` if practical |
| Structured `activeProcesses` array in body | **No** | Do NOT add — not needed for this story |
| Modifications to `ApiError` / error infra | **No** | Do NOT modify — work within existing shape |

The resulting error body follows the existing `ApiError` contract exactly:
```json
{
  "success": false,
  "error": {
    "code": "CONFLICT",
    "message": "Cannot delete supplier: 2 active workflow process(es) exist. Resolve or cancel them first."
  }
}
```

The process `id` values returned by the helper are still used: they go into the audit log `details.activeProcessIds` (for the forced override case in Task 4). They are NOT required to appear in the error response body as structured data.

### Implementation Guardrails

1. **Do not modify the workflow engine.** This story adds guards to supplier routes only.
2. **Do not modify `processInstance` schema or indexes.** The existing index is sufficient.
3. **Do not modify `ApiError` or the error response infrastructure.** Work within the existing error shape.
4. **Do not add automatic process cancellation.** If an admin wants to delete a supplier with active processes, they must cancel the processes manually first. Automatic cascade cancellation is out of scope.
5. **Do not add `SUPPLIER_STATUS_OVERRIDE` to the AuditAction enum.** Use the existing `SUPPLIER_UPDATED` action with descriptive `details`.
6. **The `force` flag applies only to PATCH status, not DELETE.** Delete is always a hard block when active processes exist.
7. **All DB queries must include tenant isolation.**

### Testing

- **Test runner:** Bun (`bun test`)
- **Test location:** `apps/api/src/routes/suppliers/__tests__/workflow-guards.test.ts`
- **Test approach: route-level integration.** Tests exercise the real `supplierDetailRoutes` through Elysia `app.handle()` with real DB fixtures.
- **Auth bypass:** Use `.derive(() => ({ user: testAdminUser, requestLogger: ... }))` to inject a real DB-backed admin user, bypassing JWT auth. The DELETE route requires `UserRole.ADMIN`; the PATCH status route requires `UserRole.ADMIN` or `UserRole.PROCUREMENT_MANAGER`.
- **DB fixture pattern:** Insert tenant, user, supplier, and process instance in `beforeAll`. Clean up by deleting the test tenant in `afterAll`.
- **For the "no active processes" test (5.3, 5.6):** Either update the process to `complete`/`cancelled` before the request, or use a second supplier that was never linked to a process.
- **For the audit log assertion (5.5):** After the PATCH request, query `auditLogs` table directly to verify the entry exists with the expected `action` and `details`.
