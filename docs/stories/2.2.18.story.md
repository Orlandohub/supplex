## Story

**As a** workflow administrator,
**I want** the workflow template builder to enforce correct configuration constraints, display published templates in a clear read-only view, and support template deletion with audit logging,
**so that** workflow templates are always valid, easy to review, and manageable throughout their lifecycle.

---

## Context

During manual testing of the workflow template builder (Epic 2.2 — Dynamic Workflows), five issues were identified that affect usability, data integrity, and lifecycle management. These fixes span the step form dialog, the published-template view, the templates list page, the validation task creation engine, and audit logging.

### Current State (from code review)
- **Multi-approver feature** (`multiApprover`, `approverCount`, `step_approver` table, `MultiApproverConfig` component, approver CRUD routes, and engine branching in `create-tasks-for-step.ts`) is fully obsolete since all approval functionality was replaced by auto-validation (Story 2.2.15). The step completion route (`complete.ts`) already only handles validation tasks — multi-approver logic is dead code.
- **Validation tasks** are created without a deadline (`dueAt: null` in `create-validation-tasks.ts` line 58). The step's `dueDays` is used for the main task but there is no separate "validation due days" field.
- **Form/document template selects** in the step dialog are optional — no validation prevents saving a "form" step without a form template or a "document" step without a document template.
- **Published templates** render in the same editor page with `canEdit=false`, which disables editing but still shows the step form dialog on click (edit/pencil button is hidden, but step detail cards don't expand to show full read-only content).
- **Delete workflow template** API exists (`DELETE /api/workflow-templates/:workflowId`) but there is no UI button or confirmation dialog, and the action is not audit-logged.

## Acceptance Criteria

### Fix 1: Complete Multi-Approver Removal (AC 1–5)

1. The multi-approver checkbox, approver count field, and `MultiApproverConfig` component are removed from the step form dialog and step detail cards in `WorkflowStepBuilder.tsx`.
2. The multi-approver branching logic in `create-tasks-for-step.ts` is removed; task creation always follows the single-task-per-step path (validation tasks handle all approval workflows via Story 2.2.15).
3. The approver CRUD API routes (`list`, `create`, `delete`, `reorder` under `workflow-templates/approvers/`) are removed.
4. `multiApprover` and `approverCount` fields are removed from step create/update API schemas and from the `WorkflowStepTemplate` / `StepApprover` TypeScript types.
5. A new migration drops the `step_approver` table and removes the `multi_approver` and `approver_count` columns from `workflow_step_template`. The Drizzle schema files are updated accordingly.
6. These changes must in no way break the workflow implementation.

### Fix 2: Validation Completion Time in Days (AC 6–8)

6. A new "Validation Due Days" numeric input appears in the step form when "Requires Validation?" is checked, allowing the admin to specify how many days validators have to complete their review.
7. The value is persisted in `validationConfig` JSON as `validationDueDays` (no new DB column needed).
8. `create-validation-tasks.ts` reads `validationDueDays` from the step template's `validationConfig` and sets `dueAt` on each validation task as `now + validationDueDays` days (same calculation pattern as `createTasksForStep`).

### Fix 3: Mandatory Template Selection for Form/Document Steps (AC 9–10)

9. When step type is `form`, the "Form Template" select is marked as required and the step cannot be saved without selecting a published form template.
10. When step type is `document`, the "Document Template" select is marked as required and the step cannot be saved without selecting a published document template.

### Fix 4: Read-Only Published Template Step View (AC 11–13)

11. When viewing a published workflow template, each step card is expandable/collapsible to show full step configuration details (task title, description, due days, assignee, form/document template name, auto-validation config) in a read-only layout.
12. The step form dialog does not open for published templates — clicking a step card toggles the expanded detail view instead.
13. Published step cards display the form template name and document template name (not just the action mode) so reviewers can see which template is linked.

### Fix 5: Delete Workflow Template with Confirmation & Audit (AC 14–17)

14. Each workflow template card on the templates list page has a delete button (trash icon) visible to Admin users.
15. Clicking the delete button opens a confirmation dialog ("Are you sure you want to delete this template? This action cannot be undone.") with Cancel and Delete buttons.
16. Confirming the deletion calls `DELETE /api/workflow-templates/:id` and shows a success/error toast.
17. The delete action is audit-logged using the existing `logAuditEvent` service with a new `WORKFLOW_TEMPLATE_DELETED` action type, recording the template ID, name, and status at time of deletion.

### Fix 1 Details — Complete Multi-Approver Removal (Full Stack)

The multi-approver feature (`multiApprover`, `approverCount`, `step_approver` table, `MultiApproverConfig` component, approver CRUD routes, engine branching) was the original approval mechanism. Story 2.2.15 introduced auto-validation as the complete replacement. The step completion route (`complete.ts`) already only handles validation tasks — multi-approver logic in `create-tasks-for-step.ts` is dead code. This fix removes the entire feature across all layers.

**Frontend removal:**
- `WorkflowStepBuilder.tsx`: Remove checkbox (lines ~881–901), conditional approver count (lines ~903–926), multi-approver badge in card (lines ~598–603), `MultiApproverConfig` rendering (lines ~612–621), and all `multiApprover`/`approverCount` form fields and defaults.
- Delete `MultiApproverConfig.tsx` entirely.

**Backend removal:**
- `create-tasks-for-step.ts`: Remove the `if (stepTemplate.multiApprover)` branch (lines ~176–274) and `stepApprover` import. Only the single-task creation path remains.
- Delete all approver CRUD routes under `workflow-templates/approvers/`.
- `steps/create.ts` and `steps/update.ts`: Remove `multiApprover` and `approverCount` from body schemas and persistence logic.
- `copy.ts`: Remove `stepApprover` row copying logic.

**Schema removal (migration `0031_remove_multi_approver.sql`):**
```sql
DROP TABLE IF EXISTS step_approver CASCADE;
ALTER TABLE workflow_step_template DROP COLUMN IF EXISTS multi_approver;
ALTER TABLE workflow_step_template DROP COLUMN IF EXISTS approver_count;
```
- Delete `packages/db/src/schema/step-approver.ts` and remove its export from `packages/db/src/index.ts`.
- In `workflow-step-template.ts`: Remove `multiApprover`, `approverCount` columns and the `approvers: many(stepApprover)` relation.

**Type removal:**
- `packages/types/src/models/workflow-template.ts`: Remove `multiApprover`, `approverCount` from `WorkflowStepTemplate`; remove `StepApprover` interface; remove `WorkflowStepTemplateWithApprovers`.

### Fix 2 Details — Validation Due Days

**Current behavior:** `create-validation-tasks.ts` line 58 hardcodes `dueAt: null` for all validation tasks.

**Target behavior:** Read `validationDueDays` from `validationConfig` JSONB and compute deadline:

```typescript
const validationDueDays = validationConfig.validationDueDays;
const dueAt = validationDueDays
  ? new Date(Date.now() + validationDueDays * 24 * 60 * 60 * 1000)
  : null;
```

This mirrors the pattern in `create-tasks-for-step.ts` lines 237–239 and 277–279.

**No migration needed** — `validation_config` is already a JSONB column. The new field is simply a new key in the JSON object.

### Fix 3 Details — Required Template Selection

**Current behavior:** Form template and document template selects are optional in the Zod schema. A user can create a "form" step without selecting any form template.

**Target behavior:** Use Zod's `.superRefine()` or conditional `.refine()` to enforce:
- `stepType === "form"` → `formTemplateId` must be non-empty
- `stepType === "document"` → `documentTemplateId` must be non-empty

### Fix 4 Details — Read-Only Expanded Steps

**Current behavior:** Published steps show a compact card with type badge, summary line (due days, assignee role, form/document mode), but no way to see full details. The edit pencil is hidden.

**Target behavior:** Step cards become expandable on click (like an accordion). Expanded view shows all fields in a clean read-only layout. This requires fetching template names — either pass them from the step list API or look them up from the already-fetched `formTemplates`/`documentTemplates` arrays.

### Fix 5 Details — Delete + Audit

**Backend:** The `DELETE /api/workflow-templates/:workflowId` route already exists and performs soft delete. Add `logAuditEvent` call after the soft delete succeeds:

```typescript
import { logAuditEvent } from "../../lib/audit/logger";
import { AuditAction } from "@supplex/types";

await logAuditEvent({
  tenantId,
  userId: user.id,
  action: AuditAction.WORKFLOW_TEMPLATE_DELETED,
  details: {
    templateId: workflowId,
    templateName: existing.name,
    templateStatus: existing.status,
  },
});
```

**Frontend:** Add a delete button per card that stops event propagation, opens a confirmation dialog, and calls the API on confirm.
