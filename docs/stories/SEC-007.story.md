# Story SEC-007: Frontend Auth Session & Access Handling Refactor

## Story

**As a** platform user,
**I want** the frontend to manage my authentication session using Supabase's recommended event-driven pattern, handle access errors consistently, and avoid stale cached auth state,
**so that** my session is reliable, auth errors are clearly communicated, unnecessary network calls are eliminated, and my role/permissions are always current.

---

## Context

The frontend has a working auth pipeline, but it is built on patterns that Supabase has deprecated or warns against. The current implementation makes redundant Supabase Auth calls, uses polling instead of events, persists stale auth state in `localStorage`, and handles 401/403 errors inconsistently across routes.

This story consolidates the frontend auth flow into a clean, event-driven architecture aligned with Supabase's recommended patterns.

| M6 | `ProtectedRoute.tsx` component exists but is unused | **Dead Code** |

### Dependencies

- **Independent** — this story does not depend on any other SEC story. The frontend auth refactor is orthogonal to the backend claims model (SEC-001), RLS (SEC-003A/B), and JWT signing (SEC-006).
- However, once SEC-001 moves claims to `app_metadata`, the `userRecord` in the Zustand store becomes the only client-side source of `role` — making the staleness fix (U5) even more important.

## Acceptance Criteria

### AC 1: Session Monitor Removed (S2, U4, P2)

1. The `session-monitor.ts` file is deleted.
2. `AuthProvider.tsx` no longer calls `setupSessionMonitoring`.
3. All session lifecycle is handled by Supabase's built-in `onAuthStateChange` listener (already partially in place in `AuthProvider.tsx`).
4. Token refresh is handled by the Supabase JS SDK automatically — no manual polling or `refreshSession()` calls.
5. Session expiration detection uses `onAuthStateChange` events (`SIGNED_OUT`, `TOKEN_REFRESHED` failure) — not a 30s `getUser()` poll.

### AC 2: Single Server-Side Auth Validation (S3, P1)

6. On an initial authenticated page load, `supabase.auth.getUser()` is called **at most once** server-side across `root.tsx` and `_app.tsx` loaders combined.
7. `root.tsx` uses `getSession()` (local session parse, no network call) for the minimal check needed to decide whether to render the authenticated layout or redirect.
8. `_app.tsx` performs the single full `getUser()` validation (via `requireAuthSecure` or equivalent). This is the authoritative server-side auth check.
9. The `userRecord` fetched by `_app.tsx` is passed to the client — `AuthProvider` hydrates from it via `useRouteLoaderData("routes/_app")` instead of re-fetching on mount.

### AC 3: Client-Side `getUser()` Eliminated on Hydration

10. `AuthProvider.tsx` no longer calls `supabase.auth.getUser()` or queries the `users` table on initial mount when the server has already provided validated auth data (via loader props).
11. The `onAuthStateChange` listener handles subsequent auth events (`TOKEN_REFRESHED`, `SIGNED_OUT`, `USER_UPDATED`) — these are the only triggers for client-side auth state updates.

### AC 4: Global 401/403 Handling (U1, U2)

12. The `api-client.ts` fetch wrapper handles **all** 401 responses (not just `TOKEN_EXPIRED`) for authenticated app API calls: clear auth state and redirect to login. Login, password-reset, and recovery flows are exempt — they do not use the authenticated API client and must not trigger redirect loops.
13. The `api-client.ts` fetch wrapper handles 403 responses with a consistent UX: display an "Access Denied" toast notification. Do not redirect — the user is authenticated but lacks permission.
14. Individual route loaders/actions that use the shared API client no longer implement ad-hoc 401/403 handling — they rely on the centralized handler. Any route-specific exceptions that remain (e.g., a full-page access-denied view) must be documented in completion notes.

### AC 5: Stale Zustand State Addressed (U5)

15. The Zustand `persist` configuration in `useAuth.ts` no longer persists `userRecord` to `localStorage`. Only minimal, non-sensitive UI hints may be persisted (e.g., `isAuthenticated` for flash prevention). **If `isAuthenticated` is persisted, it is a UX hint only — it must never be treated as the authoritative source of auth state.** The authoritative source is the server-validated session from the `_app` loader.
16. On `SIGNED_OUT` event (from `onAuthStateChange`), the Zustand store is fully cleared, including any persisted state.
17. On tab visibility change (returning to a stale tab), the auth state is revalidated via `getSession()` (local parse, no network) — if the session is missing or expired, clear auth and redirect.

### AC 6: Dead Code Removed (U3, M6)

18. `apps/web/app/lib/auth/middleware.ts` is deleted (contains `withTokenRefresh`, `withAuth`, `withAuthAction`, `scheduleTokenRefresh` — all unused).
19. `apps/web/app/components/auth/ProtectedRoute.tsx` is deleted (unused, referenced only in documentation).
20. Any documentation references to the deleted files (e.g., README files) are updated or removed.

### AC 7: No Functional Regressions

21. The auth flow works end-to-end: login, authenticated navigation, token refresh, logout.
22. Role-based UI rendering (sidebar, permissions) continues to work using `userRecord` from the `_app` loader.
23. Multi-tab behavior: logging out in one tab clears auth in other tabs (via `storage` event on Zustand or `onAuthStateChange` broadcast).
24. All existing frontend tests pass.

- [x] **3.1** In `apps/web/app/providers/AuthProvider.tsx`:
  - Remove the `useEffect` that calls `supabase.auth.getUser()` on mount.
  - Remove the `useEffect` that queries `supabase.from("users")` on mount (the `userRecord` is now hydrated from the server).
  - Keep the `onAuthStateChange` subscription — this handles `TOKEN_REFRESHED`, `SIGNED_OUT`, `USER_UPDATED` events.

  - On `SIGNED_IN`: use the `session` parameter from the callback for auth state, then fetch `userRecord` from the `users` table (the one client-side DB query needed on login).
  - On `USER_UPDATED`: refresh `userRecord` from the `users` table (the user's profile was just changed). Use `session` from the callback for the auth state update.

## Dev Notes

### Current Auth Flow (BEFORE This Story)

```
Initial page load:
  root.tsx loader → getSession() → supabase.auth.getUser()  [1st network call]
  _app.tsx loader → requireAuthSecure() → getUser()          [2nd network call]
  AuthProvider mount → supabase.auth.getUser()               [3rd network call]
  AuthProvider mount → supabase.from("users").select()       [4th: DB query]
  session-monitor → setInterval(getUser(), 30s)              [ongoing polling]
```

### Target Auth Flow (AFTER This Story)

```
Initial page load:
  root.tsx loader → getSession() (local parse, no network)   [no network call]
  _app.tsx loader → requireAuthSecure() → getUser()          [1 network call]
  _app.tsx loader → DB query for userRecord                  [1 DB query]
  AuthProvider hydration → uses server data (no client calls) [no network call]
  onAuthStateChange → handles TOKEN_REFRESHED, SIGNED_OUT    [event-driven]
```

### `onAuthStateChange` Events Reference

| Event | Trigger | Action |
|-------|---------|--------|
| `SIGNED_IN` | User logs in | Set auth state (already handled) |
| `SIGNED_OUT` | User logs out (any tab) | Clear Zustand, clear `localStorage`, redirect to `/login` |
| `TOKEN_REFRESHED` | SDK automatically refreshed the token | Update session in Zustand (no `getUser()` needed) |
| `USER_UPDATED` | `supabase.auth.updateUser()` was called | Refresh `userRecord` from DB (profile changed) |
| `PASSWORD_RECOVERY` | User clicked password reset link | Handle in auth UI (not relevant to session management) |

### `root.tsx` vs `_app.tsx` — Division of Responsibility

| Responsibility | `root.tsx` | `_app.tsx` |
|---------------|------------|------------|
| Session existence check | `getSession()` (local parse) | — |
| Full auth validation (`getUser()`) | — | `requireAuthSecure()` |
| `userRecord` from DB | — | `getAuthenticatedUser()` |
| Permissions computation | — | `hasPermission(role, ...)` |
| Layout selection (auth vs public) | Based on session existence | — |
| `env` vars for client | `window.ENV` | — |

### Global Error Handler Design

The `api-client.ts` handler should be the **single point** for API error handling:

| Status | Behavior | Rationale |
|--------|----------|-----------|
| **401 (any)** | `clearAuth()` → redirect `/login` | User is unauthenticated — session expired, token invalid, or revoked |
| **403** | Toast "Access denied" | User is authenticated but lacks permission — don't redirect, just inform |
| **Other errors** | Pass through to caller | Let route-specific handlers deal with business errors |

The current code only handles `TOKEN_EXPIRED` in the 401 body — this must be broadened to all 401 responses. After SEC-001 (claims hardening), a stale JWT with claims only in `user_metadata` could produce a non-`TOKEN_EXPIRED` 401 — the global handler must catch this.

### Zustand Persistence Decision

**Option A (recommended):** Remove `userRecord` from `partialize`, keep `isAuthenticated` only.
- Pro: No stale role/tenant in `localStorage`. Minimal flash prevention.
- Con: Brief loading state on hard refresh (until `_app` loader provides `userRecord`).
- **Important:** The persisted `isAuthenticated` is a non-authoritative UX hint. It must never gate access to routes or data — only prevent a visual flash during hydration.

**Option B:** Remove `persist` entirely.
- Pro: Simplest. No `localStorage` auth data at all.
- Con: Flash of unauthenticated UI on every hard refresh.

**Option C:** Keep `persist` but add a version and revalidation check.
- Pro: Smooth UX.
- Con: More complexity, still stores PII in `localStorage`.

The story recommends **Option A**. The dev agent should implement Option A unless they discover that it causes unacceptable UX (document the decision in completion notes).

### Implementation Guardrails

1. **Do not remove the `onAuthStateChange` listener.** It is the correct Supabase pattern. Enhance it, don't replace it with polling.
2. **Do not call `getUser()` on the client during normal navigation.** The Supabase SDK handles token refresh internally. `getUser()` is only for server-side validation.
3. **The `root.tsx` loader must NOT call `getUser()`.** It must use `getSession()` or `getSessionFast()` (local parse only).
4. **Do not break the `shouldRevalidate: false` optimization in `_app.tsx`.** This prevents redundant loader re-runs on client navigation. The `_app` loader should only re-run on form actions or explicit revalidation.
5. **The global 401 handler must not interfere with the login page itself.** If the user is on `/login` and gets a 401 (expected), it should not trigger a redirect loop.
6. **The `visibilitychange` listener must use `getSession()` (local parse), not `getUser()` (network).** The goal is lightweight revalidation, not a full server round-trip.
7. **Do not remove the `storage` event listener if one exists.** Cross-tab auth sync via `localStorage` changes is a valid pattern that complements `onAuthStateChange`.
8. **Task 4.2 (audit inline 401/403 handling) should focus on the most impactful routes first.** It is acceptable to leave some route-specific 403 handling in place if the UX genuinely differs (e.g., a full-page access-denied view for a supplier process). Document any routes left with inline handling in completion notes.

### Testing [Source: architecture/testing-strategy.md]

- **Frontend tests:** Vitest (if configured) or manual verification.
- **E2E tests:** Playwright (`apps/web/tests/e2e/`) if auth flow tests exist.
- **Key verification scenarios:**
  - Initial load: single `getUser()` in Network tab (from `_app` loader only)
  - No 30s polling visible in Network tab
  - Token refresh: silent, no full-page reload
  - Logout: Zustand cleared, `localStorage` entry removed, all tabs redirect
  - 401 from API: redirect to login
  - 403 from API: toast/notification, no redirect
  - Tab switch after idle: session revalidated without network call

### Coding Standards [Source: architecture/coding-standards.md]

- Remix loaders: use `defer` / `json` for data loading
- Supabase client: use `getBrowserClient()` for client-side, `createServerClient()` for server-side
- State management: Zustand hooks with `useShallow` for selective subscriptions
- Navigation: Remix `useNavigate()` for programmatic navigation, not `window.location.href`
