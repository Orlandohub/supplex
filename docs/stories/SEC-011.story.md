# Story SEC-011: Structured Logging & Auth Observability

## Story

**As a** platform operator,
**I want** all API runtime logging to use the structured Pino logger instead of `console.*`, and all authentication/authorization denials and rate-limit blocks to be logged with full request context,
**so that** production logs are queryable, auth failures are observable for incident response and audit, and unstructured `console.*` output no longer pollutes or escapes log aggregation pipelines.

---

## Context

The API has a well-configured Pino logger (`apps/api/src/lib/logger.ts`) with child loggers for `auth` and `workflow-health`. However, the majority of runtime code still uses `console.error`, `console.log`, and `console.warn` — approximately 160 calls across route handlers, services, queue workers, and core libraries. These unstructured calls bypass Pino's JSON output, lose context fields (correlation ID, module, timestamp format), and cannot be filtered or queried in log aggregation tools.

Additionally, two critical observability gaps exist: (a) authorization denials (403 from `requireRole`/`requirePermission`) produce **no log output at all**, making it impossible to detect privilege probing or misconfigured roles; and (b) rate-limit blocks (429) are silent, providing no visibility into abuse patterns.

## Dev Notes

### Pino Structured Logging Conventions

**Log levels:**
| Level | Use For |
|-------|---------|
| `error` | Unexpected failures, exceptions, data corruption |
| `warn` | Auth denials, rate limits, degraded service, missing optional config |
| `info` | Significant operational events: server start, connection established, job completed |
| `debug` | Detailed diagnostic info: cache hits, query details, handler entry — invisible in production |

**Context pattern:**
```typescript
logger.error({ err: error, userId, operation: "createTemplate" }, "Failed to create template");
//          ^-- context object (structured)                        ^-- message (human-readable)
```

**Request-scoped logging (required default for route handlers):**
Route handlers receive `requestLogger` from the `correlation-id` middleware. This logger automatically includes the correlation ID, HTTP method, and URL path. It is the **required default** for all route handler logs — using a module-level logger in a route handler silently loses request context:
```typescript
async ({ body, user, requestLogger }) => {
  requestLogger.info({ templateId: body.id }, "Creating template");
  // ...
  requestLogger.error({ err: error }, "Failed to create template");
}
```

**Module-level loggers (for non-request code only):**
Services, queue workers, and libraries that run outside of a request context use module-level child loggers. These do **not** have correlation IDs — that is expected for background work:
```typescript
import { logger } from "../lib/logger";
const emailLogger = logger.child({ module: "email-notification" });
```

### Auth Denial Log Format

```json
{
  "level": 40,
  "time": 1712400000000,
  "module": "auth",
  "event": "authz_denied",
  "userId": "uuid-...",
  "userRole": "viewer",
  "requiredRoles": ["admin"],
  "route": "/api/admin/email-settings",
  "method": "PUT",
  "correlationId": "corr-uuid-...",
  "clientIp": "192.168.1.1",
  "msg": "Authorization denied: insufficient role"
}
```

This format enables:
- Alerting on high-frequency denials (privilege probing detection)
- Filtering by user, role, route for incident response
- Correlation with other request logs via `correlationId`
- Client IP tracking for abuse detection

### Rate-Limit Log Format

```json
{
  "level": 40,
  "time": 1712400000000,
  "module": "rate-limiter",
  "event": "rate_limited",
  "key": "rate:192.168.1.1:/api/auth/login",
  "route": "/api/auth/login",
  "method": "POST",
  "correlationId": "corr-uuid-...",
  "clientIp": "192.168.1.1",
  "retryAfter": 60,
  "msg": "Rate limit exceeded"
}
```

### Client IP Extraction

Use headers in this precedence:
1. `x-forwarded-for` (first IP if comma-separated — set by reverse proxy)
2. `x-real-ip` (set by some proxies)
3. Connection remote address (if ElysiaJS exposes it)

```typescript
function getClientIp(request: Request): string | null {
  const xff = request.headers.get("x-forwarded-for");
  if (xff) return xff.split(",")[0].trim();
  return request.headers.get("x-real-ip");
}
```

This can be a utility in `apps/api/src/lib/logger.ts` or inline in middleware.

### Sensitive Data Rules

**Never log:**
- Full JWT tokens or `Authorization` header values
- Passwords or password hashes
- Full email addresses in high-volume logs (cache keys are OK if hashed or using user IDs)
- Supabase service role key

**OK to log:**
- User ID (UUID)
- Role
- Tenant ID
- Route/URL path
- Correlation ID
- Error messages (but not full stack traces in production — Pino handles this via `err` serializer)

### Implementation Guardrails

1. **Use the Pino `err` serializer for error objects.** Pass errors as `{ err: error }` — Pino serializes them into structured JSON with `type`, `message`, and `stack`.
2. **Do not log full stack traces in production.** Pino's default `err` serializer handles this — stack traces are included in the JSON but can be filtered by log level.
3. **`requestLogger` is the required default for route handlers.** It carries the correlation ID and request context. Module-level child loggers are for non-request code only (services, queue, libraries). If a route handler log cannot use `requestLogger`, document the reason in completion notes.
4. **Do not convert test files** (`__tests__/`, `*.test.ts`). `console.log` in tests is acceptable.
5. **Do not convert `packages/db/` CLI scripts.** These are developer tools, not runtime code.
6. **Sensitive data must not appear in logs.** See "Sensitive Data Rules" above.
7. **The `config.ts` bootstrap exception is acceptable.** If logger initialization depends on config loading, `console.error` for config validation failure is the correct fallback. Document with a comment.
8. **Log auth denials at `warn`, not `error`.** A 403 is expected behavior when a user tries something they're not authorized for. `error` level is for unexpected failures.

- **No test changes expected** — logger calls are side effects and typically not asserted in existing tests.
- **Manual verification:**
  - Start dev server → verify JSON structured output
  - Trigger 403 → check log output for denial context
  - Trigger 429 → check log output for rate-limit context

### Coding Standards [Source: architecture/coding-standards.md]

- Logging: use `@/lib/logger` — `logger.info`, `logger.error`, `logger.debug`
- Child loggers: `logger.child({ module: "name" })` for domain isolation
- Error objects: pass as `{ err: error }` to Pino (standard serializer)
- Log messages: short, descriptive, no variable interpolation in the message string (use context object)
