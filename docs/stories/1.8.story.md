## Story

**As a** procurement manager,  
**I want** to upload and manage documents for each supplier,  
**so that** all certifications and contracts are centralized and accessible.

---

## Acceptance Criteria

1. **Documents Tab** on supplier detail page displays list of uploaded documents
2. Document list shows: Filename, Document type (dropdown: Certificate, Contract, Insurance, Audit Report, Other), Upload date, Uploaded by, File size, Expiration date (optional)
3. "Upload Document" button opens file picker supporting PDF, Excel, Word, PNG, JPG (max 10MB per file)
4. Multiple files can be uploaded simultaneously with progress bar per file
5. Document metadata form captures: Document type (required), Description, Expiration date (date picker)
6. Documents stored in Supabase Storage with tenant-isolated buckets using RLS
7. Download button allows users to download document (opens in new tab for PDFs)
8. Delete button with confirmation modal (soft delete, Admin/Procurement Manager only)
9. Document expiration warning displays if expiration date is within 30 days
10. Expired documents display with red warning badge
11. Document list is sortable by upload date, expiration date, document type
12. Empty state displayed with "Upload your first document" CTA when no documents exist
13. File upload validates file type and size before upload with clear error messages
14. Virus scanning integration point documented for future implementation (Phase 2)

### Document Data Model

[Source: architecture/data-models.md, packages/types/src/models/document.ts]

**Document TypeScript Interface:**

```typescript
export enum DocumentType {
  CERTIFICATE = "certificate",
  CONTRACT = "contract",
  INSURANCE = "insurance",
  AUDIT_REPORT = "audit_report",
  OTHER = "other",
}

export interface Document {
  id: string; // UUID
  tenantId: string;
  supplierId: string;
  filename: string;
  documentType: DocumentType;
  storagePath: string;
  fileSize: number; // Bytes
  mimeType: string;
  description: string | null;
  expiryDate: Date | null;
  uploadedBy: string; // User ID
  createdAt: Date;
  updatedAt: Date;
  deletedAt: Date | null;
}
```

**Database Schema:**
[Source: packages/db/src/schema/documents.ts]
- Table name: `documents` (already defined in `packages/db/src/schema/documents.ts`)
- UUID primary key with `uuid_generate_v4()` default
- Foreign keys: tenant_id (CASCADE DELETE), supplier_id (CASCADE DELETE), uploaded_by (references users)
- Indexes on: tenant_id, supplier_id for performance
- Soft delete via `deleted_at` timestamp (must exclude from queries with `WHERE deleted_at IS NULL`)
- All columns NOT NULL except: description, expiryDate, deletedAt

**Important Notes:**
- **Storage Path Pattern:** `{tenantId}/{supplierId}/{uuid}_{sanitizedFilename}` ensures tenant isolation at storage level
- **File Size Validation:** 10MB max = 10485760 bytes
- **MIME Type Validation:** Accept only: `application/pdf`, `image/png`, `image/jpeg`, `application/vnd.ms-excel`, `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`

### Supabase Storage Configuration

[Source: architecture/tech-stack.md, apps/api/src/lib/supabase.ts]

**Supabase Storage:**
- Storage provider: Supabase Storage (managed object storage with RLS)
- Bucket name: `supplier-documents` (create if not exists)
- RLS policies: Tenant-isolated access (users can only access files from their tenant)
- CDN-backed for fast download speeds
- File access via signed URLs (time-limited, secure)

**Supabase Client Configuration:**
[Source: apps/api/src/lib/supabase.ts]
- Service role key available in `supabaseAdmin` client for backend file operations
- Anon key available in `supabase` client for public operations
- Client initialized with auto-refresh disabled for server-side operations

**Storage Operations:**
```typescript
// Upload file to Supabase Storage
const { data, error } = await supabaseAdmin.storage
  .from('supplier-documents')
  .upload(storagePath, file, {
    contentType: file.type,
    upsert: false, // Prevent overwrite
  });

// Generate signed URL for download (valid for 5 minutes)
const { data: signedUrl, error } = await supabaseAdmin.storage
  .from('supplier-documents')
  .createSignedUrl(storagePath, 300); // 300 seconds = 5 minutes

// Delete file from storage (optional - for hard delete)
const { error } = await supabaseAdmin.storage
  .from('supplier-documents')
  .remove([storagePath]);
```

**RLS Policy Creation:**
[Source: architecture/database-schema.md, architecture/security-and-performance.md]
```sql
-- Create bucket (run migration or manual SQL)
INSERT INTO storage.buckets (id, name, public) 
VALUES ('supplier-documents', 'supplier-documents', false);

-- RLS policy: Users can only access files from their tenant
CREATE POLICY "Users can access their tenant's documents"
ON storage.objects FOR SELECT
USING (
  bucket_id = 'supplier-documents' 
  AND (storage.foldername(name))[1] = auth.uid()::text
);
```

**Important:** For MVP, we'll implement application-level tenant filtering in API endpoints. RLS policies on storage buckets can be added in Phase 2 for defense-in-depth.

### API Endpoint Specifications

[Source: architecture/api-specification.md, coding standards]

**GET /api/suppliers/:supplierId/documents**
- **Purpose:** Fetch all documents for a supplier
- **Auth:** Requires authentication (any role)
- **Tenant Scoping:** Filter by tenant_id from authenticated user's JWT
- **Query Parameters:** None
- **Response:**
```typescript
{
  documents: Document[] // Array of document objects
}
```
- **Errors:**
  - 404 Not Found if supplier doesn't exist or belongs to different tenant
  - 401 Unauthorized if not authenticated

**POST /api/suppliers/:supplierId/documents**
- **Purpose:** Upload new document with metadata
- **Auth:** Requires Admin or Procurement Manager role
- **Tenant Scoping:** Automatically set tenant_id from authenticated user's JWT
- **Request Body (multipart/form-data):**
```typescript
{
  file: File; // Binary file data (max 10MB)
  documentType: DocumentType; // Required
  description?: string; // Optional
  expiryDate?: string; // Optional, ISO date format
}
```
- **Response:**
```typescript
{
  document: Document // Full document object with generated id
}
```
- **Errors:**
  - 400 Bad Request if validation fails (invalid file type, size too large)
  - 403 Forbidden if user lacks permission
  - 404 Not Found if supplier doesn't exist
  - 413 Payload Too Large if file exceeds 10MB
  - 507 Insufficient Storage if Supabase quota exceeded

**DELETE /api/documents/:id**
- **Purpose:** Soft delete document
- **Auth:** Requires Admin or Procurement Manager role
- **Tenant Scoping:** Delete only if document belongs to user's tenant
- **Response:**
```typescript
{
  success: true
}
```
- **Errors:**
  - 404 Not Found if document doesn't exist or belongs to different tenant
  - 403 Forbidden if user lacks permission

**GET /api/documents/:id/download**
- **Purpose:** Generate signed URL for document download
- **Auth:** Requires authentication (any role)
- **Tenant Scoping:** Allow only if document belongs to user's tenant
- **Response:**
```typescript
{
  url: string // Signed URL valid for 5 minutes
  filename: string
  mimeType: string
}
```
- **Errors:**
  - 404 Not Found if document doesn't exist or belongs to different tenant
  - 500 Internal Server Error if signed URL generation fails

**Implementation Notes:**
- Use Elysia's built-in file upload handling: `@elysiajs/multipart` plugin
- File received as `File` object in request body
- Automatically set `uploaded_by`, `created_at`, `updated_at`, `tenant_id` from authenticated user context
- For file size validation, use `@elysiajs/multipart` config: `maxFileSize: 10485760` (10MB)
- Return standardized error responses on failures

**ElysiaJS Multipart Plugin Setup:**
```typescript
// Add to apps/api/src/routes/documents/index.ts or upload.ts
import { Elysia } from 'elysia'
import { multipart } from '@elysiajs/multipart'

// Enable multipart/form-data handling
app.use(multipart({
  maxFileSize: 10485760, // 10MB in bytes
  maxFiles: 10, // Allow up to 10 files per request for multi-upload
}))
```

**Route Aggregation Pattern:**
```typescript
// apps/api/src/routes/documents/index.ts
import { Elysia } from 'elysia'
import { list } from './list'
import { upload } from './upload'
import { download } from './download'
import { deleteDocument } from './delete'

export const documentsRoutes = new Elysia({ prefix: '/api' })
  .use(list)
  .use(upload)
  .use(download)
  .use(deleteDocument)
```

Then register in main app:
```typescript
// apps/api/src/index.ts
import { documentsRoutes } from './routes/documents'

app.use(documentsRoutes)
```

### Security Considerations

[Source: architecture/security-and-performance.md]

**Tenant Isolation:**
- Backend: All queries MUST filter by `tenant_id` from authenticated user's JWT
- Storage: Use tenant-specific folder structure in Supabase Storage: `{tenantId}/{supplierId}/...`
- RLS Policies: Enforce tenant isolation at storage bucket level (Phase 2)
- **CRITICAL:** On upload, automatically set tenant_id from user context (never accept from client)
- **CRITICAL:** On download/delete, verify document belongs to user's tenant before allowing access

**File Upload Security:**
- Validate MIME type against whitelist (not just file extension)
- Validate file size before upload (reject files > 10MB)
- Sanitize filename to prevent path traversal attacks (remove `..`, `/`, `\`)
- Generate unique storage path using UUID to prevent filename collisions
- Use Supabase signed URLs for downloads (time-limited, no permanent public URLs)
- Implement virus scanning in Phase 2 (add TODO comment for now)

**Authorization:**
- Only Admin and Procurement Manager can upload/delete documents
- All authenticated users can view/download documents (read-only access)
- Use RBAC middleware to enforce role checks on backend
- Use `usePermissions()` hook to conditionally render upload/delete UI elements

**Input Validation:**
- Validate all metadata fields using Zod on both frontend and backend
- Validate document type is one of the allowed enum values
- Validate expiry date is a valid date and not in the past (warning only, not error)
- Return proper error responses for validation failures

**Data Integrity:**
- Store document metadata in database immediately after successful upload to storage
- Use database transactions for multi-step operations (upload + metadata insert)
- Implement soft delete (set deleted_at) to preserve audit trail
- Do not delete files from storage on soft delete (keep for compliance)
- Automatically set audit fields (uploaded_by, created_at, updated_at) server-side

### Performance Considerations

[Source: architecture/security-and-performance.md]

**Performance Targets:**
- Document list load time: < 2s
- File upload time: < 5s for 5MB file
- File download time: < 3s for 5MB file (via CDN-backed signed URL)
- Document list refresh after upload: < 1s

**Backend Optimizations:**
- Use database indexes on tenant_id and supplier_id for fast document queries
- Limit document list to 100 most recent per supplier (pagination can be added in Phase 2)
- Return only necessary document fields in list response (exclude binary data)
- Use Supabase signed URLs for downloads (offload bandwidth to CDN)

**Frontend Optimizations:**
- Use optimistic UI updates (add document to list immediately after upload starts)
- Show progress bars during upload to improve perceived performance
- Lazy load document thumbnails (for images/PDFs) using IntersectionObserver
- Debounce sort/filter actions to avoid excessive re-renders
- Use Remix's built-in caching for document list data

**File Upload Optimizations:**
- Upload multiple files in parallel (max 3 concurrent uploads)
- Show individual progress bar for each file
- Cancel support for in-progress uploads
- Retry failed uploads automatically (max 3 retries)

### File Type Validation

[Source: AC #3, #13]

**Allowed MIME Types:**
```typescript
const ALLOWED_MIME_TYPES = [
  'application/pdf',                          // PDF
  'image/png',                                // PNG
  'image/jpeg',                               // JPG
  'application/vnd.ms-excel',                 // Excel (.xls)
  'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet', // Excel (.xlsx)
  'application/msword',                       // Word (.doc)
  'application/vnd.openxmlformats-officedocument.wordprocessingml.document', // Word (.docx)
];

const MAX_FILE_SIZE = 10 * 1024 * 1024; // 10MB in bytes
```

**Validation Logic:**
```typescript
function validateFile(file: File): { valid: boolean; error?: string } {
  if (!ALLOWED_MIME_TYPES.includes(file.type)) {
    return { valid: false, error: 'File type not supported. Please upload PDF, Excel, Word, PNG, or JPG files.' };
  }
  
  if (file.size > MAX_FILE_SIZE) {
    return { valid: false, error: `File size exceeds 10MB limit. File size: ${(file.size / 1024 / 1024).toFixed(2)}MB` };
  }
  
  return { valid: true };
}
```

### Expiration Date Logic

[Source: AC #9, #10]

**Expiration Warning Thresholds:**
- **Within 30 days:** Yellow warning badge "Expires in X days"
- **Expired:** Red danger badge "Expired"
- **No expiry date:** No badge displayed

**Calculation Logic:**
```typescript
function getExpiryStatus(expiryDate: Date | null): 'expired' | 'warning' | 'valid' | null {
  if (!expiryDate) return null;
  
  const today = new Date();
  const diffInDays = Math.ceil((expiryDate.getTime() - today.getTime()) / (1000 * 60 * 60 * 24));
  
  if (diffInDays < 0) return 'expired';
  if (diffInDays <= 30) return 'warning';
  return 'valid';
}
```
