## Story

**As a** developer,  
**I want** to remove the concept of task templates and generate runtime tasks only when a workflow step starts,  
**so that** tasks act as to-do warnings and are driven by workflow execution.

---

## Acceptance Criteria

1. The engine does not use task_template
2. Tasks are created only as task_instance when a step_instance transitions to active
3. A task_instance represents a to-do warning and includes:
   - tenant_id
   - process_instance_id
   - step_instance_id
   - assignee_type (role or user)
   - assignee_role (nullable)
   - assignee_user_id (nullable)
   - title
   - description
   - completion_time_days (nullable)
   - due_at (nullable)
   - status (open, completed)
4. Tasks are not configurable via reusable templates
5. Task title/description/completion_time_days are configured at the workflow step level (template) and instantiated at runtime
6. Task visibility is tenant-isolated via tenant_id

### Previous Story Insights

[Source: Story 2.2.5 - Task Template Library and Runtime Task Model]

**Implementation Completed:**
- Created `task_template` table for reusable task definitions
- Created `task_instance` table for runtime task execution
- Implemented tenant isolation with composite indexes
- SET NULL cascade rule for task_template_id to preserve runtime tasks
- RESTRICT cascade rule for user references (audit trail)

**Why Course Correction is Needed:**
- Task templates add unnecessary complexity for MVP
- Task configuration belongs at the workflow step level (Story 2.2.6), not as standalone reusable library
- Tasks should be simple to-do warnings created when a step starts, not complex entities with template management
- Removing task_template simplifies the data model and reduces backoffice UI complexity (no need for template builder)

**Key Changes in This Story:**
1. **Remove task_template table entirely** - no more reusable task library
2. **Tasks created only at runtime** - when step_instance transitions to 'active'
3. **Configuration moves to workflow_step_template** - task title/description/completion_time_days defined at workflow template level (Story 2.2.6)
4. **Assignee flexibility** - tasks can be assigned to role (all users with role) or specific user
5. **Simplified status** - only 'open' and 'completed' (remove 'pending', 'in_progress', 'cancelled')
6. **step_instance_id now required** - all tasks belong to a specific step (no process-level tasks)

**New Model (Runtime Only):**
```typescript
export enum TaskAssigneeType {
  ROLE = "role",
  USER = "user",
}

export enum TaskInstanceStatus {
  OPEN = "open",
  COMPLETED = "completed",
}

export interface TaskInstance {
  id: string; // UUID
  tenantId: string; // FK to tenants - CASCADE delete
  processInstanceId: string; // FK to process_instance - CASCADE delete
  stepInstanceId: string; // FK to step_instance - CASCADE delete (NOT NULL - always linked to step)
  title: string; // Runtime title (from workflow step config)
  description: string; // Runtime description (from workflow step config)
  assigneeType: TaskAssigneeType; // 'role' or 'user'
  assigneeRole: string | null; // Role name if assigneeType = 'role' (e.g., 'procurement_manager')
  assigneeUserId: string | null; // User ID if assigneeType = 'user' - FK to users(id) RESTRICT
  completionTimeDays: number | null; // Days to complete (from workflow step config)
  dueAt: Date | null; // Calculated due date (createdAt + completionTimeDays)
  status: TaskInstanceStatus; // 'open' | 'completed'
  completedBy: string | null; // User ID - RESTRICT delete
  completedAt: Date | null;
  metadata: Record<string, any>; // Runtime task data
  createdAt: Date;
  updatedAt: Date;
  deletedAt: Date | null;
}
```

**Task Assignment Logic:**

**Assignee Type: 'role'**
- `assigneeType` = 'role'
- `assigneeRole` = role name (e.g., 'procurement_manager')
- `assigneeUserId` = NULL
- All users in tenant with matching role can see and complete the task
- Task appears in task list for all users with that role

**Assignee Type: 'user'**
- `assigneeType` = 'user'
- `assigneeRole` = NULL
- `assigneeUserId` = specific user ID
- Only the assigned user can see and complete the task
- Task appears in task list only for that specific user

**Due Date Calculation:**
```typescript
// When task is created from workflow step:
const dueAt = completionTimeDays 
  ? new Date(Date.now() + completionTimeDays * 24 * 60 * 60 * 1000)
  : null;
```

**Workflow Integration (Story 2.2.6 will define this):**
- workflow_step_template will include fields:
  - `task_title` (string) - copied to task_instance.title
  - `task_description` (text) - copied to task_instance.description
  - `completion_time_days` (integer, nullable) - copied to task_instance.completion_time_days
  - `assignee_type` (enum: 'role', 'user')
  - `assignee_role` (string, nullable) - used if assignee_type = 'role'
  - `assignee_user_id` (UUID, nullable) - used if assignee_type = 'user'
- When step_instance transitions to 'active', workflow engine creates task_instance from workflow_step_template config

**New Schema (This Story):**
```sql
CREATE TABLE task_instance (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    process_instance_id UUID NOT NULL REFERENCES process_instance(id) ON DELETE CASCADE,
    step_instance_id UUID NOT NULL REFERENCES step_instance(id) ON DELETE CASCADE, -- NOW NOT NULL
    title VARCHAR(300) NOT NULL,
    description TEXT,
    assignee_type VARCHAR(50) NOT NULL DEFAULT 'user', -- NEW: 'role' or 'user'
    assignee_role VARCHAR(50), -- NEW: role name if assignee_type = 'role'
    assignee_user_id UUID REFERENCES users(id) ON DELETE RESTRICT, -- NEW: user ID if assignee_type = 'user'
    completion_time_days INTEGER, -- NEW: days to complete (from workflow step config)
    due_at TIMESTAMP WITH TIME ZONE, -- RENAMED from due_date
    status VARCHAR(50) NOT NULL DEFAULT 'open', -- UPDATED: 'open' or 'completed'
    completed_by UUID REFERENCES users(id) ON DELETE RESTRICT,
    completed_at TIMESTAMP WITH TIME ZONE,
    metadata JSONB NOT NULL DEFAULT '{}',
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMP WITH TIME ZONE
);

-- NEW INDEX: Role-based task assignment
CREATE INDEX idx_task_instance_tenant_assignee_status 
    ON task_instance(tenant_id, assignee_type, assignee_role, status) 
    WHERE status = 'open' AND deleted_at IS NULL;

-- NEW INDEX: User-based task assignment
CREATE INDEX idx_task_instance_tenant_assignee_user_status 
    ON task_instance(tenant_id, assignee_user_id, status) 
    WHERE assignee_type = 'user' AND status = 'open' AND deleted_at IS NULL;

-- KEEP: Workflow task retrieval
CREATE INDEX idx_task_instance_process_step 
    ON task_instance(process_instance_id, step_instance_id) 
    WHERE deleted_at IS NULL;

-- UPDATE: Overdue task detection (use due_at)
CREATE INDEX idx_task_instance_tenant_due_at 
    ON task_instance(tenant_id, due_at) 
    WHERE status = 'open' AND deleted_at IS NULL;
```

**Migration Changes:**
1. DROP TABLE task_template CASCADE
2. ALTER TABLE task_instance DROP COLUMN task_template_id
3. ALTER TABLE task_instance DROP COLUMN assigned_to
4. ALTER TABLE task_instance ADD COLUMN assignee_type VARCHAR(50) NOT NULL DEFAULT 'user'
5. ALTER TABLE task_instance ADD COLUMN assignee_role VARCHAR(50)
6. ALTER TABLE task_instance ADD COLUMN assignee_user_id UUID REFERENCES users(id) ON DELETE RESTRICT
7. ALTER TABLE task_instance ADD COLUMN completion_time_days INTEGER
8. ALTER TABLE task_instance RENAME COLUMN due_date TO due_at
9. ALTER TABLE task_instance ALTER COLUMN step_instance_id SET NOT NULL
10. DROP INDEX idx_task_instance_tenant_assigned_status
11. CREATE INDEX idx_task_instance_tenant_assignee_status
12. CREATE INDEX idx_task_instance_tenant_assignee_user_status
13. UPDATE INDEX idx_task_instance_tenant_due_date to use due_at column

**Cascade Delete Rules:**
- `tenant_id` → CASCADE: Deleting tenant removes all task instances
- `process_instance_id` → CASCADE: Deleting process removes all task instances
- `step_instance_id` → CASCADE: Deleting step removes all task instances
- `assignee_user_id` → RESTRICT: Cannot delete user assigned to tasks (audit trail)
- `completed_by` → RESTRICT: Cannot delete user who completed tasks (audit trail)
