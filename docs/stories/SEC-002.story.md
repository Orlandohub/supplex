## Story

**As a** platform operator,
**I want** the hardcoded JWT secret fallback removed and a production guard added, and the workflow-health endpoint scoped to the requesting admin's tenant,
**so that** the API server cannot silently start with a known secret in production, and tenant admins cannot see diagnostic data from other tenants.

---

## Context

This is a small, high-priority P0 story addressing two critical security findings from the [Auth Architecture Analysis](../implementation-reports/auth-architecture-analysis-final.md). Both are quick wins that can be implemented in parallel with SEC-001.

### Analysis Report Items Addressed

| ID | Finding | Severity |
|----|---------|----------|
| C1 | **JWT secret hardcoded fallback** — server starts with known secret `"dev-secret-key-change-in-production-min-32-chars"` if env vars are not set. Zod validation passes because the fallback meets `min(32)`. No production guard exists. | **Critical** |
| C2 | **`workflow-health.ts` queries across all tenants** — a tenant admin sees every tenant's data via the health-check endpoint. | **Critical** |

### Dependencies

- **None** — this story is independent and can be implemented in parallel with SEC-001.
- No downstream dependents. After SEC-006 (JWT signing key migration to ES256), the JWT secret env var becomes irrelevant entirely, but C1 must be fixed now as an interim measure.

## Acceptance Criteria

### AC 1: JWT Secret Fallback Removed (C1)

1. The hardcoded fallback string `"dev-secret-key-change-in-production-min-32-chars"` is removed from `config.ts`.
2. In production (`NODE_ENV === "production"`), if neither `SUPABASE_JWT_SECRET` nor `JWT_SECRET` is set, `loadConfig()` throws a clear, descriptive error and the server does not start.
3. In `development` or `test` (`NODE_ENV === "development"` or `NODE_ENV === "test"`) only, if neither env var is set, the server may start with a clearly-labeled development-only secret, logged with a loud warning. This development secret must not be the same as the previously hardcoded string. The fallback must not be available in any other environment (including preview, staging, or any production-like environment).
4. The Zod schema for `jwt.secret` in production mode must not accept any default — the value must come from environment variables.

### AC 2: Workflow Health Endpoint Tenant-Scoped (C2)

5. All three diagnostic queries in `workflow-health.ts` (stuck processes, orphaned tasks, state mismatches) include a `tenantId` filter matching the authenticated admin's `user.tenantId`.
6. The handler receives the authenticated user's context from the middleware (already available via `requireRole([UserRole.ADMIN])`) and uses `user.tenantId` for filtering.
7. The response shape is unchanged — only the data scope is narrowed to the requesting tenant.
8. The existing tests in `__tests__/workflow-health.test.ts` are updated to verify tenant isolation (query results only contain data from the test tenant).

### AC 3: No Functional Regressions

9. The server starts successfully in development with no JWT secret env var set (using the dev-only fallback).
10. The server starts successfully in production when `SUPABASE_JWT_SECRET` or `JWT_SECRET` is properly configured.
11. All existing tests pass.

### C1: Current JWT Secret Loading (lines 70-75 of `config.ts`)

```typescript
jwt: {
  secret:
    process.env.SUPABASE_JWT_SECRET ||
    process.env.JWT_SECRET ||
    "dev-secret-key-change-in-production-min-32-chars",  // ← REMOVE THIS
},
```

The Zod schema validates `jwt.secret` as `z.string().min(32)`. The hardcoded fallback passes this validation because it is 48 characters long. In production, if both env vars are unset, the server silently starts with a known secret — any attacker who reads the source code can forge JWTs.

**Fix approach:** Remove the fallback. In any environment other than `development` or `test`, `undefined` fails the Zod `z.string().min(32)` check, and the server refuses to start. In `development`/`test` only, provide a different dev-only string that is clearly labeled and logged with a loud warning. Preview, staging, and production-like environments must require a real secret.

### C2: Workflow Health Cross-Tenant Queries

The three diagnostic queries in `workflow-health.ts` select from `processInstance`, `stepInstance`, and `taskInstance` without any `tenantId` filter. The route already requires `UserRole.ADMIN`, and the authenticated user context (including `user.tenantId`) is available from the middleware chain — it just isn't being used.

All three tables have a `tenantId` column (confirmed in schema files). The fix is to add `eq(table.tenantId, user.tenantId)` to each query's `where` clause.

**Query-by-query breakdown:**

| Query | Current Filter | Required Addition |
|-------|---------------|-------------------|
| Q1: Stuck processes | `status = 'in_progress' AND currentStepInstanceId IS NULL AND deletedAt IS NULL` | `AND tenantId = user.tenantId` on `processInstance` |
| Q2: Orphaned tasks | `taskInstance.status = 'pending' AND stepInstance.status NOT IN (...)` | `AND taskInstance.tenantId = user.tenantId` |
| Q3: State mismatches | `status = 'in_progress' AND NOT EXISTS (active steps)` | `AND tenantId = user.tenantId` on `processInstance` |

### Implementation Guardrails

1. **Minimize response shape changes** to the workflow health endpoint. The data scope narrows to the requesting tenant. The only acceptable structural change is optionally dropping the `tenantId` field from Query 1's projection (since all results share the same tenant).
2. **The dev-only fallback secret must be different from the original hardcoded string.** If the original string appears in any deployed environment's `auth.users` JWT tokens, those tokens would break if the fallback changes. But since this is for local development only and the real secret comes from env vars, this is safe.
3. **Do not modify the Zod schema itself** — `z.string().min(32)` is correct. The fix is in the value provided to it, not the validation rule.
4. **The `requestLogger` destructuring uses `any` type** in the health endpoint. Keep it as-is — this is addressed in SEC-008 (error standardization).

- Use the structured Pino logger (already imported in both files)
- ElysiaJS routes: child routes must NOT have prefixes (parent `/api/admin` provides it)

## Operator Notes

### Deployment Prerequisite (BLOCKING)

**Production deployment must not proceed** unless the following is confirmed:

- `SUPABASE_JWT_SECRET` or `JWT_SECRET` is present and non-empty in the target environment's configuration (env vars, secrets manager, or hosting platform config).
- This applies to **all non-development/test environments**: production, staging, preview, and any custom deployment target.

After this change, the server will **refuse to start** if the JWT secret is missing. This is intentional fail-fast behavior. Deploying without the secret configured will cause an immediate outage (server crashes on startup).

```bash
# Verify before deploying (adapt to your hosting platform)
echo $SUPABASE_JWT_SECRET   # must be non-empty
# or
echo $JWT_SECRET             # must be non-empty
```

If neither variable is set, **do not deploy**. Configure the secret first.

### After Deploying

No special rollout steps. Both changes (config guard and tenant filter) take effect immediately on server restart. No data migration, no session impact, no rollout window.
