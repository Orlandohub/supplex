## Story

**As a** platform operator,
**I want** all workflow engine state mutations to be atomic, concurrency-safe, and idempotent,
**so that** concurrent users cannot corrupt workflow state through race conditions, double transitions, or duplicate task creation.

---

## Context

An architecture assessment (`.cursor/plans/architecture_assessment_report_5c337427.plan.md`) identified that the workflow engine performs **0 transactions** across 15-25 sequential DB statements per step completion, has **0 concurrency controls** (no atomic compare-and-swap, no row locking of any kind), and **0 idempotency guards** on task creation. These are the highest-severity findings in the report (Tier 1 — C1, C2, C3 and Tier 2 — R1, R2).

### Current Problems (from assessment)

1. **C1 — Concurrent validation approval causes double transitions.** In `approve-validation-task.ts`, two validators approving simultaneously both mark their task completed, both re-read all validation tasks, both see `allComplete === true`, and both call `transitionToNextStep`. This creates duplicate next-step tasks and corrupts `current_step_instance_id`.

2. **C2 — Concurrent step completion (duplicate submit).** `completeStep` checks `step.status !== "active"` but between the read and subsequent writes, a parallel request also passes the guard. Both requests mark the step completed and both create validation tasks or both transition.

3. **C3 — Non-atomic multi-table writes.** All engine operations (instantiate, complete step, transition, approve validation, decline) perform 5-25 sequential DB statements without transactions. A crash mid-sequence leaves partial state (e.g., step marked completed but no next step activated).

4. **R1 — Duplicate task creation on re-activation.** When a validation decline resets a step to `active`, `createTasksForStep` is called without checking if pending tasks already exist. If the decline is processed twice (race or retry), duplicate action tasks are created.

5. **R2 — Duplicate validation task creation.** `createValidationTasks` inserts one task per approver role with no idempotency check. If `completeStep` races (C2), validation tasks are duplicated.

### Technical Approach

- **Wrap all engine operations in `db.transaction()`** using Drizzle's postgres-js transaction API.
- **Replace read-then-write status checks with atomic compare-and-swap (CAS)**: use `UPDATE ... WHERE status = :expected RETURNING *` — if 0 rows returned, another request already processed it. **CAS is the preferred concurrency mechanism** throughout this story. Only resort to explicit row locking (`SELECT ... FOR UPDATE`) if a specific path cannot be safely handled with CAS alone; no such case has been identified in the current engine.
- **Add idempotency guards** on task creation: check for existing pending tasks before inserting, backed by role-aware partial unique indexes.
- **Thread the transaction object (`tx`)** through all engine helper functions instead of using the module-level `db` singleton.
- **Consolidate the duplicate instantiation logic** between `routes/workflows/instantiate.ts` and `lib/workflow-engine/instantiate-workflow.ts`.
- **Add a missing partial index** on `task_instance(step_instance_id, status)` for the hot validation approval path.

## Acceptance Criteria

### AC 1: Transactional Engine Operations

1. `completeStep()`, `approveValidationTask()`, and `instantiateWorkflow()` each execute all their DB statements inside a single `db.transaction()` call. If any statement fails, all changes within that operation are rolled back.
2. `transitionToNextStep()`, `createTasksForStep()`, `createValidationTasks()`, and `seedStepDocuments()` all accept a transaction handle (`tx`) and operate within the caller's transaction — they never open their own connections.
3. The route handler `POST /workflows/steps/:stepInstanceId/complete` wraps the engine mutation logic in a single `db.transaction()` call. Event description reads and `logWorkflowEvent` calls remain **outside** the transaction (fire-and-forget, see Implementation Guardrails).
4. The route handler `POST /workflows/instantiate` delegates to the library `instantiateWorkflow()` function (eliminating the duplicate code in the route) and the entire operation runs in a single transaction.

### AC 2: Atomic Status Transitions (Compare-and-Swap)

5. Step completion uses an atomic `UPDATE step_instance SET status = 'completed' WHERE id = :id AND status = 'active' RETURNING *` instead of a separate SELECT + status check + UPDATE. If 0 rows are returned, the request is rejected with "Step already processed or not in active state".
6. Validation task approval uses an atomic `UPDATE task_instance SET status = 'completed' WHERE id = :id AND status = 'pending' RETURNING *`. If 0 rows are returned, the approval is rejected with "Task already processed".
7. After approving a validation task, the "are all validations complete?" check and subsequent step transition are serialized using an atomic CAS: `UPDATE step_instance SET status = 'validated' WHERE id = :stepId AND status = 'awaiting_validation' RETURNING *`. Only the request that successfully transitions the step row proceeds to call `transitionToNextStep()`. If 0 rows are returned (another approval already claimed the transition), the request returns success without duplicating the transition.
8. The decline path in `complete.ts` uses an atomic `UPDATE task_instance ... WHERE status = 'pending'` for the declining task and an atomic `UPDATE step_instance SET status = 'active' WHERE id = :id AND status = 'awaiting_validation'` for the step reset.

### AC 3: Idempotency Guards on Task Creation

9. `createTasksForStep()` checks for existing pending tasks of type `action` or `resubmission` for the same `step_instance_id` before inserting. If a pending task already exists, it returns the existing task(s) without inserting duplicates.
10. `createValidationTasks()` checks for existing pending validation tasks **per role** for the same `step_instance_id` before inserting. If a pending validation task already exists for a given `assignee_role`, that role is skipped. This is role-aware: a step with 2 required roles legitimately has 2 concurrent pending validation tasks.
11. A new migration adds role-aware partial unique indexes to enforce idempotency at the DB level as a safety net:
    - **Validation tasks**: `UNIQUE (step_instance_id, task_type, assignee_role) WHERE status = 'pending' AND deleted_at IS NULL` — prevents duplicate pending validation tasks for the same role on the same step, while allowing one pending task per required role.
    - **Action/resubmission tasks**: `UNIQUE (step_instance_id, task_type) WHERE status = 'pending' AND deleted_at IS NULL AND task_type IN ('action', 'resubmission')` — prevents duplicate pending action tasks on the same step. Verify this matches actual cardinality before implementation (currently one action/resubmission task per step).
    - **Performance index**: A non-unique partial index on `(step_instance_id, status) WHERE deleted_at IS NULL` for the hot validation approval path (moved from AC 5 for clarity).

### AC 4: Validation Task Semantics Preservation

The current validation model **must be preserved exactly**. This story hardens concurrency; it does not change validation behavior.

12. One validation task is created per required approver role (from `validationConfig.approverRoles`). A step with N required roles produces exactly N pending validation tasks.
13. Approval by any single user with the matching role satisfies that role's validation requirement.
14. The step does not proceed to the next step until **all** required roles have approved (all validation tasks completed).
15. If any single validation task is declined, **all remaining pending** validation tasks for that step are closed (`outcome: 'auto_closed'`) and the step returns to `active` for resubmission.
16. No duplicate validation task may be created for the same `(step_instance_id, assignee_role)` combination while a pending task already exists.

### AC 5: Instantiation Consolidation & Batch Inserts

17. The route `POST /workflows/instantiate` delegates entirely to `instantiateWorkflow()` from the engine library. The duplicate template-query, step-creation loop, and rollback logic in the route file are removed.
18. `instantiateWorkflow()` uses a single batch `INSERT INTO step_instance VALUES (...), (...), ...` instead of inserting each step in a loop with individual round trips.

### AC 6: Engine Function Signatures

19. All engine helper functions (`transitionToNextStep`, `createTasksForStep`, `createValidationTasks`, `seedStepDocuments`) accept the DB/transaction handle as a **required** parameter (no defaulting to the module-level `db` singleton). This ensures they can only be called within a transaction context.
20. The `NodePgDatabase<any>` type import in engine files is replaced with the correct `PostgresJsDatabase` type from `drizzle-orm/postgres-js` to match the actual driver used in `packages/db`.

### AC 7: Backward Compatibility

21. All existing workflow functionality (submit, approve, decline, instantiate, transition) continues to work correctly — no behavioral changes beyond atomicity and race-condition prevention.
22. Existing unit tests pass after refactoring (with mock/spy updates for the new `tx` parameter threading).

## Implementation Guardrails

**This story hardens the current implementation. It must not change product behavior.**

The following are explicitly **out of scope** and must not be altered:

- **Workflow statuses** — do not add, remove, or rename any `process_instance.status` values.
- **Step statuses** — do not add, remove, or rename any `step_instance.status` values. The existing status flow (`blocked` → `active` → `completed` → `awaiting_validation` → `validated`, plus `declined`) is preserved exactly.
- **Approval model** — do not change how approvals work. Validation tasks are role-based; one user per role approves.
- **Validation semantics** — do not change when validation tasks are created, how many are created, or what triggers transition. See AC 4 for the model that must be preserved.
- **Task assignment semantics** — do not change how `createTasksForStep` resolves assignees (supplier user → procurement manager → initiator fallback chain).
- **Workflow sequencing** — do not change step ordering, the `stepOrder + 1` transition logic, or process completion detection.
- **Event logging architecture** — event logging reliability/completeness redesign is out of scope. This story keeps `logWorkflowEvent` outside the transaction (fire-and-forget), which matches current behavior. Do not redesign the audit subsystem, add transactional event logging, or change the event logger's API.

**Concurrency mechanism preference**: Use atomic compare-and-swap (`UPDATE ... WHERE status = :expected RETURNING *`) as the primary concurrency control. Do not introduce `SELECT ... FOR UPDATE` unless a specific code path demonstrably cannot be handled with CAS alone. No such case exists in the current engine.

## Implementation Passes

Although this remains **one story**, implementation may be done in two safe passes to reduce risk:

**Pass A — Transaction ownership + atomic state transitions** (Tasks 1, 2, 3, 4, 5, 7):
Thread `tx` through all engine helpers, wrap route handlers in transactions, and replace read-then-write patterns with atomic CAS. This pass eliminates the concurrency bugs (C1, C2, C3) and the non-atomic writes.

**Pass B — Idempotency guards + consolidation + tests** (Tasks 6, 8, 9):
Add application-level idempotency checks and DB safety-net indexes, consolidate the duplicate instantiation logic, batch-insert steps, and update all tests.

The dev agent may implement both passes in a single session or split them. Either way, **all tasks must be completed** before the story is marked done. If splitting, Pass A should be completed and tested before starting Pass B.

## Dev Notes

### Drizzle Transaction API [Source: packages/db/src/index.ts]

The project uses `drizzle-orm/postgres-js` with the `postgres` driver. Transactions are supported via:

```typescript
import { db } from "@supplex/db";

const result = await db.transaction(async (tx) => {
  // All operations use tx instead of db
  const [row] = await tx.select().from(table).where(...);
  await tx.update(table).set({ ... }).where(...);
  return row;
});
// If any statement throws, the entire transaction is rolled back.
```

The `tx` object has the same API as `db` — it's a `PostgresJsTransaction` which extends `PostgresJsDatabase`. For typing engine functions, use:

```typescript
import type { PostgresJsDatabase } from "drizzle-orm/postgres-js";
type DbOrTx = PostgresJsDatabase<any>;
```

Note: The current engine files import `NodePgDatabase` from `drizzle-orm/node-postgres` which is the **wrong driver type** — the actual driver is `postgres-js`. This works at runtime due to structural compatibility but should be corrected.

### Atomic Compare-and-Swap Pattern (Preferred Concurrency Mechanism)

CAS is the **primary concurrency control** used throughout this story. Every status transition uses this pattern — do not use `SELECT ... FOR UPDATE` unless CAS is demonstrably insufficient for a specific path.

Instead of:
```typescript
// BROKEN: read-then-write race
const [step] = await db.select().from(stepInstance).where(eq(stepInstance.id, id));
if (step.status !== "active") return error;
await db.update(stepInstance).set({ status: "completed" }).where(eq(stepInstance.id, id));
```

Use:
```typescript
// CORRECT: atomic CAS
const [updated] = await tx
  .update(stepInstance)
  .set({ status: "completed", completedBy, completedDate: new Date() })
  .where(
    and(
      eq(stepInstance.id, stepInstanceId),
      eq(stepInstance.tenantId, tenantId),
      eq(stepInstance.status, "active") // CAS guard
    )
  )
  .returning();

if (!updated) {
  return { success: false, error: "Step already processed or not in active state" };
}
```

### Validation Approval Serialization (C1 Fix)

The critical fix for C1 is to serialize the "all validations complete → transition" decision. After atomically completing a validation task, attempt an atomic step status CAS:

```typescript
// Atomically mark task as completed (only if still pending)
const [completedTask] = await tx
  .update(taskInstance)
  .set({ status: "completed", completedBy: userId, completedAt: new Date() })
  .where(and(eq(taskInstance.id, taskId), eq(taskInstance.status, "pending")))
  .returning();

if (!completedTask) return { success: false, error: "Task already processed" };

// Check remaining pending validation tasks
const pendingValidations = await tx
  .select({ id: taskInstance.id })
  .from(taskInstance)
  .where(
    and(
      eq(taskInstance.stepInstanceId, completedTask.stepInstanceId),
      eq(taskInstance.taskType, "validation"),
      eq(taskInstance.status, "pending")
    )
  );

if (pendingValidations.length === 0) {
  // Attempt to atomically claim the transition
  const [transitioned] = await tx
    .update(stepInstance)
    .set({ status: "validated", updatedAt: new Date() })
    .where(
      and(
        eq(stepInstance.id, completedTask.stepInstanceId),
        eq(stepInstance.status, "awaiting_validation") // CAS guard
      )
    )
    .returning();

  if (transitioned) {
    // This request owns the transition
    await transitionToNextStep(transitioned.id, transitioned.processInstanceId, tenantId, tx);
    return { success: true, allValidationsComplete: true, nextStepActivated: true };
  }
  // Another request already transitioned — safe to return success
  return { success: true, allValidationsComplete: true, nextStepActivated: false };
}

return { success: true, allValidationsComplete: false, remainingApprovals: pendingValidations.length };
```

### Drizzle `onConflictDoNothing` API (for Task 6.3)

Use this as a secondary safety net alongside the application-level idempotency check. It leverages the role-aware partial unique indexes from Task 1:

```typescript
// For validation tasks — index is (step_instance_id, task_type, assignee_role) WHERE pending
const [task] = await tx
  .insert(taskInstance)
  .values({ stepInstanceId, taskType: "validation", assigneeRole: role, ... })
  .onConflictDoNothing()
  .returning();

// If task is undefined, a pending validation task for this role already exists

// For action tasks — index is (step_instance_id, task_type) WHERE pending AND type IN (action, resubmission)
const [actionTask] = await tx
  .insert(taskInstance)
  .values({ stepInstanceId, taskType: "action", ... })
  .onConflictDoNothing()
  .returning();
```
