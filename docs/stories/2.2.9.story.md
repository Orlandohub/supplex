## Story

**As an** admin or procurement_manager,  
**I want** to start any workflow process for a supplier from the supplier detail page,  
**so that** I can manage all supplier-related processes (qualification, audits, onboarding, compliance checks, etc.) in one central location.

---

## Acceptance Criteria

### Database Changes

1. `workflow_template` table has `active` BOOLEAN field (default: true)
2. `workflow_step_template` table has `completion_status` VARCHAR(100) field (nullable)

### Workflow Template Management

3. Admin can toggle workflow templates between active and inactive
4. Only templates with `active = true` AND `status = 'published'` appear in workflow selection dropdowns
5. Inactive workflows cannot be instantiated (existing process instances continue normally)

### Supplier Detail Page - Workflows Tab

6. Supplier detail page has "Workflows" tab (replaces "Qualifications" tab)
7. Tab shows all workflow process instances where `entity_type='supplier'` AND `entity_id=supplier.id`
8. "Start Process" button shows dropdown of active published workflow templates
9. Admin or Procurement Manager can initiate any workflow for the supplier

### Workflow Instantiation

10. Selected `workflow_template_version_id` is used to instantiate the process
11. Process is linked via: `entity_type='supplier'`, `entity_id=supplier.id`
12. Workflow status tracking follows custom completion statuses (see AC 13-15)

### Custom Workflow Status Tracking

13. When a workflow step completes:
    - If `workflow_step_template.completion_status` is NOT NULL, copy value to `process_instance.status`
    - If `completion_status` is NULL, leave `process_instance.status` unchanged
14. Example: Step "Document Review" with `completion_status="Under Review"` completes → workflow status becomes "Under Review"
15. Process status reflects the most recently completed step's completion_status

### Task Visibility

16. Supplier-facing tasks visible only to `supplier_user` of that supplier and tenant
17. Internal tasks visible only to internal users of the same tenant

### Workflows Page

18. Global workflows page at `/workflows` (replaces `/qualifications`)
19. Lists all workflow processes across all suppliers (tenant-filtered)
20. Filters: All, My Tasks, My Initiated
21. Supports search, filtering by status, export to CSV

### UI Navigation

22. Sidebar "Qualifications" link renamed to "Workflows" linking to `/workflows`

### Backward Compatibility

23. Existing process instances with `entity_type='qualification'` continue to work
24. System treats qualification as a legacy entity type
25. No data migration required

### Implementation Logic: Completion Status

**Location:** `apps/api/src/lib/workflow-engine/complete-step.ts`

**Add after step completion (line ~120):**

```typescript
// Get the workflow step template to check for completion_status
const stepTemplate = await db.query.workflowStepTemplate.findFirst({
  where: and(
    eq(workflowStepTemplate.id, stepInstance.workflowStepTemplateId),
    eq(workflowStepTemplate.tenantId, tenantId)
  ),
});

// If completion_status is defined, update process status
if (stepTemplate?.completionStatus) {
  await db
    .update(processInstance)
    .set({
      status: stepTemplate.completionStatus,
      updatedAt: new Date(),
    })
    .where(eq(processInstance.id, stepInstance.processInstanceId));
}
```

**Key Points:**
- Only update status if `completionStatus` is NOT NULL
- This is NULL-safe: if field is NULL/undefined, status unchanged
- No validation on status values (admin defines them freely)
- Status is a VARCHAR(50) in process_instance (already exists)
