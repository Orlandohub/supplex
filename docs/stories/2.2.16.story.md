## Story

**As an** admin and supplier_user,  
**I want** the workflow execution engine to correctly transition between steps, auto-complete form steps on submission, and display accurate UI information,  
**so that** workflows run end-to-end without manual workarounds and the user experience matches expectations.

---

## Context

### User Feedback (from Manual Testing of Stories 2.2.14 + 2.2.15)

During manual acceptance testing, the following issues were reported:

1. After submitting a form, the user is not redirected to the workflow page; the step is not completed, next task is not created, workflow status is not updated.
2. Dropdown fields on form templates should not be allowed without values.
3. Breadcrumbs on the form page within a workflow context show "Forms > Form" but the Forms page does not exist in that context.
4. The workflow progress visualization card does not show validation of a step as its own visual step.
5. Workflow detail page should show the workflow template name as the workflow name.
6. Form instances are created on first navigation to the form step — if that takes time, there is no loading indicator.
7. Need a new "Forms" tab on the supplier detail page to see forms filled out or in draft during workflows.
8. **CRITICAL**: Workflow breaks after step 1 — validation task was not created, step 2 showed active but no task was created, "Complete Step" button does nothing.

### Root Cause Analysis

**Critical Bug (#8 and #1):** Story 2.2.14 renamed database columns and metadata keys from `workflowTemplateVersionId` to `workflowTemplateId`, but the workflow engine files were not fully updated:

- `transition-to-next-step.ts` line 94 reads `process.metadata.workflowTemplateVersionId` — but `instantiate-workflow.ts` stores it as `workflowTemplateId`
- `transition-to-next-step.ts` line 108 queries `workflowStepTemplate.workflowTemplateVersionId` — column was renamed to `workflowTemplateId`
- `complete-step.ts` line 181 reads `step.workflowStepTemplateId` — this column does NOT exist on `step_instance`
- `approve-validation-task.ts` line 114 reads `step.workflowStepTemplateId` — same issue
- `routes/workflows/steps/complete.ts` lines 77-82 queries step templates by `tenantId + stepOrder` only, without filtering by `workflowTemplateId` — can match wrong template
- `return-to-previous-step.ts` lines 103-109 has the same missing `workflowTemplateId` filter

**Result:** After step 1 completes, `transitionToNextStep()` throws because `workflowTemplateVersionId` is undefined. The error is caught silently, step 2 gets activated but no tasks are created, and the "Complete Step" button calls an endpoint that also fails to find the right template.

## Acceptance Criteria

### Task 1: Fix Workflow Engine Step Transitions (AC 1-7)

1. `transition-to-next-step.ts` reads `process.metadata.workflowTemplateId` (not `workflowTemplateVersionId`)
2. `transition-to-next-step.ts` queries `workflowStepTemplate.workflowTemplateId` (not `workflowTemplateVersionId`)
3. `complete-step.ts` resolves the step template via `process.metadata.workflowTemplateId` + `step.stepOrder` instead of `step.workflowStepTemplateId`
4. `approve-validation-task.ts` resolves the step template via the same method (process metadata + step order)
5. `routes/workflows/steps/complete.ts` queries step template by `tenantId + workflowTemplateId + stepOrder` (adds `workflowTemplateId` filter)
6. `return-to-previous-step.ts` queries step template by `tenantId + workflowTemplateId + stepOrder` (adds `workflowTemplateId` filter; requires process metadata access)
7. After step completion, the next step is activated AND tasks are created for it

### Task 2: Form Submit Auto-Completes Workflow Step (AC 8-12)

8. When a form submission is submitted via `POST /api/form-submissions/:id/submit` and the submission is linked to a `stepInstanceId`, the workflow step is automatically completed
9. After form submit and step completion, the API returns the `processInstanceId` so the frontend can redirect
10. After successful form submission, the frontend redirects the user to the workflow detail page (`/workflows/processes/:processId`)
11. The step's associated tasks are marked as completed
12. If the step has `requiresValidation=true`, validation tasks are created automatically after form submit

### Task 3: Fix Breadcrumbs for Workflow Form Page (AC 13-15)

13. When a form is accessed from a workflow context (submission has `processInstanceId` and `stepInstanceId`), breadcrumbs show: "Workflows > [Workflow Name] > [Step Name]"
14. When a form is accessed outside workflow context, breadcrumbs show: "Forms > [Form Name]"
15. The "Back" button navigates to the workflow detail page when in workflow context

### Task 4: Show Validation as Visual Step in Progress (AC 16-19)

16. `WorkflowStepTimeline.tsx` displays `awaiting_validation` status with an orange/amber icon and "Awaiting Validation" badge
17. `WorkflowStepTimeline.tsx` displays `validated` status with a green check icon and "Validated" badge
18. For steps with validation tasks, the timeline inserts a visual "Validation" sub-step below the parent step showing the validation state
19. `WorkflowProcessDetailPage.tsx` history tab includes `awaiting_validation` and `validated` steps

### Task 5: Show Workflow Template Name (AC 20-21)

20. `WorkflowProcessDetailPage.tsx` header displays `process.metadata.workflowName` as the page title instead of `processType`
21. If `metadata.workflowName` is not available, fall back to formatted `processType` (backward compatibility)

### Task 6: Add Loading State for Form Instance Creation (AC 22-23)

22. The workflow form step route (`_app.workflows.processes.$processId.steps.$stepId.form.tsx`) shows a loading spinner/skeleton while the draft submission is being created
23. If draft creation fails, an error message is displayed instead of a blank page

### Task 7: Verify Dropdown Field Validation (AC 24-25)

24. API rejects creation of `dropdown` and `multi_select` fields with empty `options.choices` array (already implemented — verify only)
25. Frontend `AddFieldModal.tsx` and `EditFieldModal.tsx` disable save button when dropdown/multi_select has no options (already implemented — verify only)

### Task 8: Add Supplier "Forms" Tab (AC 26-30)

26. `SupplierDetailTabs.tsx` has a new "Forms" tab between "Workflows" and "History"
27. The Forms tab queries and displays all `form_submission` records linked to the supplier's workflow processes
28. Each row shows: form template name, workflow name, step name, submission status (draft/submitted), submission date
29. Clicking a submission navigates to `/forms/:submissionId` with workflow context for correct breadcrumbs
30. If no form submissions exist, show empty state: "No forms have been filled out for this supplier yet"

### Technical Context — Step Template Resolution Pattern

The core pattern that needs to be applied across all workflow engine files:

**WRONG (current broken code):**
```typescript
// step.workflowStepTemplateId does NOT exist on step_instance
const stepTemplate = await db.query.workflowStepTemplate.findFirst({
  where: eq(workflowStepTemplate.id, step.workflowStepTemplateId),
});
```

**CORRECT (fix):**
```typescript
// Get workflowTemplateId from process metadata
const [process] = await db
  .select()
  .from(processInstance)
  .where(eq(processInstance.id, step.processInstanceId));

const workflowTemplateId = (process.metadata as any)?.workflowTemplateId;

// Query by templateId + tenantId + stepOrder
const [stepTemplate] = await db
  .select()
  .from(workflowStepTemplate)
  .where(
    and(
      eq(workflowStepTemplate.workflowTemplateId, workflowTemplateId),
      eq(workflowStepTemplate.tenantId, tenantId),
      eq(workflowStepTemplate.stepOrder, step.stepOrder)
    )
  );
```

This pattern must be applied in:
- `complete-step.ts` (line 178-183)
- `approve-validation-task.ts` (line 113-114)
- `routes/workflows/steps/complete.ts` (lines 74-82, 346-351)
- `return-to-previous-step.ts` (lines 101-109)
- `transition-to-next-step.ts` (lines 94-112) — slightly different, uses metadata key

### Technical Context — Supplier Forms Tab

Query pattern for supplier form submissions:
```sql
SELECT fs.id, fs.status, fs.submitted_at, fs.created_at,
       ft.name AS form_template_name,
       pi.metadata->>'workflowName' AS workflow_name,
       si.step_name
FROM form_submission fs
JOIN process_instance pi ON fs.process_instance_id = pi.id
JOIN step_instance si ON fs.step_instance_id = si.id
JOIN form_template ft ON fs.form_template_id = ft.id
WHERE pi.entity_type = 'supplier'
  AND pi.entity_id = :supplierId
  AND pi.tenant_id = :tenantId
ORDER BY fs.created_at DESC
```
