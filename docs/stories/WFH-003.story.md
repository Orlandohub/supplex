# Story WFH-003: Manual Workflow Recovery Admin Endpoint

<!-- Powered by BMAD™ Core -->

## Status

**Approved**

Date Created: April 8, 2026

## Story

**As a** platform operator,
**I want** an admin endpoint that can diagnose and repair known partial-failure states for a specific workflow process,
**so that** stuck or inconsistent processes can be recovered manually without direct database intervention.

---

## Context

Even with WFH-001 and WFH-002 in place (transaction wrapping for document review and form submit), partial failures can still occur from:
- **Pre-WFH-001/002 data:** Processes that failed before those fixes were deployed and remain in an inconsistent state.
- **Extreme edge cases:** DB crashes during transaction commit, application crashes mid-execution, network timeouts.

The existing `GET /api/admin/workflow-health` endpoint detects some inconsistencies (stuck processes with null `currentStepInstanceId`, orphaned tasks, state mismatches where `in_progress` processes have no active steps), but provides no repair capability. For the cases it doesn't detect (form-step divergence, missing supplier status on completion), there is no visibility at all.

Section 9.3 of the architecture analysis identifies five specific partial-failure states. This story addresses the three that are correctness-related and repairable by admin action:

| # | Partial-Failure State | Currently Detected? | Recovery Action |
|---|----------------------|--------------------|-----------------| 
| 1 | Form `submitted` + Step `active` | **No** | Re-run `completeStep()` for the step |
| 2 | Step `validated`/`completed` + Process `in_progress` + No next step active | **Partially** (health endpoint finds some as "state mismatches") | Activate the correct next step, or mark process complete if last step |
| 3 | Process `complete` + Supplier status not updated | **No** | Re-read the workflow type → supplier status mapping and apply it |

**Out of scope for this story:**
- Email notification sweep (orphaned PENDING rows) — this is a background job concern, not a manual recovery action.
- Automated periodic reconciliation — the analysis explicitly scopes the first recovery story as manual-only.

### Architecture Analysis Findings Addressed

| ID | Finding | Severity | Category |
|----|---------|----------|----------|
| **C4** | No recovery path for partial failures: health endpoint detects stuck states but provides no repair mechanism | **Medium** | Operational Resilience |

### Dependencies

- **Depends on WFH-001 and WFH-002 being implemented first.** The repair endpoint uses `completeStep(tx, ...)` and `transitionToNextStep(tx, ...)` which must accept a transaction handle correctly. WFH-001 and WFH-002 ensure these engine functions are properly threaded.
- **Independent of WFH-004 and WFH-005.**

## Acceptance Criteria

### AC 1: Per-Process Repair Endpoint Exists

1. New endpoint: `POST /api/admin/workflow-repair/:processId`.
2. Admin-only (uses the existing `requireAdmin` middleware via the admin route group).
3. Requires `processId` as a path parameter (UUID).
4. Accepts optional body: `{ dryRun?: boolean }`. Defaults to `false`.
5. The endpoint is tenant-isolated: it can only operate on processes belonging to the requesting admin's tenant.

### AC 2: Diagnoses Known Partial-Failure States

6. The endpoint checks the given process for three specific partial-failure states (see Repair Types below).
7. Each detected issue is included in the response with a `type`, `description`, and affected entity IDs.
8. If `dryRun: true`, the endpoint reports what it found but does NOT apply any repairs.
9. If no issues are found, the response says so clearly.

### AC 3: Repairs Form-Step Divergence (State 1)

10. **Detection:** A `formSubmission` exists where `status = 'submitted'` AND `stepInstanceId` references a `stepInstance` with `status = 'active'` for this process.
11. **Repair:** Run `completeStep(tx, { tenantId, stepInstanceId, completedBy: adminUserId, outcome: 'completed' })` inside a `db.transaction()`.
12. **Safety:** The CAS guard in `completeStep` (`WHERE status = 'active'`) prevents double-completion. If the step was already completed by another path, the CAS fails and the repair reports "already resolved."
13. **Post-repair state:** Step progresses normally (next step activated or process completed, depending on workflow structure).

### AC 4: Repairs Stalled Transition (State 2)

14. **Detection:** A `stepInstance` exists where `status IN ('validated', 'completed')` AND `processInstance.status = 'in_progress'` AND no subsequent step (by `stepOrder`) is in `active` status.
15. **Repair strategy depends on sub-state:**
    - **Next step exists and is `pending`:** Run `transitionToNextStep(tx, stepInstanceId, processId, tenantId)` inside a `db.transaction()`. Then reconcile `completedSteps` to the actual count of completed/validated/skipped steps.
    - **Next step exists and is already `active`:** Only fix `processInstance.currentStepInstanceId` and reconcile `completedSteps` (the transition partially ran — don't re-activate or re-create tasks).
    - **No next step exists (last step):** Mark `processInstance` as `complete`, set `completedDate`, reconcile `completedSteps` to `totalSteps`, and apply the supplier status mapping (same logic as `transitionToNextStep` completion path).
16. **Safety:** The sub-state detection prevents double-activation. The `completedSteps` reconciliation uses an actual count query rather than incrementing, avoiding the double-increment problem from partial re-runs.

### AC 5: Repairs Missing Supplier Status on Completion (State 3)

17. **Detection:** `processInstance.status = 'complete'` AND `processInstance.entityType = 'supplier'` AND the chain `workflowTemplate → workflowType → supplierStatus` resolves to a status that does NOT match the supplier's current `supplierStatusId`.
18. **Repair:** Read the expected supplier status from the `workflowType → supplierStatus` mapping and apply it to the supplier.
19. **Safety:** If the mapping chain is broken (e.g., `workflowType.supplierStatusId` is null), the repair reports "mapping not found — cannot auto-repair" and does NOT modify the supplier. The admin must fix the mapping first.
20. **Edge case:** If the supplier's status was manually overridden via `force: true` (WFH-004), the repair would revert it. The repair response should note the current vs. expected status so the admin can decide. In `dryRun` mode, this is visible without any mutation.

### AC 6: Each Repair Is Logged as a Workflow Event

21. Each successful repair logs a workflow event via `logWorkflowEvent` with:
    - `eventType`: A new `WorkflowEventType.ADMIN_REPAIR` value (added to the enum).
    - `eventDescription`: Human-readable summary of what was repaired (e.g., "Admin repair: re-ran step completion for form-step divergence").
    - `actorUserId`, `actorName`, `actorRole`: The admin who triggered the repair.
    - `processInstanceId`, `stepInstanceId`: The affected entities.
    - `metadata`: `{ repairType: "form_step_divergence" | "stalled_transition" | "missing_supplier_status", dryRun: false }`.
22. Workflow events are fire-and-forget (same as all other event logging). Event logging failures do NOT fail the repair.
23. Dry-run operations do NOT create workflow events.

### AC 7: Structured Response

24. The response includes:
    ```typescript
    {
      success: true,
      data: {
        processId: string;
        processStatus: string;
        dryRun: boolean;
        issues: Array<{
          type: "form_step_divergence" | "stalled_transition" | "missing_supplier_status";
          description: string;
          stepInstanceId?: string;
          repaired: boolean;  // false in dryRun mode
          repairAction?: string;  // human-readable description of what was (or would be) done
          error?: string;  // if repair failed
        }>;
        totalIssues: number;
        totalRepaired: number;
      }
    }
    ```
25. If the process does not exist or belongs to a different tenant, return 404.
26. If the process has no issues, return 200 with `issues: []` and `totalIssues: 0`.

### AC 8: Route-Level Tests Verify Repair Behavior

27. Route-level test: Repair form-step divergence — seed a process with form `submitted` + step `active`, call repair, assert step completed and process advanced.
28. Route-level test: Repair stalled transition (next step pending) — seed a process with step `validated` + next step `pending`, call repair, assert next step activated.
29. Route-level test: Repair missing supplier status — seed a completed process with supplier status not matching the mapping, call repair, assert supplier status updated.
30. Route-level test: Dry run — seed a process with an issue, call with `dryRun: true`, assert issue reported but no state change.
31. Route-level test: No issues — call repair on a healthy process, assert `issues: []`.
32. Route-level test: Process not found → 404.

## Dev Notes

### Repair Safety Model

Each repair operation uses one of two safety strategies:

1. **CAS-protected re-run** (State 1): The engine function has a built-in CAS guard (`WHERE status = 'active'`). Re-running it is inherently safe — if the state was already resolved, the CAS fails gracefully.

2. **State reconciliation** (States 2 and 3): The repair computes the correct target state from current data and applies it directly. This avoids the "double-increment" problem that would occur from blindly re-running `transitionToNextStep` (which increments `completedSteps` with `sql\`completedSteps + 1\`` rather than computing the correct value).

The `completedSteps` reconciliation is critical for State 2 repairs. Always compute the actual count of completed/validated/skipped steps rather than incrementing. This makes the repair idempotent — running it twice produces the same result.

### Why `completeStep` Is Safe to Re-Run but `transitionToNextStep` Is Not

`completeStep` opens with a CAS update: `UPDATE step_instance SET status = 'completed' WHERE id = :id AND status = 'active'`. If the step is no longer active, the update touches zero rows and the function returns `{ success: false }`. This is a clean, safe idempotency guard.

`transitionToNextStep` does NOT have a CAS guard. It reads the current step, finds the next one, and unconditionally activates it. Its `completedSteps` update uses `sql\`completedSteps + 1\``, which double-counts on re-run. For the repair, the "next step pending" sub-case can safely call `transitionToNextStep` (since the next step is genuinely pending), but MUST follow up with a `completedSteps` reconciliation query.

### Supplier Status Repair — Manual Override Awareness

If a supplier's status was manually overridden via WFH-004's `force: true` flag, the State 3 repair would "undo" that override by applying the workflow's expected mapping. The repair response includes both `currentStatus` and `expectedStatus` so the admin can:
- In `dryRun`: see the mismatch and decide whether to proceed.
- In live mode: verify the action was correct after the fact.

This is acceptable for a manual recovery tool — the admin is explicitly choosing to reconcile the state. If the admin knows the manual override was intentional, they simply skip the repair (or don't run it for that process).

### Transaction Boundaries

- **State 1 repair:** `db.transaction(async (tx) => { completeStep(tx, ...) })`. The entire `completeStep` chain (CAS, task completion, transition) runs in one transaction.
- **State 2 repair (next step pending):** `db.transaction(async (tx) => { transitionToNextStep(tx, ...); reconcileCompletedSteps(tx, ...); })`. Transition + counter fix in one transaction.
- **State 2 repair (next step active / last step):** `db.transaction(async (tx) => { /* targeted fixes */ })`. Direct state updates in one transaction.
- **State 3 repair:** `db.transaction(async (tx) => { /* supplier status update */ })`. Single update in a transaction.

### Endpoint Placement

The endpoint goes in `apps/api/src/routes/admin/workflow-repair.ts` and is registered in the admin route group (`/api/admin`). This gives it the `requireAdmin` middleware automatically. The full path is `POST /api/admin/workflow-repair/:processId`.

### Workflow Event Type Addition

Adding `ADMIN_REPAIR = "admin_repair"` to `WorkflowEventType` is the only shared-type change in this story. It is a new enum value — no existing values are modified. The workflow event table's `event_type` column is `varchar`, not a PG enum, so no migration is needed.

### Implementation Guardrails

1. **Do not add automated background reconciliation.** This story is manual-only. A periodic sweep job is a separate later story.
2. **Do not modify the workflow engine functions.** Use them as-is (they accept `DbOrTx` correctly after WFH-001/002).
3. **Do not modify the existing `workflow-health` endpoint.** The repair endpoint handles its own detection. Health endpoint enhancements are a separate concern.
4. **Do not modify `ApiError` or the error response infrastructure.**
5. **All repairs must run inside a `db.transaction()`.** No independent writes.
6. **All DB queries must include tenant isolation.**
7. **The `completedSteps` reconciliation must use an actual count query, not an increment.** This is critical for idempotency.
8. **The `dryRun` flag must prevent ALL mutations** — no DB writes, no workflow events.

### Testing

- **Test runner:** Bun (`bun test`)
- **Test location:** `apps/api/src/routes/admin/__tests__/workflow-repair.test.ts`
- **Test approach: route-level integration.** Tests exercise the real `workflowRepairRoute` through Elysia `app.handle()` with real DB fixtures representing each partial-failure state.
- **Auth bypass:** Use `.derive(() => ({ user: testAdminUser, requestLogger: ... }))` to inject a real DB-backed admin user.
- **DB fixture pattern:** Each test seeds a specific partial-failure state. The fixture setup is more complex than previous stories because it requires creating processes in specific intermediate states (e.g., step `validated` with next step `pending`).
- **Assertion strategy:** After each repair call, verify BOTH the HTTP response (repair report) AND the DB state (process/step/supplier rows reflect the expected post-repair state). For dry-run tests, verify the DB state is UNCHANGED.
- **Workflow event assertion:** After non-dry-run repairs, query the `workflowEvent` table to verify an `admin_repair` event was logged with the correct process ID and metadata.
