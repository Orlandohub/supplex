## Story

**As a** procurement manager,  
**I want** to view complete supplier information on a detail page,  
**so that** I have all supplier context in one place.

---

## Acceptance Criteria

1. Supplier detail page displays comprehensive information in tabbed interface: Overview, Documents, History
2. **Overview Tab** displays: Company name, status badge, address, phone, email, website, categories, certifications, notes
3. Primary contact and additional contacts displayed with name, title, email, phone
4. Status change dropdown allows Admin/Procurement Manager to update supplier status
5. Status change triggers confirmation modal and records change in audit history
6. Supplier metadata displayed: Created date, Created by, Last modified date, Last modified by
7. "Edit" button navigates to edit form (visible only to users with edit permissions)
8. "Delete" button with confirmation modal (soft delete, preserves history, Admin only)
9. Mobile-responsive layout with touch-optimized buttons and readable text
10. Loading state displays while fetching supplier data
11. 404 page displays if supplier ID not found or belongs to different tenant
12. Breadcrumb navigation: Home > Suppliers > [Supplier Name]

### API Endpoint Specifications

[Source: architecture/api-specification.md]

**GET /api/suppliers/:id**
- **Purpose:** Fetch single supplier by ID
- **Auth:** Requires valid JWT (any authenticated user can view suppliers)
- **Tenant Scoping:** Automatically filtered by user's tenant_id
- **Parameters:**
  - `id` (path): UUID of supplier
- **Response:**
```typescript
{
  supplier: Supplier
}
```
- **Errors:**
  - 404 if supplier not found or belongs to different tenant
  - 404 if supplier is soft-deleted (deleted_at is not null)
  - 400 if ID is not valid UUID format
  - 401 if not authenticated
- **Performance:** Must respond in < 500ms

**Implementation Notes:**
- Use Drizzle ORM with `.where(eq(suppliers.id, id))` and `.where(eq(suppliers.tenantId, tenantId))`
- Exclude soft-deleted suppliers: `.where(isNull(suppliers.deletedAt))`
- Join with users table to get created_by and updated_by user information for display
- Return standardized error responses on failures

**PATCH /api/suppliers/:id/status**
- **Purpose:** Update supplier status
- **Auth:** Requires Admin or Procurement Manager role
- **Tenant Scoping:** Status change only within user's tenant
- **Request Body:**
```typescript
{
  status: SupplierStatus, // One of: prospect, qualified, approved, conditional, blocked
  note?: string // Optional note for audit trail
}
```
- **Response:**
```typescript
{
  supplier: Supplier // Updated supplier object
}
```
- **Errors:**
  - 404 if supplier not found
  - 403 if user lacks permission (must be Admin or Procurement Manager)
  - 400 if status value is invalid

**DELETE /api/suppliers/:id**
- **Purpose:** Soft delete supplier
- **Auth:** Requires Admin role only
- **Implementation:** Set `deleted_at` to current timestamp (do not physically delete)
- **Response:**
```typescript
{
  success: true,
  message: "Supplier deleted successfully"
}
```
- **Errors:**
  - 404 if supplier not found
  - 403 if user is not Admin

### Eden Treaty Client Usage

[Source: architecture/coding-standards.md, Story 1.5 implementation]

**CRITICAL:** Never use direct `fetch()` calls. Always use Eden Treaty client for type safety.

**Example Loader Implementation:**
```typescript
import { createEdenTreatyClient } from '~/lib/api-client';
import { requireAuth } from '~/lib/auth';

export async function loader({ request, params }: LoaderFunctionArgs) {
  const user = await requireAuth(request);
  const { id } = params;
  
  if (!id) {
    throw new Response('Supplier ID required', { status: 400 });
  }
  
  const client = createEdenTreatyClient(request);
  
  const { data, error } = await client.api.suppliers[id].get();
  
  if (error) {
    if (error.status === 404) {
      throw new Response('Supplier not found', { status: 404 });
    }
    throw new Response('Failed to load supplier', { status: 500 });
  }
  
  return json({ supplier: data.supplier });
}
```

**Example Action Implementation for Status Change:**
```typescript
export async function action({ request, params }: ActionFunctionArgs) {
  const user = await requireAuth(request);
  const { id } = params;
  const formData = await request.formData();
  const status = formData.get('status') as string;
  
  const client = createEdenTreatyClient(request);
  
  const { data, error } = await client.api.suppliers[id].status.patch({
    status
  });
  
  if (error) {
    return json({ error: error.message }, { status: error.status || 500 });
  }
  
  return json({ success: true, supplier: data.supplier });
}
```

Eden Treaty provides:
- End-to-end type safety (TypeScript inference from ElysiaJS routes)
- Automatic request/response validation
- Error handling with typed error responses

[Source: architecture/security-and-performance.md]

- Backend: All queries MUST filter by `tenant_id` from authenticated user's JWT
- Frontend: Never display tenant_id in URLs (use only supplier ID)

**Authorization:**
- All authenticated users can view supplier details (no role restriction)
- Status change requires Admin or Procurement Manager role
- Delete requires Admin role only
- Use RBAC middleware to enforce role checks on backend
- Use `usePermissions()` hook to conditionally render UI elements

**Input Validation:**
- Validate supplier ID is valid UUID format using Zod
- Validate status enum values using Zod schema
- Sanitize all inputs to prevent XSS
- Return proper error responses for validation failures

**Soft Delete:**
- Never physically delete suppliers (preserve audit history)
- Set `deleted_at` timestamp instead
- Exclude soft-deleted suppliers from all queries
- Soft-deleted suppliers should return 404 to users
