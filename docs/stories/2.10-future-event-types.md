## Overview

Story 2.10 defined 8 event types for the audit trail system. Of these, **6 are currently implemented** and **2 are reserved for future features** that don't exist in the current workflow system.

---

## Currently Implemented Event Types ✅

The following 6 event types are **fully implemented** and actively logging events:

1. **WORKFLOW_INITIATED** - Logs when a qualification workflow is created
   - **Location**: `apps/api/src/routes/workflows/initiate.ts`
   - **Metadata**: Supplier name, risk score, checklist info

2. **DOCUMENT_UPLOADED** - Logs when a document is uploaded or linked
   - **Location**: `apps/api/src/routes/workflows/upload-document.ts`
   - **Metadata**: Document name, type, whether linked vs uploaded

3. **DOCUMENT_REMOVED** - Logs when a document is unlinked from workflow
   - **Location**: `apps/api/src/routes/workflows/remove-document.ts`
   - **Metadata**: Document name before removal

4. **STAGE_SUBMITTED** - Logs when workflow is submitted for stage review
   - **Location**: `apps/api/src/routes/workflows/submit.ts`
   - **Metadata**: Stage number, assigned reviewer

5. **STAGE_APPROVED** - Logs when a workflow stage is approved
   - **Location**: `apps/api/src/routes/workflows/approve-stage.ts`
   - **Metadata**: Stage number, comments, next stage info, final approval flag

6. **STAGE_REJECTED** - Logs when a workflow stage is rejected
   - **Location**: `apps/api/src/routes/workflows/reject-stage.ts`
   - **Metadata**: Stage number, rejection reason, workflow reversion

## Future Event Types (Not Yet Implemented) ⏳

### 1. RISK_SCORE_CHANGED

**Purpose**: Track changes to workflow risk scores during qualification process

**Current Status**: Not implemented - risk scores are currently set once during workflow initiation and not modified during the qualification process.

**Future Implementation Scenario**:
- Quality manager reviews supplier documentation
- Identifies new risk factors (e.g., financial instability, quality issues)
- Updates risk assessment during Stage 2 (Quality Review)
- System logs RISK_SCORE_CHANGED event

**Implementation Requirements**:
- Add "Update Risk Score" feature to workflow review UI
- Create API endpoint: `PATCH /api/workflows/:id/risk-score`
- Integrate event logging:
  ```typescript
  await logWorkflowEvent({
    tenantId,
    workflowId,
    eventType: WorkflowEventType.RISK_SCORE_CHANGED,
    eventDescription: `Risk score updated from ${oldScore} to ${newScore}`,
    actorUserId,
    actorName,
    actorRole,
    metadata: {
      oldRiskScore: oldScore,
      newRiskScore: newScore,
      reason: updateReason,
      geographic: newAssessment.geographic,
      financial: newAssessment.financial,
      quality: newAssessment.quality,
      delivery: newAssessment.delivery,
    },
  });
  ```

**Database Schema**: ✅ Already supports this event type
- `workflow_events.eventType` accepts any varchar(100)
- `workflow_events.metadata` JSONB field stores old/new scores
- Filtering by `eventType = 'RISK_SCORE_CHANGED'` will work when implemented

**Timeline Display**: ✅ Already supports this event type
- Frontend timeline component has icon mapping (AlertTriangle icon)
- Color coding defined: Yellow (#ffcc00)
- Filtering includes this type in "all" filter

**Estimated Effort**: 4-6 hours
- 2 hours: Backend API endpoint with validation
- 2 hours: Frontend UI for risk score update
- 1 hour: Integration with event logger
- 1 hour: Testing

### 2. COMMENTS_ADDED

**Purpose**: Track standalone comments added to workflows outside of approval/rejection

**Current Status**: Partially implemented - comments are currently captured as part of STAGE_APPROVED and STAGE_REJECTED events, but there's no standalone commenting feature.

**Future Implementation Scenario**:
- User views workflow and wants to add a note
- Clicks "Add Comment" button
- Enters comment: "Discussed with supplier - documentation update expected by Friday"
- System logs COMMENTS_ADDED event

**Implementation Requirements**:
- Add "Add Comment" feature to workflow detail page
- Create API endpoint: `POST /api/workflows/:id/comments`
- Integrate event logging:
  ```typescript
  await logWorkflowEvent({
    tenantId,
    workflowId,
    eventType: WorkflowEventType.COMMENTS_ADDED,
    eventDescription: "Comment added to workflow",
    actorUserId,
    actorName,
    actorRole,
    comments: commentText,
    metadata: {
      commentLength: commentText.length,
      isPrivate: false, // If private comments feature is added
    },
  });
  ```

**Database Schema**: ✅ Already supports this event type
- `workflow_events.eventType` accepts any varchar(100)
- `workflow_events.comments` text field stores comment content
- Filtering by `eventType = 'COMMENTS_ADDED'` will work when implemented

**Timeline Display**: ✅ Already supports this event type
- Frontend timeline component has icon mapping (MessageSquare icon)
- Color coding defined: Purple (#9933cc)
- Filtering includes "comments" category
- Expandable/collapsible comments UI already built

**Estimated Effort**: 3-4 hours
- 1 hour: Backend API endpoint
- 1.5 hours: Frontend comment input UI
- 0.5 hours: Integration with event logger
- 1 hour: Testing

## Implementation Priority Recommendations

### Short-term (Next Sprint)
**Neither feature is required for current workflow functionality.** The 6 implemented event types provide complete audit trail coverage for the current system.

### Medium-term (Next Quarter)
1. **RISK_SCORE_CHANGED** - Higher priority
   - Aligns with dynamic risk management practices
   - Requested by quality managers for complex supplier evaluations
   - Compliance benefit: demonstrates ongoing due diligence

2. **COMMENTS_ADDED** - Lower priority
   - Nice-to-have collaboration feature
   - Current approval/rejection comments are sufficient for audit trail
   - Consider adding when workflow collaboration features are expanded

### Long-term Considerations
- **Notification integration**: Send emails when comments are added
- **@mentions**: Tag users in comments for attention
- **Comment threads**: Reply to specific comments
- **Audit trail filtering**: Search/filter by comment content

## Database Migration Impact

**No migration needed** - The existing `workflow_events` table schema supports these event types:
- `eventType` is varchar(100) - accepts any event type string
- `metadata` is JSONB - flexible schema for event-specific data
- `comments` is text - sufficient for long comments
- Indexes on `eventType` will optimize filtering when implemented
