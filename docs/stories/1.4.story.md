# Story 1.4: Tenant Management & User Roles

## Story

**As a** tenant administrator,  
**I want** to manage users and assign roles within my organization,  
**so that** I can control who has access and what they can do.

---

## Acceptance Criteria

1. Four roles defined and enforced: Admin, Procurement Manager, Quality Manager, Viewer
2. Role stored in `user_metadata` field of Supabase Auth user record
3. Tenant settings page shows list of all users in current tenant with their roles
4. Admin can invite new users via email with role assignment
5. Admin can change user roles for existing users within their tenant
6. Admin can deactivate/reactivate users (soft delete, preserves audit history)
7. ElysiaJS middleware validates JWT and extracts user role for authorization
8. RBAC helper functions created to check permissions (e.g., `canEditSupplier(user, action)`)
9. Frontend conditionally renders UI elements based on user role (hide edit buttons for Viewers)
10. Attempting unauthorized actions returns 403 Forbidden with clear error message
11. Audit log records all user management actions (invite, role change, deactivation)
12. Initial tenant admin is automatically assigned during sign-up

```typescript
export interface User {
```

### Supabase Auth Integration

[Source: architecture/tech-stack.md, architecture/high-level-architecture.md]

**Role Storage Strategy:**
- **Primary:** Role stored in `users` table as source of truth
- **Secondary:** Role synced to Supabase Auth `user_metadata.role` for JWT inclusion
- **JWT Claim:** JWT token includes role in payload for server-side validation

**User Metadata Structure:**
```typescript
// Stored in Supabase Auth user_metadata field
{
  role: "admin" | "procurement_manager" | "quality_manager" | "viewer",
  tenant_id: "uuid-string",
  full_name: "John Doe"
}
```

**Synchronization Pattern:**
1. When user is created, set role in both database and Supabase Auth
2. When role is updated, update both locations atomically
3. JWT refresh automatically picks up new role from user_metadata

### API Endpoints Specification

[Source: architecture/api-specification.md]

**Required Endpoints:**

**GET /api/users**
- Returns: List of users in authenticated user's tenant
- Auth: Requires Admin role
- Query params: `?status=active|inactive` (optional)
- Response: `{ users: User[], total: number }`

**POST /api/users/invite**
- Body: `{ email: string, role: UserRole, message?: string }`
- Auth: Requires Admin role
- Action: Creates Supabase Auth user + sends invitation email
- Response: `{ user: User, invitationSent: boolean }`

**PATCH /api/users/:id/role**
- Body: `{ role: UserRole }`
- Auth: Requires Admin role, cannot change own role
- Action: Updates role in database + Supabase Auth metadata
- Response: `{ user: User }`

**PATCH /api/users/:id/status**
- Body: `{ isActive: boolean }`
- Auth: Requires Admin role, cannot deactivate own account
- Action: Updates is_active flag (soft deactivation)
- Response: `{ user: User }`

**GET /api/users/:id/audit**
- Returns: Audit log for specific user's management actions
- Auth: Requires Admin role
- Response: `{ logs: AuditLog[], total: number }`

### Frontend Role-Based Rendering

[Source: Epic AC #9, architecture/components.md]

**React Component Pattern:**
```typescript
// apps/web/app/lib/rbac/RoleGuard.tsx
interface RoleGuardProps {
  allowedRoles: UserRole[];
  children: React.ReactNode;
  fallback?: React.ReactNode;
}

export function RoleGuard({ allowedRoles, children, fallback }: RoleGuardProps) {
  const { user } = useAuth();
  
  if (!user || !allowedRoles.includes(user.role)) {
    return fallback || null;
  }
  
  return <>{children}</>;
}

// Usage:
<RoleGuard allowedRoles={['admin', 'procurement_manager']}>
  <EditSupplierButton />
</RoleGuard>
```

**Permission Hook Pattern:**
```typescript
// apps/web/app/hooks/usePermissions.ts
export function usePermissions() {
  const { user } = useAuth();
  
  return {
    canManageUsers: user?.role === 'admin',
    canEditSupplier: ['admin', 'procurement_manager'].includes(user?.role),
    canCreateEvaluation: ['admin', 'quality_manager'].includes(user?.role),
    // ... more permission checks
  };
}
```

### Audit Logging Schema

[Source: Epic AC #11, architecture/data-models.md patterns]

**Audit Log Table Schema:**
```sql
CREATE TABLE audit_logs (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  user_id UUID NOT NULL REFERENCES users(id),
  target_user_id UUID REFERENCES users(id), -- User being managed
  action VARCHAR(100) NOT NULL, -- USER_INVITED, ROLE_CHANGED, USER_DEACTIVATED
  details JSONB NOT NULL, -- { old_role, new_role } or other context
  ip_address VARCHAR(50),
  user_agent TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL
);

CREATE INDEX idx_audit_logs_tenant_id ON audit_logs(tenant_id);
CREATE INDEX idx_audit_logs_target_user_id ON audit_logs(target_user_id);
CREATE INDEX idx_audit_logs_created_at ON audit_logs(created_at DESC);
```

**TypeScript Interface:**
```typescript
export enum AuditAction {
  USER_INVITED = 'USER_INVITED',
  ROLE_CHANGED = 'ROLE_CHANGED',
  USER_DEACTIVATED = 'USER_DEACTIVATED',
  USER_REACTIVATED = 'USER_REACTIVATED'
}

export interface AuditLog {
  id: string;
  tenantId: string;
  userId: string; // Who performed the action
  targetUserId: string | null; // Who was affected
  action: AuditAction;
  details: Record<string, any>;
  ipAddress: string | null;
  userAgent: string | null;
  createdAt: Date;
}
```

### Security Considerations

[Source: architecture/security-and-performance.md]

**Authorization Security:**
- Enforce role checks on both frontend AND backend (defense-in-depth)
- Never trust client-sent role, always validate from JWT
- Prevent privilege escalation: Admin cannot change own role
- Prevent account lockout: Admin cannot deactivate own account
- Rate limiting: 10 req/min for user management endpoints

**Input Validation:**
- Validate email format with Zod schema
- Validate role against enum (reject invalid roles)
- Sanitize user-provided messages before storing
- Validate tenant_id matches authenticated user's tenant

### Performance Optimization

[Source: architecture/security-and-performance.md]

**Frontend Performance:**
- Lazy load user list table (only fetch on settings page access)
- Optimistic UI updates for role changes (rollback on error)
- Cache user list in Zustand for 5 minutes
- Paginate audit logs (20 entries per page)

**Backend Performance:**
- Index on users.tenant_id for fast user list queries
- Index on audit_logs.target_user_id for audit history
- Cache permission matrix in memory (static data)
- Batch user invitations if needed in future
