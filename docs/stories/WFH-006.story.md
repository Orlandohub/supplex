## Story

**As a** platform operator,
**I want** the workflow engine functions to return all the context that mutation routes need for event logging,
**so that** routes stop re-reading 5-8 tables after each transaction just to populate event log fields, eliminating unnecessary latency and DB load.

---

## Context

After every workflow mutation (`submit`, `approve`, `decline`), the route handlers re-read multiple tables outside the transaction to get context for `logWorkflowEvent` calls. This data was either already read inside the transaction by the engine function (and discarded) or could be cheaply read there. The re-reads add 5-8 extra DB queries per request with zero correctness benefit.

**Example — `complete.ts` submit path (current, post-WFH-001/002):**
```
[INSIDE TX]  completeStep(tx, ...) → returns { stepCompleted, nextStepActivated, nextStepId, processCompleted }

[OUTSIDE TX] SELECT stepInstance (stepOrder, stepName, completedDate)   ← for event description
[OUTSIDE TX] SELECT processInstance (id via currentStepInstanceId)      ← for processInstanceId
[OUTSIDE TX] SELECT stepInstance (full re-read)                         ← redundant
[OUTSIDE TX] SELECT processInstance (full re-read)                      ← for workflowTemplateId
[OUTSIDE TX] SELECT workflowStepTemplate (stepType)                     ← for event type detection
[OUTSIDE TX] SELECT stepInstance (next step name)                       ← for STEP_ACTIVATED event
```

That's **5-6 extra queries** for event logging context. The engine function already read the step, process, and template data inside the transaction — it just didn't return them.

The decline path in `complete.ts` already demonstrates the correct pattern: the transaction returns `{ processId, stepName, taskId }` directly, and the event logging uses those values with zero post-tx reads.

**Impact:** 15-60ms of unnecessary latency per mutation request. At 100 concurrent users, this is 100+ wasted queries per second across the system.

### Architecture Analysis Findings Addressed

| ID | Finding | Severity | Category |
|----|---------|----------|----------|
| **P1** | Event logging causes excessive post-transaction DB reads: 3-8 extra queries per mutation to populate event log fields that were available inside the transaction | **Medium** | Performance |

## Acceptance Criteria

### Scope Rule: Every New Field Must Have a Concrete Consumer

Every new return field added to an engine function in this story must be directly consumed by at least one concrete use in the rewritten `complete.ts` submit path, `complete.ts` approve path, or `submit.ts`. If a field is not actually referenced in the final route code after the rewrites in this story, it must not be added. This story must not add "future-useful" or convenience fields for later stories. The purpose is to eliminate the known post-transaction re-reads listed in the Context section — not to broadly expand engine return surfaces.

### AC 1: `completeStep` Returns Event Logging Context

1. The `CompleteStepResult.data` object is expanded with the following fields:
   - `processInstanceId: string` — the process this step belongs to
   - `stepOrder: number` — the step's order in the workflow
   - `stepName: string` — the step's display name
   - `stepType: string` — the step type (e.g., `"form"`, `"document"`, `"approval"`)
   - `wasResubmission: boolean` — `true` if the step had a non-null `completedDate` before this completion (indicates a resubmission after a previous decline)
   - `nextStepName: string | null` — the name of the activated next step, or `null` if no next step (process completed or awaiting validation)
2. All new fields are populated from data read inside the transaction — no new post-transaction reads.
3. Existing fields (`stepCompleted`, `nextStepActivated`, `processCompleted`, `nextStepId`, `awaitingValidation`) remain unchanged.
4. The `wasResubmission` field is determined by reading the step row before the CAS UPDATE.

### AC 2: `approveValidationTask` Returns Event Logging Context

5. The `ApproveValidationTaskResult` is expanded with:
   - `processInstanceId: string` — the process this step belongs to
   - `stepOrder: number` — the step's order in the workflow
   - `stepName: string` — the step's display name
   - `nextStepName: string | null` — the name of the activated next step (when `allValidationsComplete` and `nextStepActivated`)
6. All new fields are populated from data already read inside the engine function — the step and process are already queried by `approveValidationTask`.

### AC 3: `transitionToNextStep` Returns `nextStepName`

7. The `transitionToNextStep` result object is expanded with `nextStepName: string | null`.
8. When a next step is activated, `nextStepName` is populated from the `stepInstance.stepName` of the activated step (already available — it's the step being activated).
9. When the process completes (no next step), `nextStepName` is `null`.
10. Both `completeStep` and `approveValidationTask` propagate this value from `transitionToNextStep` into their own result types.

### AC 4: `complete.ts` Submit Path Uses Returned Context

11. The submit path in `complete.ts` uses `result.data.processInstanceId`, `result.data.stepOrder`, `result.data.stepName`, `result.data.stepType`, `result.data.wasResubmission`, and `result.data.nextStepName` for event logging.
12. All post-transaction `SELECT` queries for event logging context are removed (currently lines 71-120 and 152-155).
13. The event logging behavior (which events are logged, their descriptions, their types) is preserved — same `WorkflowEventType` choices, same description format.
14. Estimated savings: **5-6 queries eliminated** per submit request.

### AC 5: `complete.ts` Approve Path Uses Returned Context

15. The approve path in `complete.ts` uses `validationResult.processInstanceId`, `validationResult.stepOrder`, `validationResult.stepName`, and `validationResult.nextStepName` for event logging.
16. All post-transaction `SELECT` queries for event logging context are removed (currently lines 224-234 and 267-270).
17. The event logging behavior is preserved.
18. Estimated savings: **2-3 queries eliminated** per approve request.

### AC 6: `submit.ts` Uses Returned Context (Post-WFH-002)

19. The form submission route (`submit.ts`) uses `completeStep` result context for event logging instead of its pre/post-transaction reads.
20. The `stepInstance` reads at current lines 159-162 and 197-200 are eliminated.
21. The event logging behavior is preserved.
22. Estimated savings: **2 queries eliminated** per form submit request.

### AC 7: Existing Behavior Preserved

23. The API response shapes for all affected routes are unchanged.
24. The event logging content (event types, descriptions, metadata) is preserved.
25. All existing tests pass.
26. The `complete.ts` decline path is NOT modified — it already uses the correct pattern.

### AC 8: Tests Updated

27. Existing engine tests are updated to verify the new fields are present in the result types.
28. At least one test per engine function verifies the new context fields are correctly populated.

### Data Flow: Before and After

**Before (current):**
```
Engine function → minimal result → route re-reads DB → event logging
                                   ↑ 5-8 queries
```

**After (this story):**
```
Engine function → rich result → route uses result → event logging
                  ↑ 0-1 extra reads (inside tx)     ↑ 0 post-tx reads
```

### The Pre-CAS SELECT in `completeStep`

The current `completeStep` uses a single-shot CAS: `UPDATE ... WHERE status = 'active' RETURNING`. This returns the post-update row, so `completedDate` is always the new value (can't determine resubmission). The fix adds a SELECT before the CAS to capture the pre-mutation state.

**Net query impact:** +1 SELECT inside the transaction (for `completeStep` only). Eliminates 5-8 SELECTs outside the transaction per request. **Net savings: 4-7 queries per request.**

The SELECT + CAS UPDATE pattern is safe: the CAS still protects against concurrent updates. If another request completes the step between the SELECT and UPDATE, the CAS `WHERE status = 'active'` fails (0 rows returned) and the function returns `{ success: false }`.

### Implementation Guardrails

1. **Do not change event logging behavior.** Same events, same descriptions, same types. Only the source of the data changes (engine result vs. post-tx read).
2. **Do not remove existing result fields.** All changes are additive.
3. **Do not modify the decline path in `complete.ts`.** It already works correctly.
4. **Do not modify `review.ts`.** WFH-001 already handles its event logging context.
5. **Every new field must have a concrete consumer in this story's route rewrites.** If a field is not directly referenced by a `logWorkflowEvent` call or response-formatting line in the rewritten `complete.ts` (submit/approve) or `submit.ts`, it must not be added. Do not add fields for future convenience, anticipated needs, or other stories.
6. **The pre-CAS SELECT must still be followed by the CAS UPDATE.** The SELECT does not replace the CAS — it supplements it with pre-mutation context.
7. **All new reads must be inside the transaction.** No new post-transaction reads.
