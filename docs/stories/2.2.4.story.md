## Story

**As a** supplier_user or internal_user,  
**I want** to fill in forms progressively and save my progress,  
**so that** I can complete long questionnaires over time.

---

## Acceptance Criteria

1. Forms render dynamically based on form_template_version
2. Users can save_draft without completing required fields
3. Draft submissions remain editable
4. Users can submit the form only when required fields are completed
5. After submit, form answers become read-only
6. Form answers are stored in form_submission and form_answer
7. Access to submissions is restricted by tenant_id and process context

### Data Models

#### Form Submission

**Purpose**: Represents a runtime execution of a form template version. Tracks submission status and links to workflow processes.

**Source**: [Epic 2.2, AC: 6] - "Form answers are stored in form_submission and form_answer"

**Key Attributes**:
- `id` (UUID PK): Unique submission identifier
- `tenant_id` (UUID FK → tenants): Tenant isolation (CASCADE delete)
- `form_template_version_id` (UUID FK → form_template_version): Immutable reference to published form version
- `process_instance_id` (UUID, nullable): Link to workflow process (for workflow integration in Story 2.2.9)
- `submitted_by` (UUID FK → users): User who created/submitted the form
- `status` (enum): 'draft' | 'submitted' | 'archived'
- `submitted_at` (timestamp, nullable): When form was finalized (NULL for drafts)
- `created_at`, `updated_at`, `deleted_at`: Audit timestamps

**Business Rules**:
- UNIQUE constraint on (form_template_version_id, process_instance_id) when process_instance_id IS NOT NULL - prevents duplicate submissions for same workflow step
- Draft submissions can be edited, submitted submissions are immutable (AC: 3, 5)
- Tenant isolation enforced: all queries MUST filter by tenant_id (AC: 7)

**Relationships**:
- Many-to-One with Tenant
- Many-to-One with FormTemplateVersion
- Many-to-One with User (submitted_by)
- One-to-Many with FormAnswer

#### Form Answer

**Purpose**: Stores individual field answers for a form submission. All answers stored as TEXT, parsed based on field_type.

**Source**: [Epic 2.2, AC: 6]

**Key Attributes**:
- `id` (UUID PK): Unique answer identifier
- `form_submission_id` (UUID FK → form_submission): Parent submission (CASCADE delete)
- `form_field_id` (UUID FK → form_field): Reference to field definition
- `tenant_id` (UUID FK → tenants): Tenant isolation
- `answer_value` (TEXT): Answer stored as text, interpretation based on field_type:
  - text/textarea: plain text
  - number: numeric string (e.g., "42.5")
  - date: ISO date string (e.g., "2026-01-22")
  - dropdown: selected value string (e.g., "iso9001")
  - checkbox: "true" or "false"
  - multi_select: comma-separated values (e.g., "rohs,reach,ce")
- `created_at`, `updated_at`: Audit timestamps

**Business Rules**:
- UNIQUE constraint on (form_submission_id, form_field_id) - one answer per field per submission
- Answers are upserted when saving drafts (insert if new, update if exists)
- Tenant isolation: answer.tenant_id MUST match submission.tenant_id and field.tenant_id

**Answer Format Validation** (by field_type):
- `number`: Must parse as valid number, respect validation_rules.min/max
- `date`: Must be valid ISO 8601 date string
- `dropdown`: answer_value MUST match one of field.options.choices[].value
- `multi_select`: Each comma-separated value MUST match one of the valid choices

### Component Specifications

**Form Renderer Architecture**:

**Component**: FormRenderer.tsx  
**Location**: `apps/web/app/components/form-runtime/FormRenderer.tsx`  
**Source**: [Story AC: 1] - "Forms render dynamically based on form_template_version"

**Props**:
```typescript
interface FormRendererProps {
  formVersion: FormTemplateVersionWithStructure; // includes sections + fields
  initialAnswers: FormAnswer[]; // pre-populated answers (for editing drafts)
  mode: 'edit' | 'view'; // 'view' for submitted forms (read-only)
  onSave: (answers: Map<string, string>) => Promise<void>; // callback for save draft
  onSubmit: (answers: Map<string, string>) => Promise<void>; // callback for submit
}
```

**Rendering Logic**:
1. Map `formVersion.sections` (sorted by section_order) to Section components
2. For each section, map `section.fields` (sorted by field_order) to field components
3. Dynamically select field component based on `field.field_type`:
   - 'text' → TextField
   - 'textarea' → TextareaField
   - 'number' → NumberField
   - 'date' → DateField
   - 'dropdown' → DropdownField
   - 'checkbox' → CheckboxField
   - 'multi_select' → MultiSelectField
4. Pass field metadata (label, placeholder, required, validation_rules) to each field component
5. Maintain form state as `Map<fieldId, answerValue>`
6. Apply validation rules on blur or submit

**Field Component Pattern** [Source: docs/architecture/components.md]:
- All field components use shadcn/ui primitives (Input, Textarea, Select, Checkbox, DatePicker)
- Consistent prop interface across all field types:
  ```typescript
  interface FieldProps {
    field: FormField; // includes label, placeholder, required, validation_rules
    value: string; // current answer value
    onChange: (value: string) => void; // update callback
    error: string | null; // validation error message
    disabled: boolean; // read-only mode
  }
  ```

**Validation Strategy**:
- **Client-side**: Validate field formats on blur (regex patterns, min/max, minLength/maxLength)
- **Client-side**: Validate required fields on submit (before API call)
- **Server-side**: Re-validate all data in API endpoints (never trust client)

**State Management**:
- Use React Hook Form for form state management [Source: docs/architecture/tech-stack.md]
- Custom hook `useFormSubmission` manages API calls and state synchronization

### Remix Data Loading Pattern [Source: docs/architecture/remix-patterns.md]:
- ALL data fetching MUST be in loaders (server-side), never in useEffect

- Pass data to components via props, not by fetching in components
- Use `useRevalidator()` for mutations, never manual state updates
- Example:
  ```typescript
  // Loader fetches all data
  export async function loader({ params, request }: LoaderFunctionArgs) {
    const { session } = await requireAuth({ request });
    const client = createEdenTreatyClient(session.access_token);
    
    const [submission, formVersion] = await Promise.all([
      client.api["form-submissions"][params.submissionId].get(),
      // ... fetch form structure
    ]);
    
    return json({ submission, formVersion, token: session.access_token });
  }
  
  // Component uses loader data via props
  export default function FormExecutionPage() {
    const { submission, formVersion, token } = useLoaderData<typeof loader>();
    return <FormRenderer formVersion={formVersion} ... />;
  }
  ```

### Important Implementation Notes

1. **Tenant Isolation is Critical**: EVERY database query MUST filter by `tenant_id`. This is a security requirement. Verify in code review.

2. **Immutability Enforcement**: Once a form is submitted (status = 'submitted'), the API MUST reject any edit attempts. This is enforced at application level, not database level.

3. **Answer Storage Pattern**: All answers stored as TEXT in `answer_value` column. Interpretation/parsing based on `field_type` at runtime. This provides flexibility for future field types without schema changes.

4. **Draft Upsert Logic**: When saving drafts, use INSERT ... ON CONFLICT DO UPDATE pattern to handle both new answers and updates to existing answers in single operation.

5. **Required Field Validation**: Only enforced on final submit, NOT on save draft. Client-side validation should disable Submit button if required fields empty, but server-side must also validate.

6. **Process Instance Integration**: The `process_instance_id` column is nullable and prepared for future workflow integration (Story 2.2.9). For now, it will be NULL for standalone forms.

7. **Multi-Select Answer Format**: Store as comma-separated string (e.g., "rohs,reach,ce"). Split on comma for display, join for storage. Validate each value against options.choices.

8. **Field Type Rendering**: Use a switch/case or object map pattern to dynamically select field component. This keeps FormRenderer clean and extensible.

9. **Error Handling UX**: When validation fails, show inline errors below each field AND a summary toast. Include field labels in error messages for clarity (e.g., "Email Address is required" not "field_123 is required").

10. **Read-Only Mode**: Submitted forms should render in read-only mode with all inputs disabled and action buttons hidden. Consider showing formatted values (e.g., date formatted as "January 22, 2026" instead of raw ISO string).

11. **Concurrent Edit Handling**: Use last-write-wins strategy for draft saves. When user saves draft, the API performs upsert operation which overwrites previous values. No optimistic locking is implemented for MVP (acceptable risk for single-user draft editing). If user has same draft open in multiple browser tabs, last save wins. Consider adding "Draft last saved at [timestamp]" indicator in UI to show freshness of data.

12. **Responsive Design Strategy**: Forms use single-column layout across all breakpoints for simplicity and mobile-first approach. Desktop uses constrained width (max-w-3xl) for better readability. Action buttons stack vertically on mobile, inline on desktop. All shadcn/ui components are responsive by default.

13. **Accessibility Requirements**: All form fields must have proper ARIA labels from field.label. Required fields marked with aria-required="true". Validation errors linked with aria-describedby. Ensure keyboard navigation works: Tab through fields, Enter to submit. Test with screen reader to verify field labels and error messages are announced correctly.
