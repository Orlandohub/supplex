## Story

**As a** developer,  
**I want** a flexible, tenant-isolated workflow template model with action modes for forms and documents,  
**so that** each tenant can design approval and validation loops correctly.

---

## Acceptance Criteria

1. A workflow_template entity exists with tenant_id and process_type
2. A workflow_template_version entity exists with tenant_id and versioning
3. A workflow_step_template entity exists with:
   - step_order
   - name
   - task_title
   - task_description
   - due_days (nullable)
4. A step can include an associated form with:
   - form_template_version_id
   - form_action_mode with values fill_out or validate
5. A step can include associated document requirements with:
   - document_template_id
   - document_action_mode with values upload or validate
6. A step supports multi_approver = true
7. When multi_approver = true:
   - The step defines approver_count
   - Each approver definition specifies either:
     - a role within the tenant, or
     - a specific user within the tenant
8. A step defines a decline return behavior:
   - decline_returns_to_step_offset = 1 (returns to the immediately previous step)

**Key Changes from Story 2.2.5.1:**
- Task templates were **removed** - tasks are no longer reusable library items
- Tasks are now configured at the **workflow step level** (this story defines that configuration)

- Task configuration fields on workflow_step_template:
  - `task_title` → copied to `task_instance.title`
  - `task_description` → copied to `task_instance.description`
  - `due_days` → used to calculate `task_instance.due_at`
  - `assignee_type` → copied to `task_instance.assignee_type` ('role' or 'user')
  - `assignee_role` → copied to `task_instance.assignee_role`
  - `assignee_user_id` → copied to `task_instance.assignee_user_id`

[Source: Story 2.2.1 - Database Refactor and Decommission of Legacy Qualification Workflow Tables]

**Workflow Engine Foundation:**
- `process_instance` table tracks workflow execution
- `step_instance` table tracks individual workflow steps
- Domain-agnostic design supports any process type
- Tenant isolation enforced via `tenant_id` on all tables
- CASCADE delete for tenant hierarchy
- RESTRICT delete for audit trail preservation

[Source: Story 2.2.2 - Form Template Data Model and Versioning]

**Versioning Pattern Established:**
- Template → Version → Sections → Fields hierarchy
- Draft → Published → Archived status transitions
- Published versions are immutable (enforced at application level)
- Version numbers are sequential integers (1, 2, 3, ...)
- Unique constraint on (template_id, version)
- `is_published` boolean flag with CHECK constraint for consistency

**Workflow Template Hierarchy:**

```
Tenant
└── Workflow Template (e.g., "Supplier Qualification v2")
    ├── Version 1 (draft)
    ├── Version 2 (published) ← Active version
    │   ├── Step 1: Fill Out Supplier Profile (form, fill_out)
    │   │   ├── Task: "Complete supplier information"
    │   │   └── Form: supplier_profile_form_v3
    │   ├── Step 2: Review Supplier Profile (approval, validate)
    │   │   ├── Task: "Review supplier submission"
    │   │   ├── Form: supplier_profile_form_v3 (read-only)
    │   │   └── Approvers:
    │   │       ├── Approver 1: Role = procurement_manager
    │   │       └── Approver 2: Role = quality_manager
    │   ├── Step 3: Upload Certifications (document, upload)
    │   │   └── Task: "Upload ISO 9001, ISO 14001"
    │   └── Step 4: Validate Certifications (document, validate)
    │       ├── Task: "Review certification documents"
    │       └── Approver: Role = quality_manager
    └── Version 3 (draft) ← Work in progress
```

**Form Action Modes:**

1. **fill_out** - User fills in form and submits
   - Form is editable
   - User can save draft
   - Submit requires all required fields
   - After submit, workflow proceeds to next step
   - Use case: Initial supplier profile creation, questionnaire completion

2. **validate** - User reviews existing form submission and approves/declines
   - Form is read-only
   - UI shows "Approve" and "Decline" buttons
   - On decline: Comment required, workflow returns to previous step
   - On approve: Workflow proceeds to next step
   - Use case: Manager review, quality approval, document validation

**Document Action Modes:**

1. **upload** - User uploads required documents
   - UI shows document requirements (from document_template_id)
   - User can upload multiple files
   - Submit requires all required documents
   - After submit, workflow proceeds to next step
   - Use case: Supplier uploads certifications, financial statements

2. **validate** - User reviews uploaded documents and approves/declines
   - Documents displayed in read-only mode
   - UI shows "Approve" and "Decline" buttons
   - On decline: Comment required, workflow returns to previous step
   - On approve: Workflow proceeds to next step
   - Use case: Quality manager reviews ISO certifications

**Multi-Approver Configuration:**

When `multi_approver = true`:
- `approver_count` specifies how many approvals required (e.g., 2 out of 3)
- `step_approver` records define who can approve:
  - Role-based: All users with that role can approve (any one user action counts as one approval)
  - User-specific: Only that specific user can approve
- Workflow engine creates multiple `task_instance` records (one per approver)
- Step completes when `approver_count` tasks are marked completed
- Example: Step requires 2 approvals from procurement_manager role and 1 approval from quality_manager role

**Decline Behavior:**

`decline_returns_to_step_offset` defines where workflow returns on decline:
- `1` (default): Returns to immediately previous step
- `2`: Returns to 2 steps back
- Example: Step 4 declines → returns to Step 3 (offset = 1) or Step 2 (offset = 2)
- The returned-to step becomes 'active' again
- User at returned step can modify form/document and resubmit
- Workflow proceeds forward again after resubmission

**Tenant Isolation Strategy:**
- All tables include `tenant_id` with CASCADE delete when tenant is removed
- All queries must filter by `tenant_id` (enforced in application layer)
- Indexes include `tenant_id` as first column for multi-tenant query performance

**Foreign Key Cascade Rules:**
- `tenant_id` → CASCADE: When tenant deleted, all workflow templates and related records deleted
- `workflow_template_id` → CASCADE: When template deleted, all versions deleted
- `workflow_template_version_id` → CASCADE: When version deleted, all steps and approvers deleted
- `workflow_step_template_id` → CASCADE: When step deleted, all approvers deleted
- `created_by`, `assignee_user_id`, `approver_user_id` → RESTRICT: Cannot delete users referenced in workflow templates (audit trail preservation)
- `form_template_version_id` → RESTRICT: Cannot delete form version still referenced by workflow steps (data integrity)

**Versioning and Immutability:**
- Draft versions: Mutable, can be edited or deleted
- Published versions: Immutable (enforced at application level), cannot be edited
- Archived versions: Immutable, no longer selectable for new processes
- Unique constraint on (workflow_template_id, version) prevents duplicate version numbers
- CHECK constraint ensures `is_published = true` only when `status = 'published'`
