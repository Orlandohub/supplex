## Story

**As a** user of the workflow system,
**I want** the workflow list page, detail page, and engine operations to respond faster under load,
**so that** the system remains responsive as workflow volume grows to hundreds or thousands of active processes.

---

## Context

The architecture assessment (`.cursor/plans/architecture_assessment_report_5c337427.plan.md`, Section 5) profiled the main workflow hot paths and found significant optimization opportunities:

| Hot Path | Current | Target |
|----------|---------|--------|
| Process detail page | 7-8 **sequential** queries | 3-4 **parallel** queries via `Promise.all` |
| Process list page search | `ILIKE '%...%'` — full table scan | Trigram GIN index — index-assisted |
| Step count subqueries (list) | Correlated per-row COUNT(*) | Denormalized columns on `process_instance` |
| Supplier status update (process complete) | 5 sequential lookups | 1 joined query |
| `resolveSupplierUser` | 2-4 sequential queries | 1 joined query |
| Validation task creation | Loop of individual INSERTs | Single batch INSERT |

Story 2.2.19 already handles **transaction wrapping** (which eliminates connection overhead between statements within a transaction) and **batch step inserts** in instantiation. This story covers the remaining performance items.

### Assessment Items Addressed

| ID | Finding |
|----|---------|
| P1 | List page correlated subqueries — denormalize progress counters |
| P4 | Detail page 7-8 sequential queries — parallelize with `Promise.all` |
| P5 | ILIKE search without trigram indexes |
| P6 | Missing `step_instance(process_instance_id, status)` partial index |
| 5.3 | `resolveSupplierUser` 2-4 queries — consolidate to single join |
| 5.3 | Supplier status update chain — 5 lookups to 1 joined query |
| 5.3 | Validation tasks inserted in loop — batch insert |

### Dependencies

- **2.2.19** (Transactions): Engine functions already accept `tx`. Denormalized counter updates should occur inside the same transaction as step transitions.

### Out of Scope

- **`has_overdue_tasks` maintenance logic.** The column is added in the migration with `DEFAULT false` as a schema placeholder. The update mechanism (cron job, trigger, or event-driven) is deferred to a future story.
- **Denormalization beyond `total_steps` and `completed_steps`.** This story introduces exactly two actively maintained denormalized counters. Do not add additional denormalized columns, computed fields, or materialized aggregates beyond what is specified here.
- **Full list-route subquery elimination.** The list route has additional correlated subqueries (e.g., latest comment, active step name) beyond the step-count subqueries targeted here. Those can be addressed in a follow-up performance story.

## Acceptance Criteria

### AC 1: Parallelize Process Detail Queries

1. In `routes/workflows/processes/get.ts`, the independent queries (steps, tasks, step templates, comments, form submissions, workflow step documents, supplier name) are executed concurrently using `Promise.all` instead of sequentially.
2. Only the initial process query runs first (its result is needed for the others). The remaining 6-7 queries run in parallel.
3. The API response shape is unchanged.

### AC 2: Denormalized Progress Counters on `process_instance`

4. A new migration adds three columns to `process_instance`:
   - `total_steps INTEGER NOT NULL DEFAULT 0`
   - `completed_steps INTEGER NOT NULL DEFAULT 0`
   - `has_overdue_tasks BOOLEAN NOT NULL DEFAULT false`
5. A backfill query in the migration populates these columns for all existing processes from the current step/task data.
6. `instantiateWorkflow()` sets `total_steps` on the process when creating it (equal to `stepTemplates.length`).
7. `transitionToNextStep()` increments `completed_steps` by 1 (within the transaction) when transitioning. On process completion, `completed_steps` equals `total_steps`.
8. The process list route (`processes/list.ts`) reads `total_steps` and `completed_steps` directly from the `process_instance` row instead of the correlated `totalStepCount`/`completedStepCount` subqueries.
9. The step-count subqueries (`totalStepCountSq`, `completedStepCountSq`) are removed from the list query.

### AC 3: Trigram GIN Indexes for ILIKE Search

10. A new migration enables the `pg_trgm` extension and creates GIN trigram indexes:
    - `CREATE INDEX idx_process_instance_workflow_name_trgm ON process_instance USING GIN (workflow_name gin_trgm_ops);`
    - `CREATE INDEX idx_suppliers_name_trgm ON suppliers USING GIN (name gin_trgm_ops);`
11. No application code changes are needed — PostgreSQL automatically uses the GIN index for `ILIKE '%...%'` queries.

### AC 4: Missing Partial Indexes

12. A migration adds `CREATE INDEX idx_step_instance_process_status ON step_instance (process_instance_id, status) WHERE deleted_at IS NULL;` — used by list page step-count subqueries and the transition code.
13. The Drizzle schema file `step-instance.ts` is updated to declare this new index.

### AC 5: Optimize Supplier Status Update Chain

14. In `transitionToNextStep()`, the 5-query supplier-status-update chain on process completion (lines 169-213: query process → query workflowTemplate → query workflowType → query supplierStatus → update supplier) is replaced with a single joined query:
    ```sql
    SELECT ss.name, ss.id
    FROM process_instance pi
    JOIN workflow_template wt ON wt.id = pi.workflow_template_id
    JOIN workflow_type wty ON wty.id = wt.workflow_type_id
    JOIN supplier_status ss ON ss.id = wty.supplier_status_id
    WHERE pi.id = :processId AND pi.entity_type = 'supplier'
    ```
    followed by a single `UPDATE suppliers SET status = ..., supplier_status_id = ...`.
15. If any join returns null (no workflow type, no status mapping), the update is skipped (same behavior as current code, just one query instead of five).

### AC 6: Optimize `resolveSupplierUser`

16. The `resolveSupplierUser()` function in `create-tasks-for-step.ts` is refactored from 2-4 sequential queries to a single joined query that resolves the assignment in one round trip:
    ```sql
    SELECT
      COALESCE(su.id, pm.id, pi.initiated_by) as assignee_id,
      CASE WHEN su.id IS NOT NULL THEN 'user' ELSE ... END as assignee_type
    FROM process_instance pi
    LEFT JOIN suppliers s ON s.id = pi.entity_id AND s.tenant_id = pi.tenant_id
    LEFT JOIN users su ON su.id = s.supplier_user_id AND su.is_active = true AND su.role = 'supplier_user'
    LEFT JOIN users pm ON pm.tenant_id = pi.tenant_id AND pm.role = 'procurement_manager' AND pm.is_active = true
    WHERE pi.id = :processId AND pi.tenant_id = :tenantId
    LIMIT 1
    ```
17. The fallback hierarchy (supplier user → procurement manager → initiator) is preserved.

### AC 7: Batch Validation Task Inserts

18. `createValidationTasks()` inserts all validation tasks in a single batch `INSERT ... VALUES (...), (...), ...` instead of one insert per approver role in a loop.
19. This is a straightforward change since the `for` loop already builds one row per role.

### AC 8: Backward Compatibility

20. All API responses remain unchanged in shape and content.
21. All existing tests pass after the optimizations.
22. The denormalized counters are kept in sync by the engine — they are not directly exposed as user-editable fields.

### Process Detail Page — Current vs Target Query Flow

**Current (sequential, ~7-8 round trips):**
```
1. SELECT process               → process
2. SELECT steps                 → steps (depends on processId)
3. SELECT tasks                 → tasks (depends on processId)
4. SELECT step_templates        → templates (depends on workflowTemplateId)
5. SELECT comments              → comments (depends on processId)
6. SELECT form_submissions      → submissions (depends on processId)
7. SELECT workflow_step_documents → documents (depends on processId)
8. SELECT supplier name         → name (depends on entityId)
```

**Target (parallel after first query, ~2 round trips):**
```
1. SELECT process               → process
2. Promise.all([
     SELECT steps,              ─┐
     SELECT tasks,              │
     SELECT step_templates,     │ All depend only on processId
     SELECT comments,           │ or workflowTemplateId (from #1)
     SELECT form_submissions,   │
     SELECT step_documents,     │
     SELECT supplier name       ─┘
   ])
```

### Denormalized Counter Update Locations

| Operation | `total_steps` | `completed_steps` |
|-----------|:---:|:---:|
| `instantiateWorkflow()` | SET = stepTemplates.length | — (stays 0) |
| `transitionToNextStep()` — next step activated | — | INCREMENT +1 |
| `transitionToNextStep()` — process complete | — | SET = total_steps |

> **Why no decrement on decline?** The `completedSteps` counter is only incremented in `transitionToNextStep()`, which runs only after all validators approve (or if no validation is needed). A declined step was in `awaiting_validation` — transition was never called, so the counter was never incremented. No decrement is needed.

The `has_overdue_tasks` column can be updated by a periodic cron job or on task due-at changes. For this story, it's included in the migration as a placeholder with `DEFAULT false`. A future story can implement the update mechanism.

### Trigram Index Notes

PostgreSQL's `pg_trgm` extension enables GIN indexes that support `ILIKE '%pattern%'` queries efficiently. The extension must be enabled with `CREATE EXTENSION IF NOT EXISTS pg_trgm` (requires superuser or `pg_trgm` in `shared_preload_libraries`). On Supabase, `pg_trgm` is available by default.

After creating the GIN index, `EXPLAIN ANALYZE` should show a "Bitmap Index Scan" on the trigram index instead of a sequential scan for `ILIKE` queries.

## Implementation Guardrails

1. **API response shapes must not change.** Parallelization and denormalization are internal optimizations — every response must be byte-identical in structure and content to the pre-optimization version.
2. **Business logic must not change.** Workflow transitions, validation flows, supplier status updates, and task creation must behave identically. Only the internal query strategy changes.
3. **Denormalized counters are limited to `total_steps` and `completed_steps` only.** Do not introduce additional denormalized columns, computed fields, or materialized aggregates beyond these two counters. All counter updates must happen inside the same database transaction as the workflow mutation they reflect — never outside the transaction boundary. Stale or mismatched counters are worse than the subqueries they replace.
4. **`has_overdue_tasks` is a schema placeholder only.** The column is added in the migration but must not gain any maintenance logic, triggers, or update code in this story.
5. **If correct counter maintenance would require broader workflow behavior changes, surface it as a blocker.** Do not improvise workflow logic changes to make counters work. If an edge case is discovered where maintaining accurate counters requires altering step transition semantics, validation flows, or other engine behavior, stop and flag it — do not silently expand scope.
6. **`Promise.all` must only wrap queries that are truly independent.** Every query inside the `Promise.all` must depend only on data already available from the initial process fetch (e.g., `processInstanceId`, `workflowTemplateId`, `entityId`). No query may be parallelized if its correctness depends on the result of another query running in the same `Promise.all`. Response semantics and error behavior (e.g., which errors surface to the client and in what shape) must remain unchanged.
7. **Tenant isolation must be preserved in all new/rewritten queries.** Every query must include the `tenant_id` filter.
8. **RBAC and permission checks are untouched.** This story does not alter who can access what.
9. **Do not refactor or rename functions, files, or routes** beyond what is strictly necessary for the optimization. Keep diffs minimal.
10. **Migration must be idempotent-safe.** Use `IF NOT EXISTS` for extensions and indexes where PostgreSQL supports it.
