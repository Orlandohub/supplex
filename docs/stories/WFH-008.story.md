## Story

**As a** platform developer,
**I want** `instantiateWorkflow` to use the provided `outerDb` parameter instead of ignoring it,
**so that** callers can compose workflow instantiation into a larger transaction (e.g., instantiate workflow + create supplier atomically).

---

## Context

`instantiateWorkflow` is the engine function that creates a new workflow process instance from a published template. It creates the process, all step instances, tasks for the first step, and (for document steps) seed documents — all atomically within a transaction.

The function signature accepts `outerDb: DbOrTx` as its first parameter (line 44-45), but line 60 always calls `db.transaction()` using the **module-level `db` import** (line 12), completely ignoring the `outerDb` parameter:

```typescript
import { db, ... } from "@supplex/db";      // line 12 — module-level db

export async function instantiateWorkflow(
  outerDb: DbOrTx,                           // line 45 — accepted but IGNORED
  params: InstantiateWorkflowParams,
  log?: Logger
): Promise<InstantiateWorkflowResult> {
  // ...
  return await db.transaction(async (tx) => { // line 60 — uses module-level db, NOT outerDb
    // ...
  });
}
```

This means `outerDb` is dead code. Callers cannot compose `instantiateWorkflow` into a larger transaction — even if they pass a transaction handle, the function opens a separate, independent transaction on the module-level `db`.

Every other engine function in the codebase (`completeStep`, `approveValidationTask`, `transitionToNextStep`, `createTasksForStep`, `seedStepDocuments`) correctly uses its `tx: DbOrTx` parameter. `instantiateWorkflow` is the only outlier.

**Current callers (all pass `db`):**
- `instantiate.ts` route (line 47): `instantiateWorkflow(db, { ... })`
- All tests in `instantiate.test.ts` and `integration.test.ts`: `instantiateWorkflow(db, { ... })`

No current caller passes a transaction handle, so the bug has been invisible. It becomes a problem when a future feature needs to compose instantiation with another operation atomically.

  ```typescript
  test("composes into caller transaction (rollback proves shared tx)", async () => {
    const SENTINEL = "ROLLBACK_SENTINEL";
    let createdProcessId: string | undefined;

    try {
      await db.transaction(async (tx) => {
        const result = await instantiateWorkflow(tx, {
          tenantId,
          workflowTemplateId,
          entityType: "supplier",
          entityId: crypto.randomUUID(),
          initiatedBy: userId,
        });

        expect(result.success).toBe(true);
        createdProcessId = result.data!.processInstance.id;

        // Process exists inside the transaction
        const [proc] = await tx
          .select()
          .from(processInstance)
          .where(eq(processInstance.id, createdProcessId));
        expect(proc).toBeDefined();

        // Throw sentinel to force the outer transaction to roll back
        throw new Error(SENTINEL);
      });
    } catch (e: any) {
      // Verify it was our deliberate sentinel, not an unexpected error
      expect(e.message).toBe(SENTINEL);
    }

    // After rollback, nothing was persisted — proves instantiateWorkflow
    // wrote within the caller's transaction, not a separate one
    expect(createdProcessId).toBeDefined();
    const orphans = await db
      .select()
      .from(processInstance)
      .where(eq(processInstance.id, createdProcessId!));
    expect(orphans.length).toBe(0);
  });
  ```

  **Pattern rationale:** The sentinel-throw pattern is the preferred approach because it is unambiguous — the outer `db.transaction()` rolls back on any throw, and the sentinel string proves the throw was deliberate, not an unexpected failure. This is the clearest way to prove that `instantiateWorkflow(tx, ...)` participates in the caller's transaction scope. If the codebase/test environment is already known to support `tx.rollback()` cleanly, that is an acceptable equivalent, but the sentinel-throw is the default recommendation.

## Dev Notes

### Why `outerDb.transaction()` Instead of Using `outerDb` Directly

Every other engine function (`completeStep`, `transitionToNextStep`, etc.) uses its `tx` parameter directly without calling `.transaction()`. The caller wraps in a transaction. One might expect the same pattern here.

However, `instantiateWorkflow` has a specific need: **the function must be safely callable with plain `db`** (backward compatibility — all current callers pass `db`). If we removed the `.transaction()` wrapper and used `outerDb` directly, callers passing `db` would lose atomicity — a failure midway through would leave orphaned process instances.

The `outerDb.transaction()` approach satisfies both requirements:
- **Caller passes `db`:** Opens a new transaction (identical to current behavior).
- **Caller passes `tx`:** Drizzle creates a PostgreSQL `SAVEPOINT` within the caller's transaction. The operations are within the caller's transaction scope — not a separate, independent transaction. If the caller rolls back, everything (including `instantiateWorkflow`'s writes) is rolled back.

### Savepoint Behavior When Called With `tx`

In PostgreSQL, `SAVEPOINT` is a lightweight mechanism for nested transaction control. When `outerDb.transaction()` is called on a transaction handle:
- Drizzle issues `SAVEPOINT sp_<n>`
- All writes happen within the caller's transaction
- If the inner code throws, Drizzle issues `ROLLBACK TO SAVEPOINT sp_<n>` — only the inner work is undone
- If the inner code succeeds, Drizzle issues `RELEASE SAVEPOINT sp_<n>` — the work is committed within the caller's transaction

The savepoint adds negligible overhead (microseconds) and is the standard Drizzle pattern for composable transactions.

### The `db` Import Removal

After the fix, the module-level `db` import from `@supplex/db` is unused in `instantiate-workflow.ts`. The `type DbOrTx` import (line 10) and the schema table imports (`workflowTemplate`, `workflowStepTemplate`, `processInstance`, `stepInstance`) are still needed. Only the `db` value import is removed.

### Rollback Behavior Preserved

The existing "no steps" rollback test (`instantiate.test.ts`, lines 189-220) verifies that when `instantiateWorkflow` throws `"Workflow template has no steps"` inside the transaction, the process instance insert is rolled back and no orphans remain. This behavior is preserved:
- Current: `db.transaction()` → throw → transaction rolls back → catch returns `{ success: false }`
- After fix: `outerDb.transaction()` (where `outerDb` is `db`) → same behavior
