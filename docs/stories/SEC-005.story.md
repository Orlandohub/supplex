## Story

**As a** platform operator,
**I want** all template management and cross-supplier data routes protected by role-based access control, so that `viewer` and `supplier_user` roles cannot perform configuration operations or access data outside their intended scope,
**so that** the principle of least privilege is enforced and only authorized roles can modify system configuration or access other suppliers' data.

---

## Context

The auth analysis found that several routes lack role enforcement entirely ‚ÄĒ they use only `authenticate` (which verifies identity but not role). This means any authenticated user in the tenant, including `viewer` and `supplier_user`, can access these routes.

Most template CRUD routes already have inline `user.role === UserRole.ADMIN` checks. However, two specific write routes (`copy.ts` for both workflow and form templates) have **no role check at all**, and several read routes expose configuration data (including unpublished drafts) to roles that should not see it. Additionally, `form-submissions/by-supplier.ts` allows any authenticated user to list any supplier's form submissions.

This story adds the missing role gates. It does **not** convert existing inline admin checks to middleware ‚ÄĒ that consistency improvement is deferred to SEC-008 (admin enforcement standardization).

### Analysis Report Items Addressed

| ID | Finding | Severity |
|----|---------|----------|
| A7 | `form-submissions/by-supplier.ts` ‚ÄĒ no role enforcement; any authenticated user can list any supplier's form submissions | High |
| A8 | Workflow template `copy.ts` ‚ÄĒ no role restriction; any authenticated user can deep-copy a workflow template | High |
| A9 | Form template `copy.ts`, `list.ts`, `get.ts` ‚ÄĒ no role restriction; any authenticated user can copy templates or view drafts | High |
| A10 | Principle-of-least-privilege violation ‚ÄĒ `viewer` and `supplier_user` can perform configuration operations (copy) and see draft template data | Medium |

### Dependencies

- **None** ‚ÄĒ this story is independent. The `requireRole` and `requireAdmin` middleware already exist in `apps/api/src/lib/rbac/middleware.ts`.
- This story is independent of SEC-004 (entity authorization). SEC-004 adds entity-level ownership checks for `supplier_user`; SEC-005 adds role-level gates. Both can be implemented in parallel.

### Relationship to SEC-008

SEC-008 (admin enforcement standardization) will convert all existing inline `user.role === UserRole.ADMIN` checks to `requireRole`/`requireAdmin` middleware for consistency (finding M1). This story only adds **missing** role gates ‚ÄĒ it does not refactor existing inline checks. After SEC-005 and SEC-008 are both complete, all template CRUD routes will use middleware-based role enforcement.

## Acceptance Criteria

### AC 1: Form Submissions By-Supplier Route Protected (A7)

1. `form-submissions/by-supplier.ts` is restricted to internal roles: `admin`, `procurement_manager`, `quality_manager`.
2. `viewer` and `supplier_user` receive 403 when calling `GET /api/form-submissions/by-supplier/:supplierId`.
3. The restriction is applied via `requireRole` middleware on the route, not an inline check.

### AC 2: Workflow Template Copy Route Protected (A8)

4. `workflow-templates/copy.ts` requires `admin` role.
5. Any non-admin authenticated user receives 403 when calling `POST .../workflow-templates/:id/copy`.
6. The restriction uses an inline `user.role === UserRole.ADMIN` check (consistent with the other workflow template routes in this module, pending SEC-008 migration to middleware).

### AC 3: Form Template Copy Route Protected (A9)

7. `form-templates/copy.ts` requires `admin` role.
8. Any non-admin authenticated user receives 403 when calling `POST .../form-templates/:id/copy`.
9. The restriction uses an inline `user.role === UserRole.ADMIN` check (consistent with the other form template routes in this module, pending SEC-008 migration to middleware).

### AC 4: Form Template Reads Scoped by Role (A9, A10)

10. `form-templates/list.ts` applies role-based filtering: non-admin roles see only `published` templates. Admin sees all templates (including drafts). This mirrors the existing pattern in `workflow-templates/list.ts`.
11. `form-templates/get.ts` applies a role check: non-admin roles can only access templates with `status = 'published'`. If a non-admin requests a non-published template, the route returns 404 (not 403, to avoid leaking existence of draft templates).
12. `form-templates/get-published-by-tenant.ts` remains accessible to all authenticated users (it already returns only published templates ‚ÄĒ no change needed).

### AC 5: Consistent Error Responses

13. All new 403 responses use the `requireRole` middleware's standard error shape: `{ error: { code: "FORBIDDEN", message: "Access denied. Required role(s): ..." } }` with HTTP status 403.
14. For inline admin checks (copy routes), use the exact same error shape as the existing inline checks in the same module for consistency: `{ success: false, error: { code: "FORBIDDEN", message: "Access denied. Required role: Admin", timestamp: ... } }` with status 403. SEC-008 will standardize all of these to middleware.

### AC 6: Automated Tests

15. At least one test per route verifying that an unauthorized role (e.g., `supplier_user` or `viewer`) receives 403.
16. At least one test per route verifying that an authorized role (e.g., `admin`) succeeds.
17. For the form template read routes (AC 4): test that a non-admin user cannot see draft templates but can see published ones.

### Role Decisions

This story makes the following role access decisions:

| Route | Allowed Roles | Rationale |
|-------|--------------|-----------|
| `form-submissions/by-supplier/:supplierId` | admin, procurement_manager, quality_manager | Listing another supplier's submissions is a supervision/oversight function. `viewer` should not access cross-supplier data. `supplier_user` should only see their own submissions (already handled by existing form-submission routes). |
| Template copy (both workflow + form) | admin only | Copying is a configuration write operation. Consistent with create/update/delete being admin-only. |
| `form-templates/list` | All authenticated, filtered by role | Non-admin sees published only. Admin sees all (including drafts). Mirrors `workflow-templates/list.ts` pattern. |
| `form-templates/get/:id` | All authenticated, draft access admin-only | Published templates must be accessible for form rendering in workflow steps (supplier_user fills out forms). Draft templates are admin-only configuration data. **Note:** this deliberately differs from `workflow-templates/get.ts` (which is fully admin-gated) because form template content is needed at runtime by non-admin roles for form rendering. |
| `form-templates/get-published` | All authenticated | Already returns published only ‚ÄĒ no change. |

### Why Inline Checks for Copy Routes (Not `requireRole` Middleware)?

The copy routes in both `workflow-templates/copy.ts` and `form-templates/copy.ts` should use inline admin checks (not `requireRole` middleware) for two reasons:

1. **Consistency within the module.** Every other write route in both modules uses inline `user.role === UserRole.ADMIN` checks. Mixing one `requireRole` call among inline checks creates inconsistency.
2. **SEC-008 will standardize all of them.** The admin enforcement standardization story will convert all inline admin checks across both modules to `requireRole`/`requireAdmin` middleware in one pass. Adding the missing checks now using the same inline pattern means SEC-008 can do a clean, uniform migration.

### Copy Route Path Issue

Both `copy.ts` files register their path as `POST "/workflow-templates/:id/copy"` (or `"/form-templates/:id/copy"`), which when mounted under the parent prefix creates a doubled segment. This is an existing bug unrelated to this story ‚ÄĒ it does not affect the authorization fix. If the dev agent notices this, they should document it in completion notes for a future fix, but not change the path in this story.

### Implementation Guardrails

1. **Do not convert existing inline admin checks to middleware.** That is SEC-008 scope. Only add new checks where none exist.
2. **Do not change route registration, URL patterns, or request/response shapes** beyond the authorization additions.
3. **The `by-supplier.ts` route uses `requireRole` middleware** (not an inline check) because it needs multi-role access (admin + procurement_manager + quality_manager) and it currently has no role enforcement at all ‚ÄĒ using middleware is the cleanest addition.
4. **Copy routes use inline checks** for consistency with sibling routes in the same module. See "Why Inline Checks for Copy Routes?" above.
5. **Form template `get.ts` returns 404 (not 403) for non-admin accessing drafts.** This prevents information leakage ‚ÄĒ a 403 would confirm the template exists. 404 is indistinguishable from "does not exist."
6. **Do not restrict `get-published-by-tenant.ts`.** This endpoint is needed for form rendering during workflow steps, where `supplier_user` fills out forms.
7. **Do not add new `PermissionAction` values** to the permission matrix for this story. The current role-based approach (ADMIN for config, internal roles for oversight) is sufficient. If granular permissions are needed later, they can be added when the permission matrix is extended.

### Testing [Source: architecture/testing-strategy.md]

- **Backend tests:** Bun test runner (`bun test`)
- **Test strategy:**
  - **Preferred:** Route-level or handler-level tests that exercise the actual HTTP path with different user roles, verifying that unauthorized roles receive 403 and authorized roles succeed. This is the primary proof that auth wiring is correct.
  - **Fallback (only if route-level testing is genuinely impractical):** Code-level assertions verifying the correct middleware or inline check is present. This fallback is acceptable only when the current test infrastructure makes it difficult to invoke routes with different simulated user roles.
  - **Minimum bar when fallback is used:** Even if some routes use the code-level fallback, this story must still include **at least one real route-level or handler-level test** that proves the auth wiring works in practice for at least one protected route.
  - **Documentation when fallback is used:** Add a brief code comment in the test file and a note in Completion Notes explaining why fuller route-level coverage was not feasible.
- **Key test scenarios:**
  - `supplier_user` / `viewer` ‚Üí `by-supplier` route ‚Üí 403
  - `admin` / `procurement_manager` ‚Üí `by-supplier` route ‚Üí 200
  - `viewer` ‚Üí copy workflow template ‚Üí 403
  - `viewer` ‚Üí copy form template ‚Üí 403
  - `admin` ‚Üí copy template ‚Üí success
  - Non-admin ‚Üí `form-templates/list` ‚Üí sees only published
  - Non-admin ‚Üí `form-templates/get/:id` (draft) ‚Üí 404
  - Non-admin ‚Üí `form-templates/get/:id` (published) ‚Üí 200
  - Admin ‚Üí `form-templates/get/:id` (draft) ‚Üí 200

### Coding Standards [Source: architecture/coding-standards.md]

- Authentication middleware: use `authenticate` directly on routes, not nested in wrappers
- `requireRole` includes `authenticate` ‚ÄĒ do not chain both
- Error responses for inline checks: `{ success: false, error: { code: "FORBIDDEN", message: "...", timestamp: "..." } }` with HTTP status
- Error responses for `requireRole` middleware: automatic `{ error: { code: "FORBIDDEN", message: "..." } }` with 403
- ElysiaJS validation uses TypeBox (`t.*`), not Zod
