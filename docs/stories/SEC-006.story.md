## Story

**As a** platform operator,
**I want** JWT verification migrated from legacy HMAC shared secret (HS256) to asymmetric signing keys (ES256) using the Supabase JWKS endpoint,
**so that** the application follows Supabase's recommended security architecture, enables zero-downtime key rotation, eliminates the need to manage a shared JWT secret, and aligns with enterprise compliance frameworks (SOC2, PCI-DSS, ISO27000, HIPAA).

---

## Context

This is a P2 story that migrates the JWT verification system from the legacy HMAC approach to the modern asymmetric signing keys system recommended by Supabase.

### Why This Must Be Done

Supabase documentation explicitly states:
> "There is almost no benefit from using a JWT signed with a shared secret. [...] Using this approach can expose your project's data to significant security vulnerabilities or weaknesses."
> "Not recommended for production applications."

The current implementation in `jwt-verifier.ts` uses `jose.jwtVerify(token, secret)` with the HMAC shared secret from `SUPABASE_JWT_SECRET`. This means:
- The secret must be stored as an env var and managed by the operator
- Key rotation requires redeployment and may cause downtime
- A leaked secret allows full JWT forgery (impersonate any user)
- Compliance frameworks (SOC2, etc.) are harder to align with

After migration, JWT verification uses the JWKS endpoint (`/.well-known/jwks.json`) to fetch public keys. The private key never leaves Supabase's infrastructure.

## Acceptance Criteria

### AC 1: JWKS-Based Verification (Phase 1)

1. `jwt-verifier.ts` uses `jose.createRemoteJWKSet()` with the Supabase JWKS endpoint as the **primary** JWT verification method.
2. The JWKS URL is derived from the existing `config.supabase.url` — no new env var is needed.
3. The `jose` library's built-in JWKS caching is used (automatic fetch on first use, refetch on key ID miss). No custom JWKS caching layer is needed.

### AC 2: Algorithm-Based HMAC Fallback for Transition (Phase 1)

4. Before verification, the verifier decodes the unverified JWT header (`jose.decodeProtectedHeader`) to inspect the `alg` field. The verification path is selected based on the token's algorithm — **not** by catching a generic JWKS failure:
   - `alg === "HS256"` and `config.jwt?.secret` is configured → HMAC verification (legacy transition path).
   - `alg === "HS256"` and no `config.jwt?.secret` → reject immediately with `INVALID_TOKEN` (no secret available to verify legacy token).
   - `alg` is an asymmetric algorithm (e.g., `"ES256"`) → JWKS verification only. A JWKS failure for an ES256 token must **not** trigger HMAC fallback.
   - `alg` is missing, unsupported, or unexpected → reject immediately with `INVALID_TOKEN`.
   - Malformed token (header cannot be decoded) → reject immediately with `INVALID_TOKEN`.
5. The HMAC fallback logs a warning (at `warn` level) each time it is used, so the operator can monitor the transition: `"JWT verified via HMAC fallback — legacy HS256 token detected"`.
6. The fallback is clearly isolated in the code so it can be cleanly removed in Phase 3. JWKS failures for asymmetric tokens are never masked by the HMAC path.

### AC 3: Config Changes (Phase 1)

7. `jwt.secret` in `config.ts` becomes optional: it is no longer required for the server to start, even in production.
8. The production guard from SEC-002 (server refuses to start without `jwt.secret`) is relaxed: the server starts if JWKS is available (it always is, derived from `supabase.url`). `jwt.secret` becomes optional for the HMAC fallback only.
9. A startup log message indicates which verification mode is active:
   - `"JWT verification: JWKS (primary) + HMAC fallback (transition mode)"` — when `jwt.secret` is configured.
   - `"JWT verification: JWKS only"` — when no `jwt.secret` is configured (Phase 3 state).

### AC 4: SupabaseJWTPayload Compatible

10. The `SupabaseJWTPayload` interface remains unchanged. ES256-signed JWTs contain the same payload structure as HS256-signed JWTs — only the signature algorithm and headers differ.
11. The `extractUserIdUnsafe` and `isJWTExpiredUnsafe` utility functions are unaffected (they decode without verification).

### AC 5: Auth Cache Unaffected

12. The two-tier auth cache (`auth-cache.ts`) is unaffected. It caches the resolved user context (from the database), not JWT verification results. No changes to the cache layer.

### AC 6: Automated Tests

13. Tests verify JWKS-based verification succeeds with an ES256-signed JWT (using a test key pair).
14. Tests verify HMAC fallback succeeds with an HS256-signed JWT when `jwt.secret` is configured.
15. Tests verify that when only JWKS is available (no `jwt.secret`), an HS256 token is rejected immediately (no HMAC fallback available) with `INVALID_TOKEN`.
16. Tests verify startup log messages match the configured verification mode.
17. Tests verify that an ES256 token with an invalid signature is rejected via JWKS — HMAC fallback is **not** attempted, even if `jwt.secret` is configured.
18. Tests verify that a token with an unsupported or unexpected `alg` (e.g., `"none"`, `"RS256"`) is rejected immediately with `INVALID_TOKEN`.
19. Tests verify that a malformed token (header cannot be decoded) is rejected with `INVALID_TOKEN`.

    ```typescript
    export async function verifyJWT(token: string): Promise<SupabaseJWTPayload> {
      // Step 1: Decode unverified header to determine algorithm
      let header: jose.ProtectedHeaderParameters;
      try {
        header = jose.decodeProtectedHeader(token);
      } catch {
        throw new JWTVerificationError(
          "Malformed JWT — unable to decode token header",
          "INVALID_TOKEN"
        );
      }

      const { alg } = header;
      const issuer = config.supabase.url.endsWith("/")
        ? `${config.supabase.url}auth/v1`
        : `${config.supabase.url}/auth/v1`;
      const verifyOptions = { issuer, audience: "authenticated" };

      // Step 2: Route to the correct verification path based on alg
      let payload: jose.JWTPayload;

      if (alg === "HS256") {
        // TRANSITION: HMAC fallback — remove after ES256 migration confirmed stable (Phase 3)
        if (!config.jwt?.secret) {
          throw new JWTVerificationError(
            "HS256 token received but no HMAC secret configured — legacy token cannot be verified",
            "INVALID_TOKEN"
          );
        }
        const secret = new TextEncoder().encode(config.jwt.secret);
        ({ payload } = await jose.jwtVerify(token, secret, {
          ...verifyOptions,
          algorithms: ["HS256"],
        }));
        logger.warn("JWT verified via HMAC fallback — legacy HS256 token detected");
      } else if (alg === "ES256") {
        // Primary: JWKS verification (asymmetric)
        ({ payload } = await jose.jwtVerify(token, getJWKS(), verifyOptions));
      } else {
        throw new JWTVerificationError(
          `Unsupported JWT algorithm: ${alg || "(missing)"}`,
          "INVALID_TOKEN"
        );
      }

      // Step 3: Validate required claims
      if (!payload.sub) {
        throw new JWTVerificationError(
          "Missing user ID (sub claim)",
          "MISSING_CLAIMS"
        );
      }

      return payload as SupabaseJWTPayload;
    }
    ```

## Dev Notes

### How Supabase JWKS Works

Supabase exposes public keys for asymmetric signing algorithms at:
```
GET https://<project-id>.supabase.co/auth/v1/.well-known/jwks.json
```

The `jose` library's `createRemoteJWKSet()` function:
- Fetches the JWKS from the URL on first use
- Caches keys in memory automatically
- Refetches when encountering a `kid` (Key ID) not in the cached set
- Handles key selection based on the JWT's `kid` header

This means key rotation in Supabase is automatically picked up by the app without redeployment.

### Why HMAC Secrets Are Not in JWKS

JWKS only publishes **public** keys for **asymmetric** algorithms. HMAC (HS256) uses a **symmetric** shared secret — publishing it in JWKS would compromise it. This is why:
- ES256 tokens → header says `ES256` → JWKS verification (public key)
- HS256 tokens → header says `HS256` → HMAC verification (local secret)
- During transition: the verifier inspects the token header and routes to the correct path

### Transition Timeline

| Event | Tokens in Circulation | Verification Path |
|-------|----------------------|-------------------|
| **Phase 1 deployed** (code only) | All HS256 (Supabase hasn't rotated yet) | Header says `HS256` → HMAC path → succeeds (+ warn log) |
| **Phase 2: keys rotated** | Mix of HS256 (old) + ES256 (new) | Header `ES256` → JWKS path → succeeds. Header `HS256` → HMAC path → succeeds (+ warn log) |
| **All tokens refreshed** (~1-2 hours after rotation) | All ES256 | Header `ES256` → JWKS path → succeeds. HMAC path never entered. |
| **Phase 3: HMAC fallback removed** | All ES256 | JWKS only. HS256 tokens rejected at algorithm check. |
| **Legacy key revoked** | All ES256, HS256 permanently rejected | JWKS only |

### Performance Impact

| Method | First Request | Subsequent Requests |
|--------|--------------|-------------------|
| Current (HMAC) | ~0.1ms (local crypto) | ~0.1ms |
| JWKS (after cache populated) | ~100-200ms (HTTP fetch to JWKS endpoint) | ~0.1ms (cached public key, local crypto) |

The JWKS fetch happens once and is cached. After that, ES256 signature verification is ~0.1ms — comparable to HMAC. The first-request latency is a one-time cost per server instance.

### `jose` Library Version

The project uses `jose@^5.10.0` (`apps/api/package.json`). This version fully supports `createRemoteJWKSet()`, `ES256`, and all the APIs needed for this migration. No dependency updates required.

### Implementation Guardrails

1. **Do not change the `SupabaseJWTPayload` interface.** ES256 JWTs have the same payload as HS256 JWTs — only the signature differs.
2. **Do not change the auth cache.** It caches resolved user context, not JWT verification results.
3. **Do not change the `authenticate` middleware.** It calls `verifyJWT()` — the interface is unchanged.
4. **The JWKS singleton must be lazy-initialized** (not at import time). If created at import time, it would fetch JWKS before the config is loaded.
5. **Use `jose.createRemoteJWKSet()` — do not implement custom JWKS fetching/caching.** The `jose` library handles caching, refetching, and key selection correctly.
6. **The HMAC fallback is a transition mechanism, not a permanent feature or a general recovery path.** It is entered only when the token header declares `alg: "HS256"`. JWKS failures for ES256 tokens must never trigger HMAC fallback. Mark the HMAC path with a clear comment: `// TRANSITION: HMAC fallback — remove after ES256 migration is confirmed stable (Phase 3)`.
7. **Do not remove `extractUserIdUnsafe` or `isJWTExpiredUnsafe`.** These decode without verification and are unaffected by the signing algorithm change.
8. **The dev agent's implementation scope is Phase 1 only.** Phase 2 is operator-executed using the runbook after deployment. Phase 3 (removing HMAC fallback, removing `jwt.secret` config, revoking legacy key) is out of scope — it is a follow-up task after the operator confirms ES256 is stable. Do not implement Phase 2 or Phase 3 changes in this story.

## Operator Manual Steps

These steps must be performed by the operator at specific points. The dev agent cannot perform these.

### Prerequisites

- **SEC-001 must be stable.** The JWT payload (claims model) must be settled.
- **SEC-002 must be deployed.** The JWT secret guard is in place (SEC-006 will relax it).
- **Phase 1 code must be deployed first.** The JWKS+HMAC dual verification must be running in production before you rotate keys in Supabase.

### Phase 2: Migrate and Rotate Keys in Supabase

**Step 1: Verify Phase 1 code is deployed**

Confirm the API server is running with the updated `jwt-verifier.ts`. Check the startup log for:
```
JWT verification: JWKS (primary) + HMAC fallback (transition mode)
```

If the log says "JWKS only", the secret is not configured — this is fine but means the HMAC fallback is not active. All existing HS256 tokens will be verified via JWKS during transition only if JWKS serves the legacy key (it doesn't for HS256). Ensure you understand this before proceeding.

**Step 2: Migrate JWT secret in Supabase Dashboard**

1. Go to Supabase Dashboard → Project Settings → JWT Signing Keys (or Settings → API → JWT Settings).
2. Click **"Migrate JWT secret"**.
3. This imports the existing HS256 secret into the new signing keys system.
4. Supabase simultaneously creates a new ES256 (P-256 Elliptic Curve) key in **standby** state.
5. This step does **not** change how tokens are signed — Auth still uses HS256.

**Step 3: Verify standby key is visible in JWKS**

```bash
curl https://<project-id>.supabase.co/auth/v1/.well-known/jwks.json | jq
```

Expected: The response should contain at least one key with `"alg": "ES256"`. This is the standby key. It has not started signing JWTs yet.

**Step 4: Rotate to ES256**

1. In the Supabase Dashboard → JWT Signing Keys page, click **"Rotate keys"**.
2. This makes the ES256 key **active** — Auth starts signing all new JWTs with ES256.
3. The HS256 key moves to **"Previously used"** state — existing HS256 tokens remain valid.
4. **No users are signed out.** Active sessions receive ES256 tokens on their next token refresh.

**Step 5: Monitor the transition**

After rotation, monitor the API server logs for HMAC fallback warnings:
```
JWT verified via HMAC fallback — legacy HS256 token detected
```

These warnings should decrease over time as users' tokens refresh to ES256. After approximately 1-2 hours (depending on your access token expiry setting), no more HMAC fallback warnings should appear.

**Step 6: Verify ES256 is working**

1. Log in to the application (fresh login).
2. Decode the JWT (browser DevTools → Application → Cookies, or jwt.io).
3. Verify the JWT header shows `"alg": "ES256"` and has a `"kid"` field.
4. Verify the API responds successfully (200, not 401).
5. Navigate to protected pages and verify normal operation.

**Step 7: Plan legacy key revocation**

After confirming no HMAC fallback warnings for at least 2 hours:
1. In Supabase Dashboard → JWT Signing Keys, find the HS256 key under "Previously used".
2. Click **"Revoke"** to permanently stop trusting HS256-signed tokens.
3. **Warning:** After revocation, any remaining HS256 token will be rejected. Ensure all active sessions have refreshed to ES256 before revoking.

**Important:** Revoking the legacy JWT secret may require disabling `anon` and `service_role` API keys if they were derived from it. Supabase's new "publishable and secret API keys" system replaces them. Verify your project's API key configuration before revoking. If your project still uses the legacy `anon`/`service_role` keys (which this project does), you may need to migrate to the new API keys system first. **Do not revoke the legacy secret if it would break the `anon` or `service_role` keys.**

### Phase 3: Cleanup (Follow-Up)

After ES256 has been stable for days/weeks:

1. Remove the `SUPABASE_JWT_SECRET` / `JWT_SECRET` env vars from the deployment configuration.
2. Deploy the Phase 3 code change: remove the HMAC fallback from `jwt-verifier.ts`, remove `jwt.secret` from config entirely.
3. Verify the startup log says: `"JWT verification: JWKS only"`.

Phase 3 is optional cleanup. The system is fully secure after Phase 2 — the HMAC fallback simply adds unnecessary code. Phase 3 can be deferred or bundled with SEC-010 (cleanup story).
