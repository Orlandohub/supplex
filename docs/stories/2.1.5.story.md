## Story

**As a** procurement_manager,  
**I want** to define a supplier contact during supplier creation and automatically create the corresponding user,  
**so that** suppliers can access the platform immediately after being added.

---

## Acceptance Criteria

### Core Supplier Contact Creation

1. Supplier creation includes a **supplier_contact** section
2. Supplier contact fields include `name`, `email`, `phone`
3. On supplier creation:
   * A user is created using the supplier contact information
   * The user is assigned the `supplier_user` role
   * The user is linked to the supplier
   * The user status is set to `pending_activation` (not `active`)
4. If the `email` already exists, creation is blocked

### User Onboarding and Password Setup

5. When a supplier user is created:
   * The user account is created with status `pending_activation`
   * No password is set or visible to admin or procurement_manager at any time
   * The system generates a secure, single-use invitation link associated with the user
6. The invitation link:
   * Is time-limited with a validity of **48 hours**
   * Can only be used once
   * Is token-based for security
7. When the user accesses the link and sets a password successfully:
   * The user status changes from `pending_activation` to `active`
   * The invitation link becomes invalid and cannot be reused
8. If the invitation link expires without being used:
   * The user remains in `pending_activation` status
   * A new invitation link can be generated (handled in future story)

### Pending User Invitations Page (Backoffice – Admin Only)

9. A backoffice page called `pending_user_invitations` exists
10. The page is accessible only to `admin` role
11. The page lists all suppliers that have users with status `pending_activation`
12. For each listed supplier, the page displays:
    * Supplier name
    * User name
    * User email
    * Invitation status: `pending` or `expired`
    * The invitation link (read-only, identical to links generated for email)
13. The page provides a `resend_invitation` action
14. When `resend_invitation` is triggered:
    * The previous invitation link is invalidated
    * A new invitation link is generated with new 48-hour expiry
    * The new link is displayed in the list
15. Invitation links that have been used:
    * Are immediately invalidated
    * Are removed from the `pending_user_invitations` page
16. Users with status `active` do not appear on this page
17. Users with status `deactivated` do not appear on this page

### Security and Alignment

18. Invitation links and password reset links follow the same security rules:
    * Token-based validation
    * Time-limited validity
19. Password reset functionality is blocked for users with status `deactivated`

- Supabase Admin client is imported from `apps/api/src/lib/supabase`

**From Story 1.4 (Tenant Management & User Roles):**
- User IDs must match Supabase auth.users.id (UUID)
- User records are inserted into local `users` table after Supabase Auth user creation

**User Invitations Table (New - Migration 0008):**
```sql
CREATE TABLE user_invitations (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    token VARCHAR(255) NOT NULL UNIQUE,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    used_at TIMESTAMP WITH TIME ZONE,
    created_by UUID NOT NULL REFERENCES users(id),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL
);

CREATE INDEX idx_user_invitations_token ON user_invitations(token);
CREATE INDEX idx_user_invitations_user_id ON user_invitations(user_id);

COMMENT ON TABLE user_invitations IS 
'Secure invitation tokens for user onboarding with 48-hour validity. Single-use tokens for password setup.';
```

### User Onboarding Flow

[New in Story 2.1.5]

**Complete Invitation and Activation Flow:**

1. **Supplier Creation with Contact (Procurement Manager)**
   - Procurement manager creates supplier and checks "Create platform access"
   - Fills in supplier contact details (name, email, phone)
   - Submits form

2. **User and Invitation Creation (Backend)**
   - System creates Supabase Auth user (no password)
   - System creates local user record with status `pending_activation`
   - System generates secure 64-character hex token
   - System creates invitation record with 48-hour expiry
   - System links user to supplier via `supplierUserId`

3. **Invitation Link Display (Frontend)**
   - Success modal shows invitation link
   - Procurement manager copies link
   - Link format: `https://app.supplex.com/auth/accept-invitation?token={token}`
   - Note: Email sending will be implemented in future story

4. **Invitation Acceptance (Supplier User)**
   - Supplier user clicks invitation link
   - System validates token (exists, not expired, not used)
   - User sees password setup form
   - User enters password (min 8 chars, complexity requirements)
   - User confirms password

5. **Account Activation (Backend)**
   - System validates password strength
   - System updates Supabase Auth user with password
   - System updates user status from `pending_activation` to `active`
   - System marks invitation as used (`usedAt` timestamp)
   - System invalidates invitation token

6. **Login (Supplier User)**
   - User redirected to login page with success message
   - User logs in with email and password
   - User accesses platform with `supplier_user` role permissions

**Security Features:**
- Token is 64-character hex (256-bit entropy)
- Token is single-use (marked as used after acceptance)
- Token expires after 48 hours
- No password visible to admins at any time
- Password reset blocked for deactivated users

#### Step 2: Backend User Creation Logic

[Source: apps/api/src/routes/suppliers/create.ts, apps/api/src/routes/users/invite.ts]

**Supplier Creation with User (Transactional Pattern):**
```typescript
// Inside supplier creation route handler
try {
  const tenantId = user.tenantId as string;
  const userId = user.id as string;
  let supplierUserId: string | null = null;

  // Step 1: Create supplier user if supplierContact provided
  if (body.supplierContact) {
    const { name, email, phone } = body.supplierContact;

    // Check email uniqueness within tenant
    const existingUser = await db.query.users.findFirst({
      where: and(
        eq(users.tenantId, tenantId),
        eq(users.email, email)
      ),
    });

    if (existingUser) {
      set.status = 409;
      return {
        success: false,
        error: {
          code: "USER_EMAIL_EXISTS",
          message: "A user with this email already exists in your organization",
          timestamp: new Date().toISOString(),
        },
      };
    }

    // Create Supabase Auth user
    const { data: authUser, error: authError } = await supabaseAdmin.auth.admin.createUser({
      email,
      email_confirm: true, // Skip email verification for supplier users
      user_metadata: createUserMetadata(tenantId, UserRole.SUPPLIER_USER),
    });

    if (authError || !authUser.user) {
      throw new Error(`Supabase user creation failed: ${authError?.message}`);
    }

    // Insert user record into local database
    const [newUser] = await db.insert(users).values({
      id: authUser.user.id,
      tenantId,
      email,
      fullName: name,
      role: UserRole.SUPPLIER_USER,
      isActive: true,
      createdAt: new Date(),
      updatedAt: new Date(),
    }).returning();

    supplierUserId = newUser.id;
  }

  // Step 2: Insert supplier with supplierUserId
  const [newSupplier] = await db.insert(suppliers).values({
    ...supplierData,
    supplierUserId, // null if no contact user created
    tenantId,
    createdBy: userId,
    createdAt: new Date(),
    updatedAt: new Date(),
  }).returning();

  // Step 3: Return supplier and user info
  const supplierUser = supplierUserId
    ? await db.query.users.findFirst({
        where: eq(users.id, supplierUserId),
      })
    : null;

  set.status = 201;
  return {
    success: true,
    data: {
      supplier: newSupplier,
      supplierUser,
    },
  };
} catch (error) {
  // If supplier insert fails after user creation, clean up Supabase auth user
  if (supplierUserId) {
    await supabaseAdmin.auth.admin.deleteUser(supplierUserId);
  }
  // ... error handling ...
}
```

### Security Considerations

[Source: architecture/coding-standards.md]

**Critical Security Rules:**
- **Email Uniqueness:** Email must be unique per tenant (enforced by DB constraint `users_tenant_email_unique`)
- **Tenant Isolation:** All user and supplier queries must include tenant filter
- **Role Assignment:** Supplier users MUST be assigned `supplier_user` role (not admin/procurement)
- **No Password Visibility:** Passwords are NEVER visible to admins or stored in plain text
- **Secure Token Generation:** Invitation tokens use `crypto.randomBytes(32)` for 256-bit entropy
- **Token Single-Use:** Invitation tokens can only be used once (marked with `usedAt` timestamp)
- **Token Expiry:** Invitation tokens expire after 48 hours (enforced in validation logic)
- **User Status Tracking:** Users start in `pending_activation` status, move to `active` after password setup
- **Password Reset Block:** Deactivated users cannot reset passwords (403 error)
- **User Cleanup:** If supplier creation fails after user creation, delete Supabase Auth user to prevent orphaned accounts
- **Principle of Least Privilege:** supplier_user role has minimal permissions (only VIEW_OWN_SUPPLIER, EDIT_OWN_SUPPLIER, VIEW_OWN_TASKS, MANAGE_NOTIFICATION_SETTINGS)
- **Invitation Validation:** All invitation acceptance requests must validate token, expiry, and usage status before allowing password setup

### Error Handling

**Error Scenarios:**
1. **Duplicate Email (409 Conflict):**
   - Check: Before creating Supabase Auth user
   - Response: `USER_EMAIL_EXISTS` with clear message
   - Frontend: Highlight email field, show error message

2. **Supabase User Creation Fails (500 Internal Error):**
   - Check: After `supabaseAdmin.auth.admin.createUser()` call
   - Response: `INTERNAL_ERROR` with "Failed to create supplier user"
   - Log: Log error details for debugging

3. **Supplier Insert Fails After User Creation (500 Internal Error):**
   - Check: If supplier DB insert throws error
   - Cleanup: Delete Supabase Auth user using `supabaseAdmin.auth.admin.deleteUser()`
   - Response: `INTERNAL_ERROR` with "Failed to create supplier"
   - Log: Log error details and cleanup status

4. **Invalid Email Format (400 Validation Error):**
   - Check: Zod validation on frontend and TypeBox validation on backend
   - Response: `VALIDATION_ERROR` with field-specific errors
   - Frontend: Highlight email field, show validation message

## Post-Implementation Scope Changes (December 18, 2025)

### Summary
After initial implementation and testing, significant UX improvements were made to consolidate user management functionality. These changes deviate from the original story scope but improve system usability and maintainability.

### Changes Made

**1. Pending Invitations Moved to User Management Settings**
- **Original:** Standalone `/admin/pending-invitations` page accessible via sidebar menu
- **Changed To:** Integrated as a tab within `/settings/users` page (no sidebar menu item)
- **Rationale:** Consolidates all user management in one place, reducing navigation complexity

**2. User Management Page Restructured with 3 Tabs**
- **Tab 1 - Internal Users:** Shows internal staff (admin, procurement_manager, quality_manager, viewer)
  - Includes "Invite User" button (only visible on this tab)
  - Status filter and user management actions
- **Tab 2 - Supplier Users:** Shows suppliers with platform access
  - Expandable supplier list showing associated users
  - Users can be activated/deactivated (role editing disabled for supplier users)
- **Tab 3 - Pending Invitations:** Shows ALL pending user invitations
  - Not filtered to supplier users only (supports future internal user invitations)
  - Search, copy links, resend functionality preserved

**3. API Flexibility Enhancement**
- **Modified:** `/api/users/pending-invitations` endpoint
- **Change:** Added optional `role` query parameter for filtering
- **Original Behavior:** No filtering, returns all pending invitations
- **New Capability:** Can filter by role (e.g., `?role=supplier_user`) for tab-specific views
- **Rationale:** Supports future use cases where different tabs/pages need role-specific invitation lists

**4. Supplier User Role Protection**
- **Added:** Supplier users cannot have their role changed (only activate/deactivate)
- **Rationale:** Supplier users are tied to suppliers; changing their role would break the association

**5. UI Component Enhancement**
- **Modified:** `apps/web/app/components/ui/tabs.tsx`
- **Change:** Enhanced to support both controlled (`value` + `onValueChange`) and uncontrolled (`defaultValue`) usage
- **Rationale:** Supplier detail page tabs were broken by simplified implementation; now supports all use cases
