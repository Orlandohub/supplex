# Story WFH-001: Document Review Transaction Safety & Engine Unification

<!-- Powered by BMAD™ Core -->

## Status

**Done**

Date Created: April 8, 2026

## Story

**As a** platform operator,
**I want** the document review route (`review.ts`) to execute all workflow state mutations inside a single database transaction, delegating all state-machine logic to engine functions with CAS guards,
**so that** a network failure, timeout, or concurrent request during document review cannot leave a workflow process in an inconsistent, unrecoverable state.

---

## Context

The document review route (`POST /api/workflows/steps/:stepInstanceId/documents/review`) is the most dangerous mutation path in the workflow engine. It performs 8-20+ independent database writes across 5 tables (`workflowStepDocument`, `taskInstance`, `stepInstance`, `processInstance`, `suppliers`) with **no transaction** and **no CAS guards** on status transitions. All state-machine logic is implemented inline in the route handler, duplicating patterns that already exist safely in the engine (`approveValidationTask`, `completeStep`, `transitionToNextStep`).

This means:
- A crash or timeout midway through the review leaves the workflow in a partial state (e.g., step marked `validated` but no next step activated, or process marked `complete` but supplier status unchanged).
- Two concurrent review submissions can both succeed because there is no CAS on the step status transition.
- Bug fixes, CAS guards, and transaction safety in the engine do not protect this code path.
- The `transitionToNextStep` function is called with the raw `db` handle, so even its internal writes (including the supplier status update on process completion) are non-atomic.

In contrast, `complete.ts` (which handles submit/approve/decline for form steps) wraps all mutations in `db.transaction()`, uses CAS on every status change, and delegates to engine functions. This story brings `review.ts` to the same safety standard.

### Architecture Analysis Findings Addressed

| ID | Finding | Severity | Category |
|----|---------|----------|----------|
| **C1** | `review.ts` performs 8-20 writes with NO transaction; `transitionToNextStep(db, ...)` called with raw `db` | **Critical** | Correctness |
| **C3** | `review.ts` re-implements engine transition logic inline, creating two parallel code paths for the same operations | **High** | Correctness |
| **P3** | Declined documents are double-written: first to `declined`, then immediately overwritten to `pending` | Low | Performance (co-located fix) |
| **P5** | N+1 individual UPDATE per document decision instead of batch updates | Low | Performance (co-located fix) |

## Acceptance Criteria

### AC 1: All Document Review Mutations Wrapped in a Single Transaction

1. The `review.ts` route handler wraps all document review writes (approve path and decline path) inside a single `db.transaction(async (tx) => { ... })`.
2. All engine functions called during the review receive `tx`, not `db`.
3. If any write fails, the entire transaction rolls back — no partial state is committed.
4. Event logging (`logWorkflowEvent`) remains **outside** the transaction (fire-and-forget, consistent with all other routes).

### AC 2: State Transitions Delegated to Engine Functions

5. The route handler **does not** perform inline `UPDATE stepInstance`, `UPDATE taskInstance`, `UPDATE processInstance`, or call `transitionToNextStep` / `createTasksForStep` directly.
6. A new engine function (or pair of functions) handles the document review approve and decline state machines.
7. The engine function(s) accept `tx: DbOrTx` as the first parameter and use `tx` for all reads and writes.
8. The engine function(s) return a structured result type that contains the **minimum** context the route needs to: (a) format the current response shape, (b) populate the current event log calls, and (c) distinguish success from CAS conflict. The result type must not be expanded beyond what WFH-001 requires — broader "rich engine return type" redesign for event logging optimization is WFH-006 scope.

### AC 3: CAS Guards on Every Status Transition

9. The step status transition to `validated` uses a CAS pattern: `UPDATE stepInstance SET status = 'validated' WHERE status = 'awaiting_validation'` with `.returning()`. If the CAS fails (0 rows returned), the operation returns a conflict error.
10. The step status transition back to `active` (decline path) uses a CAS pattern: `UPDATE stepInstance SET status = 'active' WHERE status = 'awaiting_validation'` with `.returning()`.
11. The task completion uses a CAS pattern: `UPDATE taskInstance SET status = 'completed' WHERE status = 'pending'`.
12. If two concurrent review submissions race, exactly one succeeds and the other receives a conflict error (409). No data corruption occurs.

### AC 4: Declined-Document Double-Write Eliminated (P3)

13. On the decline path, declined documents are written directly to `status = 'pending'` with `documentId = null` in a single UPDATE — the intermediate `status = 'declined'` write is eliminated.
14. The decline comment, `reviewedBy`, and `reviewedAt` are preserved on the document row (these record who reviewed it and why it was declined, even though the status resets to `pending`).

### AC 5: N+1 Document Updates Batched (P5)

15. Document status updates use batch operations (e.g., `inArray` with Drizzle) instead of individual UPDATE per document.
16. On the approve path: a single UPDATE sets all reviewed documents to `approved`.
17. On the decline path: one batch UPDATE for approved documents, one batch UPDATE for declined documents (reset to `pending`).

### AC 6: Route Handler Is Thin

18. After this story, `review.ts` is responsible only for: input validation (Elysia schema), authentication (`authenticate`), authorization (`verifyTaskAssignment`), calling the engine function inside a `db.transaction()`, event logging (outside tx), and response formatting.
19. No workflow state-machine logic remains inline in the route handler.

### AC 7: Existing Behavior Preserved

20. The API response shape for both approve and decline paths is unchanged (or backward-compatible).
21. The event logging behavior (which events are logged, when) is unchanged.
22. The `verifyTaskAssignment` check at the start of the route is preserved.
23. The Elysia request/response schema is unchanged.
24. All existing workflow engine tests continue to pass.

### AC 8: New Engine Function(s) Tested

25. Unit/integration tests cover: approve-all-documents → step validated → transition to next step.
26. Unit/integration tests cover: approve-all-documents on last step → process completed → supplier status updated.
27. Unit/integration tests cover: any-document-declined → step reset to active → tasks recreated for re-upload.
28. Unit/integration tests cover: concurrent review submissions → only one succeeds (CAS conflict).
29. Tests use the same database-backed integration test pattern as existing engine tests (see `__tests__/auto-validation.test.ts`).

## Dev Notes

### Engine Function Design: One Function, Not Two

The recommended approach is a **single** engine function `reviewStepDocuments()` that internally branches on `hasDeclines`. This mirrors how `completeStep()` handles both the `completed` and `declined` outcomes in one function. The rationale:

- Both paths share the same preamble (read step, read docs, validate decisions).
- The route handler should not decide which engine function to call based on the content of the decisions — that is engine logic.
- Keeping it in one function makes it impossible for the route to accidentally call the wrong path.

### CAS Pattern Reference

The CAS (Compare-And-Swap) pattern used throughout the engine is:

```typescript
const [result] = await tx
  .update(stepInstance)
  .set({ status: "validated", updatedAt: new Date() })
  .where(
    and(
      eq(stepInstance.id, stepInstanceId),
      eq(stepInstance.status, "awaiting_validation")  // CAS guard
    )
  )
  .returning();

if (!result) {
  return { success: false, error: "Step already processed or not in expected state" };
}
```

This is already used correctly in `approveValidationTask.ts` (task CAS lines 42-57, step CAS lines 93-106) and in `complete.ts` decline path (task CAS lines 378-392, step CAS lines 415-424). Follow the same pattern exactly.

### Batch Document Updates with `inArray`

Drizzle ORM supports `inArray` for batch operations:

```typescript
import { inArray } from "drizzle-orm";

const approvedDocIds = decisions
  .filter(d => d.action === "approve")
  .map(d => docMap.get(d.requiredDocumentName)?.id)
  .filter(Boolean);

if (approvedDocIds.length > 0) {
  await tx
    .update(workflowStepDocument)
    .set({
      status: "approved",
      reviewedBy: params.reviewedBy,
      reviewedAt: new Date(),
      declineComment: null,
      updatedAt: new Date(),
    })
    .where(inArray(workflowStepDocument.id, approvedDocIds));
}
```

This replaces the N individual UPDATEs in the current code (lines 76-103 of `review.ts`) with at most 2 batch UPDATEs.

### Decline Path: What to Preserve on the Document Row

When a document is declined, the current code writes `declineComment`, `reviewedBy`, and `reviewedAt` — then immediately overwrites the status to `pending` and clears `documentId`. The intent is:

- `status = 'pending'` — the document slot is open for re-upload
- `documentId = null` — the old uploaded file is unlinked
- `declineComment` — preserved so the uploader can see why it was declined
- `reviewedBy` / `reviewedAt` — preserved for audit trail

The new engine function should write all of these in a **single** UPDATE per declined document (or batch), not two sequential writes.

### What `review.ts` Should Look Like After

The route handler should be approximately 60-80 lines (down from 314), structured as:

```
1. Auth check
2. verifyTaskAssignment (outside tx)
3. db.transaction(tx => reviewStepDocuments(tx, params))
4. Error handling (check result.success)
5. logWorkflowEvent (fire-and-forget, using result data)
6. Return response
```

This matches the pattern in `complete.ts` for the approve path (lines 178-296).

### `transitionToNextStep` — Already Transaction-Compatible

`transitionToNextStep` already accepts `tx: DbOrTx` and uses `tx` for all operations. It was hardened in Story 2.2.19. The only problem was that `review.ts` was calling it with `db` instead of `tx`. The new engine function will call it with `tx`, fixing this issue without any changes to `transitionToNextStep` itself.

### `createTasksForStep` — Already Transaction-Compatible and Idempotent

`createTasksForStep` accepts `tx: DbOrTx`, has a triple-layered idempotency guard (code check + `onConflictDoNothing` + DB unique constraint), and was hardened in Story 2.2.19. The decline path in the new engine function calls it with `tx`, matching how `complete.ts` decline path already uses it.

### Event Logging: Use Result Data, Do Not Re-Read — But Keep Scope Minimal

The current `review.ts` re-reads `stepInstance` after the approve path (line 272-275) to get the next step name for the `STEP_ACTIVATED` event log. This story eliminates that post-tx re-read by having the engine function include `nextStepName` in its result.

**Scope boundary:** The engine result type includes only the fields that the current `review.ts` event log calls and response formatting actually consume. Specifically:

| Result field | Consumed by |
|---|---|
| `outcome` | Route response (`action` field) + determines which event to log |
| `approvedCount` / `declinedCount` | Route response + event description string |
| `declinedDocumentNames` | Event description + metadata on decline path |
| `stepName` | Event description (e.g., `"Step ${stepName} Approved"`) |
| `processInstanceId` | `logWorkflowEvent` call (required field) |
| `nextStepActivated` | Route response + determines whether to log `STEP_ACTIVATED` |
| `nextStepId` | `logWorkflowEvent` for `STEP_ACTIVATED` event |
| `nextStepName` | Event description (e.g., `"Step - ${nextStepName} Active"`) |
| `processCompleted` | Route response + determines whether to log `PROCESS_COMPLETED` |
| `stepCompleted` | Route response |
| `conflict` | Route error mapping (409 vs 400) |

If a field is not in this table, it should not be on the result type. Do not add fields for future WFH-006 convenience.

**How `nextStepName` is obtained:** `transitionToNextStep` currently returns `{ nextStepId }` but not `nextStepName`. The engine function should read the next step's `stepName` from `stepInstance` **inside the transaction** (using `tx`) after `transitionToNextStep` returns. This is one extra SELECT inside the tx — acceptable and scoped. Do NOT modify `transitionToNextStep`'s return type or signature — that is WFH-006/WFH-007 scope.

**This story does NOT restructure the event logging infrastructure.** It uses the existing `logWorkflowEvent` fire-and-forget pattern identically to how `complete.ts` uses it. The only improvement is that event log fields come from the engine result instead of post-tx re-reads.

### Error Mapping: Engine Result → HTTP Status

The engine function returns `{ success: false }` for all failure cases. The `conflict` flag distinguishes CAS-type failures from input validation failures. The route maps these as follows:

| Engine result | HTTP status | When |
|---|---|---|
| `{ success: false, conflict: true }` | **409 Conflict** | CAS failed: step already processed by another request, step not in `awaiting_validation` at UPDATE time, task already completed |
| `{ success: false, conflict: false/undefined }` | **400 Bad Request** | Step not found, decision references a document name that doesn't exist for this step, no documents found for step |

The engine function should set `conflict: true` **only** when a CAS `.returning()` yields 0 rows — meaning the expected precondition was met at read time but changed by UPDATE time (or the step was already processed). All other failures (missing data, invalid references) leave `conflict` unset.

This matches the existing convention in `complete.ts` where `Errors.conflict("Step or task already processed by another request")` is thrown when a CAS fails (line 476).

### Implementation Guardrails

1. **Do not modify any existing engine function** (`transitionToNextStep`, `createTasksForStep`, `completeStep`, `approveValidationTask`, `seedStepDocuments`, `returnToPreviousStep`). They are already correct. This story only adds a new engine function and rewrites the route.
2. **Do not introduce async/deferred patterns.** The transaction remains synchronous.
3. **Do not optimize `transitionToNextStep` internals** (redundant reads, parameter threading). That is WFH-007.
4. **Do not refactor event logging infrastructure.** Use the existing `logWorkflowEvent` fire-and-forget pattern. That is WFH-006.
5. **Do not change the Elysia route schema** (params, body, response type). The API contract must not break.
6. **The dynamic import of `workflowStepTemplate` in the current decline path (line 161)** should be replaced with a standard top-of-file import in the new engine function.
7. **All DB queries must include tenant isolation** (`eq(...tenantId, params.tenantId)`), consistent with existing engine functions.
