## Acceptance Criteria

1. Supabase Auth configured with email/password authentication enabled
2. Login page implemented with email and password fields, "Remember me" checkbox, and "Forgot password" link
3. Sign-up page implemented with email, password, password confirmation, and tenant name fields
4. Password validation enforces minimum 8 characters with uppercase, lowercase, and number

6. JWT tokens stored in httpOnly cookies with appropriate expiration and refresh logic
7. Logout functionality clears session and redirects to login page
8. Forgot password flow sends reset email via Supabase Auth with branded email template
9. Protected routes implemented in Remix that redirect unauthenticated users to login
10. Session management handles token refresh automatically before expiration
11. OAuth login with Google provider configured and functional (bonus: Microsoft)
12. Login/signup UI is mobile-responsive and follows Midday design system aesthetic

### Security Implementation

[Source: architecture/security-and-performance.md]

**Frontend Security:**
- CSP headers preventing XSS attacks
- No inline scripts or dangerouslySetInnerHTML
- httpOnly cookies for JWT storage
- SameSite cookie attributes for CSRF protection

**Backend Security:**
- Input validation using Zod schemas at API boundary
- Rate limiting: 10 req/min for auth endpoints  
- Parameterized queries via Drizzle ORM
- JWT verification middleware for protected routes

**Password Security:**
- Minimum 8 characters with uppercase, lowercase, number
- Password strength indicator in UI
- Secure password reset flow via email
- Supabase handles password hashing and security

### Session Management Strategy

[Source: architecture/high-level-architecture.md#architectural-patterns]

**Hybrid Session Approach:**
- **Server-side sessions**: Remix cookie-based sessions for SSR
- **Client-side state**: Zustand for reactive auth state
- **Token management**: Automatic refresh before expiration
- **Cross-tab sync**: Session state synced across browser tabs

**Cookie Configuration:**
- httpOnly: true (XSS protection)
- secure: true (HTTPS only)  
- sameSite: 'lax' (CSRF protection)
- maxAge: 7 days for refresh token

### Integration with Multi-Tenancy

[Source: architecture/high-level-architecture.md#architectural-patterns]

**Multi-tenant Auth Flow:**
1. User signs up with email + tenant name
2. System creates Supabase auth user AND tenant record (atomic transaction)
3. JWT includes tenant_id claim for RLS policies
4. All subsequent requests automatically filtered by tenant
5. User role stored in user_metadata field of Supabase auth record

**Tenant Context Extraction:**
- `requireTenant()` middleware extracts tenant from JWT
- `extractTenantContext()` validates JWT structure  
- Automatic tenant context injection into request handlers

### API Integration Pattern

[Source: architecture/tech-stack.md + high-level-architecture.md]

**Auth API Communication:**
- **Registration**: ElysiaJS endpoint creates user + tenant
- **Login/Logout**: Direct Supabase Auth SDK calls
- **Session Management**: Remix server-side sessions
- **Protected API calls**: Eden Treaty with automatic JWT inclusion

**Error Handling:**
- Standardized error responses from ElysiaJS
- User-friendly error messages in frontend
- Proper HTTP status codes (401, 403, 422, etc.)
- Error tracking via Sentry integration
