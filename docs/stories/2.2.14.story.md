## Story

**As an** admin and developer,  
**I want** template versioning removed and replaced with copy functionality,  
**so that** template management is simpler, more intuitive, and easier for users to understand.

---

## Acceptance Criteria

### Database Schema Changes (AC 1-8)

1. `form_template_version` table is dropped
2. `workflow_template_version` table is dropped
3. `form_section.form_template_version_id` FK changed to `form_section.form_template_id`
4. `workflow_step_template.workflow_template_version_id` FK changed to `workflow_step_template.workflow_template_id`
5. `form_submission.form_template_version_id` FK changed to `form_submission.form_template_id` (if exists)
6. `process_instance.workflow_template_version_id` FK changed to `process_instance.workflow_template_id` (if exists)
7. `form_template` table has `is_active` boolean field (default: true)
8. All FK constraints properly updated with correct ON DELETE behavior

### API - Copy Template Endpoints (AC 9-12)

9. `POST /api/form-templates/:id/copy` endpoint creates deep copy of form template with all sections/fields
10. `POST /api/workflow-templates/:id/copy` endpoint creates deep copy of workflow template with all steps/approvers
11. Copied templates have status='draft' and unique name (auto-appends "Copy of...")
12. Copy operations maintain tenant isolation

### API - Template Management Updates (AC 13-18)

13. `POST /api/form-templates/publish` removed or refactored to simple status toggle
14. `PATCH /api/form-templates/:id/publish` changes status from draft→published (no version creation)
15. `PATCH /api/workflow-templates/:id/publish` changes status from draft→published (no version creation)
16. All workflow-templates/versions/* endpoints removed (5 files)
17. All form template API routes updated to remove version references
18. All workflow template API routes updated to remove version references

### Frontend - Copy Template UI (AC 19-24)

19. "Copy Template" button added to Form Templates table
20. "Copy Template" button added to Workflow Templates table
21. `CopyTemplateDialog` component created for form templates
22. `CopyTemplateDialog` component created for workflow templates
23. Copy dialog allows editing name before creating copy
24. Copy action creates new draft template and navigates to edit page

### Frontend - Template Management Updates (AC 25-30)

25. `WorkflowVersionManager.tsx` component removed entirely
26. Form template builder shows draft/published status (no versions)
27. Workflow template builder shows draft/published status (no versions)
28. Template tables show status column (draft/published)
29. Publish action is simple status toggle (no version creation)
30. No version selection dropdowns anywhere in UI

### Workflow/Form Execution Updates (AC 31-34)

31. Workflow instantiation references `workflow_template_id` (not version)
32. Workflow initiation dropdown shows templates (not versions)
33. Form submission references `form_template_id` (not version)
34. Form rendering uses `form_template_id` (not version)

### TypeScript Types Updates (AC 35-38)

35. `FormTemplateVersion` interface removed from types
36. `WorkflowTemplateVersion` interface removed from types
37. All interfaces updated to reference template IDs (not version IDs)
38. Type exports updated to remove version types

### Data Migration (AC 39-42)

39. Migration script handles existing data safely (if any production data exists)
40. Latest published version data migrated to template structure
41. Historical version data exported to JSON for archival
42. No data loss during migration

### Testing (AC 43-48)

43. All database schema tests updated to reflect new model
44. All API integration tests updated to test copy functionality
45. All frontend E2E tests updated to test copy workflow
46. Copy template tests verify deep copy (sections/fields/steps/approvers)
47. Copy template tests verify tenant isolation
48. All existing tests passing with no version references

### Documentation (AC 49-52)

49. `docs/architecture/form-template-schema.md` rewritten without versioning sections

## Dev Notes

### Context: Why This Story Exists

**User Feedback:** "Form template versioning is too complicated. Workflow template versioning is too complicated. Let's just have a copy template functionality to create a new template from existing one."

**Problem:** The versioning system (Stories 2.2.2, 2.2.6) was designed for immutable version history, but users find it overly complex. They want simple template copying instead.

**Solution:** Remove versioning tables entirely, replace with "Copy Template" functionality that creates editable draft copies.

### Reference Documents

- **Sprint Change Proposal:** `docs/sprint-change-proposals/2026-03-15-epic-2.2-remove-template-versioning.md`
- **Original Versioning Stories:** 2.2.2 (Forms), 2.2.6 (Workflows)

### Data Migration Strategy

**If NO production data:**
- Simply drop version tables
- Update FK columns
- No data migration needed

**If production data EXISTS:**
1. Export all version data to JSON (archival)
2. Identify latest published version for each template
3. Copy sections/fields/steps from version to template
4. Update all process_instance and form_submission references
5. Verify no orphaned records
6. Drop version tables

### Architecture Changes

**Before (Complex):**
```
form_template (1) → form_template_version (N) → form_section (N) → form_field (N)
workflow_template (1) → workflow_template_version (N) → workflow_step_template (N)
```

**After (Simple):**
```
form_template (1) → form_section (N) → form_field (N)
workflow_template (1) → workflow_step_template (N)
```

### FK Cascade Rules (Updated)

| Foreign Key | Old Behavior | New Behavior |
|-------------|-------------|--------------|
| `form_section.form_template_id` | N/A (was version) | CASCADE (delete sections with template) |
| `workflow_step_template.workflow_template_id` | N/A (was version) | RESTRICT (prevent delete if used in processes) |
| `form_submission.form_template_id` | N/A (was version) | RESTRICT (preserve audit trail) |
| `process_instance.workflow_template_id` | N/A (was version) | RESTRICT (preserve audit trail) |

### New User Workflows

**Copy Form Template:**
1. User navigates to Form Templates page
2. Clicks "Copy" button on any template
3. Dialog opens with name pre-filled: "Copy of {original name}"
4. User can edit name
5. Clicks "Copy"
6. New draft template created
7. User redirected to edit page

**Copy Workflow Template:**
1. User navigates to Workflow Templates page
2. Clicks "Copy" button on any template
3. Dialog opens with name and description fields
4. User edits as needed
5. Clicks "Copy"
6. New draft template created with all steps/approvers
7. User redirected to edit page

### Testing Focus Areas

**Critical Tests:**
1. Copy creates true deep copy (not references)
2. Tenant isolation maintained during copy
3. FK constraints prevent orphaned records
4. Published templates cannot be edited (only copied)
5. Draft templates can be edited freely
6. Workflow instantiation works with simplified model
7. Form submission works with simplified model

### Performance Considerations

**Query Optimization:**
- Removing version joins simplifies queries
- Direct template→sections joins are more efficient
- Indexes remain tenant-first for multi-tenancy

**Copy Performance:**
- Deep copy requires transaction
- Expect 100-500ms for typical template
- Large templates (50+ fields/steps) may take 1-2s

### Rollback Strategy

**If deployment fails:**
1. Revert code to `pre-versioning-removal` tag
2. Restore database from pre-migration backup
3. Verify all services running
4. Communicate issue to users
5. Schedule retry after fixing issues
