## Fixes Applied

### ✅ DATA-001 - RESOLVED

**Solution:** Created comprehensive assigned-reviewer endpoint and integrated into data flow

**Implementation Details:**

1. **New API Endpoint Created:**
   - File: `apps/api/src/routes/workflows/assigned-reviewer.ts`
   - Route: `GET /api/workflows/:id/assigned-reviewer`
   - Returns: `{ reviewer: { id, fullName, email } }`

2. **Reviewer Assignment Logic:**
   - Implements proper fallback hierarchy:
     1. Check `tenant.settings.workflowReviewers.stage1`
     2. Fallback to first active Procurement Manager
     3. Fallback to first active Admin
     4. Return error if no reviewers available
   - Ensures tenant isolation and active user filtering

3. **Route Registration:**
   - Updated: `apps/api/src/routes/workflows/index.ts`
   - Added: `.use(assignedReviewerRoute)`

4. **Loader Integration:**
   - Updated: `apps/web/app/routes/_app.workflows.$id.tsx` (lines 73-85)
   - Added parallel fetch: `assignedReviewerResponse`
   - Passed to component: `assignedReviewer: assignedReviewerResponse.data?.reviewer || null`

5. **Component Props Flow:**
   - `_app.workflows.$id.tsx` → `WorkflowDetailPage` → `SubmitWorkflowModal`
   - Modal now displays correct reviewer name before submission

**Verification:**
- ✅ Endpoint responds correctly with reviewer data
- ✅ Loader fetches data in parallel (no performance impact)
- ✅ Modal displays reviewer name (AC 3 satisfied)
- ✅ Graceful handling when reviewer is null
