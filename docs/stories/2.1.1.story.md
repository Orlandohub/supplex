## Dev Notes

### Context from Epic 2.1

This story is the foundation for Epic 2.1 (Bug Fixing and Course Correction). The current database schema uses naming that doesn't align with the actual business process:

- **Current:** `document_checklists` table stores "templates" for required documents
- **Business Reality:** These are qualification templates that define document requirements
- **Current:** `qualification_workflows` table tracks the qualification process
- **Business Reality:** This is the actual qualification process instance

This misalignment causes confusion in code and communication. Renaming these tables will make the codebase self-documenting and prepare for subsequent bug fixes in Epic 2.1.

### Previous Story Insights

[From Story 2.10 - Audit Trail & History View]

- **Database Migrations:** Previous migrations have been successful with Drizzle. Use `pnpm db:generate` to create migrations, but review carefully as Drizzle may not auto-detect table renames.
- **Schema Changes:** When renaming tables, ensure all related indexes, foreign keys, and constraints are updated in the migration file.
- **Testing After Schema Changes:** Always run full test suite after schema changes to catch any missed references.
- **Foreign Key Cascades:** Verify that CASCADE delete behavior is preserved in renamed tables to maintain data integrity.

### Database Schema Changes

[Source: packages/db/src/schema/document-checklists.ts, qualification-workflows.ts]

**Current Tables:**

1. **document_checklists** (to be renamed to **qualification_templates**)
   - Fields: id, tenant_id, template_name, required_documents (JSONB), is_default, created_at, updated_at, deleted_at
   - Index: `idx_document_checklists_tenant_default` on (tenant_id, is_default)
   - Foreign Key: tenant_id в†’ tenants.id (CASCADE)

2. **qualification_workflows** (to be renamed to **qualification_process**)
   - Fields: id, tenant_id, supplier_id, status, initiated_by, initiated_date, current_stage, risk_score, snapshotted_checklist (JSONB), created_at, updated_at, deleted_at
   - Indexes: 
     - `idx_qualification_workflows_tenant_supplier` on (tenant_id, supplier_id)
     - `idx_qualification_workflows_tenant_status` on (tenant_id, status)
   - Foreign Keys:
     - tenant_id в†’ tenants.id (CASCADE)
     - supplier_id в†’ suppliers.id (CASCADE)
     - initiated_by в†’ users.id (RESTRICT)

**Tables with Foreign Key References:**

### Migration Strategy

[Source: docs/architecture/database-schema.md, docs/architecture/tech-stack.md]

**Tech Stack:**
- **ORM:** Drizzle 0.30+ with migration system
- **Database:** PostgreSQL 15+ on Supabase
- **Migration Tool:** Drizzle Kit (`pnpm db:generate`, `pnpm db:push`)

**Migration Approach:**

1. **Drizzle Auto-Generation Limitation:** Drizzle may not automatically detect table renames and might generate DROP + CREATE statements instead of RENAME. This would lose data.

2. **Manual Migration Required:** Create a manual migration file with explicit RENAME statements:

```sql
-- Migration: Rename qualification tables for business alignment
-- Date: 2025-12-17

-- Rename document_checklists to qualification_templates
ALTER TABLE document_checklists RENAME TO qualification_templates;

-- Rename qualification_workflows to qualification_process
ALTER TABLE qualification_workflows RENAME TO qualification_process;

-- Rename indexes for qualification_templates
ALTER INDEX idx_document_checklists_tenant_default 
  RENAME TO idx_qualification_templates_tenant_default;

-- Rename indexes for qualification_process
ALTER INDEX idx_qualification_workflows_tenant_supplier 
  RENAME TO idx_qualification_process_tenant_supplier;
ALTER INDEX idx_qualification_workflows_tenant_status 
  RENAME TO idx_qualification_process_tenant_status;

-- Foreign key constraints are automatically updated by PostgreSQL
-- when using RENAME, but verify after migration
```

3. **Verification Steps:**
   - Check that foreign key constraints still reference correct tables
   - Verify indexes exist on renamed tables
   - Test queries to ensure they work with new table names
   - Check application logs for any errors after deployment

### Coding Standards

[Source: docs/architecture/coding-standards.md]

**Naming Conventions:**
- Database Tables: snake_case (`qualification_templates`, `qualification_process`)
- TypeScript Variables: camelCase (`qualificationTemplates`, `qualificationProcess`)
- TypeScript Types: PascalCase (`InsertQualificationTemplate`, `SelectQualificationProcess`)

**Critical Rules:**
- **Type Sharing:** Update exports in `packages/types` if shared types are affected
- **Drizzle ORM:** Use parameterized queries - never string concatenation
- **Tenant Isolation:** Ensure all queries maintain tenant filtering (unchanged by rename)
- **Migration Safety:** Always use RENAME statements, never DROP + CREATE for data preservation

### Risk Assessment

**Low Risk Factors:**
- No functional behavior changes (pure refactoring)
- No API contract changes (URLs, request/response formats unchanged)
- No UI changes needed (frontend uses type names, not table names)
- Database RENAME operations are atomic in PostgreSQL

**Mitigation Strategies:**
- Thorough testing of all API endpoints post-migration
- Database backup before migration (local and staging)
- Gradual rollout: test locally в†’ staging в†’ production
- Rollback plan: Create reverse migration if needed
