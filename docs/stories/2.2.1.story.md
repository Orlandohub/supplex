## Story

**As a** developer,  
**I want** to refactor the database schema to support a reusable, tenant-isolated workflow engine,  
**so that** dynamic forms and workflows are not constrained by qualification-only tables.

---

## Acceptance Criteria

1. The following existing tables are not extended with new functionality:
   - qualification_process
   - qualification_stages
   - workflow_documents
   - workflow_events
   - qualification_templates

2. A new, domain-agnostic execution model is introduced using:
   - process_instance
   - step_instance

3. All new workflow execution logic is implemented only using the new tables

4. Existing qualification tables are treated as legacy and remain unchanged

5. No data migration from legacy tables to the new engine is required

6. All new engine tables include tenant_id and enforce tenant isolation

7. New foreign keys, events, and runtime data must reference the new engine tables only

**Table 1: process_instance**
```sql
CREATE TABLE process_instance (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    process_type VARCHAR(100) NOT NULL,
    entity_type VARCHAR(100) NOT NULL,
    entity_id UUID NOT NULL,
    status VARCHAR(50) NOT NULL DEFAULT 'active',
    initiated_by UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    initiated_date TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    completed_date TIMESTAMP WITH TIME ZONE,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMP WITH TIME ZONE
);

CREATE INDEX idx_process_instance_tenant_type_status 
    ON process_instance(tenant_id, process_type, status) 
    WHERE deleted_at IS NULL;

CREATE INDEX idx_process_instance_tenant_entity 
    ON process_instance(tenant_id, entity_type, entity_id) 
    WHERE deleted_at IS NULL;
```

**Table 2: step_instance**
```sql
CREATE TABLE step_instance (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    process_instance_id UUID NOT NULL REFERENCES process_instance(id) ON DELETE CASCADE,
    step_order INTEGER NOT NULL,
    step_name VARCHAR(200) NOT NULL,
    step_type VARCHAR(50) NOT NULL,
    status VARCHAR(50) NOT NULL DEFAULT 'pending',
    assigned_to UUID REFERENCES users(id) ON DELETE RESTRICT,
    completed_by UUID REFERENCES users(id) ON DELETE RESTRICT,
    completed_date TIMESTAMP WITH TIME ZONE,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMP WITH TIME ZONE
);

CREATE INDEX idx_step_instance_process_order 
    ON step_instance(process_instance_id, step_order) 
    WHERE deleted_at IS NULL;

CREATE INDEX idx_step_instance_tenant_assigned_status 
    ON step_instance(tenant_id, assigned_to, status) 
    WHERE status IN ('pending', 'active') AND deleted_at IS NULL;
```

**Tenant Isolation Strategy:**
- Both tables include `tenant_id` with CASCADE delete when tenant is removed
- All queries must filter by `tenant_id` (enforced in application layer)
- Future stories will add Row Level Security (RLS) policies for database-level enforcement
- Indexes include `tenant_id` as first column for optimal query performance

**Foreign Key Cascade Rules:**
- `tenant_id` → CASCADE: When tenant deleted, all processes and steps deleted
- `process_instance_id` → CASCADE: When process deleted, all steps deleted
- `initiated_by`, `assigned_to`, `completed_by` → RESTRICT: Cannot delete user who participated in workflow (audit trail preservation)

### Technical Constraints

[Source: architecture/coding-standards.md, architecture/tech-stack.md]

**Drizzle ORM Patterns:**
- **Table Definition:** Use `pgTable` from `drizzle-orm/pg-core`
- **Column Types:** 
  - `uuid()` for all IDs and foreign keys
  - `varchar()` with explicit length for enum-like fields
  - `timestamp()` with `{ withTimezone: true, mode: "date" }` for all dates
  - `jsonb()` for flexible metadata fields
  - `integer()` for ordering/counting
- **Indexes:** Use composite indexes starting with `tenant_id` for multi-tenant queries
- **Partial Indexes:** Use `.where()` clause for conditional indexes (e.g., deleted_at IS NULL)
- **Relations:** Define using `relations()` from Drizzle ORM for type-safe joins
- **Naming:** 
  - Tables: snake_case (e.g., `process_instance`)
  - Schema files: kebab-case (e.g., `process-instance.ts`)
  - TypeScript interfaces: PascalCase (e.g., `ProcessInstance`)

**Drizzle Relations Example:**
```typescript
// Example relations to define in schema files
import { relations } from "drizzle-orm";

export const processInstanceRelations = relations(processInstance, ({ one, many }) => ({
```

### Security Review

**Tenant Isolation: EXCELLENT**
- All queries will be filtered by tenant_id (enforced by schema design)
- CASCADE delete when tenant removed prevents orphaned data
- RESTRICT delete on user FKs preserves audit trail (cannot delete users who initiated/completed workflows)
- Soft deletes (deleted_at) maintain historical data for compliance

**Audit Trail: EXCELLENT**
- initiated_by, assigned_to, completed_by FKs use RESTRICT delete
- Timestamps (created_at, updated_at, completed_date) track all state changes
- Soft deletes prevent permanent data loss

**No Security Concerns Identified**

### Performance Considerations

**Index Strategy: EXCELLENT**
- All composite indexes start with tenant_id for optimal multi-tenant queries
- Partial indexes exclude soft-deleted records (reduces index size, improves performance)
- Conditional partial index on step_instance (WHERE status IN ('pending', 'active')) for efficient task lists
- Index on (process_instance_id, step_order) supports sequential step retrieval

**Query Performance: OPTIMIZED**
- Tenant-based process filtering: Uses idx_process_instance_tenant_type_status
- User task lists: Uses idx_step_instance_tenant_assigned_status
- Entity-based lookups: Uses idx_process_instance_tenant_entity
- Sequential steps: Uses idx_step_instance_process_order

**JSONB Metadata:**
- Flexible without sacrificing performance
- Can add GIN indexes later if metadata queries become frequent

**No Performance Concerns Identified**
