# Story WFH-007: Transition Query Optimization

**Approved**

## Story

**As a** platform operator,
**I want** the step transition function to avoid redundant database reads,
**so that** each workflow step transition uses 2 fewer queries, reducing latency and DB load during every workflow progression.

---

## Context

`transitionToNextStep()` is the core function called every time a workflow moves from one step to the next. It is called by `completeStep`, `approveValidationTask`, and (after WFH-001) the document review engine function. Every workflow step completion triggers exactly one call to `transitionToNextStep`.

The function currently executes **2 redundant queries** on every invocation:

### Redundancy 1: Separate Current Step + All Steps Reads

```
Query 1 (line 38-46):  SELECT * FROM stepInstance WHERE id = :currentStepInstanceId AND tenantId = :tenantId
Query 2 (line 52-61):  SELECT * FROM stepInstance WHERE processInstanceId = :processId AND tenantId = :tenantId ORDER BY stepOrder
```

The current step (Query 1) is already contained in the "all steps" result (Query 2). The only reason for Query 1 is to get `currentStep.stepOrder` so the function can find the next step. After Query 2 runs, the current step is immediately available via `allSteps.find(s => s.id === currentStepInstanceId)`.

**Fix:** Replace the two queries with a single query. Find the current step within the result.

### Redundancy 2: Process Re-read for `workflowTemplateId`

```
Line 73-81:  UPDATE processInstance SET currentStepInstanceId = :nextStepId, completedSteps = completedSteps + 1 ...
Line 83-86:  SELECT * FROM processInstance WHERE id = :processId   ← immediately re-reads the row just written
Line 92:     const workflowTemplateId = process.workflowTemplateId  ← the only field used from the re-read
```

`workflowTemplateId` is immutable — it is set when the process is created and never updated. Both callers already have this value available before calling `transitionToNextStep`:
- **`completeStep`** reads the process at line 148-151 and has `process.workflowTemplateId` at line 229.
- **`approveValidationTask`** reads the process at line 111-114 and has `process.workflowTemplateId`.
- **WFH-001 engine function** (post-merge) reads the process as part of its setup and has the template ID.

**Fix:** Accept `workflowTemplateId` as a parameter. Remove the process re-read entirely.

### Architecture Analysis Findings Addressed

| ID | Finding | Severity | Category |
|----|---------|----------|----------|
| **P4** | `transitionToNextStep` re-reads `processInstance` immediately after writing it; reads current step separately from all steps | **Low** | Performance |

### Dependencies

- **Depends on WFH-001** (merged). WFH-001 creates a new engine function for document review that calls `transitionToNextStep`. The signature change in this story affects all callers — WFH-001's new engine function must also be updated.
- **Compatible with WFH-006 in any merge order.** WFH-006 adds `nextStepName` to the return type of `transitionToNextStep`. This story only changes the input signature, internal query structure, and caller updates — it does not modify the return type. The two changes compose cleanly regardless of merge order. See the "Return Type Preservation Contract" section in Dev Notes for the explicit rules the dev agent must follow.

## Acceptance Criteria

### AC 1: `workflowTemplateId` Passed as Parameter

1. `transitionToNextStep` accepts a new required parameter `workflowTemplateId: string` after `tenantId`:

   ```typescript
   export async function transitionToNextStep(
     tx: DbOrTx,
     currentStepInstanceId: string,
     processId: string,
     tenantId: string,
     workflowTemplateId: string,
     log?: Logger
   )
   ```

2. The function uses this parameter directly where it previously used the value from the process re-read.
3. The `SELECT processInstance` re-read (current lines 83-86) is removed entirely.

### AC 2: Current Step and All Steps Merged Into Single Query

4. The two separate `SELECT stepInstance` queries (current lines 38-46 and 52-61) are replaced by a single query that fetches all step instances for the process, ordered by `stepOrder`.
5. The current step is found within the result using `allSteps.find(s => s.id === currentStepInstanceId)`.
6. The "step not found" guard is preserved: if the current step is not in the result, the function throws the same error as before.

### AC 3: All Callers Updated

7. `completeStep` passes `process.workflowTemplateId` when calling `transitionToNextStep`.
8. `approveValidationTask` passes `process.workflowTemplateId` when calling `transitionToNextStep`.
9. If WFH-001 has been merged: the document review engine function passes `workflowTemplateId` when calling `transitionToNextStep`.

### AC 4: Existing Behavior and Return Type Preserved

10. The function produces identical results for all inputs: same return values, same DB mutations, same error conditions.
11. The next-step activation, process completion, supplier status mapping, task creation, and document seeding behaviors are unchanged.
12. Tenant isolation is preserved in the merged query (the `WHERE` clause includes `tenantId`).
13. The return type of `transitionToNextStep` at implementation time is preserved exactly. This story must not add, remove, or rename any return fields. If WFH-006 has already been merged and the return type includes `nextStepName`, that field remains part of the return contract and must continue to be populated correctly.

### AC 5: Tests Updated

14. All existing test call sites for `transitionToNextStep` are updated to pass `workflowTemplateId`.
15. All existing tests pass with no changes to assertions (behavior is identical).
16. At least one new assertion verifies the preserved behavior for a falsy `workflowTemplateId` parameter: the function throws (matching the existing guard at current lines 94-98, which now checks the parameter instead of a re-read value).
17. At least one new assertion verifies the preserved behavior for a valid but non-matching `workflowTemplateId` (UUID that produces no step template match): the function does NOT throw — it returns a normal success result (`{ currentStepCompleted: true, nextStepActivated: true, ... }`) with task creation and document seeding silently skipped (matching the existing warning path at current lines 130-133).

## Dev Notes

### Query Flow: Before and After

**Before (next step exists path):**
```
1. SELECT stepInstance WHERE id = :currentStepId                     ← current step
2. SELECT stepInstance WHERE processInstanceId = :processId           ← all steps (includes current step)
3. UPDATE stepInstance SET status = 'active' WHERE id = :nextStepId
4. UPDATE processInstance SET currentStepInstanceId, completedSteps
5. SELECT processInstance WHERE id = :processId                      ← RE-READ for workflowTemplateId
6. SELECT workflowStepTemplate WHERE workflowTemplateId + stepOrder
7. createTasksForStep()  → 2-4 queries
8. [if document step] seedStepDocuments() → 3-4 queries
Total: 8-14 queries
```

**After (next step exists path):**
```
1. SELECT stepInstance WHERE processInstanceId = :processId           ← all steps (find current in result)
2. UPDATE stepInstance SET status = 'active' WHERE id = :nextStepId
3. UPDATE processInstance SET currentStepInstanceId, completedSteps
4. SELECT workflowStepTemplate WHERE workflowTemplateId + stepOrder  ← uses parameter directly
5. createTasksForStep()  → 2-4 queries
6. [if document step] seedStepDocuments() → 3-4 queries
Total: 6-12 queries
```

**Net savings: 2 queries per transition (always).**

### Process Completion Path Is Unaffected

The process-completion branch (lines 141-190 in the current code) does not use the local `workflowTemplateId` variable. It has its own 4-table join query (lines 152-167) that reads `processInstance.workflowTemplateId` directly from the join:

```typescript
.innerJoin(workflowTemplate, eq(workflowTemplate.id, processInstance.workflowTemplateId))
```

This join is correct and independent — it is not affected by the parameter change because it reads `processInstance.workflowTemplateId` from the DB row within the join, not from a local variable.

### `workflowTemplateId` Immutability Justification

`workflowTemplateId` on `processInstance` is set during `instantiateWorkflow` and is never updated by any mutation path. The entire workflow engine relies on this being stable — step templates, task creation, and document seeding all resolve relative to it. Passing it as a parameter instead of re-reading is safe because the value cannot change between the caller's read and the `transitionToNextStep` call (both are inside the same transaction).

### The `!` Non-null Assertion on `process.workflowTemplateId`

Both `completeStep` and `approveValidationTask` read the process and use `process.workflowTemplateId` before calling `transitionToNextStep`. The value is nullable in the schema (historical reason) but is always present for processes created via `instantiateWorkflow`. The non-null assertion `process.workflowTemplateId!` matches the existing assumption in the codebase (e.g., `completeStep` line 229 uses it without a null check).

If the dev agent prefers safety, they can add an explicit guard:

```typescript
if (!process.workflowTemplateId) {
  return { success: false, error: "workflowTemplateId missing on process" };
}
```

This is optional — the existing code already assumes it is present.

### Return Type Preservation Contract

This story and WFH-006 both modify `transitionToNextStep` but for different reasons. WFH-006 expands the return type (adds `nextStepName`). This story changes the input signature and internal queries. The rule for the dev agent implementing WFH-007 is:

**Preserve the current effective return type of `transitionToNextStep` at implementation time.** Concretely:

- **If WFH-006 has already been merged:** The return type includes `nextStepName`. WFH-007 must preserve it. The value is still available — `nextStep.stepName` comes from the same `allSteps.find(...)` result used after the merged query. No return type change needed.
- **If WFH-006 has NOT been merged:** The return type is `{ currentStepCompleted, nextStepActivated, nextStepId?, processCompleted }`. WFH-007 must preserve it exactly. Do not pre-add `nextStepName` or any other field — that is WFH-006 scope.
- **If both are in-flight simultaneously:** Each story touches different surfaces (WFH-006: return type + route rewrites; WFH-007: input signature + internal queries). Merge conflicts will be limited to the call sites (adding the parameter to the same call that adds `nextStepName` to the return destructuring).

**WFH-007 must not add any return fields under any circumstances.** The story's scope is exclusively: input signature, internal query consolidation, caller updates.

### Implementation Guardrails

1. **Preserve the current effective return type exactly.** This story only changes the input signature, internal query structure, and caller updates. The return type of `transitionToNextStep` must match whatever exists at implementation time — do not add, remove, or rename any return fields. If WFH-006 has already been merged and `nextStepName` is on the return type, it must remain and be correctly populated from the merged query result.
2. **Do not change the function's external behavior.** Same return values, same mutations, same error messages, same logging. Only the internal query count changes.
3. **Do not remove the `workflowTemplateId` null guard.** The guard that throws when `workflowTemplateId` is falsy should be preserved — it now checks the parameter instead of the re-read value.
4. **Do not modify the process-completion branch's status mapping join.** That join reads `processInstance.workflowTemplateId` from the DB within the join itself. It is correct and independent of the parameter.
5. **Preserve tenant isolation in the merged query.** The single `SELECT stepInstance` must include `eq(stepInstance.tenantId, tenantId)` in its `WHERE` clause.
6. **Do not change `createTasksForStep`, `seedStepDocuments`, or any other engine function.** Only `transitionToNextStep` internals and its callers are modified.
