## Story

**As a** platform operator,
**I want** the form submission route (`submit.ts`) to execute the form status update and workflow step completion inside a single database transaction, with task assignment verification,
**so that** a failure during step completion cannot leave a form permanently stuck in `submitted` status with its workflow step frozen in `active` — a dead state with no retry path.

---

## Context

The form submission route (`POST /api/form-submissions/:submissionId/submit`) is the primary path users hit from the UI when submitting a dynamic form linked to a workflow step. It has two correctness problems:

**Problem 1 — No transaction wrapping.** The route commits the form status to `submitted` (line 146-154) as an independent write, then calls `completeStep(db, ...)` (line 164) with the raw `db` handle — not inside a `db.transaction()`. If `completeStep` fails (network error, Supabase timeout, constraint violation), the error is caught and swallowed (lines 218-220). The form remains `submitted`, but the step is stuck in `active`. This is a **dead state**: the `ALREADY_SUBMITTED` guard (line 62-67) blocks any retry, and there is no manual path to re-trigger `completeStep`.

In contrast, `complete.ts` (action=submit) wraps `completeStep(tx, ...)` in a proper `db.transaction()` (lines 57-64). The same logical operation — "complete a form step" — has two code paths with different safety guarantees. The form-submissions route is the **more common and less safe** path.

**Problem 2 — Missing `verifyTaskAssignment`.** The route only checks that the user is the original form submitter (`eq(formSubmission.submittedBy, userId)` at line 48). It does not verify that the user has a pending task assignment for the step. A user who created a draft but had their task reassigned could still submit. `complete.ts` enforces `verifyTaskAssignment(user, stepInstanceId, ["action", "resubmission"], db)` (lines 46-54) before allowing submission.

### Architecture Analysis Findings Addressed

| ID | Finding | Severity | Category |
|----|---------|----------|----------|
| **C2** | `submit.ts` commits form status independently from `completeStep()`; `completeStep` called with `db` not `tx`; error caught and swallowed — creates unrecoverable dead state | **High** | Correctness |

## Acceptance Criteria

### AC 1: Form Status Update and Step Completion in a Single Transaction

1. When a form submission is linked to a workflow step (`stepInstanceId` is not null), the form status update to `submitted` and the `completeStep()` call are wrapped in a single `db.transaction(async (tx) => { ... })`.
2. `completeStep` receives `tx`, not `db`.
3. If `completeStep()` fails, the entire transaction rolls back — the form stays as `draft`, not `submitted`.
4. The dead state (form `submitted` + step `active` with no retry path) is eliminated.

### AC 2: Standalone Form Submissions Still Work

5. When a form submission is NOT linked to a workflow step (`stepInstanceId` is null), the form status update to `submitted` is still executed (in its own implicit transaction — no explicit wrapping needed since it's a single write).
6. No behavioral change for standalone forms.

### AC 3: Task Assignment Verified Before Submission

7. When the form is linked to a workflow step, `verifyTaskAssignment(user, stepInstanceId, ["action", "resubmission"], db)` is called before the transaction.
8. If the user does not have a pending task of type `action` or `resubmission` for the step, the request is rejected with 403 Forbidden.
9. This check runs **outside** the transaction (it is a read-only authorization check).
10. The existing `submittedBy` ownership check is preserved (the form must belong to the user).

### AC 4: Error-Swallowing Eliminated

11. The try/catch block that currently swallows `completeStep` errors (lines 218-220 of `submit.ts`) is removed.
12. If `completeStep()` returns `{ success: false }`, the transaction is rolled back (form stays `draft`) and the route returns an appropriate error response.
13. If `completeStep()` returns `{ success: false }` due to a CAS failure (step not in `active` state), the route returns 409 Conflict.
14. If `completeStep()` returns `{ success: false }` for other reasons, the route returns 400 Bad Request.

### AC 5: Event Logging Unchanged

15. Event logging (`logWorkflowEvent`) remains **outside** the transaction (fire-and-forget), consistent with all other routes.
16. The same events are logged as before: `FORM_SUBMITTED` / `FORM_RESUBMITTED`, `PROCESS_COMPLETED`, `STEP_ACTIVATED`.
17. Event log context reads (step name, next step name) remain outside the transaction, consistent with the existing pattern in `complete.ts`.

### AC 6: Existing Behavior Preserved

18. The API response shape is unchanged (or backward-compatible).
19. Form validation (required fields, answer format validation) still runs before any state mutation.
20. The `ALREADY_SUBMITTED` guard still rejects double submissions with 400.
21. The Elysia route schema is unchanged.
22. All existing tests continue to pass.

### AC 7: Route-Level Transaction Wiring Tested

23. At least one database-backed route-level test exercises the real `POST /:submissionId/submit` route through Elysia, proving the submit route is actually wired correctly (transaction wrapping, `verifyTaskAssignment`, error mapping).
24. Route-level test covers: workflow-linked form submit → 200 response, form `submitted` + step completed atomically in DB.
25. Route-level test covers: `completeStep` CAS fails (step not active) → transaction rolls back → form stays `draft` in DB → response is 409 Conflict.
26. Route-level test covers: `verifyTaskAssignment` fails (no pending task for user) → 403 Forbidden, form stays `draft`.
27. Route-level test covers: standalone form (no `stepInstanceId`) → 200, form `submitted`, no step completion attempted.

      .update(formSubmission)
      .set({
        status: "submitted",

  });

    .returning();

## Dev Notes

### What the Route Structure Should Look Like After

```
1. Auth check (user.id, user.tenantId)
2. Read form submission (verify ownership, tenant, not-deleted)
3. ALREADY_SUBMITTED guard (400 if already submitted)
4. verifyTaskAssignment (403 if no pending task — only for workflow-linked forms)
5. Read fields + answers (pre-validation)
6. Validate required fields + answer formats (400 if invalid)
7. IF workflow-linked:
     db.transaction(tx => {
       tx.update formSubmission → submitted
       completeStep(tx, ...)
       if !success → throw (triggers rollback)
     })
   ELSE:
     db.update formSubmission → submitted
8. Event logging (fire-and-forget, outside tx)
9. Return response
```

Steps 1-6 are all read-only — they run outside the transaction. Step 7 is the only mutation block. Step 8 is fire-and-forget side effects.

### Why the Dead State Occurs Today

Current code flow (simplified):

```
form.status = 'submitted'  ← COMMITTED (independent write, line 146-154)
try {
  completeStep(db, ...)     ← NOT in a transaction (line 164)
} catch (err) {
  log.error(err)            ← ERROR SWALLOWED (lines 218-220)
}
return { submission, stepCompleted: false }  ← USER SEES SUCCESS
```

If `completeStep` fails:
- The form is permanently `submitted` (the `ALREADY_SUBMITTED` guard at line 62-67 blocks any retry)
- The step is stuck in `active` (no one can re-trigger `completeStep` for it)
- The user sees a successful response (the error was swallowed)
- The `workflow-health` endpoint does NOT detect this (the step is legitimately `active`, just orphaned)

After this story:

```
db.transaction(tx => {
  tx.update form.status = 'submitted'
  completeStep(tx, ...)
  if (!success) throw → ROLLBACK (form stays draft, user can retry)
})
```

### `completeStep` — Already Works Correctly with `tx`

`completeStep` accepts `tx: DbOrTx` as its first parameter and uses `tx` for all operations. It was hardened in Story 2.2.19. The only problem was that `submit.ts` passed `db` instead of `tx`. Passing `tx` from the transaction context is the entire fix for atomicity — no changes to `completeStep` are needed.

### `completeStep` Returns `{ success: false }`, Does Not Throw

`completeStep` handles its own errors internally and returns `{ success: false, error: string }` instead of throwing. This means inside the transaction, the caller must check the result and explicitly throw to trigger rollback:

```typescript
const stepResult = await completeStep(tx, params);
if (!stepResult.success) {
  throw new Error(stepResult.error || "Step completion failed");
}
```

The thrown error propagates out of `db.transaction()`, which catches it, rolls back, and re-throws. The route's outer catch block then maps it to the appropriate HTTP response.

### Error Mapping: completeStep Failure → HTTP Status

| `completeStep` failure reason | HTTP status | Detection |
|---|---|---|
| CAS failed: step not in `active` state | **409 Conflict** | `stepResult.error` contains "not in active state" or "already processed" |
| Process not found | **400 Bad Request** | `stepResult.error` contains "Process instance not found" |
| Required documents missing | **400 Bad Request** | `stepResult.error` contains "required document(s)" |
| Other / unexpected | **500 Internal** | Falls through to outer route catch |

The dev agent should check `stepResult.error` string content to distinguish CAS conflicts from other failures. Alternatively, use a simple heuristic: if the error mentions "active state" or "already processed", it's a 409; everything else from completeStep is a 400.

### `verifyTaskAssignment` — What It Does

`verifyTaskAssignment(user, stepInstanceId, requiredTaskTypes, db)` checks for a pending `taskInstance` that is either:
- Directly assigned to the user (`assigneeUserId = user.id`), OR
- Assigned by role (`assigneeType = 'role'`, `assigneeRole = user.role`, `assigneeUserId IS NULL`)

It returns `{ allowed: boolean, taskId?: string }`. This is the same function used by `complete.ts` (line 46-54) and `review.ts` (line 56-59). Adding it to `submit.ts` closes the authorization gap where a user who created a draft could still submit even after their task was reassigned.

### The `submittedBy` Check Is Not Redundant

The existing `eq(formSubmission.submittedBy, userId)` check (line 48) verifies that the user who created the form draft is the one submitting it. `verifyTaskAssignment` verifies that the user has an active workflow task for the step. Both are needed:

- A user could have a task for the step but not be the form draft creator (e.g., task reassigned after draft creation by someone else — edge case but possible).
- A user could be the form draft creator but no longer have a task for the step (e.g., task reassigned away).

Both checks should pass for the submission to proceed.

### Event Logging Reads: Why They Stay Outside the Transaction

The step context reads for event logging (step name, step order, next step name) currently happen outside any transaction in `submit.ts`. The `complete.ts` submit path does the same (lines 71-167). Moving these reads inside the transaction would be a correctness improvement (they'd see guaranteed-consistent data), but it would also widen the transaction scope for a non-mutation concern.

This story preserves the existing pattern: event logging reads are outside the tx. The architecture analysis explicitly categorizes post-transaction event logging overhead as a performance concern (P1), not a correctness concern. That optimization is WFH-006 scope.

### Implementation Guardrails

1. **Do not modify `completeStep` or any other engine function.** They are already correct. This story only modifies the route.
2. **Do not move form validation inside the transaction.** Required field checks and answer format validation are read-only and should stay outside the transaction to minimize transaction duration.
3. **Do not introduce async/deferred patterns.** The transaction remains synchronous.
4. **Do not restructure event logging.** Keep the existing fire-and-forget pattern with post-tx reads. That is WFH-006 scope.
5. **Do not change the Elysia route schema.** The API contract must not break.
6. **Preserve the `ALREADY_SUBMITTED` guard.** It still serves a purpose: preventing accidental double-clicks that send two requests before the first completes. The CAS in `completeStep` is the authoritative guard, but the early 400 gives a cleaner error message.
7. **All DB queries must include tenant isolation**, consistent with the existing code.

### Testing

- **Test runner:** Bun (`bun test`)
- **Test location:** `apps/api/src/routes/form-submissions/__tests__/submit-transaction.test.ts`
- **Test approach: route-level integration.** These tests exercise the real `submitRoute` through Elysia's `app.handle()`, with real DB fixtures. This is the only way to prove that the route wiring — transaction boundaries, `verifyTaskAssignment` placement, error mapping — is correct in practice.
- **Why route-level, not engine-level:** The risk in this story is not whether `completeStep(tx, ...)` works inside a transaction (it already does — tested in `auto-validation.test.ts`). The risk is whether `submit.ts` actually wraps the form update and `completeStep` in the same transaction, whether `verifyTaskAssignment` runs before the mutation, and whether CAS failures map to 409. Only route-level tests prove this.
- **Auth bypass:** The test app uses `.derive(() => ({ user: testUser, requestLogger: ... }))` to inject a real DB-backed user, bypassing the JWT `authenticate` middleware. This is the same pattern used by the existing `submit.test.ts` (line 19-21) and is appropriate for integration tests that focus on route logic, not auth middleware.
- **DB fixture pattern:** Same as `auto-validation.test.ts` — insert real tenant/user/template/process/step/task/form rows in `beforeAll`, clean up by deleting the test tenant in `afterAll` (cascade).
- **For the rollback test (Task 5.3):** Before the POST request, directly UPDATE the step to `status = 'completed'` in the DB. The route's transaction will call `completeStep(tx)`, the CAS will fail (step not in `active`), the thrown error rolls back the transaction, and the DB query after the request confirms the form is still `draft`.
- **For the task assignment test (Task 5.4):** Create a second user (different from the task assignee) and inject it into a separate test app instance. The route will call `verifyTaskAssignment`, which will find no matching pending task, and return 403 before any mutation.
