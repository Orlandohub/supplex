# Story 2.2.20: Workflow Engine Security — Entity-Level Authorization & Upload Hardening

<!-- Powered by BMAD™ Core -->

## Status

**Done**

## Story

**As a** platform operator,
**I want** all workflow, document, and supplier API endpoints to enforce entity-level authorization beyond tenant matching,
**so that** supplier users cannot access other suppliers' data, internal workflow details are hidden from external users, and uploaded files are validated at the byte level before storage.

---

## Context

The architecture assessment (`.cursor/plans/architecture_assessment_report_5c337427.plan.md`) identified that while tenant-level isolation works (all queries filter by `tenantId`), **within-tenant authorization is largely missing**. The `requirePermission()` middleware exists but is **never used on any production route**. The permission matrix (`PERMISSION_MATRIX` in `packages/types/src/models/permissions.ts`) is effectively dead code.

The most dangerous gaps are:
- A `supplier_user` who knows any `processInstanceId` in their tenant can read the full internal workflow state (validator assignments, internal comments, all task details) via `GET /workflows/processes/:id`.
- Document download/view routes generate signed URLs for any document in the tenant — no supplier ownership or workflow relationship check.
- File uploads trust the client-provided `Content-Type` header with no server-side magic-byte validation.
- Document review has no check that the user is a designated validator.

| 4.2 | `GET /suppliers/:id` has no role check for supplier_user |

### Out of Scope

- **`GET /form-submissions/by-supplier/:supplierId`** — identified in the assessment (4.2) as lacking supplier-user authorization, but deferred to a future story to keep this story focused on workflow/document/process authorization.

## Acceptance Criteria

### AC 1: Supplier-User Process Detail Scoping (C4)

1. `GET /workflows/processes/:processInstanceId` checks if the requesting user has role `supplier_user`. If so, the endpoint verifies that `process.entityType === 'supplier'` AND `process.entityId` matches the supplier linked to the requesting user (via `suppliers.supplierUserId`). If not, the request is rejected with 403.
2. When a `supplier_user` accesses a process detail they own, the response **filters out** sensitive internal data:
   - Tasks where `assigneeRole` is NOT `supplier_user` and `assigneeUserId` is NOT the requesting user are excluded (internal validator tasks hidden).
   - Comments are returned unfiltered for now (future story will add internal/external flag).
3. Internal roles (`admin`, `procurement_manager`, `quality_manager`, `viewer`) continue to see all data for any process in their tenant.

### AC 2: Supplier-User Step & Document List Scoping

4. `GET /workflows/steps/:stepInstanceId` (if it exists as a standalone route) applies the same supplier-user scoping: verify the step's process belongs to the user's supplier.
5. `GET /workflows/steps/:stepInstanceId/documents` applies supplier-user scoping: verify the step's process belongs to the user's supplier.

### AC 3: Document Download/View Ownership Checks (C5)

6. `GET /documents/:id/download` (`routes/documents/download.ts`) and the document view route (`routes/workflows/steps/documents/view.ts`) add ownership verification:
   - If the user is `supplier_user`: the document's `supplierId` must match the supplier linked to the requesting user. Otherwise 403.
   - If the user is an internal role: tenant match is sufficient (current behavior).
7. Both routes use a shared helper function (e.g., `verifyDocumentAccess(user, document)`) to avoid duplicating the ownership logic.

### AC 4: Document Review Validator Check (4.2)

8. `POST /workflows/steps/:stepInstanceId/documents/review` verifies that the requesting user has a pending validation task for this step before allowing approve/decline. If no matching task exists, return 403 "Not authorized to review this step".

### AC 5: Step Submit Assignee Check (4.2)

9. `POST /workflows/steps/:stepInstanceId/complete` with `action: "submit"` verifies that the requesting user has a pending `action` or `resubmission` task assigned to them (by `assigneeUserId` or matching `assigneeRole`) for this step. If not, return 403.

### AC 6: Supplier Detail Route Role Check (4.2)

10. `GET /suppliers/:id` — if the requesting user is `supplier_user`, verify that `supplier.supplierUserId === user.id`. Otherwise 403. Internal roles retain full access to any supplier in the tenant.

### AC 7: Server-Side MIME Validation via Magic Bytes (C7)

11. Both upload routes (`/suppliers/:id/documents` and `/workflows/steps/:stepId/documents/:docName/upload`) validate file type using magic-byte detection (e.g., the `file-type` npm package) in addition to the `Content-Type` header check.
12. If the detected file type does not match the allowed MIME types list, the upload is rejected with 400 "File type validation failed: detected type does not match declared Content-Type".
13. The magic-byte validation is implemented as a shared utility function (e.g., `validateFileMagicBytes(file)`) reusable across both upload routes.

### AC 8: Activate `requirePermission` on Key Routes (C6)

14. The following routes use `requirePermission()` middleware instead of (or in addition to) ad-hoc role checks:
    - `POST /suppliers/:id/documents` → `requirePermission(PermissionAction.UPLOAD_DOCUMENTS)` — this intentionally aligns access with the `PERMISSION_MATRIX`, expanding the route from `admin + procurement_manager` (current inline check) to also include `quality_manager`, who is a valid uploader when assigned in a workflow template.
    - `DELETE /documents/:id` → `requirePermission(PermissionAction.DELETE_DOCUMENTS)` (if exists)
    - `POST /workflows/instantiate` → `requirePermission(PermissionAction.CREATE_QUALIFICATIONS)` (internal roles only)
15. Supplier-user-specific authorization continues to use inline checks (since their access is entity-scoped, not permission-scoped).

### AC 9: Supabase Storage RLS Policies (C8)

16. A migration or SQL script enables Row Level Security on the `storage.objects` table for the `supplier-documents` bucket.
17. The RLS policy restricts `SELECT` (signed URL generation via service role still works, but direct bucket access is tenant-scoped):
    - `(storage.foldername(name))[1] = (current_setting('request.jwt.claims', true)::json->>'tenant_id')`
18. The existing `storage-bucket-setup.sql` is updated with the active (uncommented) RLS policies.

### AC 10: Shared Authorization Helper

19. A new `apps/api/src/lib/rbac/entity-authorization.ts` module is created containing:
    - `getSupplierForUser(userId, tenantId, db)` — returns the supplier linked to a supplier_user, cached per request
    - `verifyProcessAccess(user, process)` — checks supplier_user entity scoping
    - `verifyDocumentAccess(user, document)` — checks supplier ownership
    - `verifyTaskAssignment(user, stepInstanceId, taskType, db)` — checks user has a matching pending task
20. These helpers are reused across all affected routes to keep authorization logic DRY.

### AC 11: Backward Compatibility

21. All existing functionality for internal users (`admin`, `procurement_manager`, `quality_manager`, `viewer`) is unchanged — **except** that `quality_manager` gains access to the supplier document upload route (`POST /suppliers/:id/documents`) as an intentional alignment with the `PERMISSION_MATRIX` (see AC 14).
22. Existing `supplier_user` workflows (submit forms, upload documents, view own processes) continue to work for their own supplier's processes.

## Implementation Guardrails

The following constraints **must not** be violated during implementation:

1. **No changes to workflow status/transition logic** — this story only adds authorization gates; the workflow engine state machine is untouched.
2. **No changes to the `PERMISSION_MATRIX`** — the matrix is treated as source of truth. Task 9.1's expansion of the upload route to `quality_manager` is an alignment with the existing matrix, not a matrix change.
3. **No changes to the `authenticate` or `requireRole` middleware** — only `requirePermission` is being activated on new routes.
4. **No changes to database schema** — all authorization checks use existing columns and relationships. The only DB-level change is enabling Storage RLS policies (Task 10).
5. **No removal of existing tenant-level filtering** — entity-level checks are added *on top of* existing `tenantId` filtering, never replacing it.
6. **Authorization checks run before engine calls** — all new guards must execute before any state-mutating workflow engine operation.

### Current Authorization Landscape

The `authenticate` middleware (in `middleware.ts`) provides tenant-level auth by extracting `user.tenantId` from the JWT. It's used on **every** route. The system also has:

- `requireRole(roles[])` — used on 3 routes (supplier detail PATCH/DELETE, user invitations)
- `requirePermission(permission)` — **defined** but **never used** on any production route
- `PERMISSION_MATRIX` — fully defined for all 5 roles, maps to granular `PermissionAction` enums, but effectively dead code

The `processes/list.ts` route correctly scopes `supplier_user` access (lines 58-80 — looks up `suppliers.supplierUserId = userId` and filters to `entityId = supplier.id`). This is the reference pattern for how process detail and other routes should scope supplier users.

### Supplier-User Scoping Pattern (from `list.ts`)

```typescript
if (userRole === UserRole.SUPPLIER_USER) {
  const supplier = await db.query.suppliers.findFirst({
    where: and(
      eq(suppliers.supplierUserId, userId),
      eq(suppliers.tenantId, tenantId),
      isNull(suppliers.deletedAt)
    ),
    columns: { id: true },
  });
  if (!supplier) {
    set.status = 403;
    return { success: false, error: { code: "FORBIDDEN", message: "Supplier user is not associated with a supplier" } };
  }
  // Then filter: process.entityType === "supplier" AND process.entityId === supplier.id
}
```

This pattern should be extracted into `getSupplierForUser()` and reused in the detail route and all other supplier-scoped routes.

### Task Filtering for Supplier Users (AC 2)

When returning tasks for a supplier user, filter the task list:

```typescript
const isSupplierUser = user.role === UserRole.SUPPLIER_USER;

const filteredTasks = isSupplierUser
  ? tasks.filter(t =>
      t.assigneeUserId === user.id ||
      t.assigneeRole === "supplier_user"
    )
  : tasks;
```

This hides internal validation tasks (assigned to `admin`, `quality_manager`, etc.) from the supplier while showing them their own assigned tasks. The two conditions cover both direct assignment (`assigneeUserId`) and role-based assignment (`assigneeRole`).

### Magic-Byte Validation Pattern

The `file-type` npm package detects file type from the file's magic bytes (first few bytes of the buffer):

```typescript
import { fileTypeFromBuffer } from "file-type";

export async function validateFileMagicBytes(file: File): Promise<{
  valid: boolean;
  detectedType?: string;
  error?: string;
}> {
  const buffer = Buffer.from(await file.arrayBuffer());
  const detected = await fileTypeFromBuffer(buffer);

  if (!detected) {
    return { valid: false, error: "Could not detect file type from content" };
  }

  if (!ALLOWED_MIME_TYPES.includes(detected.mime)) {
    return {
      valid: false,
      detectedType: detected.mime,
      error: `Detected file type (${detected.mime}) is not in allowed types`,
    };
  }

  return { valid: true, detectedType: detected.mime };
}
```

Note: `file-type` is an ESM-only package. Bun handles ESM natively, so no CJS compatibility issues. If the version is v19+, the import is `import { fileTypeFromBuffer } from "file-type"`.

### Supabase Storage RLS Notes

- The `supabaseAdmin` client uses the **service_role** key which **bypasses RLS by default** in Supabase. This means the API server's signed URL generation will continue to work.
- RLS policies on `storage.objects` protect against direct SDK access with the `anon` key or user JWTs, adding defense-in-depth.
- The storage path convention `{tenantId}/{supplierId}/...` makes the RLS policy straightforward using `storage.foldername(name)[1]`.

### `requirePermission` vs Inline Role Checks

The `requirePermission` middleware is already fully implemented in `middleware.ts` (lines 373-395). It:
1. Uses `authenticate` to get the user
2. Calls `checkPermission(user.role, permission)` against the `PERMISSION_MATRIX`
3. Returns 403 if the role doesn't have the permission

This is cleaner than ad-hoc `if (![UserRole.ADMIN, UserRole.PROCUREMENT_MANAGER].includes(user.role))` checks because it uses the centralized permission matrix. However, for supplier-user entity-scoping, inline checks are still needed since the RBAC matrix doesn't encode entity ownership.

### Previous Story Insights (2.2.19)

Story 2.2.19 wraps all engine operations in transactions. This story (2.2.20) should work with the transaction-wrapped functions. The authorization checks added here happen **before** the engine call, so they don't interact with the transaction logic. However, `verifyTaskAssignment` does query the DB, so it should either run inside the transaction or use a separate connection (running before the transaction is simpler and recommended).
