## Story

**As a** workflow participant (supplier user, procurement manager, or quality manager),  
**I want** to upload required documents in workflow document steps, have validators review and approve/decline each document individually with comments, and preview uploaded documents in the browser,  
**so that** document collection and review happens seamlessly within the workflow without manual file-sharing or step workarounds.

---

## Context

The workflow engine schema supports `document` step types (`step_type: "document"`, `document_template_id`, `document_action_mode: upload|validate`) but has no implementation. Users currently see a generic "Complete Step" button with no way to upload or review documents. Document templates define required documents as a JSONB array with name, description, type, and required flag.

### Existing Infrastructure
- **Supabase Storage**: `supplier-documents` bucket (10MB limit, PDF/Excel/Word/images)
- **Documents table**: stores file metadata (filename, storagePath, mimeType, fileSize)
- **Document templates**: `required_documents` JSONB defines what documents are needed
- **Supplier document upload**: working upload modal, download via signed URLs

### Key Behavior: Per-Document Validation Loop
When a validator declines individual documents:
1. Only declined documents reset to `pending` — approved documents keep their status
2. The step returns to `active` so the uploader can re-upload only the declined items
3. The upload page highlights declined documents with their decline comments
4. This loop continues until all documents are approved

## Acceptance Criteria

### Task 1: Database & Schema (AC 1-4)

1. New `workflow_step_document` table links uploaded files to workflow step instances with per-document review status
2. Each row tracks: required document name, upload status (pending/uploaded/approved/declined), decline comment, reviewer
3. Drizzle schema and TypeScript types are defined for `workflow_step_document`
4. Schema is exported from `@supplex/db`

### Task 2: API Endpoints (AC 5-10)

5. `GET /api/workflows/steps/:stepId/documents` returns all required documents for a step with upload/review status
6. `POST /api/workflows/steps/:stepId/documents/:requiredDocName/upload` uploads a file to Supabase Storage, creates a documents row, and links to workflow_step_document
7. `GET /api/documents/:id/view` returns a signed URL for viewing/downloading a document
8. `POST /api/workflows/steps/:stepId/documents/review` accepts batch review decisions (approve/decline per document with optional comment)
9. All endpoints enforce tenant isolation and proper authentication
10. Upload validates file types and size (same rules as supplier documents)

### Task 3: Workflow Engine (AC 11-15)

11. When a document step activates, `workflow_step_document` rows are seeded from the document template's `required_documents`
12. Step completion is blocked until all required documents have status `uploaded` or `approved`
13. Validation decline resets only declined documents to `pending` and returns the step to `active`
14. When all documents are `approved`, the step completes and workflow advances
15. Process detail API includes document upload progress per step

### Task 4: Frontend - Document Upload Page (AC 16-20)

16. Route `_app.workflows.processes.$processId.steps.$stepId.documents.tsx` shows a checklist of required documents
17. Each row shows: document name, description, required flag, upload button, status badge
18. Previously declined documents are highlighted with the decline comment
19. "Submit Documents" button is enabled only when all required documents are uploaded
20. Submitting triggers step completion (which may trigger validation if configured)

### Task 5: Frontend - Document Validation & Viewer (AC 21-25)

21. Validators see the same page in read-only mode with per-document Approve/Decline buttons
22. Decline opens a comment input for the reason
23. "Submit Review" sends all decisions at once
24. In-browser viewer: PDFs via `<iframe>`, images via `<img>`, Word/Excel show download link
25. Viewer opens in a modal overlay

### Task 6: ActiveStepPanel Updates (AC 26-28)

26. Document steps with `document_action_mode: "upload"` show an "Upload Documents" button
27. Steps in `awaiting_validation` show a "Review Documents" button for validators
28. Document completion progress is shown (e.g., "3/5 documents uploaded")

## Dev Notes

### Priority Order
1. Database migration and schema (foundation)
2. API endpoints (backend capability)
3. Workflow engine changes (step lifecycle)

5. Frontend validation & viewer (review flow)

### Key File Locations
- Existing document upload: `apps/api/src/routes/documents/upload.ts`
- Supabase storage: `apps/api/src/lib/supabase.ts` (`supabaseAdmin`)
- Document template schema: `packages/db/src/schema/document-template.ts`
- Workflow step template: `packages/db/src/schema/workflow-step-template.ts`
- Documents table: `packages/db/src/schema/documents.ts`
- Active step panel: `apps/web/app/components/workflow-engine/ActiveStepPanel.tsx`
- Step task creation: `apps/api/src/lib/workflow-engine/create-tasks-for-step.ts`
- Process detail API: `apps/api/src/routes/workflows/processes/get.ts`

### Storage Pattern
- Bucket: `supplier-documents`
- Path: `{tenantId}/{supplierId}/workflows/{processId}/{stepId}/{uuid}_{filename}`
- Signed URLs: 5-minute expiry for viewing

### In-Browser Viewer Support
- PDF: `<iframe src="signedUrl" />` (native browser support)
- Images (PNG, JPG): `<img src="signedUrl" />`
- Word/Excel: Download link only (no external service)

### Coding Standards
- Tenant isolation on all queries
- Use Drizzle ORM for all DB operations
- Elysia + authenticate middleware for API routes
- Eden Treaty for frontend API calls
- Remix loaders for SSR data fetching
