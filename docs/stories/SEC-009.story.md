# Story SEC-009: Custom Access Token Auth Hook for JWT Claim Injection

<!-- Powered by BMAD™ Core -->

## Status

## Story

**As a** platform operator,
**I want** a Supabase Custom Access Token Auth Hook that reads each user's `role` and `tenant_id` from the `users` table and injects them into the JWT at token issuance time,
**so that** JWT claims are always sourced from the single authoritative database record, eliminating reliance on manually synced `app_metadata` and guaranteeing that role changes propagate to the next token refresh without additional API calls.

---

## Context

SEC-001 moved authorization claims (`role`, `tenant_id`) from `user_metadata` (client-writable) to `app_metadata` (admin-only). This closed the privilege-escalation vulnerability, but the claims still rely on manual sync: every write path (`register.ts`, `invite.ts`, `update-role.ts`, `suppliers/create.ts`, `suppliers/add-contact.ts`) must explicitly write to `app_metadata` via the Supabase Admin API. If any path is missed or fails, the JWT carries stale claims until the next admin write.

Supabase's recommended pattern is a **Custom Access Token Auth Hook** — a PostgreSQL function that runs before every JWT is issued (sign-in, token refresh). The hook reads the user's current `role` and `tenant_id` directly from the `public.users` table and injects them into `app_metadata` in the JWT. This makes the database the single source of truth and eliminates the sync problem entirely.

### Dependencies

- **Depends on SEC-001** (Authorization Claims Model Hardening). The `authenticate` middleware must already read from `app_metadata`. The hook injects claims into `app_metadata` — if the middleware still reads `user_metadata`, the hook has no effect.
- **Recommended after SEC-006** (JWT Signing Key Migration). The hook works with any signing algorithm (HS256 or ES256), but SEC-006 changes the JWT infrastructure. Implementing both simultaneously increases rollout risk. Complete SEC-006 first, verify stability, then deploy SEC-009.
- **Must not overlap with SEC-006.** Both stories modify the JWT pipeline. Concurrent changes make it impossible to isolate failures.

### Task 1: Create Auth Hook Migration (AC: 1, 5)

- [x] **1.1** Create migration `packages/db/migrations/0040_custom_access_token_hook.sql`:

```sql
-- Custom Access Token Auth Hook
-- Runs before every JWT is issued (sign-in, token refresh).
-- Reads role and tenant_id from public.users and injects into app_metadata.
-- IMPORTANT: Uses jsonb_set to merge individual keys — preserves all pre-existing
-- app_metadata fields (provider, providers, etc.). Do NOT replace the whole object.
-- Reference: https://supabase.com/docs/guides/auth/auth-hooks/custom-access-token-hook

CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  claims jsonb;
  user_role text;
  user_tenant_id text;
BEGIN
  SELECT role, tenant_id::text
  INTO user_role, user_tenant_id
  FROM public.users
  WHERE id = (event->>'user_id')::uuid;

  IF user_role IS NULL OR user_tenant_id IS NULL THEN
    RETURN event;
  END IF;

  claims := event->'claims';

  IF jsonb_typeof(claims->'app_metadata') IS NULL THEN
    claims := jsonb_set(claims, '{app_metadata}', '{}');
  END IF;

  claims := jsonb_set(claims, '{app_metadata, role}', to_jsonb(user_role));
  claims := jsonb_set(claims, '{app_metadata, tenant_id}', to_jsonb(user_tenant_id));

  event := jsonb_set(event, '{claims}', claims);
  RETURN event;
END;
$$;

-- supabase_auth_admin needs EXECUTE to call the hook and SELECT to read users
GRANT EXECUTE ON FUNCTION public.custom_access_token_hook TO supabase_auth_admin;
GRANT SELECT ON TABLE public.users TO supabase_auth_admin;

-- Prevent direct invocation by application roles
REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM authenticated, anon, public;
```

## Dev Notes

### How the Hook Works

```
Token issuance (sign-in or refresh):
  1. Supabase Auth builds initial JWT claims from auth.users
     → app_metadata from raw_app_meta_data
     → user_metadata from raw_user_meta_data
  2. custom_access_token_hook(event) is called
     → Reads role, tenant_id from public.users WHERE id = user_id
     → Sets event.claims.app_metadata.role and .tenant_id
     → Returns modified event
  3. Supabase Auth signs the JWT with the modified claims
  4. Client receives JWT with hook-injected app_metadata
```

### Why Keep Admin Writes (Belt-and-Suspenders)

The `app_metadata` admin writes from SEC-001 serve two purposes even with the hook active:

1. **Initial sign-up window:** When `register.ts` creates an auth user (via `createUser` or `signUp`), Supabase may issue a token before the `public.users` row is created later in the same request. The hook fires at token issuance, finds no matching row, and returns the event unchanged. The admin write to `raw_app_meta_data` (via `updateUserById`) ensures that first JWT has claims even before the hook can source them. On the next token refresh, the `public.users` row exists and the hook takes over.

2. **Rollback safety:** If the hook is disabled (emergency rollback), the `raw_app_meta_data` on `auth.users` still contains correct claims from the admin writes. Sign-in continues to work using the pre-existing metadata.

3. **Direct DB queries:** Some admin tools or debugging workflows query `auth.users.raw_app_meta_data` directly. Keeping it in sync is useful for operational visibility.

Future cleanup (not this story): Once the hook is confirmed stable in production for at least one release cycle, the `app_metadata` admin writes can be removed from all paths except `register.ts` (where the timing issue requires the write).

### Edge Cases

| Scenario | Hook Behavior | Result |
|----------|--------------|--------|
| User exists in `public.users` | Injects `role` and `tenant_id` | Normal operation |
| User does NOT exist in `public.users` (mid-registration) | Returns event unchanged | JWT uses pre-existing `raw_app_meta_data` (from admin write) |
| User has `NULL` role or `NULL` tenant_id | Returns event unchanged | Middleware rejects with 401 `INVALID_ROLE` / `MISSING_TENANT` |
| User is deactivated (`is_active = false`) | Still injects claims | Middleware handles deactivation check separately |
| Hook function errors (syntax, permission) | Supabase Auth returns error | **All sign-ins fail** — disable hook immediately (Step 6) |

### Claim Flow After All SEC Stories

```
After SEC-001 + SEC-009:
  Source of truth: public.users table (role, tenant_id columns)

  JWT issuance path:
    auth.users.raw_app_meta_data → pre-filled by admin writes (SEC-001)
    custom_access_token_hook    → overwrites with DB values (SEC-009)
    Result JWT: app_metadata.role + app_metadata.tenant_id = always current

  Consumers:
    API middleware (authenticate)  → reads app_metadata from JWT
    RLS policies (SEC-003A/B)     → reads auth.jwt()->'app_metadata'
    tenant-context.ts             → reads app_metadata from JWT
    Storage RLS (SEC-003A)        → reads auth.jwt()->'app_metadata'->'tenant_id'
```

All four consumer paths now read from the same location (`app_metadata`) which is always populated by the hook from the same source (`public.users`). The S5/T3/M9 finding (three different claim paths) is fully resolved.

### Security Considerations

- **`SECURITY DEFINER`**: The function runs with the privileges of the function owner (typically `postgres`), not the caller. This is required because `supabase_auth_admin` needs to read `public.users` through the function.
- **`SET search_path = public`**: Prevents search-path hijacking in `SECURITY DEFINER` functions.
- **`GRANT SELECT` only**: The hook only needs to read from `public.users`. It does not need `INSERT`, `UPDATE`, or `DELETE`.
- **`REVOKE EXECUTE` from application roles**: Prevents direct invocation by `authenticated`, `anon`, or `public` — only `supabase_auth_admin` (the Supabase Auth internal role) can call it.

### Performance

The hook adds one `SELECT` query to every token issuance. Given:
- `public.users` has a primary key index on `id`
- The query is `WHERE id = (uuid)` — a single-row index lookup
- Token issuance happens on sign-in and refresh (not every API call)
- Typical refresh interval is 3600s (1 hour)

The overhead is negligible: ~1ms per token issuance, amortized over the token lifetime.

### Implementation Guardrails

1. **Do not remove `app_metadata` admin writes.** They are belt-and-suspenders. See "Why Keep Admin Writes" above.
2. **Do not make the hook reject users.** The hook should always return the event (modified or unchanged). User deactivation is handled by the middleware, not the hook.
3. **The hook must be idempotent.** Running it multiple times on the same event should produce the same result.
4. **Do not add business logic to the hook.** It should only read and inject claims. Complex authorization decisions belong in the API middleware.
5. **The hook must preserve all pre-existing `app_metadata` fields.** It may only overwrite `role` and `tenant_id`. It must not remove or clobber unrelated keys such as `provider`, `providers`, or any future fields that Supabase or other hooks may add. The SQL must use per-key `jsonb_set` (as shown in Task 1.1), not wholesale `app_metadata` replacement.
6. **The migration must be applied BEFORE the hook is enabled in Supabase.** If the hook is enabled without the function existing, all sign-ins will fail.
7. **Test the hook function in SQL Editor before enabling it via dashboard.** See Operator Step 2.

- **Integration test:** After enabling the hook, verify JWT contents via browser DevTools (Operator Steps 4-5).
- **Rollback test:** Disable the hook → verify sign-in still works using `raw_app_meta_data` (Operator Step 6).
