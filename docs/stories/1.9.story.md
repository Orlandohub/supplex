## Acceptance Criteria

1. GitHub Actions workflow configured to run on every pull request
2. CI pipeline runs: lint, type-check, unit tests, integration tests in parallel
3. Test coverage report generated and commented on PR if coverage drops below threshold (70% backend, 60% frontend)
4. Frontend (Remix) configured for automatic deployment to Vercel on merge to `main`
5. Backend (ElysiaJS) configured for deployment to Fly.io (or Vercel) on merge to `main`
6. Preview deployments created for every PR with unique URL (Vercel preview + Fly.io preview app)
7. Database migrations run automatically on deployment using Drizzle Kit
8. Environment variables properly configured for Dev, Staging, Production environments
9. Deployment status badges added to README showing build status
10. Rollback procedure documented in case deployment fails
11. Health check endpoints implemented for both frontend (`/health`) and backend (`/api/health`)
12. Monitoring alerts configured in Sentry for production errors
13. Failed builds block merge with clear error messaging

### GitHub Actions Workflow Configuration

[Source: architecture/tech-stack.md, architecture/testing-strategy.md]

**CI Workflow Structure:**

```yaml
name: CI

on:
  pull_request:
    branches: [main, staging]
  push:
    branches: [main, staging]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
        with:
          bun-version: 1.1.0
      - uses: pnpm/action-setup@v2
        with:
          version: 8.15.0
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint

  type-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
      - uses: pnpm/action-setup@v2
      - uses: actions/setup-node@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm type-check

  test-frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
      - uses: actions/setup-node@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm --filter @supplex/web test --coverage
      - uses: codecov/codecov-action@v3
        with:
          files: ./apps/web/coverage/coverage-final.json
          flags: frontend

  test-backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
      - uses: pnpm/action-setup@v2
      - run: pnpm install --frozen-lockfile
      - run: pnpm --filter @supplex/api test --coverage
      - uses: codecov/codecov-action@v3
        with:
          files: ./apps/api/coverage/coverage-final.json
          flags: backend
```

**Branch Protection Configuration:**
- Require status checks to pass before merging: `lint`, `type-check`, `test-frontend`, `test-backend`
- Require branches to be up to date before merging
- Require pull request reviews (1 approver minimum)
- Dismiss stale pull request approvals when new commits are pushed

### Database Migration Automation

[Source: architecture/tech-stack.md, packages/db]

**Drizzle Kit Migration Commands:**
- `pnpm db:generate` - Generate migration files from schema changes
- `pnpm db:migrate` - Run pending migrations against database
- `pnpm db:push` - Push schema changes directly (development only, skips migrations)

**Migration Strategy:**
- Migrations stored in `packages/db/migrations/` directory
- Migrations run automatically on Fly.io deployment via `release_command`
- Migration status tracked in `__drizzle_migrations` table
- Failed migrations prevent deployment (Fly.io rolls back)

**Rollback Procedure:**
- Drizzle does not support automatic rollbacks
- Manual rollback: Create new migration that reverses previous migration
- Emergency rollback: Revert to previous Fly.io deployment version (`flyctl releases rollback`)

**Migration Testing:**
- Test migrations locally before deploying: `pnpm db:migrate` against staging database
- Use database backups before running migrations in production
- Verify migration success via health check endpoint

### Rollback Procedures

[Source: Epic AC #10]

**Vercel Rollback:**
1. Navigate to Vercel dashboard → Deployments
2. Find previous successful deployment
3. Click "..." menu → "Promote to Production"
4. Confirm rollback
5. Verify rollback by checking application version

**Fly.io Rollback:**
1. List recent releases: `flyctl releases list -a supplex-api`
2. Identify version to rollback to (previous stable version)
3. Rollback: `flyctl releases rollback <version> -a supplex-api`
4. Verify rollback: `flyctl status -a supplex-api`
5. Check health endpoint: `curl https://supplex-api.fly.dev/api/health`

**Database Migration Rollback:**
1. Identify failed migration in `__drizzle_migrations` table
2. Create reverse migration manually (Drizzle does not auto-generate rollback)
3. Run reverse migration: `bun run db:migrate` (after creating rollback migration)
4. Verify database state using `pnpm db:studio`

**Emergency Hotfix Procedure:**
1. Create hotfix branch from `main`: `git checkout -b hotfix/critical-fix`
2. Implement fix and commit
3. Fast-track CI by ensuring all tests pass locally
4. Create PR and merge immediately (with approval)
5. Deploy automatically via CI/CD pipeline
6. Monitor Sentry for errors post-deployment
