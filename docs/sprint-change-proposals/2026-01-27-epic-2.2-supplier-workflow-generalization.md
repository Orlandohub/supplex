## Executive Summary

This proposal addresses a fundamental architectural misalignment in Epic 2.2 where workflows were narrowly scoped to "qualifications" rather than general-purpose supplier processes. The proposed changes generalize the workflow system to support any supplier-related process (qualification, audits, compliance checks, onboarding, etc.) while adding custom status tracking capabilities.

**Key Changes:**
- ✅ Generalize Story 2.2.9 from "Supplier Qualification Integration" to "Supplier Workflow Integration"
- ✅ Add workflow template active/inactive toggle
- ✅ Add custom completion status field to workflow steps
- ✅ Rename all "Qualifications" UI elements to "Workflows"
- ✅ Replace `/qualifications` route with `/workflows`

**Impact:** Medium effort (~5.5 days), High value, Low risk

**Why Now:** Story 2.2.9 has not been started yet, making this the optimal time for course correction.

## 3. Proposed Changes

### 3.1 Story 2.2.9 - Complete Rewrite

**OLD STORY:**
```markdown
Story 2.2.9: Supplier Qualification Integration (Phase 1)

As the system,
I want to use the workflow engine for supplier qualification,
so that qualification becomes configurable per tenant.

Acceptance Criteria:

1. Supplier qualification starts a process_instance by:
   - Admin selects workflow from dropdown of published workflow templates filtered by tenant_id
   - The selected workflow_template_version_id is used to instantiate the process
   - The instance is linked to [supplier_id]
   - Tasks and forms are generated from the active step

2. Supplier-facing tasks are visible only to supplier_user of that supplier and tenant

3. Internal tasks are visible only to internal users of the same tenant
```

**NEW STORY:**
```markdown
Story 2.2.9: Supplier Workflow Integration

As an admin or procurement_manager,
I want to start any workflow process for a supplier from the supplier detail page,
so that I can manage all supplier-related processes (qualification, audits, onboarding, compliance checks, etc.) in one central location.

Acceptance Criteria:

1. Database Changes:
   - workflow_template table has `active` BOOLEAN field (default: true)
   - workflow_step_template table has `completion_status` VARCHAR(100) field (nullable)

2. Workflow Template Management:
   - Admin can toggle workflow templates between active and inactive
   - Only templates with `active = true` AND `status = 'published'` appear in workflow selection dropdowns
   - Inactive workflows cannot be instantiated (existing process instances continue normally)

3. Supplier Detail Page - Workflows Tab:
   - Supplier detail page has "Workflows" tab (replaces "Qualifications" tab)
   - Tab shows all workflow process instances where entity_type='supplier' AND entity_id=supplier.id
   - "Start Process" button shows dropdown of active published workflow templates
   - Admin or Procurement Manager can initiate any workflow for the supplier

4. Workflow Instantiation:
   - Selected workflow_template_version_id is used to instantiate the process
   - Process is linked via: entity_type='supplier', entity_id=supplier.id
   - Workflow status tracking follows custom completion statuses (see AC 5)

5. Custom Workflow Status Tracking:
   - When a workflow step completes:
     - If workflow_step_template.completion_status is NOT NULL:
       - Copy completion_status value to process_instance.status
     - If completion_status is NULL:
       - Leave process_instance.status unchanged
   - Example: Step "Document Review" with completion_status="Under Review" → When step completes, workflow status becomes "Under Review"

6. Task Visibility:
   - Supplier-facing tasks visible only to supplier_user of that supplier and tenant
   - Internal tasks visible only to internal users of the same tenant

7. Workflows Page:
   - Global workflows page at `/workflows` (replaces `/qualifications`)
   - Lists all workflow processes across all suppliers (tenant-filtered)
   - Filters: All, My Tasks, My Initiated
   - Supports search, filtering by status, export to CSV

8. Sidebar Navigation:
   - "Qualifications" link renamed to "Workflows"
   - Links to `/workflows` page

9. Backward Compatibility:
   - Existing process instances with entity_type='qualification' continue to work
   - System treats qualification as a legacy entity type
   - No data migration required
```

#### 3.4.4 New Endpoint: `apps/api/src/routes/workflow-templates/toggle-active.ts`

```typescript
import { Elysia, t } from "elysia";
import { db } from "@supplex/db";
import { workflowTemplate } from "@supplex/db";
import { eq, and, isNull } from "drizzle-orm";
import { authenticate } from "~/middleware/authenticate";

export const toggleActiveRoute = new Elysia()
  .use(authenticate)
  .patch(
    "/:templateId/toggle-active",
    async ({ params, user, set }) => {
      // Admin-only check
      if (!user?.role || user.role !== "admin") {
        set.status = 403;
        return {
          error: {
            code: "FORBIDDEN",
            message: "Only admins can toggle workflow template status",
          },
        };
      }

      const { templateId } = params;

      // Fetch template
      const template = await db.query.workflowTemplate.findFirst({
        where: and(
          eq(workflowTemplate.id, templateId),
          eq(workflowTemplate.tenantId, user.tenantId),
          isNull(workflowTemplate.deletedAt)
        ),
      });

      if (!template) {
        set.status = 404;
        return {
          error: {
            code: "NOT_FOUND",
            message: "Workflow template not found",
          },
        };
      }

      // Toggle active status
      const [updated] = await db
        .update(workflowTemplate)
        .set({
          active: !template.active,
          updatedAt: new Date(),
        })
        .where(eq(workflowTemplate.id, templateId))
        .returning();

      return {
        success: true,
        data: updated,
      };
    },
    {
      params: t.Object({
        templateId: t.String({ format: "uuid" }),
      }),
    }
  );
```

**Add new section:**

```markdown
## Custom Workflow Status Tracking

The `completion_status` field enables admins to define meaningful status labels that track workflow progress:

**Example Configuration:**
- Step 1: "Submit Documents" → completion_status = "Documents Submitted"
- Step 2: "Initial Review" → completion_status = "Under Review"  
- Step 3: "Quality Check" → completion_status = "Quality Approved"
- Step 4: "Final Approval" → completion_status = "Approved"

**Runtime Behavior:**
When a step completes:
1. If `workflow_step_template.completion_status` is NOT NULL:
   - Copy the value to `process_instance.status`
   - Example: Step 2 completes → Workflow status becomes "Under Review"
2. If `completion_status` is NULL:
   - Leave `process_instance.status` unchanged
   - Useful for intermediate steps that don't represent meaningful status changes

**Benefits:**
- Admins control status labels (not hardcoded by developers)
- Status labels can match business terminology
- Each workflow can have its own status progression
- UI can display meaningful status to users
```

### Step Completion with Custom Status

When a step_instance is marked as completed:

1. Execute step completion logic (submit form / approve / validate)
2. Update step_instance.status = 'completed'
3. Mark associated task_instance(s) as completed
4. **Check for completion_status:**
   ```typescript
   if (stepTemplate.completionStatus) {

   }
   ```
5. Transition to next step OR complete workflow

This allows workflows to reflect business-meaningful statuses rather than technical states.
```
