# Sprint Change Proposal: Auto-Validation Task Creation

**SCP ID:** SCP-2026-03-17-001

## Executive Summary

User testing of Story 2.2.14 (Template Builder) revealed a UX issue: creating validation steps requires manual step creation, making workflow templates cumbersome to build. Users expect validation to be a **checkbox property** on steps, with the system automatically creating validation tasks at runtime. This proposal adds Story 2.2.15 to Epic 2.2 to implement this enhancement with minimal disruption.

**Recommendation:** APPROVE - Implement as Story 2.2.15 (estimated 2-3 days)

---

## Section 1: Identified Issue Summary

### Problem Statement

**User Feedback (from Story 2.2.14 testing):**
> "Currently when creating workflow templates I have to manually add separate validation step types. Instead, I should be able to check 'Requires Validation?' on any step, select approver roles, and let the system create validation tasks automatically during workflow execution."

### Current Implementation

**How Validation Works Today:**
1. User creates a step (e.g., "Submit Supplier Profile Form")
2. User must manually create a **second step** (e.g., "Validate Supplier Profile")
3. The validation step uses `formActionMode='validate'` or `documentActionMode='validate'`
4. This creates unnecessary template complexity and is error-prone

**Example - Current (Manual) Approach:**
```
Step 1: Submit Supplier Profile (formActionMode='fill_out')
Step 2: Validate Supplier Profile (formActionMode='validate') ← MANUAL!
Step 3: Upload Financial Documents (documentActionMode='upload')
Step 4: Validate Financial Documents (documentActionMode='validate') ← MANUAL!
Step 5: Final Approval
```

**Technical Context:**
- Database: `workflow_step_template.formActionMode` ENUM includes `validate` value
- Database: `workflow_step_template.documentActionMode` ENUM includes `validate` value
- The infrastructure for validation exists, but step creation is manual

### Proposed Solution

**How Validation Will Work (Enhanced):**
1. User creates a step (e.g., "Submit Supplier Profile Form")
2. User checks **"Requires Validation?"** checkbox
3. User selects approver role(s) from dropdown
4. At runtime, when step completes:
   - System **automatically creates validation task(s)** for selected role(s)
   - Validation occurs **before** proceeding to next step
   - No manual validation step creation needed

**Example - New (Automatic) Approach:**
```
Step 1: Submit Supplier Profile (formActionMode='fill_out', requiresValidation=true, validatorRoles=['quality_manager'])
  → Runtime: Auto-creates validation task for quality_manager
Step 2: Upload Financial Documents (documentActionMode='upload', requiresValidation=true, validatorRoles=['procurement_manager'])
  → Runtime: Auto-creates validation task for procurement_manager
Step 3: Final Approval
```

**Benefits:**
- ✅ Reduces workflow template complexity (fewer steps)
- ✅ Prevents user errors (forgetting validation steps)
- ✅ More intuitive UX (validation as property, not separate step)
- ✅ Maintains backward compatibility (existing workflows unchanged)

```markdown
### Auto-Validation Task Creation

When a step completes and has `requires_validation = true`:

**Sequence:**
```
1. User completes step (form submission or document upload)
2. Step status → 'completed'
3. System checks step.requires_validation
   ↓
4. IF requires_validation = true:
   ├─ Parse validation_config.approverRoles
   ├─ For each role:
   │  └─ Create task_instance with:
   │     - assignee_type = 'role'
   │     - assignee_role = role
   │     - Linked to current step_instance
   │     - Form/document in validate mode

   └─ Proceed to next step immediately
```

```
Validator completes validation task
   ↓
IF approved:
   ├─ Mark validation task complete
   ├─ Set step status → 'validated'
   └─ Activate next step
   ↓
IF declined:
   ├─ Create comment with decline reason
   ├─ Return to step based on decline_returns_to_step_offset
   └─ Reactivate previous step for revision

### Technical Approach

**Database Schema Addition:**
```sql
-- Migration 0021
ALTER TABLE workflow_step_template
ADD COLUMN requires_validation BOOLEAN NOT NULL DEFAULT false,
ADD COLUMN validation_config JSONB DEFAULT '{}'::jsonb;

CREATE INDEX idx_workflow_step_requires_validation
ON workflow_step_template(requires_validation)
WHERE requires_validation = true AND deleted_at IS NULL;
```

**Validation Config Structure:**
```typescript
interface ValidationConfig {
  approverRoles: string[];          // e.g., ['quality_manager', 'procurement_manager']
  requireAllApprovals?: boolean;    // Future: if true, all approvers must approve
}
```

**Workflow Engine Logic (Pseudocode):**
```typescript
async function completeStep(stepInstanceId: string) {
  // 1. Mark step completed
  await db.update(stepInstance)
    .set({ status: 'completed', completedAt: new Date() });

  // 2. Check if validation required
  const stepTemplate = await getStepTemplate(stepInstanceId);
  
  if (stepTemplate.requiresValidation) {
    // 3. Create validation tasks
    const validationConfig = stepTemplate.validationConfig;
    
    for (const role of validationConfig.approverRoles) {
      await createTaskInstance({
        stepInstanceId,
        assigneeType: 'role',
        assigneeRole: role,
        title: `Validate: ${stepTemplate.name}`,
        description: 'Review and approve or decline this step',
        // Form/document automatically in validate mode
      });
    }
    
    // 4. Set status to awaiting validation
    await db.update(stepInstance)
      .set({ status: 'awaiting_validation' });
      
    return; // Do not activate next step yet
  }
  
  // 5. No validation required - proceed to next step
  await activateNextStep(stepInstanceId);
}
```
