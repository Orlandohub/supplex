## Overview

This guide implements **Option A** from the QA review: refactoring form template schemas to use PostgreSQL `pgEnum` types instead of `varchar` + TypeScript const objects for database-level enum validation.

### Benefits of pgEnum

✅ **Database-level validation** - Invalid values rejected by PostgreSQL  
✅ **Better performance** - Enums stored as integers internally  
✅ **Type safety** - Protection against typos in direct SQL  
✅ **Documentation** - Allowed values explicit at DB level  
✅ **Third-party safety** - Tools outside your app see constraints

## Step-by-Step Implementation

### 1. Schema Changes (✅ COMPLETED)

The following changes have been applied to your schema files:

#### form-template.ts
```typescript
// Added pgEnum import
import { pgEnum } from "drizzle-orm/pg-core";

// Created enum type
export const formTemplateStatusEnum = pgEnum("form_template_status", [
  "draft",
  "published",
  "archived",
]);

// Updated table definition
status: formTemplateStatusEnum("status")
  .notNull()
  .default(FormTemplateStatus.DRAFT),
```

#### form-field.ts
```typescript
// Created enum type
export const fieldTypeEnum = pgEnum("field_type", [
  "text",
  "textarea",
  "number",
  "date",
  "dropdown",
  "checkbox",
  "multi_select",
]);

// Updated table definition
fieldType: fieldTypeEnum("field_type").notNull(),
```

**Key Design Decision**: We kept the TypeScript `const` objects (e.g., `FormTemplateStatus`) alongside the pgEnum for **compile-time safety** + **database-level validation**. This provides defense-in-depth.


The migration will:
1. Create `form_template_status` ENUM type
2. Create `field_type` ENUM type
3. Alter table columns to use ENUM types

#### Step 2b: Alternative Migration (With Existing Data)

If you have existing data, the migration file includes an alternative approach that:
1. Creates temporary columns with ENUM types

3. Drops old columns and renames new ones

## Rollback Instructions

If you need to rollback the ENUM changes:

### 1. Revert Schema Files

### 2. Rollback Database (SQL)
```sql
-- Convert columns back to VARCHAR

    ALTER COLUMN status TYPE VARCHAR(50);
    
ALTER TABLE form_template_version 
    ALTER COLUMN status TYPE VARCHAR(50);
    
ALTER TABLE form_field 
    ALTER COLUMN field_type TYPE VARCHAR(50);

-- Drop ENUM types

DROP TYPE field_type;
```

### 3. Set Defaults Again
```sql
ALTER TABLE form_template 
    ALTER COLUMN status SET DEFAULT 'draft';
    
ALTER TABLE form_template_version 
    ALTER COLUMN status SET DEFAULT 'draft';
```

## Performance Considerations

**ENUM Performance Benefits:**
- ENUMs are stored as integers (4 bytes) vs VARCHAR (variable)
- Comparisons are faster (integer comparison vs string)
- Index size is smaller (integers vs strings)
- Memory usage reduced in result sets

**Negligible Migration Overhead:**
- Enum conversion is fast with no data
- With data, conversion time scales linearly with row count
- No downtime required (can use transactional DDL)

## FAQ

### Q: Why keep the TypeScript const objects if we have pgEnum?

**A**: Defense-in-depth. The const objects provide compile-time safety in your application code, while pgEnum provides runtime safety at the database level. This catches errors in both layers.

### Q: Can I add new enum values later?

**A**: Yes! Use ALTER TYPE:
```sql
ALTER TYPE form_template_status ADD VALUE 'suspended';
```

However, removing values is harder (requires recreating the type).

### Q: Will this break existing code?

**A**: No. The TypeScript interfaces and constants remain the same. Only the database storage changes.

### Q: What if I have a typo in my application code?

**A**: TypeScript will catch it at compile time (via const objects), and PostgreSQL will catch it at runtime (via ENUM types). You're protected at both layers.
